# CDS

CDS 把“准备部署环境”变成“选择项目、批准 Agent、确认方案、打开预览”。它负责理解项目、准备运行方式、持续记录部署过程，并在新版本有问题时帮助用户恢复到成功版本。

## 2026-07 重大升级：从配置部署到授权后自动交付

这不是在原流程上增加几个按钮，而是重新调整了第一次使用 CDS 的主流程。

以前，用户要先下载技能、配置电脑、传递密钥，再填写构建、启动、端口和依赖信息。现在，用户只需要选择项目并批准一次，CDS 和 Agent 会完成其余准备工作；只有自动识别不准确时，用户才需要补充配置。

| 使用环节 | 以前 | 现在 |
| --- | --- | --- |
| 接入 Agent | 下载技能、复制密钥、配置终端 | 选择 Agent，复制无密钥口令，页面批准 |
| 权限范围 | 容易使用长期或过大的权限 | 默认只允许访问当前项目 |
| 电脑环境 | 可能修改环境变量或终端配置 | 只在当前项目保存凭据，不改系统环境 |
| 第一次部署 | 先填写完整构建和启动配置 | CDS 自动识别，用户只确认结果 |
| 查看进度 | 状态和日志分散，刷新后容易失去上下文 | 回到同一条部署记录继续查看 |
| 再次运行 | 重新拉取和构建 | 成功版本可直接再次部署 |
| 新版本异常 | 手工找旧代码重新部署 | 选择以前的成功版本恢复 |
| 部署失败 | 用户从大量日志中自行判断 | 页面先给出问题归属和处理顺序 |

## Quick Start：第一次使用 CDS

### 1. 接入你的 Agent

进入 CDS 控制台任意页面，点击左侧常驻的“接入 Agent”。入口会自动识别当前页面；在“登录与认证”等设置页，也可以直接点击页面内的“交给 Agent 配置”。

选择 Codex、Cursor 或 Claude Code，再选择“连接已有项目”或“创建一个新项目”。把页面生成的接入口令交给 Agent。口令不包含密钥，也不要求修改终端配置。

生成的接入口令会同时带上当前页面任务、验收要求和安全边界。Agent 发起申请后，在 CDS 右下角点击批准。Agent 会自动验证连接并继续工作。

### 2. 选择项目和代码分支

已有项目直接进入目标项目。新项目可以粘贴 Git 仓库地址，由 CDS 拉取代码并识别项目。

私有仓库使用该项目对应的 GitHub App 单仓库授权，不需要开放整个组织，也不需要重复粘贴 GitHub 凭据。

### 3. 确认 CDS 识别的运行方式

进入“项目设置 > 运行时 > 交付模式”，常规项目选择“托管模式”。

CDS 会检查应用目录、安装方式、构建方式、启动命令、端口、健康检查和依赖。标准 Compose 项目没有写启动命令时，CDS 会继续从项目自身信息中推导，而不是立刻要求用户手工补齐。

用户只需要确认“最终生效配置”是否符合项目实际情况。只有识别错误或项目结构特殊时，才修改“最小声明”。

### 4. 部署并打开预览

返回项目分支页，打开目标分支并点击“重新部署”。

部署页面会持续显示准备、构建、启动和检查阶段。刷新页面、暂时离开或短暂断线后，重新打开仍然是同一次部署记录。

部署成功后，点击页面给出的预览地址，检查登录、主要页面和核心操作。

### 5. 再次部署或恢复旧版本

代码没有变化时，可以在“不可变部署版本”中选择“重新部署此版本”，减少重复构建等待。

新版本有问题时，选择以前标记为可复用的成功版本，点击“回滚到此版本”。恢复过程会生成新的部署记录，仍然可以完整查看进度和结果。

## Agent 接入的权限边界

新版接入默认遵循以下规则：

- 不让用户把长期密钥粘贴到对话中；
- 不修改 `.zshrc`、`.bashrc`、系统环境变量或全局 PATH；
- 凭据只保存在当前项目的 `.cds/credentials.json`；
- 凭据文件权限为 `0600`，并加入当前仓库的本地排除规则；
- 授权默认只允许访问当前项目，其他项目不会自动继承；
- 首次创建项目使用一次性权限，项目创建成功后自动失效并换成项目专属权限；
- 等待批准时遇到短暂网络波动会继续等待，不需要重新发起；
- 测试或不再使用的权限可以撤销，不留下长期访问能力。

## 我应该选择哪个交付模式

| 项目情况 | 推荐模式 | 原因 |
| --- | --- | --- |
| 普通网站、后台服务、前后端应用 | 托管模式 | 自动识别，配置少，失败更容易理解 |
| 单个或少量常驻任务 | 托管模式 | 支持不提供网页端口的后台任务 |
| 服务很多且启动顺序特殊 | Compose 模式 | 保留明确的服务编排 |
| 使用特殊网络或自定义镜像 | Compose 模式 | 保留高级控制能力 |
| 已有稳定的复杂 Compose 配置 | 继续使用 Compose | 无需为了新版迁移成熟配置 |

两种模式可以切换。托管模式不会删除原有 Compose 配置，切回后原配置继续生效。

## 部署失败时怎么做

不要连续重复点击部署。先打开本次部署记录，按照页面给出的顺序处理：

1. 确认问题属于代码、配置、CDS、外部服务还是暂未归类；
2. 阅读当前阶段和处理建议；
3. 只修复对应问题；
4. 修复后重新部署；
5. 基础说明仍然难懂时，再使用“AI 解释”。

“AI 解释”是辅助能力。即使没有启用，基础问题判断、部署阶段和处理建议仍然可用。

## 当前发布状态

托管交付、部署记录、成功版本复用和回滚已经进入正式主流程。

“接入 Agent”、项目级页面批准、无环境变量接入、私有仓库授权复用以及标准 Compose 自动补全，已在本次重大版本开发分支完成并通过真实私有项目验收，但尚未发布到当前 `cds.miduo.org` 的主分支。因此正式环境暂时仍可能显示旧入口“下载技能包”。

正式发布后，旧入口会被“接入 Agent”替换。用户不需要同时维护两套接入方式；原有环境变量方式只作为历史兼容保留。

## 本地启动 CDS

在仓库根目录执行：

```bash
./exec_cds.sh init
./exec_cds.sh start
./exec_cds.sh status
```

首次启动时，`init` 会引导完成必要配置。需要查看运行日志时执行：

```bash
./exec_cds.sh logs
```
