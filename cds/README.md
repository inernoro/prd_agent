# CDS

CDS 用来把代码分支变成可以直接访问和验收的独立环境。你可以在一个页面里查看项目、分支、部署进度、预览地址和运行状态。

## 2026-07 新版：托管交付

托管交付新版已经进入主分支。它把常规项目的上线流程从“先学会并填写一整套部署配置”缩短为“让 CDS 自动识别，确认结果，然后部署”。

这次新版带来五项用户可直接感知的变化：

1. **自动准备上线方式**：CDS 自动识别应用目录、安装、构建、启动、端口和健康检查。
2. **刷新后继续看同一次部署**：离开页面或短暂断线，不会丢失原来的部署进度和结果。
3. **成功版本可以直接再次部署**：代码没有变化时，可以复用已经成功的产物，减少等待。
4. **新版本有问题时可以回到旧版本**：从成功版本历史中选择一个版本恢复。
5. **失败后先给结论和处理顺序**：页面会区分代码、配置、CDS 或外部服务问题，再给出下一步建议。

原有 Compose 模式继续保留。复杂网络、多服务编排或特殊启动项目不需要迁移，切换模式也不会删除原有配置。

完整说明见 [CDS 托管交付使用指南](../doc/guide.cds.managed-delivery.md)。

## 新版 Agent 接入

右上角“一键部署”菜单中的“下载技能包”已经升级为“接入 Agent”。新版接入支持 Codex、Cursor 和 Claude Code，默认只连接当前项目。

1. 点击“接入 Agent”。
2. 选择“连接已有项目”或“创建一个新项目”。
3. 复制不含密钥的接入口令给 Agent。
4. Agent 发起申请后，在 CDS 右下角点击批准。
5. Agent 自动验证连接并继续部署。

新版不会要求你把 Agent Key 粘贴到对话中，也不会修改电脑的环境变量、终端启动文件或全局 PATH。首次创建项目使用一次性权限，项目创建后自动失效并切换为该项目专属权限。

## 先确认你的 CDS 是否已经更新

CDS 可以由不同团队自行部署，因此“代码已经进入主分支”不等于每一套 CDS 实例都已经完成更新。

1. 打开 CDS，进入一个项目。
2. 点击“项目设置”。
3. 在“运行时”分组中查找“交付模式”。

看到“交付模式”表示这套 CDS 已经具备新版能力。没有看到时，请先由 CDS 管理员完成系统更新；这不是项目配置问题，不需要反复修改项目。

## Quick Start：第一次使用托管模式

建议先选一个非生产项目，并确认项目内至少有一个已经拉取完成的分支。

1. 进入“项目设置 > 运行时 > 交付模式”。
2. 选择“托管模式”。
3. 先保持默认内容，让 CDS 自动生成“最终生效配置”。
4. 检查应用目录、构建方式、启动方式、端口和依赖是否正确。
5. 返回项目分支页，打开目标分支，再打开“设置”页签，点击“重新部署”。
6. 在同一个分支抽屉的“部署”页签查看部署阶段。
7. 部署成功后，打开页面显示的预览地址完成业务检查。

只有自动识别不准确或项目结构特殊时，才需要修改“最小声明”。常规项目不需要先手写完整部署配置。

## 十分钟验证清单

下面按真实用户路径验收，不需要查看代码或调用接口。

### Phase 0：确认版本和测试对象

| 状态 | 操作 | 预期结果 |
| --- | --- | --- |
| `[ ]` | 选择一个非生产项目，并确认至少有一个分支 | 项目和分支页面可以正常打开 |
| `[ ]` | 进入“项目设置 > 运行时” | 可以看到“交付模式”入口 |

如果看不到“交付模式”，先更新 CDS。本阶段未通过时不要继续，也不要为了找入口修改项目配置。

### Phase 0.5：验证 Agent 接入没有污染电脑

| 状态 | 操作 | 预期结果 |
| --- | --- | --- |
| `[ ]` | 右上角“一键部署 > 接入 Agent” | 可以选择 Codex、Cursor 或 Claude Code 对应的项目级安装方式 |
| `[ ]` | 复制接入口令给 Agent | 口令不包含任何 Key，也不要求修改终端配置 |
| `[ ]` | 在右下角批准授权申请 | Agent 自动继续并显示连接成功 |
| `[ ]` | 检查目标仓库的 `git status` | 不出现凭据文件或需要提交的 CDS 接入改动 |
| `[ ]` | 关闭并重新打开 Agent | 当前项目仍可连接，其他项目不会继承这份权限 |

### Phase 1：启用托管模式

| 状态 | 操作 | 预期结果 |
| --- | --- | --- |
| `[ ]` | 点击“托管模式” | 页面显示“最小声明”和“最终生效配置” |
| `[ ]` | 检查自动识别结果 | 应用目录、构建、启动、端口和依赖符合项目实际情况 |

如果没有生成最终配置，先确认项目有可用分支；仍失败时，记录页面错误和项目结构，再补充最小声明。

### Phase 2：验证部署过程可续看

| 状态 | 操作 | 预期结果 |
| --- | --- | --- |
| `[ ]` | 打开目标分支，在“设置”页签点击“重新部署” | 页面持续显示准备、构建、启动和检查等阶段 |
| `[ ]` | 部署过程中刷新页面，再回到“部署”页签 | 继续显示同一次部署，而不是丢失记录或重新开始 |
| `[ ]` | 部署完成后打开预览地址 | 应用可以访问，核心业务流程正常 |

如果部署失败，先看“部署事实账本”的问题归属和建议；需要进一步排查时，再打开对应构建日志或容器日志。

### Phase 3：验证成功版本可以复用

| 状态 | 操作 | 预期结果 |
| --- | --- | --- |
| `[ ]` | 在“部署”页签找到“不可变部署版本” | 当前成功版本显示“产物可复用”或明确说明为什么只能记录 |
| `[ ]` | 对可复用的当前版本点击“重新部署此版本” | 产生一条新的部署记录，并直接使用已经成功的版本 |

如果按钮没有出现，查看版本是否标记为“仅记录”。这表示该次部署没有完整的可复用产物，需要从源码重新部署，不代表版本历史丢失。

### Phase 4：验证回滚

这一步需要至少两个成功版本。只有一个成功版本时可以暂时跳过。

| 状态 | 操作 | 预期结果 |
| --- | --- | --- |
| `[ ]` | 对较早的可复用版本点击“回滚到此版本” | 页面新增一条回滚部署记录 |
| `[ ]` | 等待回滚完成并打开预览地址 | 应用恢复为所选版本，预览地址仍可访问 |

如果旧版本标记为“仅记录”，它不能直接恢复；请选择带“产物可复用”标记的成功版本。

### Phase 5：确认旧流程没有被覆盖

| 状态 | 操作 | 预期结果 |
| --- | --- | --- |
| `[ ]` | 回到“项目设置 > 运行时 > 交付模式”，选择“Compose 模式” | 原有 Compose 配置继续存在并重新生效 |
| `[ ]` | 再次切回托管模式 | CDS 重新生成最终生效配置，原有 Compose 配置仍未被删除 |

如果切换后发现原有 Compose 配置为空，立即停止后续部署，保留页面信息和项目名称交给 CDS 管理员排查。

完成 Phase 0 到 Phase 3，即可确认新版的主流程可用；Phase 4 验证恢复能力；Phase 5 验证新版没有覆盖旧流程。

## 日常使用

| 你的目标 | 推荐操作 |
| --- | --- |
| 发布新代码 | 推送代码，进入目标分支查看自动部署；需要手动触发时使用“重新部署” |
| 刷新后继续跟进 | 回到分支的“部署”页签，查看同一条部署记录 |
| 恢复同一个成功版本 | 在“不可变部署版本”中点击“重新部署此版本” |
| 新版本有问题 | 选择较早的可复用版本，点击“回滚到此版本” |
| 部署失败 | 先看“部署事实账本”的责任侧和建议，再决定改代码、改配置或联系 CDS 管理员 |
| 项目编排很复杂 | 继续使用 Compose 模式 |

## 本地启动 CDS

在仓库根目录执行：

```bash
./exec_cds.sh init
./exec_cds.sh start
./exec_cds.sh status
```

首次启动时，`init` 会引导完成必要配置。需要查看运行日志时执行：

```bash
./exec_cds.sh logs
```
