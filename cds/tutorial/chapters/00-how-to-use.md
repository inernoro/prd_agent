# 第 0 章：这本书怎么用

## 你在做什么

你正在建立一套可重复的 CDS 学习和操作方法，而不是浏览一堆彼此独立的功能说明。CDS 同时管理项目身份、Git 分支、构建配置、容器、基础设施、域名、RemoteHost、正式发布与运行证据。只记按钮位置很容易在多项目或生产环境中做错对象；本书要求每一步都回答“我在哪个项目、操作哪个分支、目标是什么、什么证据算成功”。

## 为什么要做

CDS 的问题往往不是某条命令不会执行，而是边界没有说清。例如把全局数据库地址注入所有项目、把另一个系统的站点登记到当前项目、看到本机 `127.0.0.1` 就误以为是生产、或者只看容器健康就宣布公网页面发布成功。这些错误都能让局部步骤显示绿色，却让最终用户看到错误页面。本书把身份、执行和验收放在一条连续链上，帮助你在操作前发现错误，而不是上线后追事故。

## 开始前检查

- 你知道目标 CDS 地址，不依赖 shell 中可能遗留的全局 `CDS_HOST`。
- 你知道目标项目的 `projectId`、仓库身份和默认分支。
- AI 调用时使用项目级 `cdsp_` Key；真人通过 CDS 登录会话操作。
- 你有一个已经成功运行的预览分支，或者准备先按基础篇创建。
- 生产操作前，已经确认域名、RemoteHost、远端项目仓库路径和回滚责任人。
- 任何命令输出中的密钥、私钥、口令和带凭据 URL 都不进入截图、教程或工单。

## 跟我做

1. 打开 CDS 的项目列表，先找到目标项目，再进入它的分支页面。记录页面展示的项目名、`projectId` 和仓库身份；如果三者和你的任务不一致，立即停止。

2. 在分支页面找到默认分支或本次要验收的功能分支。确认状态为运行中，并记录完整 commit。短 SHA 适合阅读，但发布证据必须保留完整 SHA。

3. 打开项目设置，依次看构建配置、基础设施、环境变量和 GitHub 连接。此时只观察，不修改。把每项配置归类为项目级、分支级或 CDS 控制面级。

4. 打开发布中心，观察顶部当前项目、活跃发布目标、生产主目标、发布方式和已归档目标。发布中心出现陌生系统名称时，不要尝试发布；先确认目标的项目身份快照和创建来源。

5. 选择自己的阅读路线。第一次接入走基础篇；已有预览要做正式发布，从第 9 章开始；处理生产故障，从第 14 至 17 章建立证据后再进入对应运维章。

6. 为本次学习建立一份操作记录，至少包含项目、分支、commit、预览地址、目标章节、成功证据和未解决问题。后续每章都在同一记录中追加，不另起互相矛盾的笔记。

## 看到什么算成功

你能够不依赖猜测回答：当前 CDS 地址是什么、当前项目是谁、仓库是什么、分支和 commit 是什么、预览地址是什么、正式发布目标是什么。发布中心只展示当前项目的活跃目标；陌生或错误目标在归档区有原因和操作者。你也知道容器健康、预览健康和公网正式入口健康是三种不同证据。

## 失败怎么办

### 项目名看起来相似，不知道选哪个

不要按显示名猜。比较 `projectId`、项目 slug、GitHub 仓库全名和默认分支。仍不确定时只做只读查询，不创建发布目标。

### 页面记住了上一次项目

发布中心会记住最近项目。切换项目后重新观察顶部项目选择器和卡片中的“所属项目”；不要因为 URL 没变化就默认上下文已正确。

### AI Key 能看到多个项目

这通常表示使用了全局 Key。全局 Key 可以用于只读运维，但不能写发布目标。为目标项目签发项目级 Key后重试。

### 只有容器状态，没有公网证据

当前最多只能写“运行链路已启动”。继续取得预览或正式域名的 HTML、入口资源和 API 证据，不能声称发布完成。

## 本章小结

CDS 的正确使用顺序是“锁定项目身份，锁定不可变版本，选择执行路径，观察持续进度，验证最终入口，保留回滚证据”。后续章节所有按钮和命令都服从这个顺序。
