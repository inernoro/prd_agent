# 第 15 章：建立真正的正式发布系统

## 你在做什么

你要把一个已经验收的 CDS 预览分支配置成正式发布目标，并让发布过程同时支持项目现有脚本、没有脚本的 Compose 项目、没有脚本的静态站。最终结果不是“一条能跑的 shell”，而是项目身份、发布策略、不可变 commit、预检、实时日志、最终入口、回滚版本和归档证据组成的发布合同。

## 为什么要做

传统“发布脚本输入框”会把所有风险压进一段不透明命令：脚本属于哪个项目、是不是误用了另一系统目录、执行的是本机还是远端、页面资源是否完整、失败后能否回滚，都要等出事故才知道。更危险的是，全局 AI key 可以把错误目标写入任意项目，发布中心再把多个环境平铺显示，用户很容易选错。本章使用 CDS 服务端策略发现和身份锁定，把这些错误前移到创建与预检阶段。

## 开始前检查

- 目标分支状态为运行中，完整 commit 已明确，预览页面已通过验收。
- 发布中心顶部项目、项目 slug 和仓库身份与任务一致。
- AI 使用项目级 Agent Key；全局 AI key 只做只读诊断。
- RemoteHost 已由管理员配置，目标项目有权引用对应 `privateKeyRef`。
- 远端 `appPath` 是当前项目的 Git 仓库，不是静态在线目录，也不是另一个系统的仓库。
- 已确认最终生产域名和健康检查路径。
- 静态发布时，Web Server 根目录已准备指向 `publicDirectory/current`。

## 三种发布策略

### 项目现有脚本

适用于仓库已经维护可信发布脚本的项目。CDS 检查脚本存在与执行权限，注入项目、分支、commit、目标、发布编号等标准环境变量，实时转发日志，最后执行公网健康检查。脚本仍是执行逻辑 SSOT；CDS 不会擅自改写项目脚本。

### CDS 动态 Compose 发布

适用于没有发布脚本但有 Compose 文件的项目。CDS 每次按完整 commit 在远端仓库同级的 `.cds-releases/<targetId>/worktrees/<releaseId>` 建立隔离 worktree，生成确定性 Bash 脚本，执行 `docker compose up -d --build --remove-orphans`，再用 Python `os.replace` 原子更新 `current/previous`。运行目录不会出现在项目的 `git status` 中；生成脚本只存在于本次运行，SHA256 写入运行记录，不污染项目仓库。

### CDS 动态静态站发布

适用于能明确构建命令和产物目录的前端项目。CDS 在 commit 隔离 worktree 中安装锁定依赖并构建，把产物复制到非在线临时目录，检查 `index.html`、实际引用的 JS/CSS 和文件非空，归一目录/文件权限为 `755/644`，再原子切换 `current` 并保留 `previous`。最终入口失败时自动尝试恢复上一成功版本。

## 跟我做

1. 进入“发布中心”，先在顶部选择项目。检查每张活跃卡片的“所属项目”和“生产主目标”标记。如果看到不属于当前项目的系统名称，不发布、不直接删除，记录目标 ID 后走归档。

2. 点击“添加站点发布”。第一步选择由当前项目使用的服务器。项目级 Key 不能把一个从未由本项目使用的 RemoteHost 引入进来；出现 `remote_host_scope` 时，让系统管理员先预置，不要换全局 Key绕过。

3. 填写站点名称、远端项目仓库路径和生产域名。仓库路径用于 Git fetch 和 commit worktree；静态在线目录由动态静态策略的 `publicDirectory` 单独配置，二者不能混为一处。

4. 进入“确认发布方式”。阅读项目身份和检测分支。CDS 会扫描当前分支工作区中的发布脚本、Compose 文件、`package.json`、锁文件和静态产物信号，再给出候选策略。推荐只是基于事实的起点，不能代替对现有脚本安全性的检查。

5. 如果选择“项目现有脚本”，确认命令只引用当前项目脚本，并且不会直接清空在线目录。若脚本使用漂移 tag、没有回滚或没有最终入口探测，先修脚本或改选动态安全策略。

6. 如果选择“CDS 动态 Compose 发布”，确认 Compose 文件是仓库相对路径，不能包含 `..`；确认 Compose 项目名只含小写字母、数字、连字符和下划线。远端必须有 Git、Bash、base64、Python 3、Docker 和 Docker Compose。

7. 如果选择“CDS 动态静态站发布”，确认构建命令使用项目规定的包管理器和锁文件；确认产物目录是仓库相对路径；确认静态发布根目录是绝对路径，且 Web Server 指向其 `current`。

8. 配置最终上线地址与健康路径。动态策略自带上一成功版本回滚；现有脚本模式只有项目确实提供独立回滚脚本时才填写。不要勾选一个不存在的 `rollback.sh` 来制造虚假的安全感。

9. 保存目标后回到分支列表，点击发布并先运行“发布前检查”。逐项确认分支运行、commit 明确、项目身份锁定、发布策略完整、SSH 可连接、脚本或动态依赖可用、上线地址与回滚版本状态。

10. 预检无 blocking fail 后，再确认“发布目标、服务器、目录、上线地址和执行计划”。点击开始发布，持续观察连接、计划、执行、健康检查和记录阶段。动态策略日志会输出脚本摘要和 SHA256。

11. 运行进入成功状态后，从生产域名请求根 HTML，从 HTML 解析真实 JS/CSS 并逐个请求，再验证 API 和专项服务。只有这些公网证据都通过，才能声明正式发布完成。

12. 若配置了错误目标，调用归档而不是删除。归档原因明确写“不属于当前项目”或具体配置错误。刷新后断言该目标从活跃区消失、在“已归档发布目标”中保留，并且其他项目和目标没有变化。

## 发布前检查详解

| 检查 | 通过标准 | 失败时不要做什么 |
|---|---|---|
| 分支部署成功 | 分支为 running | 不从 error/starting 分支发布 |
| commit 明确 | 有完整 pinned 或 GitHub SHA | 不用 latest 代替 |
| 可发布产物 | commit 与预览地址同时存在 | 不跳过预览验收 |
| 项目身份锁定 | ID、slug、仓库与目标快照一致 | 不手改 target.projectId |
| 远端仓库身份 | `appPath` 是 Git 根且 origin 与项目仓库一致 | 不只凭目录名判断系统归属 |
| 发布策略完整 | 三种策略字段满足服务端合同 | 不在前端临时拼命令 |
| 目标主机可连接 | RemoteHost 存在，SSH 成功 | 不输出私钥排错 |
| 动态依赖可用 | Git/Bash/base64 及对应工具存在 | 不把缺依赖误报成项目代码错 |
| 上线地址可访问 | 非首次发布时现网保持可访问 | 不先停旧站再发布 |
| 可回滚版本 | 找到上一成功运行 | 首次发布明确无回滚点 |

## 页面混乱如何避免

发布中心只加载当前项目的活跃目标；归档目标单独折叠。每张卡同时展示项目身份、发布方式、生产主目标、服务器、远端仓库、上线地址、当前版本和最近操作者。同一项目同一环境只允许一个启用的主目标。这样“名字像生产”不再是选择依据，用户可以用身份、环境和实际入口交叉确认。

动态脚本不是页面临时生成。前端只提交结构化策略；服务端负责校验相对路径、绝对发布根、Compose 项目名和项目身份，运行开始时生成脚本并固化哈希。目标保存后再修改配置，也不能改变已经开始的运行快照。

## 看到什么算成功

发布目标带有服务端生成的 `projectIdentity`；卡片清楚显示所属项目、发布方式和唯一生产主目标。没有发布脚本的项目能通过 Compose 或静态策略完成预检。发布运行记录包含完整 commit、策略模式、脚本 SHA256、阶段日志、最终状态和上一成功版本。生产根 HTML、真实入口 JS/CSS、API 与专项服务都返回预期结果。错误目标只出现在归档区，并保留原因、操作者和时间。

## 失败怎么办

### 出现 project_key_required

当前调用使用全局 AI key。停止写入，为目标项目签发 `cdsp_` Key；不要通过管理员权限绕过，因为这正是跨项目污染的主要入口。

### 项目身份检查失败

目标保存的项目 ID、slug 或仓库与当前 Project 不一致，或者远端 `appPath` 的 origin 指向另一个系统。归档目标并从正确项目重新执行策略发现；不要只改名称或目录名掩盖身份错误。项目未绑定 Git 仓库时，普通 SSH 目标也会被阻断，因为系统无法证明远端目录归属；先补项目仓库身份。

### 动态发布依赖不可用

根据检查信息安装明确缺失的 Git、Docker、Compose、Bash、base64 或 Python 3。若生产主机不允许安装，改用经过审查的项目脚本或单独执行器，不要删除依赖检查。

### 静态站切换前失败

常见原因是构建命令错误、`artifactDirectory` 不对、`index.html` 缺失或 HTML 引用的入口资源不存在。旧 `current` 尚未切换，应保持在线。修配置后重新从预检开始。

### 最终入口失败并触发自动恢复

查看 `auto-restore` 阶段。恢复成功时，任务仍然是发布失败，但旧版本已恢复；交付要同时写“新版本失败”和“上一版本已恢复”。恢复失败时立即按生产发布安全规则进入人工 previous 回切和公网复验。

### 发布中心仍看到陌生目标

确认顶部项目和请求的 `?project=`。如果目标确实属于当前项目但名称陌生，查看 `projectIdentity`、创建人、服务器和域名；没有证据前不要发布。确认错误后归档，随后检查是否还有全局变量或全局 AI key造成的其他污染。

## 本章小结

真正的发布系统不是“能执行脚本”，而是把身份、版本、策略、执行、最终入口、回滚和审计连成闭环。现有脚本是其中一种执行器；没有脚本时，CDS 通过确定性模板补足执行能力，但不会放弃项目边界和生产安全门。
