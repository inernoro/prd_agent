LLM Gateway 目标架构：统一协议入口、模型池权威和请求治理层

MAP 继续负责业务协议、run、会话、画布和素材生命周期；所有 AI 请求进入独立 LLM Gateway。 GW 负责多协议适配、统一 IR、appCaller 注册、模型池调度、平台密钥、上游发送、请求日志和操作审计。

  • GW-owned 模型池、平台、模型、Exchange
  • auto、pool、pinned 路由策略

appCaller registry model pools request governance
关联键 用途
requestId 端到端追踪同一次请求。

appCallerCode 绑定模型池、预算、限流、日志归因和控制台治理。

3. 路由模式

auto

调用方不指定模型。GW 根据 appCallerCode 和 requestType 的默认池调度。

默认业务模式 统一控价

pool

调用方指定某个 GW 模型池。GW 仍负责池内健康、fallback 和能力过滤。

客户隔离 实验池

pinned

调用方指定模型或 platformId + modelId。必须精确调用，但仍经过 GW 日志、密钥和审计。

4. 视觉创作生命周期

1

页面提交

用户在 MAP 页面发起 text2img、img2img、vision、video 或 ASR。

2

MAP 创建 run

MAP 校验业务参数，保存 run、画布、素材引用和状态。

3

Worker 执行

Worker 管理重试、进度、SSE 和资产生命周期。

4

GW Native 请求

MAP 只提交 appCallerCode、requestType、输入和 trace。

5

GW Router

GW 按模型池、能力、健康、预算和参数策略选上游。

6

Provider Adapter

适配上游协议，丢弃或拒绝不支持参数，记录 attempts。

7

MAP 保存结果

结果资产、业务状态和用户可见进度仍回到 MAP 生命周期。

尺寸、画布和素材是 MAP 的业务生命周期；模型选择、上游发送和失败解释是 GW 的治理职责。

5. 控制台应该回答的问题

问题 控制台证据

6. 与 OpenRouter 的关系

OpenRouter 心智 本系统采用方式
Unified API 四类入口统一转换为 GW Request IR。
Model fallback GW model pools、provider attempts、fallback reason。
Provider routing 健康、成本、能力、参数策略和 appCaller 绑定共同决策。
Router metadata 日志详情展示 routerTrace、actual model、pool、provider attempts。
Require parameters strict-require 对能力和字段级参数执行 fail closed。

本系统不是复制 OpenRouter。目标是统一治理 MAP 和外部系统的 AI 请求，防止业务系统直接分散请求上游。
