LLM Gateway Target Architecture

This page describes the target architecture. 模型池最终归属是 GW. MAP keeps business protocol and lifecycle. MAP does not own model routing in target state.

MAP / external systems -> GW ingress adapter -> GW Request IR -> appCaller registry -> GW router -> GW model pools -> provider adapter -> upstream model provider

Callers

MAPBusiness API, run lifecycle, sessions, canvas, assets, and user-facing workflow state.
External systemsUse GW-compatible protocols instead of calling MAP internals or provider keys.

Ingress and IR

GW ingress adapterNormalizes GW Native, OpenAI-compatible, Claude-compatible, and Gemini-compatible requests.
GW Request IRPreserves appCallerCode, requestType, modelPolicy, modelPoolId, pinned target, parameterPolicy, and droppedParameters.
appCaller registryPassive discovery, configured/active state, budget, rate limit, owner, and drift evidence.

Gateway routing

GW routerOwns auto, pool, pinned, fallback, strict parameter policy, and provider retry decisions.
GW model poolsOwns appCaller-specific pools, pool members, capability snapshots, provider and exchange binding.
provider adapterMaps protocol-specific bodies, auth, raw/multipart rehydration, and response normalization.

Upstream

OpenAI / Claude / GeminiProvider APIs reached only through Gateway-owned keys and routing decisions.
OpenRouter / Volcengine Ark / Doubao ASRExchange transformers and provider adapters keep raw protocols outside MAP.
Request linkagerequestId, sessionId, runId, appCallerCode.
GW observabilityrouterTrace, providerAttempts, droppedParameters, transport.
Migration evidenceshadow comparisons, release gates, rollout ledger.
Runtime gatesconfig authority execution, active MAP fallback exit, http-full acceptance, legacy cleanup.
