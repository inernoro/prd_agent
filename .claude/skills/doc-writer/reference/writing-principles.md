# 写作原则与合并历史

> 被 SKILL.md 引用。包含写作原则、受众分层、代码使用规范、类型合并历史。

---

## 核心理念：为人写，不为机器写

design 文档的读者不只是开发者。**老板、其他部门领导、观察者**都可能打开它。
如果一个人不写代码也能理解你的方案，这份文档就写对了。

---

## 写作原则

1. **受众分层** — 前四节（管理摘要 + 问题背景 + 设计目标 + 核心决策）必须让非技术读者看懂，这四节禁止出现代码；技术章节允许代码但必须有上下文
2. **结论先行** — 前两段让读者 30 秒内判断"跟我有没有关系"；每个决策先写结论再写分析
3. **代码有节制** — 能用表格说清的不放代码；放代码必须有前置说明（这是什么、为什么放）；全文代码总量不超过 **30%**
4. **单一职责** — 一份文档只解决一个主题
5. **可扫描** — 多用表格、编号列表、粗体关键词；避免大段散文

## 受众分层指引

| 读者 | 关注点 | 应能从哪些章节获取信息 |
|------|--------|----------------------|
| **老板/高管** | 为什么做、值不值、风险多大 | 管理摘要（30 秒读完） |
| **其他部门领导** | 影响我的团队吗、需要我配合什么 | 管理摘要 + 影响范围 |
| **产品/项目经理** | 做什么、怎么验收、排期影响 | 问题背景 + 设计目标 + 影响范围 |
| **开发者（本团队）** | 怎么实现、数据怎么存、接口怎么调 | 全文 + 附录 |
| **开发者（其他团队）** | 对我的接口有变更吗 | 接口设计 + 影响范围 |

**黄金法则**：从第一节读到第四节，不看任何代码，也能完整理解方案的 What / Why / How（概念层）。

## 代码使用规则

### 前四节（管理摘要 → 核心决策）：禁止代码

这四节是给老板和非技术读者看的，必须用纯文字 + 表格表达。

### 技术章节（整体方案 → 接口设计）：允许代码，但要节制

| 场景 | 推荐做法 | 何时放代码 |
|------|----------|-----------|
| 数据模型 | 字段表格 | 字段间有复杂嵌套关系时，可补一段精简结构体 |
| API 接口 | 方法 + 路径 + 用途表格 | 请求/响应结构复杂时，可补关键 JSON 示例 |
| 业务流程 | 编号步骤列表 | 不需要代码 |
| 状态机 | 状态转换表格 | 不需要代码 |
| 架构图 | ASCII 图 + 文字 | ASCII 图本身是代码块，这是允许的 |
| 关键算法/模式 | 先用文字说清思路 | 文字说不清时放精简代码 |

### 代码纪律

1. **每段代码前必须有一句话** — 说明这是什么、为什么放在这里
2. **精简** — 只放关键片段，省略 import/using/样板代码
3. **总量控制** — 全文代码不超过 **30%**（当前 37 篇平均 35%，目标逐步降到 30% 以下）
4. **禁止代码堆砌** — 连续超过 3 个代码块之间没有文字说明 = 代码堆砌，必须拆解

## 合并历史（为什么是 6 个而不是 11 个）

项目曾有 11 种文档前缀，但其中 5 种各只有 1 个文件，类型过度细分。2026-03-08 合并为 6 种：

| 已废弃前缀 | 合并到 | 理由 |
|-----------|--------|------|
| `story.*` | `spec.*` | 用户故事是规格的子视角 |
| `agent.*` | `spec.*` | Agent 产品文档本质是规格 |
| `analysis.*` | `design.*` | 技术分析是设计的前置研究 |
| `audit.*` | `rule.*` | 审计发现最终沉淀为规则 |
| `memo.*` | `guide.*` | 备忘录是非正式的参考 |

**如遇旧前缀引用**：自动替换为新前缀，不做兼容保留。
