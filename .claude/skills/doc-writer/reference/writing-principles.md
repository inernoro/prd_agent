# 写作原则与合并历史

> 被 SKILL.md 引用。包含写作原则、代码使用规范、类型合并历史。

---

## 写作原则

1. **设计优先，代码最少化** — 用流程图、状态表、决策矩阵描述设计意图；代码示例仅在"无法用文字清晰表达"时才出现
2. **结论先行** — 前两段让读者 30 秒内判断"跟我有没有关系"
3. **单一职责** — 一份文档只解决一个主题
4. **可扫描** — 多用表格、编号列表、粗体关键词；避免大段散文

## 代码使用原则

| 场景 | 是否放代码 | 替代方式 |
|------|-----------|----------|
| 数据模型 | 否 | 字段表格 |
| API 接口 | 否 | 方法+路径+用途表格 |
| 业务流程 | 否 | 编号步骤列表 |
| 状态机 | 否 | 状态转换表格或 ASCII 图 |
| 配置格式 | 视情况 | 用户需要复制的才放 |
| 架构模式 | 视情况 | 优先 ASCII 图 + 文字 |
| 编码规范 | 是 | 需展示正确/错误对比 |

**底线**：删掉代码块后核心信息没损失 → 不该放。

## 合并历史（为什么是 6 个而不是 11 个）

项目曾有 11 种文档前缀，但其中 5 种各只有 1 个文件，类型过度细分。2026-03-08 合并为 6 种：

| 已废弃前缀 | 合并到 | 理由 |
|-----------|--------|------|
| `story.*` | `spec.*` | 用户故事是规格的子视角 |
| `agent.*` | `spec.*` | Agent 产品文档本质是规格 |
| `analysis.*` | `design.*` | 技术分析是设计的前置研究 |
| `audit.*` | `rule.*` | 审计发现最终沉淀为规则 |
| `memo.*` | `guide.*` | 备忘录是非正式的参考 |

**如遇旧前缀引用**：自动替换为新前缀，不做兼容保留。
