# 验证方法详解

> 被 SKILL.md Step 4 引用。按需阅读，无需全量加载。

## 1. 逆向验证（Reverse Verification）

从预期结果反推，代码是否能产生这个结果？

**步骤**：
1. 明确期望的输出/行为
2. 从返回点反向追踪代码路径
3. 检查是否有遗漏分支导致不同结果

**模板**：
```
期望: {预期行为}
追踪:
- 返回点在哪？→ {函数:行号}
- 调用链是否完整？→ {A → B → C}
- 有遗漏分支吗？→ {检查结果}
```

## 2. 边界条件测试（Boundary Testing）

故意输入"刁难"的数据。

**检查清单**：
- [ ] 空值 / null / undefined
- [ ] 空字符串 vs null
- [ ] 空数组 vs null vs 单元素数组
- [ ] 数字: 0, -1, MAX_INT, 小数, NaN
- [ ] 字符串: 超长, 特殊字符, emoji, 换行符
- [ ] 日期: 时区边界, 闰年, epoch
- [ ] 并发: 同时两个请求修改同一数据

## 3. 用户场景模拟（User Scenario Simulation）

假装自己是真实用户，按正常流程操作。

**步骤**：
1. 列出 3-5 个典型场景
2. 对每个场景逐步追踪执行路径
3. 检查用户在每一步会看到什么

**模板**：
```markdown
## 场景: [名称]
**角色**: [普通用户/管理员/新用户]
**前置条件**: [状态]

1. 用户执行 [操作] → 代码路径: [调用链] → 预期: [结果] → ✅/❌
2. ...
```

## 4. 数据流追踪（Data Flow Tracing）

追踪关键数据字段从入口到出口的完整路径。

**模板**：
```
字段: {fieldName}
入口 → Controller(request.field)
       ↓ 类型一致？
处理 → Service(field)
       ↓ 转换正确？
存储 → Repository.Insert(entity.field)
       ↓ 赋值正确？
输出 → Response { field }
       ↓ 与输入一致？
```

## 5. 双向推导（Bidirectional Reasoning）

分别从"代码做了什么"和"需求是什么"两边推导。

**步骤**：
1. **正向**：读代码，总结实际行为
2. **反向**：读需求/注释，理解期望行为
3. **对比**：标注差异（遗漏/多余/不一致）

**输出**：
```markdown
| 行为 | 代码 | 需求 | 差异 |
|------|------|------|------|
| 参数校验 | 只校验 A | 校验 A 和 B | ⚠️ 遗漏 B |
| 审计日志 | 无 | 需要 | ⚠️ 遗漏 |
```

## 6. API 契约验证（API Contract Verification）

验证 API 的输入输出是否符合约定。

**检查项**：
- [ ] 请求参数是否都有验证？必填字段标记 [Required]？
- [ ] 响应格式是否与类型定义一致？
- [ ] 错误响应有正确的状态码和消息？
- [ ] 是否处理了所有可能的异常？

## 7. 状态机验证（State Machine Verification）

验证状态转换是否完整。

**步骤**：
1. 列出所有可能的状态
2. 绘制转换图
3. 检查每个合法转换有对应代码
4. 检查非法转换是否被阻止

**模板**：
```
[Created] --submit--> [Pending] --approve--> [Approved]
                                --reject-->  [Rejected]
⚠️ [Created] 能直接到 [Approved] 吗？→ 不应该
⚠️ [Rejected] 能重新提交吗？→ 确认需求
```

## 8. 行为对比验证（Behavior Comparison）

适用于重构场景。

**模板**：
```markdown
| 场景 | 重构前 | 重构后 | 一致？ |
|------|-------|-------|--------|
| 正常输入 | 返回 A | 返回 A | ✅ |
| 空输入 | 返回 null | 抛异常 | ❌ |
```

## 9. Devil's Advocate（对抗式质疑）

详见 SKILL.md Step 3。按优先级（正确性→错误处理→安全→性能→可维护性）逐级质疑，每级执行"质疑→辩护→裁决"三步。
