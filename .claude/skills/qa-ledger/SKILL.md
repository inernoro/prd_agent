# QA Ledger — 对话台账

在面向用户的最终回复**开头**固定追加一张「对话台账」表格,把当轮用户提的问题逐条登记。让用户一眼看到"我提了哪些事 / AI 处理到哪一步 / 还有什么没做"。

## 为什么要这个

历史血泪:用户连续反馈"AI 回答有时稀里糊涂、不够结构化"。根因是 AI 把多个问题揉成一段散文,用户读完不知道哪些被回答了、哪些被遗漏了。台账强制把每个问题落到一行,处理到哪是哪。

## 适用场景

- **默认开启**:任何最终回复(除纯工具输出 / 一句话确认)。
- 用户一次问了 2 个及以上的事(明显需要分项回应)。
- 多轮交付中需要追踪进度("上次提的 3 件事完成度?")。
- 触发词显式激活:"台账"、"用台账格式"、"结构化回答"、"/ledger"、"qa-ledger"。

## 不适用场景

- 用户只问一句、AI 一句话就答完(如 "这个文件几行?" → 直接答数字)。
- 纯工具输出(命令结果、日志贴回)。
- 用户明确说"简短回答"、"不要表格"。

## 执行流程

生成最终回复时按以下步骤:

1. **先把当轮用户消息里所有诉求逐条拆出来**(数字列表 / 问号 / "另外" / "顺便" / "还有" 都是分隔信号)。
2. 在回复**开头**追加二级标题: `## 本轮对话台账`
3. 紧跟一张 5 列 markdown 表格,列头固定:
   - `#` (序号,1 起)
   - `你的问题` (用户原话核心,≤ 30 字)
   - `我的解决方式` (一句话说做了什么,≤ 60 字)
   - `是否解决` (枚举: `已解决` / `进行中` / `未解决` / `已答复` / `阻塞`)
   - `其他备注` (路径 / commit / 风险 / 待确认事项,可空)
4. 表格之后再写**详细解释**(展开每行表格条目),细节按需展开,不必每条都长篇。
5. 如果本轮 push 了代码,在台账尾部独立一行写:
   `【push】commit <短 hash> · 预览 <preview-url>`

## 输出模板

```markdown
## 本轮对话台账

| # | 你的问题 | 我的解决方式 | 是否解决 | 其他备注 |
|---|---------|-------------|---------|---------|
| 1 | <用户问题1精简> | <做法1> | 已解决 | <文件:行 / commit / 风险> |
| 2 | <用户问题2精简> | <做法2> | 进行中 | <待确认事项> |
| 3 | <用户问题3精简> | <已答复说明> | 已答复 | (无代码改动) |

---

(展开详细说明,按表格序号对应)

### 1. <第 1 项详细>
...

### 2. <第 2 项详细>
...
```

## 状态枚举速查

| 状态 | 何时用 |
|---|---|
| `已解决` | 代码 / 文档 / 配置确实改动 + 已 push 或可直接看到效果 |
| `进行中` | 已开始但还有后续动作(如等待 CDS 部署、等待用户确认) |
| `未解决` | 没动手,理由必须写在备注里(权限不够 / 信息不足 / 用户要求先讨论) |
| `已答复` | 用户问的是事实/咨询类问题,只需要回答而无需改代码 |
| `阻塞` | 想做但被外部因素挡住(SDK 缺失、依赖未安装、需要密钥) |

## 质量自检

提交回复前逐条核对:

- [ ] 用户消息里的每一项诉求都有对应一行
- [ ] 状态列填的是上面 5 个枚举之一,没有自创词
- [ ] "我的解决方式"这一列没有空格糊弄(如"已处理"、"完成") — 必须说做了什么
- [ ] "其他备注"在涉及代码改动时含路径或 commit
- [ ] 涉及 push 时尾行有 commit + 预览地址
- [ ] 表格后展开的详细说明能被序号回溯到表格行

## 与其他技能的关系

| 技能 | 关系 |
|---|---|
| `dev-completion-report` 的 200 字总结 | 互补。本技能在**开头**给"已经做了哪些",完工报告 Part 1 在**结尾**给"小白通俗总结"。前后呼应。（原独立的 `cn-brief-summary` 已合并进完工报告并下线） |
| `dev-completion-report` | 取代关系。`dev-completion-report` 是开发完成后的三段式报告(总结 + 改动清单 + 行业对比),粒度更粗。台账每轮都用,完工报告是整个任务结束时一次性输出。 |
| `task-handoff-checklist` | 互补。台账记录"对话过程中提了哪些问题",交接清单记录"代码层面交付了哪些东西",两者维度不同。 |

## 历史背景

2026-05-08 用户反馈"我提问的清单,你的回答,有时候总是稀里糊涂的,不够结构化"。当场决定建本技能,杜绝同类问题再发生。
