## 定位

本技能从产品评审智能体（`review-agent`）的核心业务逻辑抽离而来，用于对产品方案、需求方案、设计草案做严格评审。

抽离范围：
- 保留：评分维度、通过线、反堆砌规则、清单类规则核查、系统派生结论、输出结构。
- 不保留：文件上传、Attachment 文本提取、SSE 流式事件、MongoDB 状态机、Webhook 通知、申诉受理、排行榜统计。

核心结论：这是一个“评审口径 Skill”，不是一个“任务运行系统”。拿到方案正文后即可运行。

## 触发词

- "产品评审"、"方案评审"、"评审这份方案"、"帮我给方案打分"
- "这份 PRD 能不能过评审"、"按产品评审员标准看一下"
- "review product proposal"、"score this proposal"
- `/product-review`

## 适用场景

| 场景 | 输入 | 输出 |
|------|------|------|
| 评审会前自检 | 产品方案 Markdown | 总分、是否通过、分项评分、修改建议 |
| PRD 草稿把关 | 需求描述或设计草案 | 风险缺口、缺失章节、可测试性反馈 |
| 方案质量校准 | 多份候选方案 | 按同一评分口径横向比较 |
| 规则清单核查 | 带合规/平台规则表的方案 | 逐项判断是否涉及、是否覆盖、是否有正文证据 |

不适用：
- 只问功能价值是否值得做：优先用 `skill-validation`。
- 只评技术变更风险：优先用 `risk-matrix`。
- 只检查代码实现：优先用 `human-verify`。

## 输入要求

最低输入：

```markdown
方案标题：
方案正文：
```

推荐输入：

```markdown
方案标题：
产品线/模块：
迭代类型：
目标用户：
现状背景：
问题陈述：
用户范围：
实现思路：
验收标准：
全局规则检查清单：
```

如果用户只给了碎片内容，也要先给出评审，但在“缺失信息”中明确扣分原因，不要反问用户补齐后再开始。

## 执行流程

```text
执行进度：
- [ ] Step 1: 识别方案结构与关键论点
- [ ] Step 2: 按全局规则检查清单做二段判断
- [ ] Step 3: 按八个维度严格评分
- [ ] Step 4: 系统派生总分与是否通过
- [ ] Step 5: 输出优先级修改建议
- [ ] Step 6: 做自检，避免高分虚高或结论矛盾
```

### Step 1: 识别方案结构

先提取以下要素：
- 方案目标：要解决什么业务问题。
- 用户与场景：谁受影响，核心诉求是什么。
- 当前问题：现状、根因、数据或事实证据。
- 方案做法：结构归属、功能边界、关键流程。
- 成功标准：上线后如何证明有效。
- 风险和依赖：权限、旧数据、外部系统、运维、回滚。

缺少任何要素都要在对应维度扣分。

### Step 2: 全局规则检查清单

默认清单维度满分 30 分。每个检查项都按“是否涉及 / 方案是否包含”二段判断。

| 分类 | 检查项 |
|------|--------|
| 安全与权限类 | 风控接入（黑名单）、权限控制、应用权限配置规则、操作日志写入、用户授权（移动端）、手机号验证组件、3+2 账号单点登录、用户注销、用户协议 |
| 组件与框架类 | 新增子系统/应用的新 UI 框架顶部导航规范、移动端底部“米多技术支持”与“投诉”组件 |
| 业务功能类 | 短信费、短信签名、消息通知、门店一对多个上级经销商、门店账号一个手机支持多个门店 |
| 系统边界与集成类 | 跨子系统/应用母体、能力依赖、数据互通、入口挂载 |
| 数据与存量类 | 旧数据处理、历史存量、迁移、兼容、清洗、字段结构变化影响 |

逐项判断规则：

| involvedChecked | coverageChecked | solutionFound | 系统判定 |
|---|---|---|---|
| none | 任意 | 任意 | 不通过，视为未勾选 |
| no | 任意 | 任意 | 通过，用户声明不涉及 |
| yes | none | 任意 | 不通过，涉及但未声明是否包含 |
| yes | no | 任意 | 不通过，用户自认未包含 |
| yes | yes | true | 通过，且方案正文确有对应设计 |
| yes | yes | false | 不通过，勾选已包含但正文找不到证据 |

清单得分公式：

```text
清单得分 = floor(30 * 通过项数 / 总项数)
```

注意：
- “声明不涉及”是合规通过，不得写成“未覆盖”或“缺失”。
- 若用户没有提供清单表格，所有项按 none 处理。
- 若用户勾选“已包含”，必须在正文找到对应解决方案，否则按作弊或误填处理。

### Step 3: 评分维度

总分 100 分，通过线固定为 80 分。

| # | 维度 | 满分 | 评分重点 |
|---|------|------|----------|
| 1 | 全局规则检查清单 | 30 | 平台硬性业务/技术规则是否逐项判断并覆盖 |
| 2 | 文档规范完整性 | 8 | 文档头、项目目的、现状问题、用户范围、实现思路是否齐全 |
| 3 | 内在自洽性 | 14 | 目的、问题、用户诉求、方案是否闭环且无矛盾 |
| 4 | 问题陈述质量 | 11 | 现状是否具体，问题是否指向根因，需求是否可执行 |
| 5 | 用户价值清晰度 | 10 | 商业目标、客户价值、用户角色和诉求是否清楚 |
| 6 | 实现思路可行性 | 10 | 结构归母是否明确，设计是否具体，风险是否可控 |
| 7 | 需求可测试性 | 7 | 是否有可验证完成标准，可推导验收用例 |
| 8 | 表达质量与凝练度 | 10 | 信息密度、简洁度、术语一致性、反堆砌表现 |

### Step 4: 严格评分纪律

必须遵守：
1. 多数合格方案应落在 75-89 分，不要轻易给 90 分以上。
2. 90 分以上必须同时满足：
   - 内容完整；
   - 表达凝练；
   - 至少 5 处具体数据、链接、截图、对比或配置项；
   - 实施路径可直接落地；
   - 有非显而易见的洞察。
3. 95 分以上只给行业范本级方案；不能对外作为范本，就降到 90-94。
4. 内容缺失或质量差的维度应给 0-60% 得分率，不得“意思意思”给高分。
5. 每个 comment 都必须指出具体证据、缺失章节或原文问题。

反堆砌硬规则：
- 同一观点跨章节反复展开，必须在“表达质量与凝练度”扣分。
- 空话套话占字数，必须扣分。
- 一句话能说清却拆成多段，必须扣分。
- 列表项高度重叠，必须扣分。
- 大段背景资料与方案无关，必须扣分。
- 若方案可删减 30% 且不损失信息，“表达质量与凝练度”按 50% 得分率封顶。
- 若通篇为空话套话无具体抓手，“表达质量与凝练度”按 30% 得分率封顶。
- 凝练但短不扣分；质量看信息密度，不看字数。

### Step 5: 系统派生结论

```text
totalScore = 各维度 score 之和
isPassed = totalScore >= 80
```

结论必须以系统派生分数为准，不得让总结文字与分数矛盾。

必须在总结末尾追加：

```text
[系统结论] 最终得分 {totalScore}/100，{已通过|未通过}。
```

## 端到端示例

输入：

```markdown
方案标题：移动端消息通知升级

目标：提升消息触达率。
实现思路：新增短信提醒和站内信提醒。
验收：上线后用户能收到通知。
```

输出节选：

```markdown
## 评审结论

| 项 | 结果 |
|----|------|
| 方案 | 移动端消息通知升级 |
| 总分 | 54/100 |
| 结论 | 未通过 |
| 主要判断 | 方案涉及短信费、短信签名、消息通知和移动端用户授权，但未给出规则清单、费用归属、签名申请、授权链路与可量化验收指标。 |

## 主要问题

| 优先级 | 问题 | 影响 | 修改建议 |
|--------|------|------|----------|
| P0 | 涉及短信能力但未说明短信费扣平台还是扣商户，也未说明短信签名 | 全局规则检查清单、实现思路可行性 | 补充计费归属、签名来源、失败兜底、配置入口 |
| P0 | 移动端通知涉及用户授权，但方案未写授权弹窗、拒绝后的降级路径 | 全局规则检查清单、用户价值清晰度 | 补充授权流程、拒绝状态、再次引导策略 |
| P1 | 验收标准只有“能收到通知”，不可量化 | 需求可测试性 | 改为触达率、延迟、失败率、覆盖渠道等可验证指标 |

[系统结论] 最终得分 54/100，未通过。
```

## 自检清单

输出前必须检查：
- [ ] 是否给出了总分和是否通过。
- [ ] 每个维度是否都有具体依据，不是泛泛评价。
- [ ] 清单类规则是否按 truth table 派生，而不是让 LLM 主观判定。
- [ ] “声明不涉及”是否被正确视为通过。
- [ ] 总结文字是否与系统派生分数一致。
- [ ] 90 分以上是否满足高分纪律。
- [ ] 是否指出了可执行的修改建议。
- [ ] 输出中没有使用 emoji。

## 上下游技能

```text
skill-validation -> product-review -> risk-matrix -> plan-first
                         |
                         +-> acceptance-checklist
```

推荐组合：
- 方案刚提出：先用 `skill-validation` 判断需求是否值得做，再用本技能评审文本质量。
- 方案准备上线：本技能通过后，用 `risk-matrix` 查技术/上线风险。
- 需要真人验收：用 `acceptance-checklist` 把通过的方案转为 UAT 清单。
