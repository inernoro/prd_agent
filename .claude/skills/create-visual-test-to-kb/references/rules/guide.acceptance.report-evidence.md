# MAP 验收报告与证据交互 · 指南

> **版本**：v1.0 | **日期**：2026-07-07 | **状态**：已落地

关联规范: `doc/rule.acceptance.map-enterprise.md`。

## 1. 目标

本规范解决报告“看不懂、点不到、找不到失败、图文断链、引用敷衍”的问题。企业级报告必须做到:

- 首屏看得出 Verdict，失败必须明显。
- 左侧、正文、右侧证据栏联动。
- 表格内图号、用例号、缺陷号、章节锚点可点击。
- 截图能独立说明证明点。
- 引用先用自己的话解释，再给来源。

## 2. 信息架构

| 区域 | 必须内容 | 交互要求 |
|---|---|---|
| 首屏结论区 | Verdict、失败数量、阻断项摘要、验收对象、分支、commit、预览地址 | 不通过必须红色主视觉，首屏无需滚动即可看到 |
| 左侧章节导航 | 速览、需求对应、用例、证据、缺陷、截图回读、覆盖缺口、门禁结果 | 点击任一项必须滚动或切换到对应章节，并高亮当前章节 |
| 主内容区 | 审计正文、需求逐条对应、用例表、缺陷表、覆盖矩阵 | 表格内图号、用例号、缺陷号必须可点击 |
| 右侧证据栏 | 当前章节关联截图、证据卡、失败截图、前后对比图 | 随左侧章节切换；点击缩略图打开大图 |
| 图片对比区 | 左图/右图对比，例如改造前/改造后、预期/实际、暗色/亮色 | 必须显示标题、图号、时间、关联用例 |
| 底部审计区 | 链接检查结果、截图回读结果、未覆盖项、验收门禁结论 | 任一门禁失败，整份报告不得判通过 |

## 3. 证据对象模型

每个证据对象至少包含:

```ts
type Evidence = {
  id: string;
  kind: "screenshot" | "api" | "log" | "video" | "manual-note";
  title: string;
  claim: string;
  userRequirement: string;
  expected: string;
  actual: string;
  verdict: "pass" | "conditional" | "fail" | "uncovered";
  severity?: "P0" | "P1" | "P2" | "P3";
  chapterId: string;
  caseId?: string;
  imageUrl?: string;
  anchor: string;
  annotations: Array<{
    label: string;
    color: "red" | "orange" | "blue" | "green";
    target: string;
  }>;
  sourceRef?: {
    text: string;
    url?: string;
    line?: string;
  };
};
```

`claim`、`expected`、`actual`、`userRequirement` 不能为空。只写“图01”“已验证”“页面正常”不合格。

## 4. 失败呈现

| 场景 | 呈现方式 |
|---|---|
| 总 Verdict 不通过 | 首屏红色状态条、红色 Verdict、失败数量、P0/P1 摘要 |
| 某用例失败 | 用例行红底或红色左边框，状态为 fail，证据列跳失败截图 |
| 某截图证明失败 | 图内红框标注，caption 明确写失败原因 |
| 可点击项不可点 | 标为 P0/P1，列出点击目标、点击方式、实际无响应或错误 |
| 章节点击无效 | 标为 P1，必须截图证明左侧点击后未切换 |
| 引用不足 | 标为报告质量缺陷，不能作为通过证据 |
| 证据缺失 | 标为 uncovered，不允许包装成 pass |

颜色统一:

| 颜色 | 含义 |
|---|---|
| 红色 | 阻断、失败、P0/P1 |
| 橙色 | 风险、体验缺陷、P2/P3 |
| 蓝色 | 路径、环境、可达性说明 |
| 绿色 | 通过证据 |

同一种颜色不能同时表示通过和失败。

### 4.1 视觉问题定位三件套

报告写出 P0/P1/P2 视觉问题时，必须同时满足:

| 要素 | 要求 | 不合格例子 |
|---|---|---|
| 缺陷文字 | 写清页面区域、异常物、被影响对象、用户影响 | `遮挡问题` |
| 图内标注 | 在截图里用红/橙框或圈标出具体区域，标签写严重级和现象 | 只有 caption，没有框 |
| 证据跳转 | 缺陷行链接到对应图号，点击后能看到该截图和章节上下文 | `见图`、裸 `图02` |

合格表达示例: `P1: 右下缺陷通知卡遮挡知识库卡片操作区，影响卡片按钮重复操作；[图02](#fig-02-document-store)`。

如果问题是轻微观感且不影响主要路径，标为 P3 或 observation，并说明为什么不影响 Verdict。

## 5. 截图回读

每张截图必须完成回读:

| 字段 | 要求 |
|---|---|
| 截图名 | 稳定编号和语义名，例如 `04-report-link-popup` |
| 对应测试 | 关联测试单元或用例 |
| 用户心智 | 读者应从图中看懂什么 |
| 页面上下文 | 页面标题、导航或关键区域入镜 |
| 关键区域 | 指向性截图必须画框或圈 |
| 加载状态 | 不得是骨架屏、空白 iframe 或未加载完内容 |
| 图文一致 | caption 声称的内容必须出现在图中 |
| 结论贡献 | pass、conditional、fail 或 uncovered |

失败截图必须保留。若失败是产品问题，截图作为缺陷证据；若失败是截图操作问题，必须重拍并记录重试。

## 6. 链接可用性

| 链接类型 | 要求 |
|---|---|
| 左侧章节链接 | 点击后必须到对应章节，当前章节高亮 |
| 图号链接 | `[图01](#fig-xxx)` 必须跳到对应图片，不允许裸“图01” |
| 缺陷链接 | 缺陷表里的证据列必须跳到失败截图或失败章节 |
| 用例链接 | 用例表里的证据必须跳到截图、日志或 API 证据 |
| 预览链接 | 必须是可直接验收的深链，不是根域名 |
| 图片链接 | 缩略图和大图都必须可打开，不能 404、空白、跨域失败 |
| 外部引用 | 必须包含上下文，不能只写“官方说明”“见文档” |

链接检查必须全量执行。任一核心失败项链接不可达，报告不能判通过。

## 7. 可点击项验收

对“能点击”的要求必须有三段证据:

1. 点击前: 目标元素可见，位置和语义明确。
2. 点击动作: 用真实左键或键盘触发，不只检查 href。
3. 点击后: 新页面、章节滚动、弹窗、状态变化或错误提示可见。

若因浏览器 sandbox、CSP、iframe 或路由阻止导致点击无效，不能写“链接存在”。必须标为失败或缺口。

## 8. 右侧图片与章节联动

当报告采用左侧导航、主内容区、右侧证据栏布局时:

- 左侧点击章节后，主内容必须滚到该章节标题稍上方。
- 右侧必须显示当前章节关联证据。
- 若当前章节没有图片，右侧应显示“本章节无图片证据”及原因，不能空白。
- 图片缩略图应有标题、图号、对应用例和 Verdict。
- 失败章节默认优先显示失败截图。

## 9. 反例

| 反例 | 为什么不合格 |
|---|---|
| Verdict 写“不通过”，但顶部没有红标 | 用户第一眼看不出失败 |
| 左侧目录能显示，但点击后主内容不跳转 | 目录只是摆设 |
| 右侧有图片，但与当前章节无关 | 图文断链 |
| 证据列只写“图01”且不能点 | 无法定位证据 |
| 引用只写几个字加链接 | 没有上下文，不能支撑结论 |
| caption 写“所有可点项正常” | 结论过大，缺少逐项点击证据 |
| 失败截图没有红框，只在图外说明 | 读者看图时无法定位问题 |
| 用 API 200 证明 UI 交互通过 | 内部证据不能替代用户可见行为 |
| 临时 HTML 上传成每日验收报告 | 绕过标准模板，读者会看到不一致的样式、空白证据栏或不可点击链接 |

## 10. 报告交付门禁

报告归档或交付前必须全部通过:

1. 首屏 Verdict 明确；不通过必须红标。
2. 用户反馈逐条进入需求一一对应表。
3. 每条通过结论至少关联一条有效证据。
4. 每条失败结论必须有红色标注截图或可复现记录。
5. 左侧章节导航全量可点。
6. 右侧图片栏或证据栏必须随章节联动。
7. 所有可点击项必须完成点击验证。
8. 所有图号、用例号、缺陷号、章节锚点必须可跳转。
9. 所有图片必须可打开，缩略图和大图一致。
10. 引用必须是完整语义句，说明来源、上下文和证明点。
11. 截图回读必须全量完成。
12. 存在 P0/P1、核心链接不可达、核心证据缺失时，报告只能判不通过。
13. P0/P1/P2 视觉问题必须通过图内框选定位；读者只看截图也应能知道问题在哪里。
14. 报告必须有规范一致性自测：实际技能链、深度标签、证据门禁和引用文档必须对得上。
15. 执行类验收 HTML 必须带标准模板标记或历史标准结构；CDS 拒收临时手写 HTML。

## 11. 模板血统门禁

每日验收和视觉验收报告的 HTML 是阅读层，不是自由发挥的页面。正确链路是:

```text
Markdown 写作源 + manifest + 截图资产 -> archive_report.py -> 标准交互 HTML -> CDS /api/reports
```

标准交互 HTML 必须带 `map-acceptance-template` 标记，或保留历史标准结构 `.layout`、`.hero`、`.evidence-nav`、`#reportBody`。CDS 在创建或更新报告时会识别执行类验收 HTML；若缺少这些标记，返回 `422 acceptance_html_template_required`。

这条门禁只约束执行类验收 HTML。普通 HTML 附件、Markdown 规范文档和 `MAP自动化测试规范` 文件夹里的 SSOT 文档不受影响。规范文档统一以 Markdown 发布，执行报告才使用标准交互 HTML。
