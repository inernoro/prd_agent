# MAP 验收标准 v2（Acceptance Standard v2）

> 本文件是"下限基线"。任何功能/PR 验收**至少**满足本标准,才能声称"已验收"。
> 主纲在 `SKILL.md`,本文件是按需加载的完整规则。项目无关,配置见 `acceptance.config.json`。

---

## 0. 为什么需要这个标准（根因)

历史教训(2026-05-26):一次知识库分享功能的验收,归档报告在阅读器目录里标题显示成 `---`,验收者看不懂;且取证全程用 `page.goto(深链)` 直达页面,**绕过了导航**,根本测不出"功能做了但入口没加进菜单"这类真人必撞的问题。根因是**没有标准**——连报告格式、命名、取证路径都是临时拍的。

本标准把这些隐性坑显式化,定义可执行的下限。

---

## 1. 核心原则

| 原则 | 含义 |
|------|------|
| **模拟人类** | 取证走真人路径:点击导航进入,不用地址栏直达。这样才能在过程中发现"入口缺失/路径断裂"等新问题。 |
| **人类可读优先** | 报告首屏是给人看的"验收速览"(Verdict + 一句话结论 + 元信息表),不是机器字段(YAML)。 |
| **证据可追溯** | 每条结论挂一张截图;截图带 caption 说明"证明了什么";命名可排序。 |
| **定义下限** | 验收者无法穷举未知问题,但标准保证已知维度全覆盖(下限),不漏检。 |

---

## 2. 命名标准（强制)

- **报告条目标题 / 文件名**(固定结构,用户定 2026-05-27):`{项目} · {模块} · {功能} · {操作方式} · 验收报告`
  - 例:`prd-agent · 网页托管 · SaaS空间模型多账号MECE · 修复 · 验收报告`
  - 操作方式 = `新增功能` / `优化` / `修复`。脚本按 `--module/--feature/--type` 拼装,空段自动跳过。
  - **状态(通过/不通过)绝不进标题**——通过 `tags` 标记(脚本自动写 `[verdict_cn, type, tier]`)。不靠改名表达状态,这样同一份报告复测翻转结论时改 tag 即可,标题恒定可检索。
- **report_id**(机读,frontmatter):`acc-{project}-{YYYYMMDDHHmm}-{slug}`
- **截图文件名**:`{两位序号}-{动作}.png`,如 `03-after-subscribe.png`

### 2.1 防 `---` 硬规则（必须遵守,实测根因)

实测链路(2026-05-26 三次撞坑后定死):
1. `PUT /entries/{id}/content` 端点用**正文起始字符**重算 `entry.summary`(会覆盖建条目时传的 summary);
2. 分享阅读器 `LibraryShareReader` 的目录显示名 = `entry.summary` 的**首个非空行**。

所以**正文(content)绝对不能以 YAML frontmatter `---` 开头**——否则 summary 首行 = `---`,目录里所有报告都显示 `---`(这就是历史 bug 的真因)。

**硬规则**:报告正文**第一行必须是 `# {完整报告名}`**(ATX H1)。机读字段(report_id/verdict/...)放正文**末尾的 HTML 注释** `<!-- acceptance-meta ... -->`(不渲染、不进 summary、可被工具解析)。`archive_report.py` 已内置:`content = "# {title}\n\n" + body + "<!-- acceptance-meta ... -->"`。手工归档也必须照此,**禁止 frontmatter 打头**。

### 2.2 防「断头报告」硬规则（必须遵守,实测根因)

doc-store 归档是两步:`POST /entries`(建标题条目)→ `PUT /entries/{id}/content`(写正文)。若 PUT 撞上预览环境 524/重启而丢失,就留下一条**有标题、点开却"暂无可预览的内容"**的空壳条目——用户看得到名字看不到内容,这是真实踩过的坑(2026-05-27)。

**硬规则**:建条目写正文后,**必须** `GET /entries/{id}/content` 校验 `hasContent==true`;写不进时**自动删除空壳条目并报错**,绝不把半截条目留在库里。`archive_report.py` 已内置(校验→重写一次→仍失败则删条目 + 抛错,main 打印「归档失败」exit 3)。**结论:要么一条完整可看的报告,要么干脆没有,不留断头。**

## 3. 验收档位（下限定义)

| 档 | 适用 | 最少用例 | 最少截图 | 必查维度 | 双主题 | 负面/回滚 |
|----|------|---------|---------|---------|--------|----------|
| **L0 冒烟** | 任何 push 后的最低保障 | 1（目标页打开+核心动作不报错） | 1 | 功能适合性 | 否 | 否 |
| **L1 单项** | 单个改动点 | ≥3 | ≥3 | ≥3 个 ISO25010 | 推荐 | 推荐 |
| **L2 全链路** | 多功能串成用户旅程 | ≥6 | ≥5 | ≥4 个 ISO25010 | 强制 | 强制 |

声称"已验收"= 至少跑到 L1;声称"功能完成端到端"(CLAUDE #8)= L2。

## 3.5 准入门槛（入口准则)——不达标直接拒收

> 输入不对,输出不可能对。归档前 `archive_report.py` 会做**强制准入校验**,任一条不达标**直接退出报错、不写入知识库**(`--force` 可显式越过,但会打印告警)。目的:杜绝"什么测试都能进"污染质量资产库。

| # | 准入项 | 通过条件 | 不通过=拒收原因 |
|---|--------|---------|----------------|
| 1 | **验收目标有意义** | `--target` 非空、≥4 字、非占位垃圾(`test`/`测试`/`xxx`/`demo`/`tmp`) | "目标无意义,说不清在验什么" |
| 2 | **档位/Verdict 合法** | `tier ∈ {L0,L1,L2}`;`verdict ∈ {pass,conditional,fail}` | "档位或结论非法值" |
| 3 | **截图数达档位下限** | L0≥1、L1≥3、L2≥5(见 §3) | "证据不足,不到该档下限" |
| 4 | **证据完整** | 每张截图文件存在、非空(>1KB)、`caption` 非空且非占位 | "截图缺失/空白/无说明" |
| 5 | **报告结构完整** | 含 `Verdict` 行 + 验收用例(表) + 缺陷清单段;含 `{{IMG:<name>}}`(ZZ 逐步) 或 `{{EVIDENCE}}`(集中) 至少一种占位 | "报告缺核心段落,截图无处可挂" |
| 6 | **无半成品残留** | 正文无未替换模板占位(`{YYYY`/`{target}`/裸 `{{`)、无 `TODO`/`待填`/`待补` | "报告还没写完" |

**与业界对齐**:这就是 ISTQB / ISO·IEC·IEEE 29119 的**入口准则(entry criteria)**——开始/接受测试前必须满足的前置条件。第 7 节 Verdict 规则是**出口准则**(能不能判通过);本节是**入口准则**(够不够格被归档)。两道闸一前一后。

> Verdict 一致性(verdict=通过 却存在未决 P0/P1)属语义校验,自动校验只做结构层(1-6),语义层由 §7 Verdict 规则人工/工具把关。

## 4. 浏览器操作标准（模拟人类)——本标准的核心升级

1. **禁止地址栏直达 app 内页**:`page.goto` 只允许用于(a)登录页、(b)**外部分享/深链**(用户本就是从站外点链接进来的,如 `/library/share/{token}`)。**app 自有的可导航页面**(知识库、设置、各 agent)一律从首页**点击导航**进入(`gotoByClick`)。判据:这个 URL 用户平时是"从菜单点进去"还是"从外部链接点进来"?前者必须点击,后者可直达。
2. **找不到入口 = 一条缺陷**:若从导航点不到目标页(未注册菜单/Cmd+K/导航顺序),记 **P1**(功能做了但用户找不到)。这是 `page.goto` 永远测不出、而真人一定会撞的问题。
3. **按可见文本/role 定位**,不按写死 index(DOM 一变 index 就错)。
4. **先 snapshot 再操作**:每步操作前等关键元素渲染(`waitForSelector`),不裸 `sleep`。
5. **登录走表单**:填用户名/密码 + 点提交,不注入 token(token 注入可能漏掉 SPA 初始化的其他状态)。

## 5. 截图标准（强制)

- 基线视口 `1440×900`,`deviceScaleFactor: 2`(高清)。
- 每个关键步骤一张;关键断言区(toast / 卡片 / 页签激活态 / 空状态)必须清晰可见。
- **页面 header / 标题区必须入镜**(证明"在对的页面")。
- 每张图配中文 caption:**"这张证明了什么"**,不是"这是什么页面"。
- **双主题**(L1 推荐 / L2 强制):核心页面暗 + 亮各一张。应用无亮色主题时,单张 + 在报告里注明"应用暗色 only"。
- **测试矩阵**:覆盖维度沿用 issues 视觉验收的四轴 —— **视口 × 主题 × 状态 × 交互态**(见 `doc/rule.issues-system.md` §5 / 元 issue #605)。L2 必须按矩阵展开,不是只截一个 happy path。

## 5.1 硬约束与严重级:复用既有 SSOT(不重写)

视觉验收的**方法论**(测什么、什么算缺陷、怎么分级)已是项目 SSOT,本标准**引用**而非重写,避免两套漂移:

- **10 条项目硬约束**:`doc/rule.issues-system.md` §5.1(无 emoji / 白天无暗色 modal / Modal 三约束 / 全高布局 / 空状态 CTA / LLM 模型名可见 / 禁 >2s 静止 / 输入零摩擦 / 画布手势统一 / MAP Loader)。报告 §7 逐条核查,按改动类型勾或标 na。
- **P0-P3 严重级 + 项目实例**:`doc/rule.issues-system.md` §5.2(P0 白天黑底/emoji/Modal 撑破/登录断/空状态无 CTA;P1 模型名缺/加载不统一/双主题不一致;P2 配色间距;P3 动效文案)。
- **真值入口**:模板与矩阵的唯一可改入口是元 issue #605,§5 是离线镜像。本技能与 issue 体系**共用同一份方法论**。

## 6. 报告结构（写入知识库的 Markdown)

### 6.1 首屏"验收速览"卡(可见,非 YAML)

报告正文**第一行是 H1 标题**(见 §2.1 防 `---`),紧接人类可读速览,顺序:

```
# {项目} · {时间} · {目标}              ← 必须第一行,禁止 frontmatter 打头

> Verdict: 通过 / 有条件通过 / 不通过        ← 带状态词,醒目
> {一句话结论}

| 项目 | 目标 | 分支 | commit | 预览 | 验收人 | 日期 | 缺陷 P0/P1/P2/P3 |
|---|---|---|---|---|---|---|---|
| ... | ... | ... | ... | ... | ... | ... | 0/0/0/1 |
```

机读字段(report_id / standard / verdict / dimensions 等)放正文**末尾 HTML 注释** `<!-- acceptance-meta ... -->`:不渲染、不进 summary、可被工具 grep。**绝不用 YAML frontmatter 打头**(会让目录显示 `---`,见 §2.1)。

### 6.2 九段正文

1. **目标与价值**:做什么 + 为什么做(修复/新增的意义)。
2. **范围与不覆盖**:测了什么、明确没测什么(后续波次)。
3. **验收路径**:从登录后首页**点击进入**的完整路径(证明走的是真人路径,不是直达)。
4. **完成标准 DoD**(CLAUDE #8,逐项勾)。
5. **自测路径**(CLAUDE #8.1,至少一条 + 结论)。
6. **验收用例表**:`# | Phase(前置/执行/验证/回归/回滚) | 维度(ISO25010) | 操作 | 预期 | 实际 | 状态 | 严重级 | 证据图`。这是"测试用例大全集"。
7. **硬约束 + 截图标准核查**:项目前端硬约束 + 本标准 §4/§5 自查勾选。
8. **缺陷清单**:P0-P3,每条含 **现象 + 修复意义 + 注意点**(无则写"无")。
9. **证据 + 结论整改**:内联截图(引用上传后的 URL) + 对照 Verdict 规则的最终结论 + 非阻塞整改项。

### 6.3 ZZ 照做风（默认报告风格,用户定死)

报告写给"同岗位的下一个人照着做一定能复现",所以默认走 **ZZ 照做风**(模板 `templates/zz-report.md`)。九条铁律:

1. **全是大标题,没有小标题**——每一步是一个 `## 步骤 N · 做什么`,不再往下分级。
2. **能一句话讲清就不用两句**——每步正文一句话,讲"点哪、到哪、看到什么"。
3. **讲流程不讲思想**——只写操作序列,不写设计动机/架构权衡(那些进 §1 目标段)。
4. **同岗位照做必成**——步骤之间不跳跃,前一步的产物是后一步的输入。
5. **一条分支走完再讲下一条**——若流程分叉(如 A 账号 / B 账号两条线),把 A 全部步骤讲完,再用"续上一步"承接讲 B,禁止两条线交错(对应"14 续 13"的清晰承接)。
6. **每步尽量配图**——用 `{{IMG:<截图name>}}` 占位,`archive_report.py` 内联该步截图。
7. **切换页面必截图**——只要 URL/主视图变了就留一张,证明"真到了那一页"。
8. **变化处必画框**——点击目标、激活态、新出现的卡片/toast 都画红框 + 序号。harness 的 `stepClick` / `stepShot` / `box` 已自动注入(`__acc_box`),driver 直接调用即可。
9. **文字在上,图片在下**——靠 `{{IMG:}}` 占位天然成立(占位放在该步文字之后)。

**两种占位二选一(或并用)**:`{{IMG:<name>}}` 逐步内联(ZZ 默认) / `{{EVIDENCE}}` 集中堆到 §9 证据段(旧版)。准入校验要求至少有一种。

## 7. 国际标准对照（裁剪来源)

| 国际标准 | 取用 | 落点 |
|---|---|---|
| IEEE 829 / ISO·IEC·IEEE 29119-3 | Test Report 骨架 | §6.2 九段 |
| ISO/IEC 25010 | 8 大质量特性词表 | 用例表"维度"列 |
| ISO·IEC·IEEE 29119 | 通过准则 + 缺陷分级 | §8 Verdict + P0-P3 |
| 项目 /uat | 五段式执行 | 用例表"Phase"列 |
| 项目视觉验收 | P0-P3 + 双主题 | §5 + §8 |
| CLAUDE #8 / #8.1 | DoD + 自测路径 | §6 第 4/5 段 |

**Verdict 规则**(推导,不靠感觉):
- **通过 Pass**:无 open P0/P1 且 DoD 全勾 且核心用例全 pass
- **有条件通过 Conditional**:仅剩 P2/P3,列整改项
- **不通过 Fail**:存在 open P0/P1 或 DoD 缺项 或核心用例 fail

**严重级**:P0 阻塞 / P1 必修 / P2 可延后 / P3 优化建议(实例见 `doc/rule.issues-system.md` §5.2)。

## 8. 与 issues 视觉验收体系的关系（方法论共用,工作流互补)

项目里有三条视觉验收路径,**共用一套方法论(本标准 §5.1 指向的 SSOT),工作流各管一段**,不重叠、不删:

| 能力 | 工作流 | 何时用 |
|------|--------|--------|
| `issues-visual-create` + `issues-visual-run` | **异步 · 多 agent · GitHub 工单制**:开 `[visual-test]` issue → 24h 执行者 agent 接单跑矩阵 → 评论 P0-P3 / `/visual-pass`,label 状态机留痕 | 想派单给别的 agent 异步验、要 GitHub 上可追溯的工单 |
| 本技能 `create-visual-test-to-kb` | **同步 · 单 agent · 即时归档**:我开发+自测完,当场走真人路径验收 → 报告归档进知识库 + 分享链 | 我自己当场验完要一份结构化存档报告 |
| `acceptance-checklist`(/uat) | 执行**前**生成真人逐步打勾清单 | 交给真人手动验 |

**为什么不合并成一个**:工作流本质不同(异步工单 vs 即时归档 vs 人工清单),强合会把三种节奏塞进一个技能。**真正该统一的是方法论**——矩阵 / 10 硬约束 / P0-P3 已抽到 SSOT(§5.1),三者都引用,改一处全同步。这才是正确的"结合"。

**从 issue 体系吸取并已并入本标准的经验**:四轴测试矩阵(§5)、10 条硬约束逐条核查(§5.1)、P0-P3 带项目实例(§5.1)、失败回报表格化(§6.2 用例表)、"结果先落地再改状态"的鲁棒性原则(归档脚本先写正文再出分享链)。

## 9. 合规

- 全中文;禁 emoji(CLAUDE §0)。
- 截图引用式(上传拿 URL 内联),不塞 base64。
- 一份报告不可变:重测出新报告(report_id 递增),不改旧的。
- 预览地址走 cdscli,禁手拼(规则 #11)。
