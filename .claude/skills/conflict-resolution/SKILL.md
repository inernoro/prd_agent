# Conflict Resolution - 安全合并冲突解决

安全、保守地解决 Git 合并冲突，**绝不丢弃任何一方的代码**，除非有充分理由且已获人类确认。

## 核心安全原则

```
┌────────────────────────────────────────────────────┐
│              冲突解决三条铁律                         │
│                                                    │
│  1. 禁止静默丢弃  ── 任何一行被删除都需要有理由      │
│  2. 不确定就问人  ── 宁可打断人类，也不要猜错        │
│  3. 保留优先于删除 ── 当无法判断时，两边都保留        │
│                                                    │
│  违反任何一条 = 技能失败                              │
└────────────────────────────────────────────────────┘
```

## 触发词

- "解决冲突"、"处理冲突"、"合并冲突"
- "fix conflicts"、"resolve conflicts"、"merge conflict"
- "/resolve"

## 冲突分类与处理策略

### Level 1: 自动合并（AI 可独立处理）

**条件**：冲突双方意图明确且不矛盾，可以安全合并。

| 冲突类型 | 处理方式 | 示例 |
|---------|---------|------|
| import 新增 | 保留双方所有 import | 双方各加了不同的 import |
| 非交叉的函数新增 | 保留双方所有新增函数 | A 加了 funcX，B 加了 funcY |
| 注释/文档更新 | 保留更新内容更丰富的一方 | A 改了注释，B 也改了同一处注释 |
| 配置项新增（非冲突 key） | 合并所有新增项 | A 加了 keyX，B 加了 keyY |
| 格式化差异（空格/换行） | 采用格式化后的版本 | 仅 formatting 不同 |

**处理后必须**：列出每个自动合并决策及理由，供人类确认。

### Level 2: 需人类确认（AI 给出建议但不执行）

**条件**：双方修改了同一段逻辑，AI 能理解意图但无法确定优先级。

| 冲突类型 | 处理方式 |
|---------|---------|
| 同一函数的不同修改 | 展示双方差异 + AI 建议，等人类确认 |
| 业务逻辑分歧 | 展示两个版本的行为差异，由人类决策 |
| API 签名/DTO 变更 | 列出上下游影响，由人类决策 |
| 数据模型字段冲突 | 分析数据迁移影响，由人类决策 |
| 配置值冲突（同一 key 不同值） | 展示两个值及其来源，由人类决策 |

### Level 3: 必须人类介入（AI 禁止处理）

**条件**：冲突涉及高风险区域，AI 不得自行决定。

| 冲突类型 | 原因 |
|---------|------|
| 安全相关代码（认证、授权、加密） | 错误合并可能引入安全漏洞 |
| 数据库迁移脚本 | 错误顺序可能导致数据丢失 |
| 金额/计费/配额逻辑 | 错误合并可能造成资金损失 |
| CI/CD 管道配置 | 错误合并可能导致部署事故 |
| 环境变量/密钥配置 | 可能泄露敏感信息 |
| 删除 vs 修改冲突 | 一方删除代码，另一方修改代码，必须人类判断 |

## 执行流程

### Phase 1: 冲突扫描与分类

```bash
# 1. 获取所有冲突文件
git diff --name-only --diff-filter=U

# 2. 查看冲突内容
git diff --check

# 3. 逐文件读取冲突标记
# 搜索 <<<<<<<, =======, >>>>>>> 标记
```

**对每个冲突块**：
1. 读取 `<<<<<<<` 和 `>>>>>>>` 之间的完整内容
2. 理解 **ours**（当前分支）和 **theirs**（合入分支）各自的修改意图
3. 分类到 Level 1 / Level 2 / Level 3

### Phase 2: 上下文分析

对每个冲突文件：

1. **读取完整文件**（不只是冲突块），理解冲突的上下文
2. **检查 git log**，理解双方的 commit 意图：
   ```bash
   # 查看 ours 的提交历史
   git log --oneline HEAD -- <file>

   # 查看 theirs 的提交历史
   git log --oneline MERGE_HEAD -- <file>
   ```
3. **检查关联文件**：如果冲突涉及接口变更，检查调用方是否也有变更

### Phase 3: 安全解决

#### Level 1 冲突 — 自动解决

```
对每个 Level 1 冲突块：
  1. 执行合并
  2. 记录决策：
     - 冲突位置（文件:行号）
     - ours 内容摘要
     - theirs 内容摘要
     - 合并策略（保留双方/采用较新版本）
     - 理由
```

#### Level 2 冲突 — 请求确认

```markdown
## 需要您确认的冲突

### 冲突 #1: [文件名]:[行号范围]

**当前分支 (ours) 的修改**:
> [代码片段 + commit message 说明意图]

**合入分支 (theirs) 的修改**:
> [代码片段 + commit message 说明意图]

**AI 分析**:
- ours 的意图是：[分析]
- theirs 的意图是：[分析]
- 两者的矛盾点：[分析]

**AI 建议**: [建议方案]

**请选择**:
- A: 采用当前分支 (ours)
- B: 采用合入分支 (theirs)
- C: 采用 AI 建议的合并方案
- D: 手动处理（我来改）
```

#### Level 3 冲突 — 强制通知

```markdown
## 必须由您处理的冲突

### [文件名] — 涉及 [安全/数据迁移/计费] 相关代码

**冲突内容**:
[完整展示冲突块]

**风险说明**:
[为什么 AI 不应自行处理]

**建议**: 请您手动解决此冲突，或在确认后指示 AI 如何处理。
```

### Phase 4: 验证

解决所有冲突后：

```bash
# 1. 确认无残留冲突标记
grep -rn "<<<<<<< \|======= \|>>>>>>> " --include="*.cs" --include="*.ts" --include="*.tsx" --include="*.json"

# 2. 编译检查
# 后端
cd prd-api && dotnet build --no-restore 2>&1 | grep -E "error CS" | head -20

# 前端（按项目）
cd prd-admin && pnpm tsc --noEmit 2>&1 | head -20
cd prd-desktop && pnpm tsc --noEmit 2>&1 | head -20

# 3. 快速冒烟测试
cd prd-api && dotnet test PrdAgent.sln --filter "Category!=Integration" --no-build 2>&1 | tail -5
```

## 禁止行为清单

以下行为**绝对禁止**，违反任何一条即视为技能执行失败：

| # | 禁止行为 | 原因 |
|---|---------|------|
| 1 | **静默选择一方、丢弃另一方** | 代码是开发者的劳动成果，不能不问就扔 |
| 2 | **用 `git checkout --ours` 或 `--theirs` 批量解决** | 这是最危险的操作，会丢弃整个文件的一方修改 |
| 3 | **对不理解的代码做合并** | 不理解就不要碰，交给人类 |
| 4 | **跳过编译/类型检查验证** | 合并后不验证 = 埋定时炸弹 |
| 5 | **修改冲突之外的代码** | 解决冲突时只动冲突块，不"顺手"改别的 |
| 6 | **在冲突解决中引入新功能或重构** | 冲突解决是手术，不是装修 |
| 7 | **删除看似"重复"的代码而不确认** | 两段相似代码可能服务于不同场景 |
| 8 | **自行决定 Level 3 冲突的解决方案** | 高风险冲突必须人类决策 |

## 安全防护机制

### 1. 变更审计

每次解决冲突前后，自动记录文件快照：

```bash
# 解决前：记录冲突状态
git stash push -m "conflict-snapshot-before" -- <conflicted-files>
git stash pop

# 解决后：可用 git diff 审查所有变更
git diff --cached
```

### 2. 代码行数守恒检查

```
解决冲突后，检查：
  ours 独有行数 + theirs 独有行数 + 共同行数 ≈ 合并后总行数

如果合并后行数明显少于双方之和 → 发出警告：
  "合并后代码减少了 [N] 行，请确认是否有代码被意外丢弃"
```

### 3. 函数/类完整性检查

```
解决冲突后，扫描：
  - ours 中的所有函数/类名 → 合并后是否都存在？
  - theirs 中的所有函数/类名 → 合并后是否都存在？

如果有函数/类消失 → 立即停止并通知人类：
  "以下函数在合并后消失：[列表]，请确认是否为预期行为"
```

### 4. 回滚预案

```bash
# 如果合并结果不理想，可以回滚
git merge --abort    # 合并过程中
git reset --merge    # 合并后但未提交
```

## 与其他技能的协作

```
发现冲突
  │
  ▼
"/resolve" → 【冲突解决技能】
  │
  ├─ Level 1 冲突自动解决
  │
  ├─ Level 2/3 → 通知人类，等待决策
  │
  ├─ 解决完成后 → "/verify" → 【人工验证技能】(验证合并正确性)
  │
  └─ 涉及复杂逻辑 → "/trace" → 【路径追踪技能】(理解代码链路)
```

## 特殊场景处理

### 场景 1: lock 文件冲突 (pnpm-lock.yaml, packages.lock.json)

```bash
# lock 文件不手动解决，重新生成
# 采用 theirs 的 package.json 后重新 install
git checkout --theirs pnpm-lock.yaml
pnpm install
```

### 场景 2: 自动生成文件冲突 (*.Designer.cs, *.g.cs)

```
自动生成的文件 → 采用最新版本，然后重新生成
不要尝试手动合并自动生成的代码
```

### 场景 3: 大量冲突（超过 10 个文件）

```
超过 10 个冲突文件时：
1. 先输出冲突概览（文件列表 + 冲突块数量）
2. 询问人类是否要逐个处理，还是按优先级分批
3. 建议是否考虑 rebase 而非 merge
```

### 场景 4: 二进制文件冲突

```
二进制文件（图片、字体、编译产物）→ 必须询问人类选择哪个版本
AI 无法分析二进制内容的差异
```

## 注意事项

1. **耐心比速度重要**：宁可多花时间分析，也不要草率合并
2. **保守优于激进**：不确定的冲突，宁可保留两边代码让人类裁定
3. **上下文很重要**：不要只看冲突块，要读完整文件理解改动目的
4. **commit message 是线索**：通过 commit message 理解双方的修改意图
5. **测试是底线**：合并后必须通过编译和测试，否则不算完成
6. **透明度**：所有决策都要记录并展示给人类，没有"偷偷"的操作
