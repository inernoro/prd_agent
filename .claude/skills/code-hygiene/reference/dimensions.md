# 十类检测维度详解

> 被 SKILL.md 引用。每个维度包含：定义、信号（项目特有的检测线索）、修复策略。

---

## ① 死字段 / 死属性（Dead Fields）

> 类型/接口中定义了字段，但无任何代码读写它

**信号**：
- 字段有注释 `// reserved`、`// TODO`、`// unused`
- 字段初始值为空（`{}`、`[]`、`null`、`''`）且从未被赋其他值
- 字段仅在序列化/展示中出现，从未被业务逻辑消费

**修复**：删除字段定义及所有初始化代码

---

## ② 死条件分支（Dead Conditional Branches）

> switch/if 中的某个分支永远不会执行，或执行后直接返回空值

**信号**：
- `case 'xxx': return null;`
- `if (type === 'xxx') { /* not implemented */ }`
- 分支内只有注释说明"由其他地方处理"

**修复**：删除死分支。如果是 enum/union 类型成员导致的，同步删除类型定义中的成员

---

## ③ 兼容垫片 / 向后兼容别名（Compatibility Shims）

> 为旧版本保留的降级逻辑、环境变量别名、旧配置文件名回退

**信号**：
- 环境变量用 `||` 链连接新旧名称：`process.env.NEW_NAME || process.env.OLD_NAME`
- 配置加载中有 fallback 到旧文件名
- 注释中有 `backward compat`、`legacy`、`deprecated`、`旧版兼容`

**修复**：保留新名称，删除旧名称回退。公开 API / 环境变量先在 CHANGELOG 声明废弃，下一版本再删

---

## ④ 命名残留（Naming Artifacts）

> 变量名、文件名、注释中仍然使用旧概念的名称

**信号**：
- 变量名包含旧概念（如 `providerConfig` 而非 `platformConfig`）
- 注释中提到已不存在的模块
- 文件名与内容功能不匹配

**修复**：
- 代码引用 → 重命名为新名称
- 注释 → 更新或删除
- 数据库集合名 → 保留（迁移成本高，需单独计划）

**项目特有**：从 CLAUDE.md「已废弃概念」表获取旧名→新名映射，Grep 时大小写不敏感

---

## ⑤ 冗余参数 / 冗余类型（Redundant Parameters & Types）

> 函数参数声明了但从未使用；类型定义了但无任何变量使用该类型

**信号**：
- 参数名以 `_` 开头（有意标记未使用）
- 函数体内 destructure 了对象参数，但某些属性从未使用
- TypeScript 类型导出了但只在定义文件中出现

**修复**：
- 未使用参数 → 删除（注意接口/抽象方法的兼容）
- 未使用类型 → 删除类型定义和导出

---

## ⑥ 过时注释 / 误导注释（Stale Comments）

> 注释描述的行为与代码实际行为不符

**信号**：
- `// TODO: implement xxx` — 但 xxx 已实现
- `// Returns the slug` — 但实际返回原始名称
- JSDoc 参数与函数签名不匹配
- 注释中引用的类名/函数名不存在

**修复**：更新注释使其与代码一致，或删除不再有价值的注释

---

## ⑦ 防御性迁移代码（Defensive Migration Code）

> 为兼容旧版数据格式而添加的运行时校验/填充，在数据已全部迁移后变成死代码

**信号**：
- 注释中写明 `// migration`、`// v1 compat`、`// 旧数据兼容`
- 连续多个 `if (!field) field = default` 语句（批量迁移填充）
- 加载数据后立即执行格式转换

**判断原则**：
- 能确认生产数据已完整 → 可删除
- 无法确认（本地缓存、离线数据） → 保留，添加 `// @migration-guard: min-version=X.Y.Z`

---

## ⑧ 过度抽象 / 一次性抽象（Over-abstraction）

> 为只有一个调用者的逻辑创建了辅助函数/工具类/中间层

**信号**：
- `xxxHelper`、`xxxUtils` 函数只被一处调用
- 接口 `IXxxService` 只有一个实现 `XxxService`，且无测试 mock
- 中间层只做透传，不增加任何逻辑

**修复**：内联到唯一调用处（除非接口是为了可测试性而存在的）

---

## ⑨ 近似重复代码（Near-Duplicate Code）

> 两段代码逻辑几乎相同，仅在个别细节上有差异（80%+ 代码行相同）

**信号**：
- `parseXxxFile()` 和 `parseXxxString()` 仅入口不同
- 两个 Controller 端点代码大量雷同
- 复制粘贴后只改了部分（可能漏改）

**修复**：提取公共逻辑为内部函数，差异部分参数化

---

## ⑩ 配置-代码漂移（Config-Code Drift）

> 配置文件（YAML/JSON/注释）使用了运行时代码中不存在或已移除的语法、字段、占位符

**信号**：
- 配置文件引用了代码不解析的自定义扩展字段（如 `x-cds-inject`、`x-custom-*`）
- 占位符语法与解析器不匹配（配置写 `{{host}}`，代码只解析 `${CDS_HOST}`）
- 配置注释描述了不存在的功能（如文件头注释说"支持 x-cds-name"，但 parser 从未实现）
- 多个配置文件中同类服务版本不一致（compose A 用 `mongo:7`，compose B 用 `mongo:8`）
- 配置文件仍沿用已在 git 历史中被清除的格式/约定

**检测方法**：
1. 提取配置文件中的所有自定义字段/占位符
2. 在运行时代码中搜索对应的解析逻辑
3. 若解析逻辑不存在 → 配置-代码漂移（确定性：确认死配置）
4. 若解析逻辑存在但语法不匹配 → 配置-代码漂移（确定性：确认不兼容）

**典型案例**：
- `cds-compose.yml` 使用 `x-cds-inject` + `{{host}}:{{port}}`，但 `compose-parser.ts` 只解析 `${CDS_HOST}:${CDS_*_PORT}`
- `appsettings.json` 中定义了配置节，但代码中已无对应的 Options 类绑定
- `.env.example` 列出了环境变量，但代码中已移除对应的 `process.env.XXX` 读取

**修复**：用对应的生成工具（如 `/cds-scan`）重新生成配置，或手动对齐到代码实际解析的格式。删除代码不消费的配置字段
