# Code Hygiene - 代码卫生审计

迁移完成后，代码里不应残留任何"过渡期痕迹"。本技能系统性扫描指定范围，检测并清理所有过渡期残留物。

## 核心原则

> **迁移完成 ≠ 功能能跑。迁移完成 = 代码里看不出曾经迁移过。**

一次干净的迁移，读者应该完全不知道旧系统存在过。任何提示"曾经有个旧系统"的痕迹，都是待清理的技术债。

## 触发词

- "代码卫生"、"代码体检"、"代码清理"
- "清理残留"、"迁移清理"
- "hygiene"、"code hygiene"、"cleanup audit"
- "/hygiene"

## 适用场景

| 场景 | 例子 |
|------|------|
| **迁移后清理** | Provider→Platform 重命名完成后，扫描残留引用 |
| **重构后审计** | 架构重构后检查死代码和过时抽象 |
| **定期体检** | 每月/每季度跑一次，防止技术债累积 |
| **PR 审查辅助** | 大 PR 合入后，扫描是否引入了不必要的兼容层 |
| **新人接手** | 接手陌生模块前，先做一次卫生审计了解技术债 |

## 九类检测维度（MECE）

所有代码卫生问题归入以下 9 类，不重叠、不遗漏：

### ① 死字段 / 死属性（Dead Fields）

> 类型/接口中定义了字段，但无任何代码读写它

**检测方法**：
1. 在类型定义中找到所有字段
2. Grep 每个字段名，统计被引用的位置
3. 如果只出现在定义处和初始化处（如 `= {}`、`= []`），即为死字段

**信号**：
- 字段有注释 `// reserved`、`// TODO`、`// unused`
- 字段初始值为空（`{}`、`[]`、`null`、`''`）且从未被赋其他值
- 字段仅在序列化/展示中出现，从未被业务逻辑消费

**修复**：删除字段定义及所有初始化代码

---

### ② 死条件分支（Dead Conditional Branches）

> switch/if 中的某个分支永远不会执行，或执行后直接返回空值

**检测方法**：
1. 扫描 switch/case、if/else if 链
2. 检查每个分支的触发条件是否有实际的数据来源
3. 检查分支体是否只有 `return null`、`break`、`// TODO` 等空操作

**信号**：
- `case 'xxx': return null;`
- `if (type === 'xxx') { /* not implemented */ }`
- 分支内只有注释说明"由其他地方处理"

**修复**：删除死分支。如果是 enum/union 类型成员导致的，同步删除类型定义中的成员

---

### ③ 兼容垫片 / 向后兼容别名（Compatibility Shims）

> 为旧版本保留的降级逻辑、环境变量别名、旧配置文件名回退

**检测方法**：
1. 搜索 `||` 链中包含旧命名的回退：`process.env.NEW_NAME || process.env.OLD_NAME`
2. 搜索旧配置文件路径：`path.resolve(..., 'old-name.config.json')`
3. 搜索条件加载：`if (fs.existsSync('old-path')) { ... }`

**信号**：
- 环境变量用 `||` 链连接新旧名称
- 配置加载中有 fallback 到旧文件名
- 注释中有 `backward compat`、`legacy`、`deprecated`、`旧版兼容`

**修复**：保留新名称，删除 `||` 后的旧名称回退。如果是外部用户可能依赖的（如公开 API、环境变量），先在 CHANGELOG 中声明废弃，再在下一版本删除

---

### ④ 命名残留（Naming Artifacts）

> 变量名、文件名、注释中仍然使用旧概念的名称

**检测方法**：
1. 维护一个"旧名→新名"映射表（从 CLAUDE.md "已废弃概念"获取）
2. Grep 所有旧名（大小写不敏感）
3. 区分：代码引用 vs 纯注释 vs 数据库集合名（DB 名通常保留兼容）

**信号**：
- 变量名包含旧概念（如 `providerConfig` 而非 `platformConfig`）
- 注释中提到已不存在的模块
- 文件名与内容功能不匹配

**修复**：
- 代码引用：重命名为新名称
- 注释：更新或删除
- 数据库集合名：保留（迁移成本高，需单独计划）

---

### ⑤ 冗余参数 / 冗余类型（Redundant Parameters & Types）

> 函数参数声明了但从未使用；类型定义了但无任何变量使用该类型

**检测方法**：
1. 对函数签名中的每个参数，搜索函数体内是否引用
2. 对 type/interface 定义，搜索是否有变量声明使用该类型
3. 检查导出但未被其他文件导入的类型

**信号**：
- 参数名以 `_` 开头（有意标记未使用）
- 函数体内 destructure 了对象参数，但某些属性从未使用
- TypeScript 类型导出了但只在定义文件中出现

**修复**：
- 未使用参数：从函数签名中删除（注意接口/抽象方法的兼容）
- 未使用类型：删除类型定义和导出

---

### ⑥ 过时注释 / 误导注释（Stale Comments）

> 注释描述的行为与代码实际行为不符；重复的 JSDoc；TODO 中引用已完成的功能

**检测方法**：
1. 搜索 `TODO`、`FIXME`、`HACK`、`XXX` 注释，检查是否已完成
2. 检查 JSDoc 与函数签名是否匹配（参数个数、返回类型）
3. 检查注释中引用的函数/类/模块是否仍然存在

**信号**：
- `// TODO: implement xxx` — 但 xxx 已实现
- `// Returns the slug` — 但实际返回原始名称
- 重复的 JSDoc 块（复制粘贴后忘记删旧的）
- 注释中引用的类名/函数名不存在

**修复**：更新注释使其与代码一致，或删除不再有价值的注释

---

### ⑦ 防御性迁移代码（Defensive Migration Code）

> 为兼容旧版数据格式而添加的运行时校验/填充，在数据已全部迁移后变成死代码

**检测方法**：
1. 搜索 `if (!xxx) xxx = defaultValue` 模式（数据填充）
2. 搜索 `|| []`、`|| {}`、`|| ''` 模式（空值防御）
3. 检查这些防御是否针对已不存在的旧格式

**信号**：
- 注释中写明 `// migration`、`// v1 compat`、`// 旧数据兼容`
- 连续多个 `if (!field) field = default` 语句（批量迁移填充）
- 加载数据后立即执行格式转换

**判断原则**：
- 如果能确认所有生产数据已包含该字段 → 可删除
- 如果无法确认（如用户本地缓存、离线数据） → 保留但添加 `// @migration-guard: min-version=X.Y.Z` 注释标记清理时间

**修复**：确认数据完整性后删除防御代码；无法确认的标记版本号

---

### ⑧ 过度抽象 / 一次性抽象（Over-abstraction）

> 为只有一个调用者的逻辑创建了辅助函数/工具类/中间层

**检测方法**：
1. 对 helper/util 函数，统计调用次数
2. 对抽象层（adapter/wrapper），检查是否只有一个实现
3. 对接口定义，检查是否只有一个实现类

**信号**：
- `xxxHelper`、`xxxUtils` 函数只被一处调用
- 接口 `IXxxService` 只有一个实现 `XxxService`，且无测试 mock
- 中间层只做透传，不增加任何逻辑

**修复**：内联到唯一调用处（除非接口是为了可测试性而存在的）

---

### ⑨ 近似重复代码（Near-Duplicate Code）

> 两段代码逻辑几乎相同，仅在个别细节上有差异

**检测方法**：
1. 在同一文件或相邻文件中搜索相似的函数签名
2. 比较函数体结构——如果 80%+ 的代码行相同，即为近似重复
3. 检查差异部分是有意还是遗漏

**信号**：
- `parseXxxFile()` 和 `parseXxxString()` 仅入口不同
- 两个 Controller 端点代码大量雷同
- 复制粘贴后只改了部分（可能漏改）

**修复**：提取公共逻辑为内部函数，差异部分参数化

## 执行流程

### Phase 1: 确定扫描范围

```
用户说 "代码卫生 cds/" 或 "/hygiene src/services"
  │
  ├─ 指定了路径 → 扫描该路径下所有源代码文件
  ├─ 指定了模块名 → 扫描该模块目录
  └─ 未指定 → 询问用户要扫描的范围
```

**范围确认**：
1. 统计目标路径下的源文件数量和总行数
2. 如果文件数 > 50 或行数 > 5000，建议分模块扫描
3. 识别项目语言（TypeScript/C#/Rust/Python 等），选择对应的检测策略

### Phase 2: 建立检测上下文

```bash
# 1. 获取"旧名→新名"映射（从 CLAUDE.md 或用户提供）
# 2. 获取最近的迁移 commit（了解迁移了什么）
git log --oneline --all --grep="migrate\|rename\|refactor\|deprecat" | head -20

# 3. 识别配置文件、类型定义文件、入口文件
```

### Phase 3: 逐维度扫描

按 9 个维度逐一扫描，**每个维度完成后记录发现，不要等全部扫完再汇总**。

扫描时使用 Agent 工具进行并行搜索以提高效率。

### Phase 4: 分类评估

对每个发现进行评估：

| 评估维度 | 选项 |
|---------|------|
| **确定性** | 确认死代码 / 疑似死代码 / 需人工确认 |
| **影响范围** | 仅本文件 / 跨文件 / 跨模块 |
| **修复难度** | 可直接删除 / 需重构 / 需数据迁移确认 |
| **自动修复** | 可自动 / 需人工 |

### Phase 5: 输出报告 + 自动修复

## 语言特定检测策略

### TypeScript / JavaScript

```bash
# 死导出：导出了但无其他文件导入
grep -r "export " src/ --include="*.ts" | # 提取导出名
  # 对每个导出名 grep 其他文件的导入

# 死字段：接口/类型字段无引用
# 读取 interface/type 定义，逐字段 grep

# 未使用的 npm 依赖
# 检查 package.json dependencies vs 实际 import

# 旧命名
# grep -i "oldName" --include="*.ts" --include="*.tsx"
```

### C# (.NET)

```bash
# 死字段/属性
# 读取 class 定义中的 public 属性，grep 整个 solution

# 未使用的 using
# dotnet build 会报 CS8019 警告

# 旧命名
# grep -ri "oldName" --include="*.cs"

# 死 Controller 端点
# 检查 [HttpGet/Post/Put/Delete] 标记的方法是否被前端调用
```

### Rust

```bash
# 死代码
# cargo build 会报 dead_code 警告
# #[allow(dead_code)] 标记的需要人工确认是否仍需保留

# 未使用的依赖
# cargo udeps (需安装)
```

```
  └─ 修复超过 3 文件 → /handoff 生成交接清单
```

## 自动修复安全规则

1. **只删确认的死代码**：疑似死代码必须标记为"需人工确认"，不自动删
2. **每次修复后编译/类型检查**：修复一批后运行 `tsc --noEmit` 或 `dotnet build`，确认无破坏
3. **每次修复后跑测试**：确认无回归
4. **分批提交**：按维度分 commit，方便 revert
5. **不碰数据库**：DB 集合名/字段名的重命名不在本技能范围内，需单独迁移计划

---

## 注意事项

1. **区分"死"和"预留"**：有 `// reserved` 或 `// @planned` 注释的字段，不算死字段，但应确认计划是否仍然有效
2. **区分"兼容"和"公开 API"**：对外暴露的环境变量/API 别名，删除前需通知用户，不能静默删
3. **DB 字段保留兼容**：MongoDB 集合名、Redis key 前缀等持久化命名，迁移成本高，不在自动修复范围
4. **测试代码不漏扫**：测试文件中的 mock、helper 也可能有过时引用
5. **不追求零警告**：某些"防御性代码"是合理的（如 `?? []` 防止 null），不应为了干净而削弱健壮性
