# Dev Completion Report — 开发完工报告

每次开发任务（commit + push 之后），自动输出三段式总结报告，让非技术用户也能验收，让技术用户能快速评估质量。

## 适用场景

| 场景 | 说明 |
|------|------|
| 功能开发完成 | commit + push 后自动触发 |
| Bug 修复完成 | 修复并推送后触发 |
| 用户要求总结 | "帮我总结一下"、"开发总结"、"/dev-report" |
| 多轮迭代收尾 | 一次会话中多次修改后的最终总结 |

**不触发场景**：纯研究/探索性对话、只读操作（查日志、查配置）、用户明确说不需要。

## 执行流程

```
报告进度：
- [ ] Part 1: 生成 200 字简明总结
- [ ] Part 2: 生成总结清单
- [ ] Part 3: 生成行业对比分析
```

### Part 1: 简明总结（200 字内）

用二级标题 `## 简明总结` 开头，一段话覆盖 5 个信息点：

1. **理解** — 我理解你的目标是什么
2. **思考** — 我判断关键问题在哪里
3. **方法** — 我用了什么办法解决
4. **结果** — 现在完成了什么、状态是什么
5. **后续** — 下一步建议或待确认事项

**规则**：
- 140-200 中文字，不超过 200
- 口语化，无术语堆砌
- 让完全不懂技术的人也能读懂

### Part 2: 总结清单

用二级标题 `## 总结清单` 开头，包含 4 个子表格：

#### 2.1 改动范围

| 文件 | 改动说明 |
|------|---------|
| `path/to/file` | 一句话说明改了什么 |

#### 2.2 注意事项和风险

| 风险 | 等级 | 说明 |
|------|------|------|
| 例：API 兼容性 | 低/中/高 | 简要说明 |

无风险时写「本次改动风险低，无需特别关注」，不要留空。

#### 2.3 测试用例

| # | 测试步骤 | 预期结果 |
|---|---------|---------|
| 1 | 具体操作步骤 | 预期看到什么 |

至少 3 条，覆盖主流程 + 边界情况。

#### 2.4 验收标准

- [ ] 标准 1（具体可验证的条件）
- [ ] 标准 2
- [ ] 标准 3

用 checkbox 格式，验收者可逐项勾选。

### Part 3: 行业对比分析

用二级标题 `## 行业对比分析` 开头，包含 3 个子节：

#### 3.1 与行业最佳实践的核心差异

| 维度 | 行业最佳 | 当前状态 | 差距 |
|------|---------|---------|------|
| 例：XX | 描述 | 描述 | 小/中等/较大 |

只列与本次改动相关的维度，3-5 行即可。

#### 3.2 不足/不闭环清单

按优先级从高到低列出，格式：
1. **P0 — 标题**：说明
2. **P1 — 标题**：说明

每项一句话说明问题和影响，不展开方案。

#### 3.3 后续建议

2-3 条可操作的建议，每条说明「做什么 → 带来什么价值」。

## 质量自检

输出前逐项检查：

- [ ] Part 1 是否 ≤ 200 字且覆盖 5 个信息点
- [ ] Part 2 改动范围是否与实际 git diff 一致
- [ ] Part 2 测试用例是否具体可执行（不是"测试功能正常"）
- [ ] Part 2 验收标准是否可量化/可验证
- [ ] Part 3 行业对比是否与本次改动相关（不泛泛而谈）
- [ ] Part 3 不足清单是否有明确优先级
- [ ] 整体是否让非技术用户也能理解 Part 1
