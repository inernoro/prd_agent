# 产品文档生成器

## 工作流程（分阶段执行）

处理产品文档时，**必须先确认模板结构，再严格按照 Part A / Part B 两阶段流程执行**：

### 阶段零：模板选择（必须先执行）

**目的**：产品文档存在多种模板结构，开始生成或优化前必须先明确使用哪一种模板，避免套错结构。

**执行步骤**：

1. **识别用户是否已指定模板**：如果用户明确说明使用某个模板、文档结构或文件路径，则直接采用对应模板。
2. **未指定时必须先询问用户**：在读取正文并生成内容前，先询问用户选择哪种模板结构：
   - **工程版 1+N 主-子文档模板**：适用于复杂项目、跨页面/跨模块方案、需要主文档 + 页面子文档拆分的需求。读取 `scripts/template.md`、`scripts/template-main.md`，如需页面子文档再读取 `scripts/template-sub.md`。
   - **新城扩建 / 旧城改造扩展敏捷版模板**：适用于新城扩建类项目、旧城改造中有新独立页面或模块的项目、小版本迭代、单一页面优化或业务逻辑简单的小闭环功能。读取 `scripts/template-Greenfield-or-Brownfield_Expansion.md`。
3. **用户确认后再继续**：只有在模板明确后，才能进入阶段一生成 Part A。
4. **模板优先级**：后续章节名称、层级、必填/可选项、示例格式均以用户选择的模板文件为准；本 `SKILL.md` 中的章节清单只作为默认理解，不得覆盖模板文件中的实际结构。

### 阶段一：Part A - 价值层（先执行）

**目的**：价值对齐与立项决策，确保方向和思路正确。

**执行步骤**：

1. **分析现有内容**：识别文档结构、关键信息和缺失部分
2. **读取选定模板**：根据阶段零的选择读取对应模板文件，了解整体文档结构和格式要求
3. **生成Part A内容**：按照选定模板生成价值层章节。默认工程版主文档包含：
   - 一、文档头（文档名称、产品线/模块、版本、迭代类型、作者、审批人）
   - 二、项目目的（3个核心要点阐述商业目标与客户价值）
   - 三、现状背景与问题陈述（现状背景、问题、需求、TAPD链接）
   - 四、用户范围（核心用户角色及诉求）
   - 五、实现思路（产品定位、结构归母、设计理念、设计思路）
   - 六、版本迭代规划（分阶段迭代计划）
   - 七、需求变更（如有）
4. **优化表达**：改进文字表述，提升清晰度和专业性
5. **识别不足**：检查Part A描述是否完整、准确
6. **适当扩写**：在关键部分补充必要的业务逻辑和价值说明

**重要**：完成Part A后，**必须暂停并等待用户确认**。向用户展示Part A内容，询问：

- "请确认Part A的方向和思路是否符合预期？"
- "如有需要调整的地方，请指出。"

**只有用户确认Part A无误后，才能继续执行阶段二。**

### 阶段二：Part B - 全局设计与规范层（确认后执行）

**目的**：定义全局的设计和规范，用于AI开发。

**执行步骤**（仅在Part A确认后执行）：

1. **继续使用选定模板**：不得在Part B阶段擅自切换模板；如用户要求更换模板，需要重新确认并说明会影响已生成结构
2. **生成Part B内容**：按照选定模板生成设计层/全局设计与规范层章节。默认工程版主文档包含：
   - 八、页面清单与关系（页面清单、页面关系）
   - 九、整体业务流程（阶段、角色、动作、页面、产出物）
   - 十、全局规则库（业务规则、权限规则）
   - 十一、全局非功能性需求（性能、安全、兼容性、用户体验）
   - 十二、调用公共组件和UI规则库
3. **补全技术细节**：补充接口定义、数据模型等技术实现所需信息
4. **优化表达**：确保技术描述清晰、准确、可执行
5. **整合输出**：生成完整的符合AI开发要求的产品文档

## 文档处理原则

### 内容完整性

- 确保所有必需章节都已包含
- 功能描述包含输入、处理、输出
- 业务规则和约束条件明确
- 异常场景和边界情况有说明

### 表达清晰性

- 使用准确的技术术语
- 避免歧义和模糊表述
- 逻辑结构清晰，层次分明
- 关键信息突出显示

### AI开发友好性

- 接口定义包含完整的请求/响应格式
- 数据模型有明确的字段说明和类型
- 业务流程有清晰的步骤和状态转换
- 技术实现有足够的上下文信息

## 文档结构检查清单

处理文档时，确保包含以下要素：

- [ ] 产品概述和目标
- [ ] 功能需求清单
- [ ] 用户角色和权限
- [ ] 核心业务流程
- [ ] 接口定义（如有）
- [ ] 数据模型（如有）
- [ ] 业务规则和约束
- [ ] 异常处理和错误码
- [ ] 性能要求（如有）
- [ ] 安全要求（如有）

## 优化策略

### 补全缺失内容

- 参考模板结构，识别缺失章节
- 根据上下文推断合理内容
- 保持与现有内容的一致性

### 优化表达方式

- 将口语化表述改为专业术语
- 将模糊描述改为具体说明
- 将长句拆分为清晰的短句
- 统一术语使用

### 识别和修复不足

- **描述不完整**：补充缺失的关键信息
- **逻辑不清晰**：重新组织内容结构
- **技术细节不足**：添加必要的实现说明
- **边界情况缺失**：补充异常场景处理

### 适当扩写

- 在关键功能点补充详细说明
- 为复杂业务逻辑添加示例
- 为技术实现提供上下文信息
- 确保AI能够理解并实现需求
- **重要原则：简洁不失真** - 在保证信息完整准确的前提下，避免冗余描述，确保文档简洁清晰

## 输出格式

最终输出的文档应：

- 使用Markdown格式
- 结构清晰，层次分明
- 包含完整的目录结构
- 关键信息使用表格、列表等格式突出显示
- 代码示例和接口定义使用代码块

## 文档模板结构

根据用户选择遵循对应模板结构，不得默认套用单一模板：

### 工程版 1+N 主-子文档模板

适用于复杂项目、跨页面/跨模块方案、需要主文档 + 页面子文档拆分的需求。

### Part A: 价值层

- 用于价值对齐与立项决策
- 包含：文档头、项目目的、现状背景、用户范围、实现思路、版本迭代规划、需求变更

### Part B: 全局设计与规范层

- 用于定义全局的设计和规范
- 包含：页面清单与关系、整体业务流程、全局规则库、全局非功能性需求、公共组件和UI规则库

### 编写原则

- **强调具体化**：清晰表达、尽量量化
- **强调因果性**：如果...则...；因为...所以...
- **强调顺序性**：开始...结束...；1、2、3...
- **强调分段描述**：按具体功能闭环描述

### 新城扩建 / 旧城改造扩展敏捷版模板

适用于新城扩建类项目、旧城改造中有新的独立页面或模块的项目、小版本迭代、单一页面优化或业务逻辑简单的小闭环功能。该模板为单文档、高内聚结构，使用 `scripts/template-Greenfield-or-Brownfield_Expansion.md` 中的章节为准。

## 模板和脚本

文档模板和辅助脚本位于 `scripts/` 目录：

- `template.md`: 工程版模板使用说明和整体结构总览（选择工程版时读取，了解"1+N"主-子文档模式）
- `template-main.md`: 工程版主文档模板 - 已优化为标准markdown（选择工程版并生成主文档时参考）
- `template-sub.md`: 工程版子文档模板 - 已优化为标准markdown（选择工程版并生成子文档时参考）
- `template-Greenfield-or-Brownfield_Expansion.md`: 新城扩建 / 旧城改造扩展敏捷版模板（选择敏捷版单文档结构时读取）
- `validator.py`: 文档验证脚本（可选工具）

**模板使用顺序**：

1. **开始时**：先确认用户选择的模板结构；用户未指定时必须先询问，不得直接生成。
2. **选择工程版时**：读取 `scripts/template.md` 了解整体结构，再参考 `scripts/template-main.md`；如需要页面子文档，再参考 `scripts/template-sub.md`。
3. **选择敏捷版时**：读取 `scripts/template-Greenfield-or-Brownfield_Expansion.md`，按单文档结构生成。
4. **生成过程中**：始终以选定模板为准，禁止混用不同模板的章节。

### 关于验证脚本

`validator.py` 是一个**可选的辅助工具**，用于自动检查文档是否符合模板要求。**没有脚本也可以正常工作**，skill会按照模板要求生成文档。

**脚本功能**：

- 检查必需章节是否存在
- 验证文档头字段完整性
- 检测未替换的占位符
- 验证markdown格式规范性

**使用方法**：

```bash
python scripts/validator.py <文档路径>
```

**注意**：脚本主要用于文档生成后的质量检查，不是必需步骤。skill在生成文档时会自动遵循模板结构，无需脚本也能生成符合要求的文档。

## 文档生成目标

产品文档的目的是让AI生成符合要求的技术分析方案以及开发工作，设计内容充分考虑**简洁不失真**的原则。

## 使用示例

**输入**：不完整的产品需求描述

**输出流程**：

1. **模板确认**：如用户未指定模板，先询问使用工程版 1+N 模板还是新城扩建 / 旧城改造扩展敏捷版模板
2. **第一阶段输出**：Part A - 价值层完整内容
3. **等待确认**：用户确认方向和思路
4. **第二阶段输出**：Part B - 按选定模板生成设计层或全局设计与规范层完整内容

**处理时重点关注**：

1. **Part A阶段**：
   - 将零散需求整理为结构化的价值层文档
   - 明确项目目的、用户范围、实现思路
   - 确保商业价值和产品定位清晰

2. **Part B阶段**（确认后）：
   - 补充技术实现所需的关键信息
   - 定义页面关系、业务流程、全局规则
   - 优化表述使其易于AI理解和实现
   - 确保文档可直接用于AI开发任务

## 关键提醒

注意：生成文档时，**必须先确认模板结构**；未指定模板时必须先询问用户。模板确认后，先完成Part A并等待用户确认，确认方向和思路没有大偏差后，再开始Part B的生成。不要一次性生成完整文档。
