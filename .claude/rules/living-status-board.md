# 大型工程必须维护「活的」状态看板（Living Status Board）

> 任何跨多阶段、多天、多轨道的大型工程（如「LLM 网关剥离」），必须有**一份**随状态实时更新的看板文档作为唯一进度真相（SSOT）。用户任何时刻打开它，30 秒内就能答出「现在到哪了 / 卡在哪 / 下一步是什么 / 凭什么说这步过了」，**不需要再来问 AI**。
> 触发：启动任何预计 ≥3 个阶段或 ≥3 天或需要并线的工程时。

---

## 历史背景（本次痛点）

2026-06/07 网关剥离期间，用户反复发问「现在剥离到哪一步了」「波2.5 是不是做完了」「还差什么才能上线」。根因是进度散落在 commit message、多份 design/debt 文档、AI 的口头汇报里，**没有一处是活的、结构固定、随状态翻转即更新的**。用户被迫每次追问、AI 每次重新口述，双方都在浪费时间——这正是 `expectation-management.md`「不知道现在在干嘛 / 不知道还要多久」两种失控的叠加。本规则把「维护一块活看板」固化为大型工程的开工前提。

---

## 强制条款

### 1. 一个工程 = 一块看板，放在 `doc/plan.{appname}.{topic}.md` 顶部

- 看板是该工程 `plan.*` 文档的**第一屏**（标题下第一节，pin 在最上），不是散落各处。网关剥离的看板就在 `doc/plan.llm-gateway.rollout.md` 顶部。
- 命名遵守 `doc/rule.doc.naming.md`：`plan.` 前缀 + appname 优先。禁止另起一份 `status-xxx.md` 裸文件（会和 plan 漂移）。
- 看板是 SSOT：design 文档写「怎么做」、debt 文档写「欠什么」、看板只写「现在在哪、卡在哪、下一步、证据」。三者互相链接，不互相抄。

### 2. 固定结构（六列，缺一不可）

看板主体是一张阶段表，每个阶段（波次/轨道）一行，固定六列：

| 列 | 含义 | 硬要求 |
|---|---|---|
| **阶段** | 波次 / 轨道名 | 稳定标识，不随意改名 |
| **进度%** | 该阶段完成度 | 数字，禁止长期停在某个百分比不动（停住即 blocker 信号） |
| **状态** | 未开始 / 进行中 / 已部署 / 已验收 / 阻塞 | 枚举，不写自由文案 |
| **当前 blocker** | 卡住的外部输入 / 未决项；无则写「无」 | 有 blocker 必须一句话说清「缺什么、需要谁给」 |
| **下一步** | 下一个可执行动作 | 具体到「改哪个文件 / 翻哪个 flag / 跑哪条验收」 |
| **验收证据** | 可点击的链接 | CDS 报告深链 / 预览 URL / 测试绿灯 / 影子比对读端点；禁止只写「已测」无链接 |

看板顶部还必须有一行元信息：`最后更新：YYYY-MM-DD HH:mm | 更新人/轨道 | 距离可发布：一句话`。

### 3. 「有意义的状态变化就更新」——但只在真变化时更新

- **必须更新**的时机：阶段进度翻转、blocker 出现或消除、下一步改变、拿到新验收证据、可发布判据变化。
- **禁止**无状态变化的文档校准（呼应 `blocked-state-circuit-breaker.md`：同一份 plan/status 被反复重写而 blocker 未翻转 = 计划 thrashing，是熔断信号，不是进展）。看板更新次数应约等于真实状态翻转次数，不该远大于它。
- 每次交付/收尾若触及该工程，**必须**同步看板对应行，再报「完成」。看板与实际不符视为交付缺陷。

### 4. 用户视角自足

看板必须让**不看代码的用户**独立读懂：术语给一句大白话、每个「已验收」都挂得回可点击证据、blocker 写清「在等谁」。做不到自足 = 不合格。

## 自查清单（开工时 + 每次收尾）

- [ ] 这个工程有没有一块活看板？没有 → 开工前先在 `doc/plan.{appname}.*.md` 顶部建。
- [ ] 看板六列齐全（阶段/进度%/状态/blocker/下一步/证据），且顶部有「最后更新时间 + 距离可发布」？
- [ ] 本次收尾触及的阶段行，进度/状态/下一步/证据都同步了？
- [ ] 每个「已验收」都挂了可点击证据链接（非「已测」二字）？
- [ ] 有没有出现「进度停在同一个百分比多次更新却没动」？有 → 触发 `blocked-state-circuit-breaker.md` 自查。
- [ ] 用户不看代码能独立读懂看板、无需追问？

---

## 与既有规则的关系

- `expectation-management.md`（预期管理总纲）：本规则是它在「工程级进度」切面的落地——消灭「不知道在干嘛 / 不知道还要多久」。
- `blocked-state-circuit-breaker.md`：看板的「进度停滞 / 反复重写却无状态翻转」正是熔断触发信号；本规则提供可观测的判定面。
- `doc-types.md` / `doc/rule.doc.naming.md`：看板归 `plan.*`，appname 优先，禁裸文件。
- `debt.*`：blocker 中「已知边界 / 后续可补」要落对应 `debt.{module}.md`，看板只做索引，不重复台账。
- 本次工程的看板实例：`doc/plan.llm-gateway.rollout.md`。
