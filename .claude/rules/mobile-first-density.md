# 手机端密度优先（Mobile-First Density）

> 手机寸土寸金。桌面优先 + 手机端只是「缩小」会导致大量留白、卡套卡、控制条堆叠，**屏幕利用率低**。
> 本规则把「手机端要把空间用满、内容优先」固化为系统纪律。触发：编辑 `prd-admin/src/**/*.tsx`、共享原语（`AppShell`/`GlassCard`/`PageHeader`）、`styles/tokens.css`。
>
> 历史背景：2026-06-20 用户在真机上反复指出「明明可以利用到的地方都没利用到，利用率不高」，根因是系统从来没有手机端密度纪律——每页桌面优先，手机端只是 shrink。本规则由此诞生。

---

## 一、系统已落地的密度杠杆（改一处、全站生效，勿重复造）

| 杠杆 | 位置 | 行为 |
|------|------|------|
| 全局内容外边距 | `styles/tokens.css` 的 `--mobile-padding`（默认 10px，≤479px 8px） | AppShell 主区手机端左右 gutter。**不要在页面里硬编码 16px+ 的外层 padding**，用这个 token 或更小。 |
| 卡片密度 + 嵌套去 chrome | `components/design/GlassCard.tsx` | 手机端整体收紧一档内边距；**嵌套卡片（depth>0）自动**再收紧 + 缩小圆角 + 去外投影。靠 `GlassCardDepthContext` 自动判定，无需手动传参。 |
| 轻量 isMobile | `hooks/useBreakpoint.ts` 的 `useIsMobile()` | 只在跨 768px 阈值时重渲染，可在高频组件里安全调用。需要响应式分支用它，别自己 matchMedia。 |
| 页头紧凑 | `components/design/PageHeader.tsx` | 手机端 `px-3`，桌面 `sm:px-4`。 |

---

## 二、写任何页面/组件时的手机端纪律

1. **满铺、单层 padding**：手机端内容尽量满铺，外层只留一层 gutter（走 `--mobile-padding`）。禁止「页面 padding + 卡片 padding + 内层卡片 padding」三层叠加吃掉半屏宽。
2. **不堆嵌套卡片 chrome**：能用 GlassCard 嵌套就用（已自动去 chrome）；若手写嵌套容器，手机端自己去掉边框/圆角/投影，别让「卡中卡」框线浪费空间。
3. **进内容前 ≤ 1 条控制条**：手机端，用户在看到真正内容（图表/列表/地图）之前，最多经过 1 条控制条。多个工具条（tab / 筛选 / 模式切换 / 图例）在手机端要**合并**成一条，或收进可展开的 sheet / 横向滚动条，不要竖向堆 3~5 条。

   **控制条收纳决策表（2026-07-12 全站审计后固化，逐类照做，不要临场发明）**：

   | 控件类型 | 手机端处理 |
   |----------|-----------|
   | 视图/模式切换（图表类型、卡片⇄时间树、密度档位） | 并入唯一横滚条；对手机无价值的（如卡片密度档）直接 `hidden sm:flex` |
   | 多组筛选 chips（类型 + 权属 + 标签…） | 合并为**一条** `overflow-x-auto` 横滚条，组间加 `w-px` 细分隔 |
   | 次要动作（刷新/导出/上传/接入 AI/自动化） | 图标化并入唯一条尾部，或收进「更多」/ bottom sheet |
   | 图例 legend | 手机端 `hidden sm:inline-flex`（颜色语义靠块色自明），或收进全屏视图 |
   | 状态说明文字行（计数/采集说明/摘要标签） | 隐藏或并入横滚条内的小字，不许独占一行 |
   | 大标题 / Hero 区 | 高度至少减半，不得把核心内容挤出首屏（参照原则 4 的 ≥60%） |
   | 卡内头部（标题 + 控件两行） | 合并为单行 `overflow-x-auto`，标题 `shrink-0 whitespace-nowrap` |

   参考实现：`pages/team-activity/InsightsPanel.tsx` 的 `renderMobileSingle`（视图切换 + 子模式 + AI 分析合并为一条横滚）与 `ExperienceMap.tsx` 卡头（单行横滚 + 图例手机隐藏）。
4. **内容占视口 ≥ 60%**：核心内容（用户来这页要看的东西）在手机首屏应占据 ≥60% 高度。chrome（页头/工具条/卡片框）压到最少。
5. **横排不下就横向滚动**：成排的 chips / 段控 / 阶段条手机端放不下时，`overflow-x-auto` 单行横滚 + 子项 `shrink-0 whitespace-nowrap`，不要换行成多行或逐字竖排。
6. **桌面零回归**：所有手机端密度处理走 `useIsMobile()` / `sm:`,`lg:` 断点渐进增强，宽屏维持原布局观感。

## 三、交付前手机端自审（任何 UI 改动）

- [ ] 375px 宽下，外层只有一层 gutter，没有三层 padding 叠加？
- [ ] 嵌套容器在手机端没有多余边框/圆角/投影框感？
- [ ] 看到核心内容前控制条 ≤1 条（多条已合并/横滚/收纳）？
- [ ] 核心内容占首屏 ≥60%？
- [ ] 成排控件放不下时是横向滚动，不是换行/竖排？
- [ ] 桌面端观感未变（断点渐进增强）？

两条以上不达标 = 手机端利用率不合格，返工。

## 四、与其他规则的关系

- `full-height-layout.md`：撑满高度（桌面向）；本规则补「手机端把宽度也用满 + 内容优先」。
- `chief-designer-usability.md`：好用四原则的手机落地——密度是「可视化/不遮挡/够明显」在小屏的前提。
- `frontend-modal.md`：浮层在手机端用 `width:min(Npx, 92vw)` 自适应，本规则的密度纪律同样适用浮层内部。
- `MobileCompatGate`（`lib/mobileCompatibility.ts`）：是「复杂页给 PC 门槛」的兜底；本规则的目标是**让更多页不需要兜底**——能做密就做密，少 gate。
