# 验收必须闭环（禁止断头验收）

> 验收的终点是「用户的最终产物可见且正确」，不是「流程走了一半截图收工」。
> 任何「流程跑完 → 产物还没出来」就判 pass 的验收，都是断头验收，直接不合格。

---

## 什么是断头验收

断头验收的典型特征：

- 触发了生成（点击「生成 PPT」「发布」「导出」「提交」），但**没等到产物真的出现**就截图结束
- 截图上写「渲染完成」，图里显示的却是「正在生成…已等待 Xs」
- 只验证了「有没有触发」，没验证「产物对不对」
- 用超时（timeout）掩盖了「最终状态未知」——timeout 之后截个图就叫「已验收」

判定口诀：**截图上能看到最终产物吗？** 不能 → 断头，返工。

---

## 强制规则

### 1. 生成类功能的完整闭环必须包含「产物可见」截图

| 功能类型 | 闭环终点 | 截图必须展示 |
|---|---|---|
| PPT 生成 | reveal.js 幻灯片在 iframe 中完整渲染 | 右侧预览区出现真实幻灯片内容（标题/内容可见） |
| 图片生成 | 生成图片在预览区完整展示 | 图片像素可见，非 spinner/进度条 |
| 报告生成 | 报告文本完整展示 | 正文可读，非「正在生成」 |
| 文件导出 | 文件下载到本地或预览链接可打开 | 下载提示或预览页面截图 |
| 发布/部署 | 目标环境可访问 | 访问成功的页面截图 |

### 2. timeout 不等于「验收通过」

等待超时后的截图只能用于**记录超时现象**，不得用于宣称「产物已生成/渲染完成」。

```
✅ 正确处理超时：
  - 截图并标注「等待 120s 超时，产物未渲染，P1 缺陷：PPT 生成超时」
  - verdict 降为 conditional 或 fail
  - 不得在截图 caption 里写「PPT 渲染完成」

❌ 禁止：
  - 超时截图 → caption 写「PPT 渲染完成」→ verdict 仍为 pass
  - 用超时的截图充当「产物可见」证据
```

### 3. 等待时间必须足够

验收脚本的等待超时必须基于功能的**实际 P95 耗时**，不是一个随便写的数字。

- LLM 生成类功能（PPT、长文、报告）：最少等待 **300s**，建议 **600s**
- 图片生成：最少 **60s**
- 普通 API：最少 **30s**
- 如果 P95 耗时未知，先跑一次记录真实耗时，再设超时

### 4. 「完整生命周期」= 入口 → 触发 → 等待 → 产物 → 可选：刷新后产物仍可见

```
入口可见
  ↓
触发生成（确认按钮/发送/提交）
  ↓
等待直到产物真实出现（非 spinner）
  ↓ ← 这一步断头了就不算验收
产物内容可读可用
  ↓（可选，如有持久化需求）
刷新/重进页面 → 产物或历史仍可见
```

中间任何一步跳过或超时 → 整条链路 = 未验收。

---

## 对存量验收报告的影响

发现历史验收报告中存在断头截图（截图展示的是生成中状态，caption 却写「已完成」），应：

1. 在报告的「缺陷清单」补一条 P1：「XXX 步骤验收截图显示仍在生成，产物未经真实验证」
2. 重新运行完整验收闭环
3. 用新截图替换旧证据后归档新版本报告

## 历史背景

2026-06-09，MD 转网页 PPT 验收（acc-prd-agent-202606091338）中：
- 步骤「PPT 渲染完成」截图实际展示「正在渲染生成幻灯片…已等待 36s」
- 步骤「刷新后 PPT 预览区」截图实际展示「正在设计版式与配色…已等待 17s」
- 驱动脚本等待 120s 超时后直接截图，caption 写「PPT 渲染完成」
- Verdict 被判为 pass，但核心产物（reveal.js 幻灯片）从未在任何截图中出现

用户发现并指出断头：「生成之后的结果你没有完成，你违反了一条原则：验收必须闭环，不能验收一半，做一个断头验收。」

本规则由此诞生。
