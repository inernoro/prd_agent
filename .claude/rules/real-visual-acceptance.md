# 用户可见功能：声称「完成」前必须做真视觉验收

> 「编译过了 / 接口 200 / 测试绿」都不等于「功能对」。任何**用户会亲手操作**的功能（网关观测台、模型可见性、灰度开关面板、任何 UI/端到端流程），在声称「完成」前必须走一次**真视觉验收**：模拟真人浏览器路径、对**真实产品界面**截图取证、闭环到产物可见，再报完成。
> 触发：交付任何有 UI 入口或端到端用户流程的功能之前。

---

## 历史背景（本次痛点）

网关剥离期间多次出现「AI 报完成 → 用户一操作就发现和预期不符」：观测台某个 chip 没渲染、模型名没透出、灰度开关点了没反馈、页面塌成窄条。根因是 AI 把「后端逻辑跑通 / 单测绿 / 我通读自审过」当成了验收终点，从没以真人身份打开那一屏点一遍。这既违反 `CLAUDE.md §8`（骨架完成就报「已实现」），也违反 `closed-loop-acceptance.md`（断头验收）。用户不是 QA——把「点一下看对不对」丢给用户，就是把验收责任反向转嫁。

---

## 强制条款

### 1. 「完成」的验收终点是「真实界面上产物可见且正确」

- 声称完成前，必须在**真实部署的预览域名**上，以真人路径（登录 → 点导航进入，禁地址栏直达）打开该功能那一屏，确认产物真的出现、内容正确、状态符合预期。
- 后端接口 200 / 数据落库 / 单测绿只是必要条件，**不是**验收本身（`e2e-verification.md`：API 200 ≠ 功能正常）。

### 2. 中大型 / UI 密集功能走 `/验收`（create-visual-test-to-kb）闭环

- 网关这类涉及多屏、双主题、需要留证的功能，用 `/验收` 技能跑工业级视觉验收全流水线：模拟人类浏览器取证 → 双主题截图 → 报告归档进 CDS 验收中心 → 出可达深链。
- **闭环**（`closed-loop-acceptance.md`）：生成/加载类功能必须等到产物**真的渲染出来**才截图，禁止用 timeout / spinner / 「加载中」冒充「已完成」。截图 caption 必须与截图内容一致。
- 双主题（白天/黑夜）都要过（呼应 `cds-theme-tokens.md`），不允许「只在黑夜测过」。

### 3. 证据挂回看板 + 用最新部署取证

- 验收产出的深链/截图挂到该工程的状态看板「验收证据」列（见 `living-status-board.md`）。
- 截图必须来自**最新** sha 的部署（页脚 CDS 徽章入镜），旧图一律重拍（`miduo-review-lens.md` 镜头 6：证据闭环）。

### 4. 自测优先，不得把校验先甩给用户

- 穷尽自测路径（Vitest/xUnit 集成测试 → `/cds-deploy` → Playwright 直连预览域名 → WebFetch 打真实端点）至少跑通一条并写明（`CLAUDE.md §8.1`）。
- 自测跑不出来就**明说「自测未跑通，问题可能仍在」**，禁止在这种状态下声称「已修复/已完成」。

---

## 自查清单（报「完成」前）

- [ ] 我以真人路径在**预览域名**打开这一屏了吗（点导航进入，非地址栏直达）？
- [ ] 产物**真的渲染出来**了吗，还是停在 spinner/加载中就截图了（断头验收）？
- [ ] 截图 caption 与截图内容一致吗？没有「图里在加载、字写渲染完成」的谎？
- [ ] 双主题都过了吗？
- [ ] 截图来自最新部署 sha 吗（CDS 徽章入镜）？
- [ ] 证据深链挂回状态看板了吗？
- [ ] 我写清了自测走的是哪条路径、断言了什么吗？

任一为否，不许说「完成」。

## 与既有规则的关系

- `closed-loop-acceptance.md`：本规则是它在「网关/用户可见功能」上的强制适用；断头验收判据完全复用。
- `CLAUDE.md §8 / §8.1`：完成标准 + 自测优先；本规则把「真视觉验收」钉成完成的必过项。
- `e2e-verification.md`：API 200 ≠ 功能正常，验收必须打开真实页面。
- `miduo-review-lens.md`：镜头 6 证据闭环 + 最新部署取证。
- `cds-theme-tokens.md`：双主题都要验。
- `living-status-board.md`：验收证据回挂看板「验收证据」列。
- 执行技能：`/验收`（create-visual-test-to-kb）、`/cds-deploy`、`/preview`。
