# 好用四原则（首席设计师视角）

> "好不好用"不能靠感觉。本规则把"好用"拆成**四条可执行的判定标准**，任何前端功能交付前都要逐条过。
> 触发：编辑 `prd-admin/src/**/*.tsx`、`prd-desktop/src/**/*.tsx`、`cds/web/**` 等任何用户直接操作的界面。
>
> 历史背景：2026-06-02 用户对"再加工里的视觉/文学创作"发出灵魂拷问——"好用吗？"。审下来四项里两项不及格（等待体感、信息过载）。根因是一直在堆功能，没站在首席设计师角度"剃"。本规则把那次审视固化成标准。

---

## 核心主张

**做加法谁都会，首席设计师做减法。** 功能能跑 ≠ 好用。好用 = 用户**最快**拿到结果、**最少**步骤、**最不**被打断、**最不**需要学习。下面四条是下限，不是加分项。

---

## 第一原则：快速启动，无需等待

等待就是偷用户的时间。

**判定**
- [ ] 从"想做"到"能做"几次点击？（理想 0-1，超过 2 要剃）
- [ ] 有没有"开局空白等待"？（首屏 loading、要先选一堆东西才能开始）
- [ ] 不可避免的等待（LLM/生图）有没有**边等边看**？（逐字流式 / 进度 / 预估耗时 / 可中止），静止 spinner 超 2 秒即不及格（呼应 `CLAUDE.md §6` 禁止空白等待）
- [ ] 能预取的提前预取（异步预热模型/拉选项），别让用户点了才开始转

**怎么剃**：默认就绪（免选直接能用）、乐观更新、流式反馈、预取。

**反面案例**：再加工选视觉创作要"点选择器→点视觉创作"两击才到面板；生图 30s 只有一个 spinner。

---

## 第二原则：强大且奥卡姆——复杂可控，但剃掉不需要人类的地方

强大不等于把所有旋钮怼到用户脸上。

**判定**
- [ ] 首屏只暴露**80% 场景需要**的控件；其余收进"高级/更多"渐进展开
- [ ] **只有一个选项的选择器一律不显示**（没得选就别假装能选）
- [ ] 能自动推断的不要问人类（默认值、智能预填、从上下文取）
- [ ] 复杂能力要"够得着但不挡路"——专家能展开细调，新手用默认就出活
- [ ] 每多一个控件先问："这个决策能不能由系统替用户做？"能就剃掉

**怎么剃**：渐进式披露（折叠高级项）、智能默认、单选项隐藏、上下文预填。

**反面案例**：视觉 mini 面板把千行的 `WatermarkSettingsPanel` 全摊开；模型只 1 个还显示下拉；参考图/水印一上来全亮出来，信息过载。

---

## 第三原则：可视化、不遮挡、够明显、够顺畅、合心智

用户的眼睛和直觉是第一交互界面。

**判定**
- [ ] 关键操作/结果**不被任何浮层遮挡**（z-index 查表，模态高于全局 FAB；见 `frontend-modal.md` / `cds-theme-tokens.md`）
- [ ] 主操作**一眼可见**（大小/颜色/位置符合视觉权重），次操作弱化
- [ ] 结果**看得清**（生成的图/数据不能小到看不清，要能点开/全屏）
- [ ] 操作**顺手**（点击区够大、反馈即时、不跳变、滚动不穿透）
- [ ] 合**用户心智**：这个页面是什么、现在什么状态、下一步做什么，3 秒内懂（呼应 `guided-exploration.md`）

**怎么剃**：视觉层级、状态可见、结果可放大、空状态有引导。

**反面案例**：发送按钮曾被铃铛 FAB 遮住（已修）；生成的图在 640px 抽屉里太小看不清，没法点开。

---

## 第四原则：顺用户习惯，不杜撰长链路，能短就短（但笃定 > 步数）

每多一步，就多一处流失——**但更要紧的是每一步都笃定无歧义**。

> Krug 第二定律修正（来自 `ux-heuristics` 技能）：用户不在乎点几下，在乎的是**每下点击是否轻松、明确、建立信心**。**3 次笃定的点击胜过 1 次困惑的点击**。所以别为了减步数把多个决策硬塞进一屏制造困惑——先保证每步笃定，再压步数。

**判定**
- [ ] 每一步是否**笃定**（点之前就知道会发生什么、自己在哪、对不对）？困惑比步数更致命
- [ ] 数一数完成核心任务**几步**？在不制造困惑的前提下能不能合并/省略？（理想 2-3 步）
- [ ] 有没有"为了演示而硬造的长链路"？（自动跑一串智能体、强行多阶段）
- [ ] 顺**已有习惯**（同类操作在别处怎么做，这里就怎么做，别另搞一套）
- [ ] "选中即就绪、输入即可执行、产出即可用"——把中间确认/跳转压到最少

**怎么剃**：合并步骤、就地操作（不跳页）、复用既有交互范式、默认串好下一步。

**反面案例**：配图链 文学→为这段配图→面板→生成→插入 = 5 步；E9 早期版本默默自动跑一串智能体（被用户批"杜撰长链"）。

---

## 交付前自审（每个 UI 功能必过）

```
好用四原则自审：
- [ ] 一原则 快启动：到能用 ≤1 击？等待有流式/进度/可中止？
- [ ] 二原则 奥卡姆：单选项隐藏？高级项折叠？能自动的没问人类？
- [ ] 三原则 可视化：不被遮挡？主操作明显？结果看得清？合心智？
- [ ] 四原则 短途：核心任务 ≤3 步？没杜撰长链？顺已有习惯？
```

四条里任一**不及格**，不许声称"好用"，回去剃。

## 评分线

- 四条全及格 = 合格（可上线）
- 有"加分项"（预取/乐观更新/一键直达/可放大预览）才算"好用"
- 两条以上不及格 = 不好用，必须返工（如 2026-06-02 视觉/文学 mini 面板）

## 与 `ux-heuristics` 技能的关系（established 框架 + 项目强调）

`.agents/skills/ux-heuristics`（从技能生态 `npx skills add wondelai/skills@ux-heuristics` 安装，3.1K installs，Krug「Don't Make Me Think」+ Nielsen 10 启发式 + WCAG + dark-patterns + 0-10 评分）是**通用、经过实战检验的可用性框架**；本规则是它之上的**项目强调**：

- **跑评分用 `ux-heuristics`**（触发词：`usability audit` / `UX review` / `heuristic evaluation`），它给 0-10 分 + 严重度 + 信息架构建议。
- **守项目底线用本规则**（四原则里的「奥卡姆剃刀剃掉不需人类处」「不杜撰长链」是本系统反复踩坑后的强调，通用框架未必突出）。
- 二者互补：先用 ux-heuristics 量分找问题，再用本规则的「怎么剃」给项目化的修法。

## 与其他规则的关系

- `zero-friction-input.md`（输入零摩擦）+ `guided-exploration.md`（3 秒知道做什么）+ `CLAUDE.md §6`（禁止空白等待）是本规则的**子集落地**；本规则是它们之上的"首席设计师总纲"。
- `frontend-modal.md` / `full-height-layout.md` / `cds-theme-tokens.md` 保证第三原则（不遮挡/撑开/对比度）。
- 本规则只定"判定标准"，具体实现手法去对应子规则。
