# 并发闸/队列组件设计纪律（Concurrency Gate Discipline）

> 任何「限流并发、其余排队」的组件（信号量/构建闸/任务队列/连接池/工作池）都必须在**第一版**就带齐
> 五件套：可取消、有身份、只锁真实资源、有周期收敛、有健康不变量。缺任何一件，队列迟早堵死，
> 且堵死时无人能回答「谁占着、卡多久、是不是坏了」。
> 触发：新增或修改任何并发闸/队列/池/长任务等待组件（`build-gate` 类、BullMQ 封装、连接池、Run/Worker 队列）。

---

## 历史背景（2026-07-16 CDS 构建队列堵死事故）

线上 build-gate（上限 3）堵死：`active=3 / queued=54`，消化约 12/小时，agent 部署排队 50 分钟以上，
用户反馈「agent 总是出问题」。第一版闸门只有 60 行（FIFO + 槽位转移），看似正确，实际六个根因叠加：

1. **排队不可取消**：被 supersede 的部署僵尸排队，占 FIFO 位置、把全局等待数撑到 54，其 15s 排队
   刷新定时器还不断给 run 账本写心跳，把幽灵 run 养活 24 小时以上。
2. **槽位锁了不吃资源的等待**：闸门为 CPU 而生，却把「容器起了、纯等就绪探测（下限 1200s）」也锁在
   槽里；崩溃容器白占槽 20 分钟。
3. **零编译的路径也过闸**：极速版（docker pull）部署无条件占槽。
4. **兄弟闭包脱管**：层内 `Promise.all` 首败即走 finally 释放分支租约，排队中的兄弟继续僵尸运行 →
   同分支叠出 3 个并发部署 x 6 服务 = 18 个队列位。
5. **重试风暴正反馈**：排队久 → agent 重试 → 撞车 409 → 再重试，每次重试塞一整层服务进队列。
6. **账本无周期收敛**：`reconcileInterrupted` 只在启动跑一次，幽灵 run 只能靠重启清。

修复 PR：#1160（含 Codex 三轮审查的 7 条意见）。本规则把教训固化，**新组件按此设计，别再事后补**。

## 五件套（第一版就必须有）

### 1. 等待可取消（Cancellable Waiting）
- `acquire` 必须接受 `AbortSignal`（排队中即时踢出）+ `isCancelled` 回调（授予时惰性跳过）。
- 排队方的「租约/任务被取代」信号必须能传导到队列：周期刷新 tick 首句查租约，失效即 abort。
- 判定：**任何一个排队者的发起方死了/被取代了，它能在一个刷新周期内离开队列吗？**

### 2. 持有者身份（Holder Identity）
- Waiter/Holder 必须携带业务身份（branchId/taskId/发起方），状态接口透出 `holders[]/waiters[]` 明细。
- 没有身份的队列堵死时只能看到数字，无法回答「这 3 个槽被谁占着、占了多久」——排障从小时级退化到猜。

### 3. 只锁真实资源（Lock Only Real Contention）
- 想清楚闸门保护的是什么（CPU/连接数/内存），**不吃该资源的阶段不许持锁**：
  纯等待（就绪探测/轮询/sleep）持锁 = 白占；零消耗路径（缓存命中/纯拉取）直接跳闸。
- 等待阶段必须有「对象已死就提前退出」的活性检查（如容器 exited 即放槽），不许空等到超时。
- 释放点收在 finally 单点；转移语义防超额（active 只在授予时 +、只在无人可转移时 -）；
  **上限运行时下调后，释放要先查当前上限、缩减而不转移**（Codex P1）。

### 4. 周期收敛（Periodic Reconciliation）
- 与队列关联的账本（run/任务记录）必须有**周期性**的心跳过期收割，不能只在启动时跑一次。
- 反向义务：**活着的长静默阶段要打心跳**（节流即可），否则收割器误杀慢启动（Codex P2）。
- 两边对齐口径：僵尸出队（#1）→ 心跳自然停 → 收割器收尸；三者构成闭环，缺一环就有幽灵。

### 5. 健康不变量 + 常态回归（Health Invariants + Standing Regression）
- 提供纯函数健康判定（积压阈值/持有超时/幽灵计数/账目不变量）+ 探针端点（健康 200/退化 503）。
- 账目不变量要选**真不变**的：`active === holders.length`（计数器泄漏）可以；
  `active <= max` 不行——上限下调后的自然收敛就是合法的 active > max（Codex P2）。
- 三层拦截：CI 单测（含事故值必红断言）+ 进程内看门狗（写可见告警事件）+ 定时任务探测探针。

## 队列语义的两条延伸纪律

- **重试请求要合并不要叠加**：同目标的重复请求（agent 重试）合并为 last-writer-wins 的 pending，
  而不是 409 逼对方重试或叠加新任务。但合并要认清在途操作语义：**终止类操作（stop/delete）在途时
  不许合并**（否则 pending 在停止完成后自动派发，抵消运维意图）；**携带特殊配置（版本/参数）的请求
  不许合并**（重放会丢配置）——都维持拒绝（Codex P2 x2）。
- **fan-out 必须等全体落地**：并行子任务用 allSettled + 共享 abort（首败广播踢出排队兄弟），
  绝不允许「首败即返回、兄弟脱管续跑」——那是重复租约/重复占位的根源。

## 等待体验（与 expectation-management 对齐）

- 排队/等待页的进度必须**基于该操作自己的历史耗时**（独立样本桶，成功即记录），禁止硬编码百分比、
  禁止借别类操作的中位数（重启借构建中位 = 15s 被压成 3%）。无样本时给持续移动的时间曲线 +
  「正在积累历史数据」，绝不静止。
- 被合并/排队的请求要如实告知「已排上号、尚未执行」，不许报「已完成」。

## 交付前自审

- [ ] 排队者的发起方被取消后，它能在一个刷新周期内离开队列？
- [ ] 状态接口能回答「谁占着、占多久」？
- [ ] 纯等待阶段不持锁 / 有活性早退？零消耗路径跳闸？上限下调后释放先查上限？
- [ ] 关联账本有周期收割 + 活跃长静默有节流心跳？
- [ ] 有健康纯函数 + 探针端点 + 事故值必红的回归测试？不变量不会把合法状态误报？
- [ ] 重试会合并而不是叠加？合并排除了终止类在途与带特殊配置的请求？
- [ ] fan-out 等全体落地才释放上级资源？
- [ ] 等待 UI 走该操作自己的历史耗时、始终在动、合并态如实展示？
