# 端到端验收原则

功能开发完成后，必须以用户视角完整验收，禁止仅通过 API 调用确认"数据存了"就认为完成。

## 强制规则

1. **验收必须打开真实页面**：API 返回 200 ≠ 功能正常。必须实际访问前端页面，检查渲染结果是否符合设计稿
2. **对照设计稿逐项核查**：如有布局图/线框图，逐个区域检查是否实现完整（tab 数量、数据展示、排序顺序）
3. **"数据存了"≠"功能完成"**：后端存储了字段 ≠ 前端展示了字段。必须确认前端类型定义接收了所有必要字段，且 UI 有对应展示入口
4. **验证新旧数据兼容**：新功能上线后，旧数据（没有新字段）的 fallback 路径也必须正常展示，不能只测新创建的数据
5. **视觉排序必须用浏览器验证**：CSS 布局（columns/grid/flex）可能在视觉上重排元素。API 返回顺序 ≠ 页面渲染顺序

## 反面案例

| 错误做法 | 正确做法 |
|----------|----------|
| 调 API 拿到 200 就认为排序正确 | 打开页面检查第一行第一个是否是最新/最热的 |
| 后端存了水印字段就认为水印展示正常 | 打开详情页水印 tab 确认有内容，非"未使用" |
| 只测新创建的投稿 | 同时打开旧投稿验证 fallback 路径 |
| CSS columns 看代码觉得"顺序应该对" | 实际打开页面发现 columns 按列填充打乱顺序 |

## 检查清单

功能涉及前端展示时，提交前必须过以下清单：

- [ ] 打开页面确认布局/排序符合预期
- [ ] 新数据 + 旧数据都能正常展示
- [ ] 所有 tab/面板都点击查看过
- [ ] 前端类型定义覆盖后端返回的所有字段
- [ ] fallback/兜底路径有完整数据（而非精简版）
