# 无根之木禁令 & 借用法则

> 系统不假定无根之木。做不到的事不装能做，而是明确暴露 + 借用外力。

## 原则〇：不接受虚幻的有根之木

**看起来有根，但根是假的，比无根更危险。**

### 判断标准

一棵树看起来有根（代码存在、能编译），但如果这个根：
- 只在特定场景下成立，换个场景就失效 → 虚幻
- 是硬编码的快照，不能自动跟随真实状态更新 → 虚幻
- 绑死了一个具体实现，无法适应需求变化 → 虚幻

### 案例

| 虚幻的根 | 为什么虚幻 | 真实的根 |
|---------|-----------|---------|
| `SystemCapabilities = "硬编码30行"` | 系统变了它不变 | 运行时从注册表动态扫描 |
| 注册表扫描只能扫本系统 | 用户想分析别的系统就废了 | 种子文档 = 主上下文，注册表 = 可选辅助 |
| `if (hasFeature("RAG"))` 但没人维护这个 flag | 永远是 false | 运行时检测能力是否存在 |

### 强制规则

设计任何「基于真实数据」的功能时，检查：
- [ ] 这个"真实数据"的来源是动态的还是静态的？
- [ ] 如果换一个上下文（不同系统/不同用户/不同场景），这个数据还成立吗？
- [ ] 如果不成立，是否提供了替代路径（用户上传 / 手动输入 / 可选开关）？

## 原则一：无根之木禁令

**任何系统能力的声明都必须有根。没有根的能力 = 幻觉。**

### 强制规则

1. **不假定能力存在**：代码中引用某个能力（如"多文档扫描"、"语义搜索"、"实时协同"）时，必须确认该能力已实现。未实现的能力不能当成已有能力使用
2. **未知数必须暴露**：当系统缺乏某个能力时，必须以可见的方式告知用户，而非静默降级或假装能做
3. **暴露格式**：在 UI 上以提示信息展示，说明「当前缺乏 XX 能力」并给出替代方案

### 暴露方式

| 场景 | 错误做法 | 正确做法 |
|------|---------|---------|
| AI 生成依赖了系统没有的能力 | 直接展示，让用户以为能做 | 节点标注「⚠ 需要：多文档扫描能力（当前不具备）」 |
| 涌现出的功能需要 Embedding 服务 | 写「使用 Embedding 搜索」 | 写「需借用外部 Embedding 服务（当前系统未集成）」 |
| Prompt 声称系统有 RAG 能力 | 硬编码「已有 RAG」 | 从运行时扫描，确认没有则不列出 |

### 判断清单

写代码或 Prompt 时逐条检查：

- [ ] 这个能力是运行时可验证的（注册表里有 / API 存在 / 集合已创建）？
- [ ] 如果不可验证，是否明确标注为「未实现」或「需外部支持」？
- [ ] AI prompt 中列出的每一条能力，都能在 SystemCapabilityScanner 中扫描到？
- [ ] 用户看到的每一个功能建议，都标注了它的实现前提？

## 原则二：借用法则

**知道自己几斤几两。做不到的不强做，合理借用外部 Agent 扩充己身。**

### 系统定位

```
本系统 = 思考中枢（分析、规划、决策、涌现）
       ≠ 执行中枢（不是什么都自己干）

能做的：理解文档、分析需求、规划功能、涌现创意、生成方案
不能做的：多文档向量检索、实时代码扫描、图片生成、音视频处理
```

### 借用而非强做

| 自己缺什么 | 错误做法（强做） | 正确做法（借用） |
|-----------|-----------------|-----------------|
| 多文档语义扫描 | 自己写一个半成品 RAG | 借用外部 Embedding Agent，结果传给本系统 |
| 代码库实时分析 | 在 .NET 里写代码解析器 | 借用 Claude Code Agent 扫描后传入能力清单 |
| 图片生成 | 集成一个简陋的生成服务 | 通过 ILlmGateway 路由到 Visual Agent |
| 数据可视化 | 自己画图表 | 借用 ECharts 组件（已有） |

### 借用的实现方式

```
方式 1：技能借用（Skill）
  本系统调用其他技能 → 技能返回结果 → 本系统使用结果
  例：涌现前调 /cds-scan 技能扫描项目结构

方式 2：Agent 委托
  本系统委托其他 Agent 执行 → Agent 返回产物 → 本系统整合
  例：委托 Claude Code Agent 扫描代码库 → 传回能力清单

方式 3：数据管道
  外部系统定期推送数据 → 本系统存储 → 需要时直接读取
  例：CI/CD 推送构建产物清单 → 涌现时读取最新状态

方式 4：用户中转
  用户从其他工具获取数据 → 上传/粘贴到本系统 → 本系统分析
  例：用户上传 PRD 文档 → 本系统解析并涌现
```

### 借用的 UI 表达

当涌现探索器生成了需要借用外力的节点时：

```
┌─ 节点：智能文档语义搜索 ─────────────────┐
│  描述：基于文档内容的语义检索              │
│  锚点：DocumentEntry + Attachment          │
│                                            │
│  ⚠ 需要外部支持：                         │
│  • Embedding 向量服务（当前未集成）         │
│  • 建议：借用外部 Embedding Agent           │
│    或接入 Atlas Search                     │
└────────────────────────────────────────────┘
```

## 与其他原则的关系

```
涌现时 →「无根之木禁令」确保 AI 不编造能力
      →「借用法则」为缺失能力提供替代路径
      →「反向自洽」确保每个节点可回溯
      →「零摩擦」确保用户借用时不发呆
      →「2 秒原则」确保借用过程有反馈
```

五个原则形成闭环：**不编造 → 暴露缺失 → 提供替代 → 回溯验证 → 流畅体验**。
