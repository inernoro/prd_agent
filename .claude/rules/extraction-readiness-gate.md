# 剥离干净度记分卡 + 可安全发布 Gate

> 「剥离干净没有」不能靠感觉、靠一句「差不多了」。必须维护一张**可机器核对**的剥离干净度记分卡，并定义一个由硬判据组成的「可安全发布 Gate」。Gate 全绿才允许把默认态切到生产；任一判据红灯，就是「还不能发布」，且明确指出红在哪。
> 触发：任何「把旧实现剥成独立部件 + 逐步切换默认态」的工程（如网关从 MAP 剥离）。

---

## 历史背景（本次痛点）

用户对网关剥离最后一步的顾虑是**「不敢发布」**——因为「验收方式不透明、剥离到底干不干净说不清」。AI 说「核心命题距离只差翻一个 flag」，但用户无法独立核验：默认还是不是 inproc？有没有哪个调用还偷偷走老路径？影子比对到底跑了多少样本、差异率多少？容器里有没有静默的解密失败 / 401？这些都散在日志和 AI 口述里，没有一处汇成「客观、可核对、够不够发布」的判据。本规则把「剥离干净度」变成一张任何人都能照着核对的记分卡 + 一个明确的发布闸门。

---

## 强制条款

### 1. 维护一张剥离干净度记分卡（挂在看板旁）

记分卡列出「剥离要达成的每一件事」及其**当前客观值 + 目标值 + 数据来源**，例如网关剥离：

| 维度 | 判据 | 目标 | 当前值来源 |
|---|---|---|---|
| 默认路径 | `LlmGateway:Mode` 默认态 | 目标态（如 `http`）已按计划切换 | appsettings / 配置端点 |
| 调用去老路 | 仍走进程内老实现的 appCallerCode 数 | 0（或仅白名单内） | `HttpAppCallerAllowlist` + 影子比对来源统计 |
| 配置面迁移 | 模型池/调度/日志配置是否已迁到 serving 侧 | 100% 迁完 | serving 配置读端点 |
| 影子证据 | `llmshadow_comparisons` 样本量 / allMatch 率 | 样本 ≥阈值 且 diff 率 ≤阈值 | `GET /gw/v1/shadow-comparisons` |
| 容器健康 | serving 容器近 N 分钟解密失败 / X-Gateway-Key 401 数 | 0 | 容器日志 |
| 回归 | 全量测试 + navCoverage | 全绿 | CI / `pnpm test` / `dotnet test` |

记分卡的每个「当前值」必须来自**可查询的客观来源**（端点/日志/测试结果），禁止填「AI 认为差不多」。

### 2. 判据必须可机器核对（不靠主观）

- 每条判据都是「读某个端点 / 数某个日志 / 看某个测试退出码」就能得出的**布尔或数值**，任何人（含用户）照着能自己核一遍。
- 阈值（影子样本量下限、diff 率上限、健康窗口 N 分钟）写死在记分卡里，不临场拍。

### 3. 棘轮：A 类问题数只能单调下降（Ratchet）

- 定义「A 类问题」= 会导致行为不一致 / 数据错乱 / 静默降级的严重项（如影子比对 diff、解密失败、选 A 给 B）。
- A 类问题数是**棘轮**：每次核对只能 ≤ 上次，禁止回升。回升即视为剥离退化，Gate 立即变红，且不许用「新功能引入的」当借口——先修回 0 再谈发布。

### 4. 「可安全发布 Gate」——全绿才放行

发布（把默认态切到生产 / 宣布剥离完成）的**充要条件**是下列判据同时成立，缺一不可：

1. 默认 `Mode` 已按计划切到目标态；
2. 棘轮 A 类问题 = 0；
3. 配置面已全部迁移到 serving 侧；
4. 影子比对样本量 ≥ 阈值 且 diff 率 ≤ 阈值；
5. 关键容器近 N 分钟无解密失败、无 X-Gateway-Key 401；
6. 全量回归 + navCoverage 全绿；
7. 关键用户可见面已过真视觉验收（`real-visual-acceptance.md`）并留证。

任一红灯 → 输出**「不可发布 + 红在哪 + 补救下一步」**，禁止「先发再补」。

## 自查清单（考虑发布时）

- [ ] 记分卡每个「当前值」都来自可查询的客观来源，不是 AI 主观判断？
- [ ] 每条判据都能被用户独立照着核对？
- [ ] 棘轮 A 类问题数是 0 且未回升？
- [ ] Gate 七条判据逐条核过、全绿？
- [ ] 若有红灯，我是否明确输出「不可发布 + 红在哪 + 下一步」，而不是含糊「差不多能发」？

## 与既有规则的关系

- `living-status-board.md`：记分卡 + Gate 结论挂在看板顶部「距离可发布」一行，是它的量化底座。
- `real-visual-acceptance.md`：Gate 第 7 条判据；用户可见面必须过真视觉验收。
- `compute-then-send.md` / `llm-call-trace`：影子比对捕捉的「选 A 给 B」正是 A 类问题的典型，棘轮直接复用其判据。
- `cross-project-isolation.md`：容器解密失败 / 密钥 401 与共享密钥通道强相关，Gate 的容器健康判据是它的运行时哨兵。
- `CLAUDE.md §8`：Gate 是「完成标准」在剥离型工程上的硬化——没到 Gate 全绿不算完成。
- 本次实例数据源：`GET /gw/v1/shadow-comparisons`、`llmshadow_comparisons` 集合、`LlmGateway:Mode`、`HttpAppCallerAllowlist`、`doc/plan.llm-gateway.rollout.md`。
