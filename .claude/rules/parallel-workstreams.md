# 大型多部件工程默认并线开发（Parallel Workstreams）

> 一个工程能拆成多条**互不阻塞**的独立轨道时，默认用 Workflow 并行 fan-out（每条轨道独立 worktree 隔离）同时推进，而不是一条一条串行做。串行是慢的默认，并行是快的默认——能并就并。
> 触发：规划任何可拆成 ≥2 条弱耦合轨道的大型工程时。

---

## 历史背景（本次痛点）

网关剥离横跨 serving 服务、MAP 侧 HttpClient、观测前端（prd-llmgw-web）、影子比对、CDS 多容器/子域路由、验收取证等多个部件，其中很多部件彼此不阻塞（前端观测台不必等灰度开关做完、CDS 子域路由不必等影子比对做完）。但实际是一条线串着做，用户直接反馈**「太慢，应该并线」**。根因是没有把「可并行」当成规划时的默认审查项，习惯性串行。本规则把「先问能不能并」固化为大型工程开工的强制步骤。

---

## 强制条款

### 1. 开工前先做「可并行性拆分」

规划大型工程时，先把它拆成轨道，逐条判断耦合度：

- **弱耦合（可并行）**：改动文件集基本不重叠、无「A 完成才能开始 B」的硬依赖、可各自独立验收。→ 必须并行。
- **强耦合（须串行）**：B 依赖 A 的产物/接口/数据结构，或改同一批核心文件。→ 串行，并在看板标注依赖箭头。

判定口诀：**这两条轨道能不能同时开工、各自独立跑到验收而不用等对方？** 能 → 并行。

### 2. 满足任一条件时「必须并行」，不许默认串行

- 工程含 ≥2 条弱耦合轨道，且预计总工期 ≥2 天；
- 用户已明确要求「快 / 并线 / 别串着做」；
- 某条轨道正卡在外部 blocker（等镜像/等用户输入），而存在不依赖该 blocker 的其它轨道——此时**必须**切并行推进其它轨道，而不是干等（呼应 `blocked-state-circuit-breaker.md` 第二节「切换到不依赖该 blocker 的纯代码工作」）。

### 3. 并行轨道必须 worktree 隔离

- 每条并行轨道用独立 git worktree / 独立分支，物理隔离工作区，避免互相踩改动、避免半成品交叉污染。
- 轨道边界对齐 `scope-check`：每条轨道声明自己 owned 的文件集，跨轨道共享文件（shared）改动要显式协调，禁止一条轨道越界改另一条的 owned 文件。
- 各轨道**独立验收、独立出证据**（走 `real-visual-acceptance.md`），在看板各占一行。

### 4. 并行 ≠ 失控

- 每条轨道在状态看板（`living-status-board.md`）单独成行，标清进度/blocker/下一步/证据，且标注轨道间依赖箭头。
- 合流点（多轨道汇入 main）前跑 `/resolve` 预合并 + 全量回归，确保并行没引入冲突或行为回归。

---

## 自查清单（规划大型工程时）

- [ ] 我把工程拆成轨道并逐条判过耦合度了吗？
- [ ] 有 ≥2 条弱耦合轨道吗？有 → 是否已安排并行（而非习惯性串行）？
- [ ] 某轨道卡外部 blocker 时，我是切去并行推进其它轨道，还是在干等？
- [ ] 每条并行轨道是否 worktree 隔离、owned 文件集清晰、无越界？
- [ ] 每条轨道在看板单独成行、依赖箭头标清？
- [ ] 合流前跑了 `/resolve` + 全量回归？

---

## 与既有规则的关系

- `blocked-state-circuit-breaker.md`：卡 blocker 时「切不依赖该 blocker 的工作」正是并行的一种；本规则把「并行」从被动逃生升级为主动默认。
- `living-status-board.md`：并行轨道各占看板一行，看板承载轨道视图与依赖。
- `real-visual-acceptance.md`：每条轨道独立验收出证据。
- `scope-check` 技能 / `cross-project-isolation.md`：轨道边界与 owned/shared 划分、避免并行踩踏。
- `conflict-resolution`（`/resolve`）：合流前预合并解冲突。
