# 产品管理系统设计理念

> 基于 MAP 产品管理智能体（product-agent）已落地能力与 `doc/design.product-agent.md` 整理。  
> 主题：**产品管理系统** —— 把研发主线对象串成一张可操作的关系网。

---

## 1. 一句话定位

**文档即共识的反面是「关系即主线」**：产品管理系统不是再写一份 PRD，而是让「需求从哪来、归哪个版本、落成哪些功能、被哪些缺陷追溯、沉淀进哪个知识库」在系统里有载体、有流程、有视图。

与 **项目管理智能体（pm-agent）** 分工明确：

| 维度 | 产品管理系统（product-agent） | 项目管理（pm-agent） |
|------|------------------------------|----------------------|
| 管理对象 | 持续演进的产品 | 临时性项目 |
| 核心轴 | 版本迭代 + 需求/功能/缺陷追溯 | 任务看板 + 干系人评价 |
| 时间观 | 长期演进（T 立项号 / V 上线号） | 项目周期与里程碑 |

## 2. 设计原则

### 2.1 复用，不重建

- **缺陷**：不复制 defect-agent，只做 `TracedProductId` / `TracedRequirementId` 追溯引用。
- **知识库**：不新建文档引擎，产品/版本挂载 DocumentStore（MRD / SRS / PRD 分型）。
- **图谱**：复用 ReactFlow 画布，对象为节点、关系为边。
- **权限与 LLM**：走平台统一 RBAC、`ILlmGateway`、Run/Worker + SSE，不另起炉灶。

### 2.2 通用引擎，最小 schema

两类横切能力服务全部核心对象：

1. **表单模板引擎** —— `ProductFormTemplate` + 实例 `FormData`（key = 字段 Key）。
2. **流程定义引擎** —— `ProductWorkflowDefinition` + `CurrentState` + 统一 `POST /transition`。

P0 只保证六类对象够用；不为 hypothetical 场景过度抽象。

### 2.3 默认可编辑，SSOT 在后端

- 前端不硬编码 `switch(type)`，用 `*_REGISTRY / *_MAP` 注册表。
- 列表反规范化字段（计数、负责人名）由写操作维护，读路径不二次拼装业务规则。
- 工作流、表单、导航顺序均可配置，但**运行时 SSOT 是 MongoDB 里的定义 + Controller 校验**。

### 2.4 零摩擦输入 + 有引导的探索

- 能导入就不手敲：CSV / RTF / Excel 历史数据导入，相同外部 ID 更新而非重复造数。
- 陌生页面 3 秒内知道做什么：空状态必须有下一步（立项向导、导入、新建）。
- LLM 交互禁止空白等待：评审、AI 补全、助手问答一律 SSE 流式 + 阶段提示。

### 2.5 负责人「认领制」，非「谁导入谁负责」

- 产品 **OwnerIds** 可为空，创建/导入默认不绑定当前登录用户。
- 支持 **多位负责人**；列表无负责人时展示「待认领」。
- 团队成员、产品管理员与负责人权限分层，避免「导入历史数据就把 ownership 写死」。

## 3. 核心对象与关系

```text
产品 (Product)
  ├── 内部版本 / 立项 (ProductInitiation, T 号)
  ├── 正式版本 / 上线 (ProductRelease, V 号)
  ├── 需求 (Requirement)
  ├── 功能 (Feature)
  ├── 客户 (Customer)
  ├── 团队 (MemberIds / AdminIds / OwnerIds)
  └── 知识库 (KnowledgeStoreId)

关系链（RTM 主线）：
  客户 → 需求 → 功能 → 版本
  缺陷 ──追溯──→ 需求 / 功能 / 产品
  知识库 ←── 沉淀 ── 版本 / 产品
```

**分级**：产品类型（系统/子系统/应用/插件 + 可扩展）、需求/功能/缺陷统一 P0–P3。

**编号**：产品编号 = 类型前缀 + 全局序号；需求/缺陷 ID 对齐 TAPD 纯数字规则，导入保留外部 ID。

---

## 4. 版本流程（T / V 双轨）

承接原 TAPD + 语雀 + 产品委员会流程，在 MAP 内闭环：

| 轨道 | 含义 | 关键能力 |
|------|------|----------|
| 内部版本 (T) | 立项与三稿评审 | 三步向导、Agent 评审打分、负责人同意、生成 T 号 |
| 正式版本 (V) | 上线与公告 | 绑定已批准立项申领 V 号、功能 manifest、上线公告地址 |

列表与详情 **不再区分「系统 / 应用」两列**，统一为 **「产品」** 列（兼容历史导入的 systemName/appName）。

---

## 5. 需求主线

### 5.1 工作流

内置「米多需求收集工作流」七态（可配置扩展）：

待评审 → 待规划 → 已立项 / 已排期 → 开发中 → 已上线  
（分支：已拒绝、转为缺陷）

### 5.2 创建与导入

- **新建**：TAPD 风格左文右栏；默认状态 **待评审**（`new`），禁止 `state` 占位泄漏。
- **CSV 导入**：见同目录 `产品管理系统-需求测试数据.csv` 表头规范。
- **RTF 导入**：TAPD 批量导出按 ID 拆条，保留 `SourceSnapshot` 供状态中文兜底展示。

### 5.3 来源与类型

- 需求来源：客户反馈 / 内部规划 / 运营活动 / 竞品调研 / 其他（联动补充字段）。
- 需求类型：内置 + 可扩展目录，AI 补全可读 `definition` 识别。

## 6. 权限模型（摘要）

| 角色 | 能力 |
|------|------|
| MAP 应用管理员 | 全产品可见、历史导入、debug reset、应用配置 |
| 产品负责人 (OwnerIds) | 删产品、指派产品管理员、成员管理 |
| 产品管理员 (AdminIds) | 增删普通成员 |
| 产品成员 (MemberIds) | 访问单产品数据、参与流转 |
| 全局 `product-agent.manage` | 模板/流程/删除等管理操作 |

工作流流转按 `allowedRoles`（owner / assignee / product_admin / member）与 `isProductOwner` 多负责人逻辑校验。

## 7. 界面信息架构

**管理层（OverviewShell）**  
概览 → 产品 → 需求 → 功能 → 版本 → 缺陷 → 客户 → 知识库 → 图谱 → 应用配置 → 设置

**单产品（SingleProductView）**  
同上分 Tab，数据 scoped 到当前 `productId`。

**入口**  
百宝箱 / 左侧导航「产品管理」→ 选产品 → 进入单产品或总览各区块。

---

## 8. 与测试数据的关系

同目录 **`产品管理系统-需求测试数据.csv`** 提供 50 条以「产品管理系统」为主题的需求，用于：

1. 总览 → 需求 → **导入历史数据**（选归属产品「产品管理系统」）。
2. 验证 CSV 列解析、外部 ID 去重更新、等级/状态映射、列表与详情展示。
3. 后续可在此基础上补功能、缺陷、版本导入，跑 RTM / 图谱 / 工作流 smoke。

导入路径：**产品管理 → 需求（总览）→ 导入历史数据 → CSV**（需应用管理员权限）。
