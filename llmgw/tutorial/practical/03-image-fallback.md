# 实战 03：配置 Offering 负载、限流与故障切换

这一篇只调整同一逻辑模型内部的上游路线。不要通过修改视觉创作模型列表或复制模型池来实现负载均衡。

## 主备模式

1. 逻辑模型的路由策略选择 `priority`。
2. 主 Offering 优先级设为 10，备用依次设为 20、30。
3. 为每条 Offering 填写真实的最大并发和 RPM；未知时留空，不填猜测值。
4. 保存后发起一次安全的假上游或隔离环境请求，在日志中确认主 Offering 命中。

## 负载分担模式

1. 只有多个上游都完成质量和成本验收后，才切换到 `weighted`。
2. 权重表达相对份额，例如 70 与 30，不要求总和为 100。
3. 权重只决定主候选，失败后的其余候选仍是有界重试列表。
4. 限流在 Gateway serving 中执行。多实例共享同一 Mongo 分钟窗口，不能靠单进程内存计数。

## 故障切换验收

1. 在测试环境使用可控假上游，让第一条 Offering 返回 429 或 503。
2. 同一次请求应尝试第二条 Offering；如果协议不同，第二次请求的 endpoint 和 body 必须符合第二条协议。
3. 打开请求日志“路由”页签，核对逻辑模型未改变、OfferingId 与实际 Provider 已改变，并看到两次上游尝试。
4. 第一条 Offering 的连续失败和健康状态被更新；第二条成功只更新自己的成功状态。
5. 候选全部失败时返回结构化错误，不静默改成另一个逻辑模型。

## 限流验收

- 同一租户、同一 Offering 达到 RPM 后返回 `PROVIDER_RATE_LIMIT_EXHAUSTED`，并可尝试下一个 Offering。
- 不同 Offering 的窗口互相独立。
- 不同租户即使使用同一 Provider，也不能共享租户级 Offering 计数。
- 模型或平台更粗粒度的并发总闸仍可同时生效。

## 看到什么算成功

用户始终选择同一个 PublicId；Gateway 能解释每次实际上游。正常主备、限流切换和协议重建都有 requestId 证据，应用无须知道上游发生了变化。

## 常见失败

- 第一条 429 后直接失败：确认逻辑模型还有启用候选，并且该请求类型提供 canonical request。
- 健康一直不变：检查日志是否带 OfferingId；逻辑模型路径不能只按旧 ModelGroupId 更新健康。
- 多实例 RPM 超发：检查唯一索引与原子 upsert，不得改成每实例内存计数器。
- 备用成功但页面显示另一逻辑模型：日志和 UI 主标题应使用 LogicalModelPublicId，实际上游只作为次级信息。
