# 实战 04：从视觉创作验证图片模型并回查日志

配置完成后，先用测试桩验证免费链路，再用真实图片模型做一次最小付费验证。不要通过连续出图证明“应该可用”。

## 验证顺序

1. 免费链路：选择 `stub-image`，发起一次测试请求，确认页面、队列、Gateway 和日志链路闭环。
2. 真实链路：先选择 `openai/gpt-image-2`，使用简单提示词只生成一张图片；另外两款先以可见、可选择和路由命中为验收项，避免连续付费出图。
3. 日志回查：记录 requestId，确认实际模型、Provider、模型池、耗时、费用状态和上游尝试。

## 跟我做

1. 打开视觉创作，新建一个测试工作区。
2. 在模型选择器中确认至少出现两款可用模型，先选择 `stub-image`。
3. 输入不含敏感业务信息的简单提示词，只生成一张测试图。页面应持续显示排队、生成和完成阶段，不应长时间静止。
4. 记录完成结果里的 requestId；如果业务页只显示 runId，到“请求记录”按时间、appCaller 和模型缩小范围。
5. 切换到“OpenAI GPT Image 2”，再次只生成一张图，不开启批量模式。
6. 复制这次真实请求的 requestId。
7. 进入 LLMGW“工作区 → 请求记录”，在顶部唯一的 requestId 搜索框粘贴完整编号并查找。

![请求记录使用一条紧凑工具栏按 requestId 定位](https://cds.miduo.org/api/reports/assets/2c0565db3416c7b60a2bd11f212b1fa2a201a3df4b0a5d0e0c657b1046907709.jpg)

8. 在“概览”核对状态、应用、实际模型、Provider、总耗时和费用状态。
9. 在“路由”核对期望模型、实际模型、模型池、参数策略和上游尝试。若发生回退，记录原因。
10. 在“审计”核对币种和价格快照。不知道价格时应为 unknown，而不是 0。

![请求详情概览先回答实际模型、Provider 和上游尝试次数](https://cds.miduo.org/api/reports/assets/461587f1666bbdd38fab1b2c96bbefbd1aa81576478f15d41497dc978070f199.jpg)

## 看到什么算成功

- 业务页能分别选择 GPT Image 2、Nano Banana 2 和 Nano Banana 2 Lite，并由日志记录精确的期望模型；GPT Image 2 完成一张真实图片。
- 日志实际模型与业务页选择一致；若不一致，详情明确说明回退原因。
- Provider 指向新建或复用的有效图片供应商，不是已不存在的旧平台。
- 请求状态、耗时和费用状态有明确值，密钥只显示前缀或遮盖信息。
- 本轮真实供应商测试只有一次，不做无意义的批量付费复测。

## 常见失败

- 业务页没有新模型：回实战 02 检查 generation 池成员和 appCaller 绑定。
- 请求未进入日志：检查业务调用是否经过 Gateway，以及 appCallerCode 是否为 generation。
- 401 或 403：检查业务接入 key、团队、appCaller 和租户权限；上游 Provider key 不用于登录 Gateway。
- 404 model not found：上游模型标识错误，或供应商账号没有该模型权限。
- 400 unknown parameter：检查实际模型适配器和请求参数；不要把某一供应商支持的可选参数无条件发送给另外两款模型。
- 生成成功但尺寸与选择不同：自适应模型可能按提示词决定尺寸；用日志中的实际模型和有效尺寸判断，不要把它当成路由错误。

## 回滚与交接证据

保留变更时间、Provider 名称、模型标识、池名称、appCallerCode、一次真实 requestId 和回滚前池 ID。不要保存通讯密钥、完整请求正文或含业务隐私的图片。
