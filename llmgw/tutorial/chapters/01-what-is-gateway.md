# 第 1 章 什么是模型网关

## 你在做什么

这一章不创建数据，只建立一条简单认识：业务应用把请求交给 Gateway，Gateway 依据租户、密钥、appCaller 和模型池选择模型，再通过 Provider 访问上游。调用结果回到业务应用，同时留下请求、费用和审计证据。

## 为什么要做

很多配置错误来自把不同对象当成一回事。例如把 Provider 密钥发给业务应用，或以为模型池本身就是模型。正确理解后，你会知道哪里出错该看哪里：进不了门看接入密钥，业务归类不对看 appCaller，选错模型看模型池和 Exchange，上游连接失败看 Provider。

## 开始前检查

- 保持登录，但不要修改任何页面。
- 在纸上写四个方框：业务应用、Gateway、模型池、Provider 与模型。
- 记住租户边界来自服务端登录会话或接入密钥，调用方不能通过请求自报 `tenantId` 切换租户。

## 看到什么算成功

你能用自己的话复述：客服 Agent 使用“教程咖啡店”的 test key，以 `tutorial.gateway-book::chat` 表明用途，Gateway 从“默认对话池”选择“教程聊天模型”，再由“教程假上游”处理。你不会把登录密码、租户 key 和 Provider 通讯密钥混用。

## 失败怎么办

- 学习中心入口不可见：先刷新并确认登录会话有效；这个入口对五种控制台角色都开放，仍看不到时记录当前角色交给管理员排查。
- 仍分不清模型和模型池：把模型理解为一名具体员工，把模型池理解为一组可按优先级候选的员工；池能有多个模型。
- 仍分不清 appCaller 和 key：同一个业务用途可以在 test 与 production 各用一把 key，但 appCaller 仍是同一个用途名称。
- 看到请求模型与实际模型不同：先不要判定错误；模型池回退或 Exchange 映射都可能使实际模型不同，后续章节会教你核对。

## 本章小结

Gateway 是统一入口和可审计的调度层。业务应用不直接保管 Provider 凭据，也不自行决定租户；路由、权限、费用和日志在同一条服务端链路中留下证据。
