# 第 18 章 看懂健康、优先级和回退

## 你在做什么

这一章不改配置，只读“默认对话池”和“默认视觉池”的健康、优先级与最近回退信息。你要学会回答：这次为什么选它，失败后又去了哪里。

## 为什么要做

池里有多个模型，不等于每次随机抽一个。Gateway 会同时考虑用途、参数能力、启用状态、健康和池策略。看不懂这些信息时，用户容易把正常回退误认为“模型被偷偷更换”，或者在一个成员短暂故障时把整个池删掉。

## 开始前检查

- 第 17 章的两个默认池仍在。连续路线可能每池只有一个成员，因此本章不强求制造回退。
- 最近请求页已有安全测试产生的 requestId；安全测试只证明接入链路，不代表真实上游质量。
- 不准备批量调用付费模型。官方教程桩没有“返回失败”开关，本章禁止假装页面可以制造故障。
- 普通租户管理员只做页面观察；项目验收人员可在隔离测试环境补做自动回退验证。

## 跟我做

1. 进入左侧“模型池”，打开“默认对话池”。先读池策略，例如“优先级”或“故障转移”，再读每个成员的优先级。
2. 查看成员旁的健康状态。健康表示 Gateway 最近掌握的可用情况，不是永久承诺；“不可用”也不等于配置一定被删除。
3. 回到“请求记录”，打开一条带 requestId 的记录，比较“期望模型”“实际模型”“模型池”和 Provider 信息。
4. 若详情带“已降级”标记，把鼠标停在标记上读取期望模型，再核对列表中的实际模型。只有真实存在的记录才可作为回退证据。
5. 如果没有回退记录，明确写下“当前没有回退样本”。不要寻找不存在的失败开关，也不要改坏 Provider 来凑数据。
6. 回模型池查看成员健康、近 7 天请求和成功率，再到“请求记录”看“回退请求”汇总。以后真实故障出现时，再按 requestId 补做这一章的回退观察。
7. 普通管理员到此结束。项目验收人员在专用测试环境运行仓库已有的健康与回退套件，它使用内存假上游，不需要任何真实 key：

```bash
dotnet test prd-api/PrdAgent.sln --no-restore \
  --filter "FullyQualifiedName~PoolHealthTrackerTests|FullyQualifiedName~FailFastStrategyTests|FullyQualifiedName~ModelPoolSchedulingTests"
```

8. 只有套件全部通过时，验收人员才能补充“不可用成员被跳过、降级成员按规则选择、成功后健康恢复”的完整证据。

## 看到什么算成功

你能指出池策略、成员顺序和当前健康。若已有回退样本，你能用 requestId 说清期望与实际模型；若没有，你明确标记“尚无回退证据”。项目验收人员还应拿到健康与回退套件全绿的替代证据，不在页面伪造故障。

## 失败怎么办

- 所有成员都显示不可用：检查第 6 章“教程假上游”Provider、模型启用状态和公开教程桩地址；先恢复一个兼容成员，不要删除整个默认池。
- 实际模型与预期不同：顺着“请求记录 → appCaller → 模型池 → 成员健康 → Exchange”排查，不要只看 SDK 里的模型名。
- 没有回退明细：Quickstart dry-run 不走模型路由，官方教程桩也没有失败开关；保留“暂无样本”，等真实故障或专门的隔离验收环境提供证据。
- 回退到了能力不兼容模型：停止该池的生产使用，核对成员用途和参数能力，记录 requestId 后让管理员修正。
- 回退率突然升高：按 Provider 和模型分组观察，不要立刻批量付费复测。

## 本章小结

健康是当前证据，优先级是选择顺序，回退必须由真实记录证明。没有第二成员或失败样本时，本章只完成读图能力，不宣称回退已经实测。
