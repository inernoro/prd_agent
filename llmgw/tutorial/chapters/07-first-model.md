# 第 7 章 配置第一个模型

## 你在做什么

你将把“教程聊天模型”和“教程视觉模型”登记到“教程假上游”中，分别声明文字对话与图片理解用途，并明确价格已知或未知。

## 为什么要做

Provider 只是连接，模型才是可以实际选择的能力。模型用途决定它可以追加到哪种默认模型池；价格与币种决定费用能否估算。未知价格必须保持 unknown，不能为了让报表好看填成 0；CNY 和 USD 也不能在没有可审计汇率时相加。

## 开始前检查

- Provider 列表中“教程假上游”处于启用且密钥已配置。
- 平台公开测试桩的两个模型标识固定为 `stub-chat` 和 `stub-vision`；显示名称仍使用本书的人类名称。
- 如果没有可信价格表，准备选择“不填写价格”，而不是猜一个数。

## 跟我做

> 每做完一个编号步骤，就核对紧接在步骤下方的圈选图。同一步有两张图时，第一张确认入口或全貌，第二张确认字段或结果。

**图 029 点击“添加模型”进入模型配置**

**图 031 模型用途必须明确勾选，避免加入不兼容模型池**

## 看到什么算成功

模型列表出现两个教程模型，平台均为“教程假上游”。聊天模型带对话用途，视觉模型带图片理解用途。未填写价格时费用状态明确为未知或未覆盖，不显示 0。关联预览里的最终协议、上游模型标识和列表一致。保存提示说明只向匹配的默认池追加，没有匹配用途的池保持原样。

## 失败怎么办

- “添加模型”不可用：先确认 Provider 已存在且当前角色有路由写权限。
- 保存按钮不可用：聊天模型核对 `stub-chat` 与“对话”，视觉模型核对 `stub-vision` 与“图片理解”。
- 保存后进入错误的默认池：检查用途是否误勾；不要批量覆盖池，先更正模型用途再由管理员核对追加结果。
- 不知道价格：保持未知并记录“待供应方价格表”，绝不填 0 或在前端自行换算币种。

## 本章小结

模型把上游标识、用途、协议、Provider 和价格口径放在一起。现在 Gateway 有候选能力，但还需要模型池表达如何选择。
