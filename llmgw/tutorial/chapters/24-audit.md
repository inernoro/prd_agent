# 第 24 章 审计团队操作

## 你在做什么

这一章查找前面创建成员、轮换 key、修改模型池、保存 PromptPolicy、创建 Exchange 和导入账单留下的审计记录，并把每条变化与操作者、租户和目标对象对应起来。

## 为什么要做

配置发生变化时，只看当前结果无法知道变化过程。审计像咖啡店的交接簿：谁做的、何时做的、针对哪个对象、结果如何都要留下。它也必须遵守数据隔离，不能成为跨租户搜索入口。

## 开始前检查

- 当前账号是 Owner 或 Admin，并处于“教程咖啡店”。Developer、Viewer、Billing 看不到审计入口是正常权限结果。
- 准备一个 Exchange id、一个 appCaller 名称和一条 requestId 作为定位线索。
- 不把密钥明文、PromptPolicy 正文或用户密码当搜索词。

## 看到什么算成功

每个关键操作都能按目标或时间找到，记录包含当前 TenantId 对应边界、操作者、动作、目标和结果。敏感明文不出现，跨租户目标即使猜到 id 也不能读到。

## 失败怎么办

- 找不到审计入口：检查角色，只有具备治理权限的成员才能查看；不要给 Billing 临时加路由写权限来解决。
- 成功配置没有审计：停止继续修改，记下对象 id 和时间，交给管理员核对；关键写入必须有审计证据。
- 审计长期为 pending：先确认业务对象是否实际写入，再按同一操作标识排查收口失败，不能简单重放造成重复。
- 详情出现密钥或 PromptPolicy 正文：立即停止分享并报告安全问题。
- 搜索到其他租户记录：这是严重隔离问题，保存最小证据后停止操作，不要继续浏览。

## 本章小结

你能用审计回答“谁、何时、对什么、做了什么、结果怎样”，也知道审计不是泄露秘密或跨租户查询的后门。
