# 豆包 ASR 模型中继接入 · 指南

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：已落地

平台通过模型中继把豆包语音识别接入统一模型调度。业务 Agent 不直接保存豆包凭据，也不重复实现协议客户端。

## 1. 接入模式

| 模式 | 适用场景 | 执行事实源 |
|---|---|---|
| HTTP submit/query | 离线文件、长音频 | Exchange 转换器与 Gateway |
| WebSocket 流式 | 实时录音、低延迟字幕 | `DoubaoStreamAsrService` |

两种模式都应通过已登记的 ASR AppCaller 选择模型配置，并由服务端持续向前端推送进度或文本。

## 2. 前置条件

- 已在火山引擎获得合法的 API 凭据和资源权限。
- 已在平台模型中继中创建或导入豆包 ASR 配置。
- 调用 Agent 已登记 `ModelTypes.Asr` 的 AppCaller。
- 模型组已绑定可用的 ASR 配置。
- 音频格式、大小和访问地址满足上游要求。

凭据只进入受控秘密存储，不写入文档、日志、前端或请求示例。

## 3. 导入与配置

在模型中继管理页面选择豆包 ASR 模板，填写：

- 模式和上游地址。
- 对应的 API key 或 App ID 与 Access Token。
- 资源标识、超时和轮询参数。
- 支持的音频格式和大小限制。

保存后先执行配置测试，再绑定到业务 AppCaller。模板标识和字段以 Exchange 注册表为事实源。

## 4. HTTP 异步模式

执行链路：

1. Gateway 将标准 ASR 请求转换为豆包 submit 请求。
2. 服务端保存请求标识和上游日志标识。
3. Worker 按受控间隔查询状态。
4. 成功后把文本和分段转换为平台结果。
5. 超时、上游失败或取消进入明确终态。

业务 Agent 只提交音频引用和业务参数，不自行轮询豆包接口。

## 5. WebSocket 流式模式

执行链路：

1. 服务端建立带认证的 WebSocket。
2. 发送音频与识别配置。
3. 按协议切分音频并控制发送节奏。
4. 持续解析部分结果并通过 SSE 推给页面。
5. 最后一帧确认后保存完整文本与分段。

二进制帧、序列号、压缩和结束标识由 `DoubaoStreamAsrService` 维护，文档不复制协议实现。

## 6. 业务 Agent 接入

新增调用方时：

1. 在 `AppCallerRegistry` 登记稳定的 ASR 调用点。
2. 使用 `ILlmGateway` 或共享 ASR 服务。
3. 通过服务端权威 Run/Worker 管理长任务。
4. 通过 SSE 展示准备、上传、识别、整理和完成阶段。
5. 保存结果来源、模型和失败信息。

业务代码不得直接读取 Exchange 的加密凭据。

## 7. 验证

至少准备一个短音频和一个较长音频，验证：

- 配置测试成功且模型绑定正确。
- HTTP 模式可以从提交进入完成。
- WebSocket 模式持续返回部分文本。
- 最终文本、分段和时长合理。
- 取消、超时、无权限和无效音频有明确错误。
- 日志中没有凭据和完整敏感音频内容。

真实验收应走业务页面或公开调用路径，不以直接访问上游成功代替平台闭环。

## 8. 故障定位

| 现象 | 检查 |
|---|---|
| 认证失败 | 凭据类型、资源权限、Exchange 配置 |
| submit 成功但不完成 | 请求标识、轮询状态、超时 |
| WebSocket 立即断开 | 地址、header、帧格式和音频参数 |
| 页面没有增量 | 服务端 SSE、部分结果事件和代理超时 |
| 文本为空 | 音频格式、声道、采样率和上游结果 |

设计边界见 `doc/design.speech-agent.md`，产品行为见 `doc/spec.speech-agent.md`。
