# CDS FU-02 MapAuthStore（Mongo 后端） · 设计

> **版本**：v1.0 | **日期**：2026-07-17 | **状态**：已落地

## 管理摘要

- **解决的问题**：内存认证存储在 CDS 重启后丢失用户和会话，也无法在多个实例之间共享认证状态。
- **核心方案**：`MongoAuthStore` 与 `MemoryAuthStore` 实现同一 `AuthStore` 契约，通过环境变量选择后端。
- **当前状态**：Mongo 实现、启动选择、用户与会话持久化、团队工作空间数据及单元测试均已落地。
- **回滚方式**：切回默认的 `memory` 后端并重启；认证数据不会自动在两个后端之间迁移。

## 问题背景

`MemoryAuthStore` 适合本地开发和无外部依赖启动，但进程退出会清空登录态，也不能支持共享状态的多实例部署。认证数据与 CDS 业务状态是两套独立存储：前者管理用户、会话、工作空间成员与邀请，后者管理分支、部署和基础设施状态，不能混为一个后端开关。

## 设计目标

1. 保持 `AuthStore` 上层服务和路由契约不变。
2. Mongo 后端能够在重启后保留用户、会话和协作空间数据。
3. 本地开发默认仍可零依赖使用内存后端。
4. 配置错误在启动时明确失败，不以不可用状态继续运行。
5. 两种后端的关键行为由相同服务测试和各自存储测试覆盖。

## 核心决策

| 决策 | 说明 |
|------|------|
| 接口不变 | `AuthService` 只依赖 `AuthStore`，不感知具体存储 |
| 默认内存 | 未设置 `CDS_AUTH_BACKEND` 时使用 `memory`，方便开发和测试 |
| Mongo 显式启用 | 只有 `CDS_AUTH_BACKEND=mongo` 才连接 MongoDB |
| 启动即校验 | Mongo 模式缺少连接串时直接终止启动 |
| 不迁移内存状态 | 从 memory 切到 mongo 后，所有用户需要重新认证一次 |
| 独立数据库 | 认证库通过专用变量配置，可与 CDS state 后端独立演进 |

## 存储范围

Mongo Auth Store 当前承载的不只是最初设计中的用户和会话，还包括协作能力所需的数据。

| 数据 | 用途 |
|------|------|
| 用户 | GitHub 用户、本地用户、状态、组织信息与最后登录时间 |
| 会话 | 登录令牌、来源信息、创建与过期时间 |
| 活动 | 用户活动记录及容量约束 |
| 工作空间 | 团队工作空间及基本属性 |
| 成员 | 用户在工作空间中的角色与权限 |
| 邀请 | 待接受邀请及生命周期 |

具体集合命名与索引要求以 `mongo-handle.ts` 和部署配置为准，文档不复制可漂移的完整 Schema。

## 启动选择

| 配置 | 行为 |
|------|------|
| 未设置或 `memory` | 由服务端创建 `MemoryAuthStore` |
| `mongo` | 读取 Mongo 配置，连接数据库并注入 `MongoAuthStore` |
| 未知值 | 记录警告并回退到 `memory` |
| `mongo` 且缺少连接串 | 启动失败并明确指出缺少 `CDS_MONGO_URI` |

Mongo 数据库名优先读取 `CDS_AUTH_MONGO_DB`，其次读取 `CDS_MONGO_DB`，最后使用认证库默认值。敏感连接信息不得写入日志或文档样例。

## 关键行为

### 用户

- GitHub 用户按不可变的 GitHub ID 查找与更新，登录名和组织信息可随登录刷新。
- 本地用户名创建包含进程内串行保护，并依赖数据库唯一索引处理跨实例重复。
- 用户停用状态由认证服务统一判断，存储层只负责持久化。

### 会话

- 会话令牌随机生成并持久化，不把可逆凭据放入用户记录。
- 校验时同时确认会话存在、未过期且用户仍有效。
- 撤销、列出和清理行为在两种后端保持一致。

### 工作空间协作

- 工作空间、成员和邀请复用同一 `AuthStore` 契约。
- 权限判定留在服务层，存储实现不自行发明角色规则。
- Mongo 实现必须保持与 Memory 实现相同的重复检查和返回语义。

## 切换与回滚

从内存后端切换到 Mongo 时：

1. 停止 CDS，配置 Mongo 连接和认证数据库。
2. 设置 `CDS_AUTH_BACKEND=mongo` 后启动。
3. 用户重新完成认证；内存中的旧会话不会导入。
4. 验证用户、会话、工作空间和邀请读写。
5. 出现故障时切回 `memory` 并重启，接受再次登录和临时状态不持久的限制。

生产启用前必须确认唯一索引和会话过期策略已按部署要求配置。应用不会以“自动改数据库结构”为前提完成启动。

## 验证重点

| 场景 | 预期 |
|------|------|
| 首次与重复登录 | 首次创建，重复登录更新资料且不复制用户 |
| 会话校验 | 有效会话返回用户，过期或孤儿会话失效 |
| 本地用户并发创建 | 相同用户名只保留一个有效账户 |
| 用户状态变化 | 被停用用户不能继续通过认证 |
| 工作空间成员与邀请 | Mongo 与 Memory 行为一致 |
| 后端切换 | 配置选择明确，错误配置启动失败或按规则回退 |

自动测试位于 `cds/tests/infra/auth-store/mongo-store.test.ts`，通用认证服务测试使用 Memory 实现验证契约行为。
