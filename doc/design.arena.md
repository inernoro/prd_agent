# AI 竞技场 · 设计

> **版本**：v3.1 | **日期**：2026-07-17 | **状态**：已落地

## 管理摘要

- **问题**：模型选型常被品牌印象影响，用户难以用同一问题直接比较多个模型的真实输出。
- **决策**：竞技场按管理员配置的槽位并行发起匿名对战；用户先盲评，再揭晓模型身份和记录结果。
- **价值**：提供可复现的模型对比体验，并沉淀平台内的评测偏好与对战数据。

## 产品边界

| 角色 | 能力 |
|---|---|
| 管理员 | 管理对战分组、槽位和可用模型 |
| 普通用户 | 选择分组、提交问题、阅读匿名回答、投票或揭晓 |
| 系统 | 并行执行、隔离真实模型身份、保存对战与偏好记录 |

竞技场不是模型配置页：用户不能在对战过程中修改平台配置；管理员也不通过竞技场绕过模型池、权限或 LLM Gateway。

## 核心决策

1. **槽位由管理员显式配置**：不自动扫描模型，保证一次对战的可用性和成本边界可控。
2. **先匿名后揭晓**：回答在盲评阶段只使用稳定的匿名标签，真实模型信息只在揭晓后返回。
3. **对战独立记录**：分组、槽位、回答、投票和揭晓状态作为独立业务数据保存，便于审计和统计。
4. **统一调用链**：并发模型调用仍经过平台的模型解析、网关和日志体系，不在前端直接访问模型提供商。

## 执行流程

1. 用户选择可用分组并提交问题。
2. 服务端按分组槽位创建对战，校验权限和模型可用性。
3. Worker 并行执行槽位请求，持续推送匿名回答进度。
4. 用户在盲评阶段阅读、比较和投票。
5. 用户主动揭晓后，服务端返回槽位对应的真实模型及对战结果。

失败槽位必须明确显示失败原因，不得以其他模型回答填充或伪装成功。

## 数据与接口边界

| 对象 | 用途 |
|---|---|
| ArenaGroup | 可选对战分组和展示策略 |
| ArenaSlot | 分组内的模型槽位与排序 |
| ArenaBattle | 一次问题、匿名回答、揭晓和评测记录 |

用户侧入口负责分组、开始对战和揭晓；管理侧入口负责分组与槽位配置。具体字段、权限和流式事件以 Controller 与前端契约为准。

## 风险与验收

- 并发对战受模型池容量和配额限制，界面必须展示槽位级进度与失败。
- 匿名标签、槽位排序和真实模型映射必须由服务端保持一致。
- 揭晓前不得在回答、日志摘要或页面辅助信息中泄露模型身份。
