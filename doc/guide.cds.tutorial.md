# 从零开始使用 CDS · 指南

> **版本**：v2.0 | **日期**：2026-07-15 | **状态**：已落地

这份教程带你从一个 Git 仓库出发，完成 Agent 接入、项目创建、自动识别、部署、预览和版本恢复。默认流程不要求复制长期密钥、不修改电脑环境变量，也不要求先学会 Docker 或 Compose。

## 适用场景

- 第一次使用 CDS；
- 希望把普通网站、后台或前后端项目快速部署起来；
- 使用 Codex、Cursor 或 Claude Code 协助部署；
- 想从简单项目逐步理解数据库、缓存和多服务项目。

如果你的项目已经有稳定且复杂的 Compose 配置，可以直接阅读 [CDS 部署方式选择](guide.cds.deploy-three-paths.md)。

## 第一部分：三分钟完成接入

### 1. 在 CDS 选择 Agent

打开 CDS 项目列表，点击右上角“一键部署 > 接入 Agent”，选择 Codex、Cursor 或 Claude Code。

接着选择：

- 连接已有项目；
- 创建一个新项目。

页面会生成一条无密钥接入口令。把它交给当前 Agent 即可。

### 2. 在页面批准

Agent 发起申请后，CDS 右下角会出现授权请求。确认 Agent 名称和目标项目无误后批准。

批准完成后：

- 权限只属于当前项目；
- 凭据保存在项目内的 `.cds/credentials.json`；
- 凭据不会显示在对话和正常命令输出中；
- CDS 不修改 `.zshrc`、`.bashrc`、系统环境变量或全局 PATH。

新项目使用一次性创建权限。项目创建成功后，一次性权限自动失效，并切换为新项目专属权限。

### 3. 检查接入结果

接入完成后只需确认：

- Agent 可以识别当前 CDS 项目；
- `.cds/credentials.json` 没有出现在 `git status`；
- 切换到其他项目时，需要重新批准；
- 项目级技能文件不含凭据，是否提交由团队决定。

## 第二部分：部署第一个项目

### 1. 选择仓库

在“一键部署项目”中粘贴 Git 仓库地址，也可以从已经授权的 GitHub 仓库中选择。

私有仓库优先使用 GitHub App 单仓库授权。授权过一次后，项目扫描、构建和后续部署可以复用，不需要反复粘贴 GitHub Token。

### 2. 让 CDS 自动识别

点击“检测仓库并自动填好配置”。CDS 会尝试识别：

- 应用所在目录；
- 前端、后台或后台任务；
- 安装和构建方式；
- 启动命令和端口；
- 数据库、缓存等依赖；
- 判断服务就绪的方法。

先看识别结果，再决定是否修改。普通项目通常只需确认；特殊目录、多入口或自定义镜像项目才需要调整。

### 3. 选择交付模式

| 项目情况 | 建议 |
| --- | --- |
| 普通网站、后台、前后端应用 | 托管模式 |
| 单个后台任务或少量常驻任务 | 托管模式 |
| 已有成熟的复杂 Compose 配置 | Compose 模式 |
| 特殊网络、复杂启动顺序、自定义镜像 | Compose 模式 |

托管模式负责自动准备常见运行配置；Compose 模式保留完整控制。切换模式不会删除原有 Compose 配置。

### 4. 先做试运行

配置把握不高时，点击服务卡片上的“试运行验证”。CDS 会使用一次性环境验证镜像、启动命令和端口，结束后自动清理。

根据结果处理：

| 结果 | 含义 | 下一步 |
| --- | --- | --- |
| 配置可用 | 服务已启动并响应 | 直接创建和部署 |
| 需确认 | 进程仍在，但端口尚未响应 | 检查端口、启动耗时和健康路径 |
| 不通过 | 命令、目录、镜像或依赖存在问题 | 按页面建议修改后再次验证 |

### 5. 创建并部署

确认后点击创建。CDS 会依次完成准备、构建、启动和检查。页面会持续显示当前阶段，不需要靠刷新猜测进度。

短暂断网、刷新或离开页面不会丢失本次部署。重新打开分支详情时，会继续显示同一条部署记录。

### 6. 打开预览

部署成功后点击页面给出的预览地址，至少验证：

- 首页或服务入口可以访问；
- 登录和一个核心操作正常；
- 后台接口能够返回；
- 数据库或缓存等依赖确实可用；
- 当前预览对应正确项目和分支。

## 第三部分：按复杂度逐步练习

仓库内保留四个现有示例，可以按顺序学习，不需要一次掌握全部能力。

| 练习 | 示例目录 | 重点 |
| --- | --- | --- |
| 静态网页 | `cds/examples/tutorial-01-static-web/` | 构建、预览和版本记录 |
| 网页加后台 | `cds/examples/tutorial-02-web-and-backend/` | 前后端路由和健康检查 |
| 加 MongoDB | `cds/examples/tutorial-03-web-backend-mongo/` | 依赖注入和数据连接 |
| 多服务和多基础设施 | `cds/examples/tutorial-04-fullstack-infra/` | Compose 编排和服务隔离 |

每个练习都遵循同一顺序：接入项目、识别配置、试运行、部署、打开预览、验证核心操作。

## 第四部分：日常使用

### 推送新代码

项目已连接 GitHub 后，推送目标分支即可触发自动部署。打开 CDS 对应分支，可以看到触发来源、当前阶段和结果。

### 再次运行成功版本

代码没有变化，只想恢复服务或重复验证时，在分支详情的“不可变部署版本”中选择成功版本并重新部署。标记为可复用的版本不需要再次构建。

### 恢复以前的版本

新版本有问题时，从成功版本中选择目标版本并点击“回滚到此版本”。恢复操作会生成新的部署记录，仍然可以查看全过程。

### 部署失败

不要连续点击部署。先打开本次部署记录，依次确认：

1. 问题属于代码、配置、CDS、外部服务还是暂未归类；
2. 失败发生在准备、构建、启动还是检查阶段；
3. 页面建议优先修改什么；
4. 修复后再重新部署。

## 完成标准

| 检查项 | 通过标准 |
| --- | --- |
| Agent 接入 | 不复制长期密钥，不修改系统环境 |
| 项目识别 | 应用目录、命令、端口和依赖与仓库一致 |
| 部署过程 | 刷新后仍能看到同一条部署记录 |
| 用户预览 | 页面和核心操作正常 |
| 版本复用 | 成功版本可以判断是否能直接再次部署 |
| 故障恢复 | 可以选择以前的成功版本恢复 |
