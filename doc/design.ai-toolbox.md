# AI 百宝箱 · 设计

> **版本**：v1.2 | **日期**：2026-07-17 | **状态**：已落地

## 管理摘要

- **问题**：多个智能体和工具分散在不同入口，用户难以判断该使用哪项能力，也难以追踪一次复杂任务的执行状态。
- **决策**：百宝箱提供统一入口、工具目录、运行记录和流式交互；真实业务能力通过适配器或领域入口执行，不把内置智能体降级为单纯提示词。
- **价值**：用户可从一个入口发现、启动和复盘能力；平台可统一执行、权限、模型调用和结果沉淀。

## 产品边界

| 能力 | 百宝箱负责 | 领域系统负责 |
|---|---|---|
| 发现与启动 | 工具目录、快捷入口、输入与上下文收集 | 定义自身可执行能力 |
| 调度 | 意图识别、适配器选择、Run 创建和进度汇聚 | 执行领域业务规则 |
| 运行观察 | 流式文本、阶段、错误、历史和结果入口 | 写入业务事实与制品 |
| 自定义工具 | 创建、编辑、权限和知识库关联 | 提供可调用的稳定能力 |

百宝箱不是第二套领域实现：涉及视觉生成、缺陷处理、知识库写回等动作必须调用相应领域的真实入口。

## 核心决策

1. **统一 Run 生命周期**：长任务创建可追踪 Run，由 Worker 执行并以 SSE 持续反馈。
2. **适配器隔离**：每种内置能力通过适配器声明可处理的请求和结果，编排器不硬编码领域实现细节。
3. **服务端权威调度**：意图、权限、模型解析和工具选择由服务端完成；前端只表达用户意图和观察状态。
4. **工具与知识分离**：工具配置可关联知识库，但知识注入、权限和引用范围由后端控制。
5. **结果可回到业务上下文**：生成的文本、文件或任务链接应指向实际领域对象，不形成孤立副本。

## 运行流程

1. 用户从工具目录、快捷入口或对话输入发起任务。
2. 服务端校验权限与输入，识别可用工具或选择领域适配器。
3. 编排器创建 Run，必要时交给 Worker 执行，并持续发出阶段和结果事件。
4. 前端逐段显示文本、思考和进度；失败时显示可操作的错误结论。
5. 最终结果写入对应领域或作为受控制品保存，运行记录保留可追溯摘要。

## 数据与接口边界

| 对象 | 用途 |
|---|---|
| ToolboxItem | 自定义工具的配置、可见范围、提示词和可用能力 |
| ToolboxRun | 一次执行的状态、输入摘要、结果和时间线 |
| AgentAdapter | 内置能力的调用边界与结果适配 |
| Attachment / KnowledgeBase | 受控上下文与文档引用，不等同于任意全文外泄 |

公开接口、字段和流式事件以 `AiToolboxController` 与前端服务契约为准。模型调用必须经 LLM Gateway，不能在工具组件内直连提供商。

## 风险与验收

- 未登记能力不得被假定可调用；缺能力时界面应引导用户选择已有工具或创建配置。
- 流式与批量执行必须持续反馈，不能静态等待。
- 自定义工具、知识库关联和运行记录必须按用户与组织权限隔离。
