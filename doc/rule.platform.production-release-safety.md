# 生产发布表面健康与可追溯性 · 规则

> **版本**：v1.0 | **日期**：2026-07-12 | **状态**：已落地

## 核心原则

**一句话**：生产发布必须从用户最终入口证明页面、静态资源、API 和专项服务同时可用；容器存活或单一健康端点通过不等于发布成功。

本文件是 MAP 生产发布安全合同的唯一文档 SSOT。`AGENTS.md`、Agent 路径触发规则、发布脚本、CI 和监控只能引用本文件，不得复制出不同版本。

## 为什么需要这条规则

2026-07-12，生产主站出现 nginx 500/502。API `/health` 与 LLM Gateway 页面仍返回 200，但主站静态目录权限为 `700`，nginx worker 无权读取 `index.html`，最终触发 SPA fallback 内部重定向循环。现有发布验证偏重 API、Gateway 和容器健康，没有访问用户真正打开的主页面及其入口资源，因此故障未在发布门禁中被发现。

已经证实的直接原因是静态目录不可遍历；首次把目录设置为 `700` 的具体进程无法从现有证据中确定。系统级根因是发布脚本没有固定权限后置条件、没有原子切换、没有产品表面验证，也没有保存足够的权限变化证据。

同日还确认了发布命令兼容性退化：旧流程中 `./exec_dep.sh release` 等价于部署 latest，新参数解析会把 `release` 当作版本 ref，继而请求不存在的产物并输出多层错误。发布入口改变语义时没有兼容别名和迁移提示，也属于本规则治理范围。

## 规则条目

### 1. 发布前必须验证静态产物

发布脚本必须在非在线目录中完成下载、校验和解压，并验证：

- `index.html` 存在且非空。
- HTML 至少引用一个实际存在、非空的入口 JS；引用 CSS 时同样验证。
- 静态根目录和所有子目录可被 nginx worker 遍历，目录权限不得严于 `755`。
- 普通静态文件可读，权限不得严于 `644`。
- 产物 SHA-256 与发布目标 commit 可审计关联。

禁止在验证完成前清空或覆盖当前在线目录。

### 2. 静态站必须原子切换并保留回滚点

新产物只能按“临时目录解压 → 权限归一化 → 离线验证 → 切换 current 指针”的顺序上线。上一版必须保留为 previous；切换后任一强制探针失败，应自动恢复 previous，而不是让线上停留在半成品目录。

### 3. 发布后必须验证完整产品表面

发布后验证必须从公网域名执行，至少覆盖四层：

| 层级 | 强制断言 |
|---|---|
| 主页面 | `GET /` 返回 200、`Content-Type` 为 HTML，正文包含应用入口 |
| 静态资源 | 从实际 HTML 解析入口 JS/CSS，请求返回 200、MIME 合理且大小大于 0 |
| API | `GET /health` 返回 200，业务状态为 healthy |
| 专项服务 | LLM Gateway 等本次发布涉及的独立页面和健康端点返回预期结果 |

容器内 curl、Docker healthcheck、API 200 只能作为分层诊断证据，不能替代公网产品表面验收。

### 4. 旧发布命令必须兼容迁移

已被生产人员使用的命令语义不得静默改变。需要调整时必须：

1. 保留至少一个明确兼容周期。
2. 旧命令继续完成原行为，并输出一次可操作的弃用提示。
3. `--help` 同时展示旧命令、推荐命令和最终发布目标。
4. 参数解析回归测试覆盖旧命令。

`./exec_dep.sh release` 的兼容合同是部署 latest，不得把 `release` 静默解释为产物 tag。

### 5. 错误必须收敛为一个主要原因

发布失败时先输出失败阶段、主要原因和一条恢复动作，再提供展开详情。禁止把下载、校验、gate、容器和探针的派生错误同时弹给操作者，却不说明首个失败点。

### 6. 每次发布必须留下结构化证据

发布证据至少包含：

- 操作者、开始与结束时间、完整目标 commit、实际命令语义。
- 产物 URL、SHA-256、HTML 与入口资源 hash。
- 静态目录发布前后 owner、group、mode。
- current 与 previous 指向。
- 四层公网探针的状态码、耗时和响应摘要。
- 首个失败阶段、是否回滚、回滚结果。

证据应写入不可覆盖的按发布编号文件；生产静态目录还应保留权限或替换操作的系统审计记录，确保能回答“谁在何时通过什么进程改变了什么”。

### 7. 健康监控必须独立于发布动作

外部监控应周期性请求主页面、实际入口资源、API 与专项服务。告警必须区分“页面不可用但 API 正常”“API 不可用”“专项服务不可用”和“HTML/资源版本不匹配”，避免所有故障都被压成笼统 502。

### 8. 完成声明必须附带最终入口证据

涉及生产发布的任务只有在公网最终入口、静态资源和相关 API 都通过后才能声明完成。只提供 CI、镜像构建、容器 healthy 或内部健康端点证据时，状态必须写为“部署链路通过，用户表面未验收”。

## 强制回归矩阵

| 场景 | 必须结果 |
|---|---|
| `umask 077` 下创建发布目录 | 最终在线目录仍为 `755`，文件为 `644` |
| 缺少或清空 `index.html` | 切换前失败，旧页面保持 200 |
| HTML 引用的入口资源不存在 | 切换前失败，旧页面保持 200 |
| 新版切换后公网探针失败 | 自动恢复 previous，记录回滚证据 |
| 执行 `./exec_dep.sh release` | 兼容部署 latest，并输出迁移提示 |
| API 健康但主页面不可读 | 产品表面门禁失败并明确归类为静态站故障 |

## 判断标准速查

| 场景 | 是否适用 | 处理方式 |
|---|---|---|
| 只构建镜像、不触发部署 | 适用产物与命令兼容规则 | CI 验证，不能声称线上完成 |
| CDS 分支预览 | 适用页面、资源、API 表面检查 | 使用分支最终深链验证 |
| 生产后端维护发布且复用静态站 | 仍适用 | 先证明现有静态目录可读，再验证公网主页面 |
| 纯文档修改 | 不触发生产探针 | 仍须保持本规则和索引一致 |

## 例外情况

紧急恢复可以先执行范围最小、可逆的权限修复或 previous 回切，但必须立即验证公网表面，并在后续变更中补齐长期防回归。紧急情况不允许跳过证据记录，也不能把临时恢复当作永久修复。

## 关联债务

- `debt.platform.production-release.md`：记录尚未完成的原子静态切换、发布证据账本、外部监控和命令兼容整改。

## 关联技能与闭环

生产发布不能由单个技能包办。以下技能共同引用本规则，并各自只承担一段责任：

| 阶段 | 技能 | 责任 | 不能替代 |
|---|---|---|---|
| 风险识别 | `risk-matrix` (`/risk`) | 识别正确性、兼容性、运维部署和用户体验风险 | 不能证明代码或页面可用 |
| 对抗复审 | `human-verify` (`/verify`) | 逆向验证、边界测试和用户场景审查 | 不能替代真实部署 |
| 灰度部署 | `cds-deploy-pipeline` (`/cds-deploy`) | 部署分支、等待就绪、读取日志和分层 smoke | 容器 running 不能替代页面表面验收 |
| API 冒烟 | `smoke-test` (`/smoke`) | 验证 API 契约和核心 CRUD/业务链 | API 200 不能证明 HTML、JS/CSS 和渲染正常 |
| 预览入口 | `preview-url` (`/preview`) | 从 CDS SSOT 获取真实预览域名 | 只生成地址，不证明页面健康 |
| 真人验收 | `acceptance-checklist` (`/uat`) | 从最终入口检查页面、资源、交互、回归和回滚 | 不能替代 CI 和 API 契约测试 |
| 正式热修 | `production-hotfix-release` (`/hotfix-prod`) | 基于生产 revision 做最小发布并执行公网验证 | 不得绕过本规则和 open 债务 |
| 交接收口 | `task-handoff-checklist` (`/handoff`) | 汇总证据、风险、回滚和未偿债务 | 不能把缺失证据包装成完成 |

标准链路为：

```text
/risk → /verify → /cds-deploy → /smoke → /preview → /uat → /hotfix-prod → /handoff
```

若是线上故障，可从 `/hotfix-prod` 进入最小恢复，但恢复后仍必须补齐公网表面证据，并由 `/handoff` 把未偿机制写入债务台账。
