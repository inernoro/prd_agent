## 债务主题

把当前"单技能内联标准"的验收，演进为**「标准创建 Agent」+「验收执行 Agent」+「知识库枢纽」**的三方架构：一个 Agent 创建并维护验收标准、绑定到知识库；另一个 Agent 从知识库读取标准、执行验收、把结果关联回知识库。对应设计文档 `design.acceptance.system` §10.2/§10.3/§10.4（验收即 gate、跨 Agent 独立验收、全栈追溯）的具体落地。

## 目标架构（三角色 + 知识库枢纽）

```
标准创建 Agent ──写──▶ 知识库:standard 条目(structured + 可读, 有 id/version)
                              │
验收执行 Agent ──读 standard_id──┘
      │  按标准取证(harness 模拟人类)→ 生成报告
      └──写──▶ 知识库:report 条目(meta 记录 standard_id@version)
                              │
                        反查:某标准下所有结果 / 某功能历次验收
```

打通 = 三条数据绑定：
1. **标准 → 知识库**：标准本身是 KB 条目，有 `standard_id` + 版本，可治理、可演进。
2. **执行 → 标准**：执行时声明 `standard_id`，按该标准的"准入门槛"校验输入。
3. **结果 → 知识库 + 标准**：报告归档 KB，meta 里 `standard_id@version` 回指，实现"这份结果是按哪版标准验的"。

## 我的评估（值不值得 + 难在哪）

**价值**：标准从"硬编码在技能脚本里"升级为"知识库里可版本化、可治理的资产"；执行读活标准；结果可全栈追溯（标准 ↔ 结果）。多团队/多项目可各自定制标准而共用执行引擎。

**前置难点（必须先解决，否则就是花架子）**：
1. **标准必须机器可读**。现在准入门槛/档位/硬约束是 `archive_report.py` 里的硬编码 + `standard-v2.md` 的散文。要让 Agent "读标准来校验"，必须把它抽成**结构化 schema**（如 `standard.schema.json`：`tier_min_shots / junk_targets / required_sections / hard_constraints / verdict_rules`），执行 Agent 读 schema 而非读散文。
2. **版本绑定**。报告必须记录 `standard_id@version`；标准变更不得让历史报告失去可解释性。
3. **标准创建 Agent 需要"元标准"**。谁保证它产出的标准不太松（什么都能过）也不太严（什么都过不了）？否则只是把"输入不对"问题上移一层。
4. **防过度工程（最大风险）**。3 Agent + KB 绑定远重于 1 技能。**单项目单标准阶段，现在的 1 技能完全够用**。只有当"标准频繁演进 + 多方各自定制 + 结果要跨标准聚合分析"同时成立时，拆 Agent 才回本。

## 分阶段开发要求（建议的还债顺序，先地基后编排）

| 阶段 | 要做什么 | 价值 | 依赖 |
|------|---------|------|------|
| **0 数据层**（最高价值，先做） | 把准入门槛/档位/硬约束从脚本硬编码抽成 `standard.schema.json`；报告 meta 增加 `standard_id@version` 字段 | 标准可机读、结果可追溯——这是后续一切的地基 | 无 |
| **1 知识库绑定** | 标准作为 KB 条目存储（有 id/版本）；执行时按 id 拉取标准 schema | 标准进入"可治理资产"形态 | 阶段 0 |
| **2 角色拆分** | 标准创建 Agent（人 + `feature-emerge` 共同维护标准）+ 执行 Agent（读标准跑验收，结果回写） | 解耦：标准演进不动执行；跨 Agent 独立验收 | 阶段 1 |
| **3 聚合涌现** | 跨标准/跨结果的质量看板、回归基线、语义检索 | 资产库的复利兑现（design §9/§10） | 阶段 2 + 报告语料积累 |

## 边界：现在明确不做

- 本次不写任何代码，仅记录债务与要求。
- 单项目阶段不强拆 3 Agent（YAGNI）；若先动手，**只做阶段 0**（schema 化 + 版本绑定），不碰编排。
- 不引入新的存储模型，复用文档空间 `document_stores` 承载标准与结果。

## 触发条件（何时该还这笔债）

满足以下任一，再启动：
- 出现**第 2 个项目/团队**要用本验收体系，且标准需求不同 → 需要标准可定制（阶段 0-1）。
- 验收报告积累到**可做聚合分析**（几十份以上），想要质量看板/回归基线 → 阶段 3。
- 要把验收接进 CI Gate / 无人值守闭环（design §10.1/§10.2）→ 需要执行 Agent 独立可编排（阶段 2）。

在此之前，现状（单技能 + 内联标准 + 强制准入门槛）是匹配当前规模的正确选择，不提前还债。
