# CDS ORM 接入 · 指南

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：已落地

CDS 扫描项目时识别常见 ORM，并在服务启动命令前加入数据库等待和 migration。用户应先使用扫描结果，不手工复制通用启动模板。

## 1. 支持范围

当前识别能力以 cdscli 扫描器源码和测试为准，主要覆盖 Prisma、EF Core、TypeORM、Sequelize、Rails 和 Flyway。未识别框架按普通服务处理，不猜测 migration 命令。

## 2. 接入

```bash
python3 .claude/skills/cds/cli/cdscli.py scan /path/to/project
```

检查生成的 compose 内容：

- ORM 类型识别正确。
- 数据库服务和连接变量存在。
- 启动命令先等待数据库，再迁移，最后启动应用。
- 生产模式不自动执行破坏性 seed。
- 用户已有的 wait 或 migration 没被重复注入。

确认后再按 CDS 标准流程应用扫描结果。

## 3. 部署模式

| 模式 | 行为 |
|---|---|
| `prod` | 等待数据库并执行安全 migration，不注入示例数据 |
| `dev` | 在项目明确提供安全 seed 时允许执行 |

默认使用 `prod`。是否支持 seed 由项目事实决定，不因为处于预览分支就自动猜测。

## 4. 多分支数据库

- 每分支隔离的数据库应使用稳定、可追踪的命名。
- 共享数据库必须明确 schema 冲突和迁移并发风险。
- migration 失败时服务不应伪装为就绪。
- 回收分支前确认数据保留和清理策略。

## 5. 扩展新 ORM

维护者需要同时补充：

- 稳定的检测文件或依赖特征。
- migration 与可选 seed 命令。
- 数据库就绪条件。
- 幂等注入和已有命令识别。
- dev、prod 与多分支测试样例。

框架特征和命令维护在 cdscli 扫描器，不在文档复制 Python 配置表。

## 6. 禁止事项

- 只因目录名相似就判定 ORM。
- 把 `db push` 等破坏性命令默认用于生产。
- 在用户已有命令前重复注入 migration。
- 数据库未就绪时直接启动应用并依赖无限重启。
- 为通过健康检查跳过真实 schema 验证。

## 7. 验证

至少验证：

1. 扫描结果命中正确 ORM。
2. 新空数据库能自动建立 schema。
3. 第二次部署保持幂等。
4. 生产模式不写入测试 seed。
5. migration 失败能阻止错误版本进入就绪状态。

具体支持矩阵和命令以扫描器测试为事实源。
