# AI 技能工作流指南 — 从需求到上线的完整技能链

> **版本**：v1.0 | **日期**：2026-03-20 | **状态**：已落地

## 适用场景

- 新成员入项目，需要了解 AI 辅助开发的完整工作流
- 开发者想知道"这个阶段该用什么技能"
- 团队想建立标准化的质量保障流程

## 技能链总览

```
需求提出 ──→ /validate ──→ 方案设计 ──→ /risk ──→ /trace
                                                      │
                                                    编码实现
                                                      │
              /weekly ←── /handoff ←── /preview ←── /cds-deploy ←── /verify
                │
              /doc-sync
```

每个技能解决一个不可替代的问题，串联起来覆盖从需求到交付的完整生命周期。

---

## 阶段一：需求验证

### `/validate` — 需求气味检测

| 项目 | 说明 |
|------|------|
| **解决什么问题** | 需求模糊、重复造轮子、价值不明确 |
| **什么时候用** | 拿到新需求后、写代码之前 |
| **输出什么** | 七维度打分 + 气味检测报告 + 雷同功能排查 + go/no-go 建议 |

**使用方式**：

```
/validate 用户希望在缺陷报告中添加 AI 自动修复建议功能
```

**真实案例**：

- W10 上线此技能后，用于验证"配置市场"需求，检测出与现有提示词管理功能有 40% 重叠，调整了方案范围避免重复建设

---

## 阶段二：方案评审

### `/risk` — MECE 六维度风险评估

| 项目 | 说明 |
|------|------|
| **解决什么问题** | 方案盲点、遗漏的非功能需求 |
| **什么时候用** | 技术方案确定后、开发开始前 |
| **输出什么** | 正确性 / 兼容性 / 性能 / 安全 / 运维 / 体验六个维度的风险矩阵 |

**使用方式**：

```
/risk 我准备给 Session 模型新增 DocumentIds 字段支持补充文档
```

**真实案例**：

- Session.DocumentIds 需求通过 `/risk` 评估后，发现 DocumentsController 的权限校验未覆盖新引用路径，在开发前就修复了设计缺陷。这个案例后来成为 `data-audit.md` 规则的起源

### `/trace` — 全链路数据流追踪

| 项目 | 说明 |
|------|------|
| **解决什么问题** | 跨层数据断裂、字段名不一致、序列化陷阱 |
| **什么时候用** | 涉及前后端联动的功能设计完成后 |
| **输出什么** | 从 API 入口到数据库到前端展示的完整数据流图 |

**使用方式**：

```
/trace 缺陷分享功能的数据流：从创建分享链接到外部用户查看
```

**真实案例**：

- 缺陷分享功能开发时，`/trace` 追踪发现 `defect_share_links` 集合的 token 字段在 Controller 层用 `ShareToken`、Service 层用 `Token`、前端用 `shareToken` 三种命名，在编码前统一为 `shareToken`

## 阶段三：编码实现

这个阶段主要是人工编码（或 AI 辅助编码），以下技能在编码过程中按需使用：

### `/visibility` — LLM 交互可视化审计

| 项目 | 说明 |
|------|------|
| **解决什么问题** | LLM 调用时用户面对空白等待，违反项目强制规则 |
| **什么时候用** | 开发了涉及大模型调用的功能后 |
| **输出什么** | 合规报告：哪些 LLM 调用缺少流式输出 / 进度反馈 / 思考过程展示 |

**使用方式**：

```
/visibility 检查 DefectAgent 的 AI 分析功能
```

**真实案例**：

- W11 上线 SSE 基础组件库后，用 `/visibility` 审计全部 Agent 模块，发现视频 Agent 的生成流程缺少阶段提示，补充了"准备中 → 分析中 → 生成中 → 完成"四阶段推送

### `/doc` — 文档类型守护

| 项目 | 说明 |
|------|------|
| **解决什么问题** | doc/ 目录文档格式混乱、前缀不统一 |
| **什么时候用** | 需要在 doc/ 下创建或修改文档时 |
| **输出什么** | 自动套用 6 种标准模板（spec/design/plan/rule/guide/report） |

**使用方式**：

```
/doc                          # 查看类型速查表
直接创建 doc/design.xxx.md    # 自动触发模板校验
```

## 阶段四：验证

### `/verify` — 多角度模拟验证

| 项目 | 说明 |
|------|------|
| **解决什么问题** | 开发者视角盲区，编译器和 linter 无法发现的逻辑问题 |
| **什么时候用** | 功能开发完成、准备部署前 |
| **输出什么** | 魔鬼辩护 + 逆向验证 + 边界测试 + 用户场景模拟 |

**使用方式**：

```
/verify 检查本次 PR 的所有改动
```

### `/smoke` — 链式冒烟测试

| 项目 | 说明 |
|------|------|
| **解决什么问题** | API 端点回归、接口契约变更 |
| **什么时候用** | 后端 API 改动后、部署前后 |
| **输出什么** | 自动生成的链式 curl 命令（前一步输出 ID 传给后续请求） |

**使用方式**：

```
/smoke DefectController
```

**真实案例**：

- 移动端适配计划 (`plan.mobile-adaptation.md`) 中明确要求：Phase 1 完成后跑 `/smoke` 验证 API 层未因布局改动产生回归，发版后再跑一次线上冒烟

## 阶段五：部署

### `/cds-deploy` — CDS 灰度部署流水线

| 项目 | 说明 |
|------|------|
| **解决什么问题** | 手动部署繁琐、缺少自动化验证 |
| **什么时候用** | 代码开发完成、验证通过后 |
| **输出什么** | 自动完成 Push → Pull → Deploy → 就绪检查 → 冒烟测试 → 报告 |

**前置条件**：

```bash
export CDS_HOST=cds.miduo.org       # CDS 地址
export AI_ACCESS_KEY=your-key        # 推荐，零交互认证
```

**使用方式**：

```
/cds-deploy              # 标准触发
"帮我部署到灰度"           # 自然语言触发
"容器报错了"               # 触发诊断场景
"重启 API"                # 触发重启场景
"清理灰度"                # 触发清理场景
```

**真实案例**：

- W11 首次实战部署验证时发现 8 个陷阱并全部写入技能定义：
  1. CDS 仅支持 Cookie 认证（非标准 Bearer）
  2. deploy 后状态更新有 3 秒延迟
  3. `starting` 状态假死但容器实际已就绪
  4. Cloudflare 将 401 透传为 500
  5. container-exec 中嵌套 JSON 引号问题
  6. AI Access Key 的 Impersonate 限制
  7. 沙箱环境无 jq
  8. 沙箱环境无 xxd

### `/preview` — 预览验收地址

| 项目 | 说明 |
|------|------|
| **解决什么问题** | 部署后找不到预览地址 |
| **什么时候用** | 部署完成后、需要人工验收时 |
| **输出什么** | `https://{branch-slug}.miduo.org/` 格式的可访问 URL |

**使用方式**：

```
/preview
```

**真实案例**：

- 自动集成到 `/handoff` 交接清单中，PR 描述里直接附带预览地址，评审者一键打开验收

## 阶段六：交付

### `/resolve` — PR 前冲突预合并

| 项目 | 说明 |
|------|------|
| **解决什么问题** | 多分支并行开发时 PR 冲突 |
| **什么时候用** | 准备提 PR 前 |
| **输出什么** | 自动合并 main 分支 + 三级冲突分类 + AI 解决方案 |

**使用方式**：

```
/resolve
```

### `/handoff` — 任务交接清单

| 项目 | 说明 |
|------|------|
| **解决什么问题** | 交接信息不全、验收时反复追问 |
| **什么时候用** | 开发完成后（3+ 文件变更时自动触发） |
| **输出什么** | 8 维度结构化报告：导航 / 文档 / 规则 / 工作流 / 测试 / 风险 / 代码质量 / 后续事项 |

**使用方式**：

```
/handoff
```

**触发规则**：涉及 3 个以上文件变更、API 端点变更、或 UI 页面变更时，AI 必须主动生成（CLAUDE.md 强制规则）。

**真实案例**：

- PR #234 包含 35+ commits，`/handoff` 自动生成交接清单后，评审者零追问通过

### `/weekly` — 周报生成

| 项目 | 说明 |
|------|------|
| **解决什么问题** | 手动统计一周工作量，容易遗漏 |
| **什么时候用** | 每周五收尾时 |
| **输出什么** | 从 git 历史自动生成：概览 → 完成事项 → 关键数据 → 下周方向 |

**使用方式**：

```
/weekly
```

**真实案例**：

已生成 5 份周报，数据持续积累：

| 周报 | 提交数 | 文件变更 | 代码变化 | PR 合并 |
|------|--------|---------|---------|---------|
| W06 | 早期 | - | - | - |
| W07 | 增长期 | - | - | - |
| W09 | 成熟期 | - | - | - |
| W10 | 312 | 406 | 大量 | 多个 |
| W11 | 208 | 234 | +28,262 / -27,114 | 31 |

完成后自动触发 `/doc-sync` 同步文档索引。

## 常见问题

### Q: 技能必须按顺序全部跑一遍吗？

不需要。小修改（1-2 个文件）直接 `/verify` → 提 PR 即可。完整链路适用于中大型功能开发。

### Q: 技能可以组合使用吗？

可以。常见组合：
- **方案评审三件套**：`/validate` → `/risk` → `/trace`
- **部署验收三件套**：`/cds-deploy` → `/preview` → `/handoff`
- **周五收尾二件套**：`/weekly`（自动触发 `/doc-sync`）

### Q: /cds-deploy 需要什么环境变量？

```bash
CDS_HOST=cds.miduo.org       # 必填，CDS 地址
AI_ACCESS_KEY=your-key        # 推荐，零交互认证
```

### Q: /handoff 什么时候自动触发？

涉及 3+ 文件变更、API 端点变更、或 UI 页面变更时，AI 必须主动生成（CLAUDE.md 强制规则）。1-2 个文件的小修改无需生成。
