# 系统涌现：从基础组件到协同智能

> **版本**：v1.0 | **日期**：2026-03-28 | **状态**：已实现
>
> 取用之于基础，串联其协同，落地之实际，编之于幻想。

## 一、管理摘要

- **这篇文档是什么**：不是某个功能的设计文档，而是整个 PrdAgent 系统的**灵魂说明书**——展示基础组件如何组合出远超预期的能力
- **核心观点**：6 个独立 Agent + 1 个工作流引擎 + 1 个事件总线 = 无限可能的协同网络。单个 Agent 是工具，组合起来是智能体生态
- **业务价值**：让客户理解"为什么这不只是 6 个 AI 工具的简单拼凑"——系统的价值在于**涌现**，即组合后产生的能力远大于部分之和
- **读者**：老板向客户讲解系统愿景时的底稿；团队理解"我们在建造什么"的北极星

## 二、什么是涌现

蚂蚁不懂建筑学，但蚁群建造了精密的巢穴。每只蚂蚁只遵循简单规则，涌现出复杂的集体智慧。

PrdAgent 也是如此——

| 层级 | 看到什么 | 涌现了什么 |
|------|----------|-----------|
| **单个 Agent** | 一个能聊天、能生图、能写周报的 AI 工具 | 独立完成特定领域任务 |
| **Agent + 工作流** | Agent 可以调用工作流扩展自己的能力边界 | 从"AI 工具"变成"可编程的智能助手" |
| **工作流 + 外部系统** | 工作流连接 TAPD、语雀、Git、邮件、Webhook | 从"内部系统"变成"企业神经中枢" |
| **事件驱动 + 自动化** | Agent 产生事件 → 触发其他 Agent 的工作流 | 从"被动响应"变成"主动协同" |

**一句话**：我们不是在做 6 个 AI 产品，我们在做一个**会自我组合的智能体生态**。

## 三、四层架构：从基础到涌现

```

│  → Agent B 的结果 → 回调 → 更新 Agent A 的状态           │
│  能力不是设计出来的，是组合出来的                          │
└────────────────────────┬────────────────────────────────┘
                         │ 事件驱动
┌────────────────────────▼────────────────────────────────┐
│              第三层：编排层（Orchestration）               │

│  一个工作流可以调用任意 Agent、任意外部 API               │
└────────────────────────┬────────────────────────────────┘
                         │ 服务调用
┌────────────────────────▼────────────────────────────────┐
│              第二层：能力层（Capabilities）                │
│  6 个 Agent + 基础服务                                    │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐            │
│  │Visual  │ │Literary│ │Defect  │ │Report  │            │

│  ┌────────┐ ┌────────┐ ┌────────────────────┐           │
│  │Video   │ │PRD     │ │ Open Platform      │           │
```

## 四、基础组件：每一块都很简单

### 4.1 六个 Agent——各司其职

| Agent | 做什么 | 一句话 |
|-------|--------|--------|
| **Visual Agent** | AI 生图、编辑、组合 | 你的 AI 设计师 |
| **Literary Agent** | 文学创作、文章配图 | 你的 AI 作家 |
| **Defect Agent** | 缺陷提报、分析、修复追踪 | 你的 AI 测试员 |
| **Report Agent** | 日报打点、周报生成、AI 汇总 | 你的 AI 助理 |
| **Video Agent** | 视频脚本、分镜、代码生成 | 你的 AI 导演 |
| **PRD Agent** | 需求文档解读、评审 | 你的 AI 产品经理 |

如果故事到这里结束，我们只是做了 6 个独立的 AI 工具。但故事不会在这里结束。

### 4.2 工作流引擎——万能胶水

工作流引擎是整个系统的**编排中枢**，它不做具体业务，但它能把任何东西串起来。

**25+ 种舱类型，4 大分类**：

| 分类 | 舱类型 | 能力 |
|------|--------|------|
| **触发器** | 定时器、Webhook 接收、手动触发、文件上传、事件触发 | 定义"什么时候开始" |
| **处理器** | HTTP 请求、智能 HTTP、LLM 分析器、脚本执行器、TAPD 采集器、数据提取/合并/转换/聚合 | 定义"怎么处理" |
| **控制流** | 延时、条件分支 | 定义"按什么逻辑走" |
| **输出器** | 报告生成、网页生成、文件导出、Webhook 发送、通知推送、视频生成、站点发布、邮件发送 | 定义"结果去哪" |

**关键设计**：工作流的每个节点的输出（Artifact）自动流入下游节点的输入，形成数据管道。

### 4.3 事件总线——隐形纽带

`AutomationHub` 是系统的事件神经网络：

1. 任何 Agent 都可以发布事件（如 `defect-agent.report.created`）
2. 事件匹配自动化规则（精确匹配 + 通配符）
3. 匹配后触发动作：发 Webhook、发通知、**触发工作流**
4. 被触发的工作流又可以调用其他 Agent

这形成了一个**闭环**：Agent → 事件 → 工作流 → Agent → 事件 → …

## 五、组合涌现：1+1 远大于 2

### 场景 1：周报自动采集——Agent 调用工作流

**基础能力**：Report Agent 管理团队周报，工作流引擎有 TAPD 采集舱和 Git 采集能力。

**涌现**：管理者配置一个数据采集工作流 → Report Agent 每周自动触发 → 工作流从 TAPD 拉取本周任务完成情况 + 从 Git 拉取代码提交 + 从语雀拉取文档更新 → 数据自动注入成员周报素材 → AI 汇总生成团队摘要。

```
管理者只做了一件事：配置工作流
系统自动完成了：数据采集 → 归属映射 → 素材注入 → AI 汇总 → 团队报告
```

**没有工作流之前**：成员手动去 TAPD 截图、手动统计 Git 提交、手动写进周报。
**有了工作流之后**：零人工，数据自动流转，AI 自动汇总。

### 场景 2：缺陷驱动修复——事件触发跨 Agent 协同

**基础能力**：Defect Agent 管理缺陷，Visual Agent 能分析截图，工作流能发邮件。

**涌现**：
1. 测试员在 Defect Agent 提交缺陷（附截图）
2. 系统发布事件 `defect-agent.report.created`
3. AutomationHub 匹配规则 → 触发工作流
4. 工作流第一步：调用 Visual Agent 的 Vision 能力分析截图 → 自动识别 UI 异常
5. 工作流第二步：调用 LLM 分析器 → 结合缺陷描述和截图分析，生成修复建议
6. 工作流第三步：邮件发送舱 → 将修复建议发给对应开发者
7. 工作流第四步：Webhook 发送舱 → 更新 TAPD 工单状态

```
测试员只做了一件事：提交缺陷 + 截图
系统自动完成了：截图分析 → 修复建议 → 通知开发 → 更新工单
```

### 场景 3：内容创作流水线——Agent 串联 Agent

**基础能力**：Literary Agent 写文章，Visual Agent 生配图，Video Agent 做视频。

**涌现**：
1. 用户给 Literary Agent 一个主题："写一篇关于春天的散文"
2. Literary Agent 生成文章，自动标注配图位置
3. 工作流触发 → Visual Agent 为每个标注位置生成配图
4. 配图完成后 → 工作流触发 Video Agent → 将文章 + 配图转为短视频
5. 视频生成后 → 网页生成舱 → 自动生成展示页面 → 站点发布舱 → 一键上线

```
用户只说了一句话："写一篇关于春天的散文"
系统产出了：文章 + 配图 + 短视频 + 展示网页
```

### 场景 4：抖音内容工厂——工作流调用外部 + 内部

**基础能力**：工作流有抖音解析舱、视频下载舱、语音转文字舱、文案改写舱。

**涌现**：
1. 用户通过苹果快捷指令分享一个抖音链接
2. Channel Adapter 接收 → 触发工作流
3. 抖音解析舱 → 提取视频信息
4. 视频下载舱 → 下载视频
5. 语音转文字舱 → 提取口播文案
6. LLM 分析器舱 → 改写为原创文案
7. 报告生成舱 → 汇总成内容分析报告
8. 邮件发送舱 → 推送到用户邮箱

```
用户只做了一件事：在抖音点了"分享"
系统产出了：原始文案 + 改写文案 + 分析报告 + 邮件推送
```

### 场景 5：总裁日报——多源数据自动汇聚

**基础能力**：Report Agent 有团队汇总，Defect Agent 有缺陷统计，Executive Dashboard 有数据聚合。

**涌现**：
1. 每日 18:00，定时触发工作流
2. 工作流从 Report Agent 拉取各团队当日工作进展
3. 从 Defect Agent 拉取当日新增/关闭缺陷数
4. 从 Git 数据源拉取当日代码提交量
5. LLM 分析器汇总 → 生成"今日公司技术动态"
6. 网页生成舱 → 生成精美的日报页面
7. 站点发布舱 → 发布到内部站点
8. 通知推送舱 → 推送给总裁

```
没有人写日报，但总裁每天 18:00 准时收到
```

## 六、涌现的技术基础：为什么能这样组合

### 6.1 统一的 Run/Worker 模式

所有 Agent 和工作流共享同一套异步执行架构：

| 组件 | 职责 |
|------|------|
| `IRunQueue` | Redis 队列，任务持久化，进程重启不丢失 |
| `*RunWorker` | 后台消费者，每种任务一个 Worker |
| `IRunEventStore` | SSE 事件流，支持 `afterSeq` 断线续传 |

这意味着**任何长任务都不会因为用户关闭浏览器而中断**——服务器权威性设计保证了任务必达。

### 6.2 AppCallerRegistry——能力注册表

每个 Agent 在 `AppCallerRegistry`（1085 行）中声明自己的能力，工作流和工具箱通过注册表发现和调用 Agent。

这是一个**能力市场**：新 Agent 只要注册进来，立刻可以被工作流和其他 Agent 调用。

### 6.3 IAgentAdapter——适配器模式

4 个 Agent 适配器（PrdAgent、VisualAgent、LiteraryAgent、DefectAgent）实现统一接口，工具箱编排器（`IToolboxOrchestrator`）通过意图分类自动路由到对应 Agent。

用户不需要知道"该找哪个 Agent"——说出需求，系统自动分配。

### 6.4 DAG 编排——任意复杂的组合

工作流引擎基于 DAG（有向无环图）执行，支持：
- 同层节点并行执行（`Task.WhenAll`）
- 条件分支（`condition` 舱）
- 延时等待（`delay` 舱）
- 节点间数据自动传递（Artifact 管道）

这意味着**任何你能画出流程图的业务，都可以用工作流实现**。

## 七、从现实到幻想：涌现的边界在哪

### 已经落地的涌现（现实）

| 场景 | 涉及模块 | 状态 |
|------|----------|------|
| 周报数据自动采集 | Report Agent + Workflow + Git 数据源 | ✅ 已实现 |
| 抖音内容解析链 | Shortcuts + Workflow + LLM | ✅ 已实现 |
| 缺陷截图 AI 预分析 | Defect Agent + Vision 模型 | ✅ 已实现 |
| 文章自动配图 | Literary Agent + ImageGen | ✅ 已实现 |
| 视频场景代码生成 | Video Agent + LLM | ✅ 已实现 |
| 事件驱动自动化 | AutomationHub + Workflow | ✅ 已实现 |

### 触手可及的涌现（近期）

| 场景 | 需要什么 | 复杂度 |
|------|----------|--------|
| 缺陷提报 → 自动通知开发 → 跟踪修复 | Defect Agent + Workflow + 邮件舱 | 低 — 组合现有舱类型 |
| 语雀文档变更 → 自动更新 PRD → 通知团队 | Webhook 触发 + HTTP 舱 + LLM 分析 + 通知舱 | 中 — 需配置语雀 Webhook |
| 竞品监控 → 定时爬取 → AI 分析 → 报告推送 | 定时触发 + HTTP 舱 + LLM 分析 + 报告生成 | 中 — 需编写采集逻辑 |
| 多 Agent 协同创作（文 + 图 + 视频一条龙） | Literary + Visual + Video + Workflow 编排 | 中 — Agent 间数据格式对齐 |

### 需要想象力的涌现（幻想）

| 场景 | 为什么是幻想 | 距离现实有多远 |
|------|-------------|---------------|
| **自编排工作流**：用户用自然语言描述需求，AI 自动设计工作流 DAG | 需要 LLM 理解舱类型语义并生成合法 DAG | 技术上可行，需要 prompt 工程 + 验证层 |
| **Agent 自我进化**：Agent 分析自己的失败日志，自动调整 prompt | 需要闭环反馈机制 + prompt 版本管理 | 基础设施已有（LLM 日志 + prompt 管理），缺闭环逻辑 |
| **跨企业协同**：通过 Open Platform，A 公司的 Defect Agent 发现 bug → 自动提交到 B 公司的修复工作流 | 需要跨租户的事件路由 + 权限隔离 | Open Platform 架构已支持，缺跨租户安全模型 |
| **知识自增长**：每次 Agent 交互的优秀结果自动沉淀为知识库 → 下次类似问题自动引用 | 需要 RAG/Embedding（知识库的未实现部分） | 知识库框架已有，RAG 未实现 |

## 八、一句话总结

> **PrdAgent 不是 6 个 AI 工具的合集。它是一个由基础组件自由组合、通过工作流编排、由事件驱动协同的智能体生态。每新增一个 Agent 或一种舱类型，系统的能力不是线性增长，而是指数级涌现。这就是为什么我们不只是在做产品——我们在做一个会"长大"的系统。**
