# Claude Code 云开发教程

> **版本**：v1.0 | **更新日期**：2026-02-25 | **适用范围**：全体开发人员与管理者
>
> **定位**：科普 + 上手教程 + 管理规范。阅读本文后，任何人都应能独立完成"对话 → 提交 → PR → 合并"的完整闭环。

## 一、什么是云开发

**一句话定义**：在 Anthropic 托管的云端隔离虚拟机中，通过自然语言对话驱动 AI 完成编码任务。

开发者不需要在本地打开 IDE、不需要手动写代码。你只需要：

1. 打开 [claude.ai/code](https://claude.ai/code)
2. 用自然语言描述你要做什么
3. Claude 在云端 VM 中自动：克隆仓库 → 分析代码 → 修改 → 测试 → 推送分支
4. 你审查 Diff → 创建 PR → Code Review → 合并

**与本地开发的核心区别**：

| 维度 | 本地开发 | 云开发 |
|------|---------|--------|
| 代码在哪里改 | 开发者本机 | Anthropic 云端 VM |
| 编码方式 | 手动写代码 | 对话驱动 AI 编码 |
| 过程可见性 | 黑盒（只有最终提交） | 白盒（对话全程可观测） |
| 环境依赖 | 需要本地配置 | 云端预装通用工具链 |
| 并行能力 | 受限于个人精力 | 可同时启动多个独立会话 |

### 适用场景

- Bug 修复和例行维护
- 有明确需求的功能开发
- 多任务并行推进
- 代码重构和优化
- 不在本地的远程仓库开发

## 二、为什么推行云开发

### 2.1 过程可观测：从黑盒到白盒

**以前**（传统开发）：

```
开发者在本地编码（不可见）→ ... → 若干天后提交代码（仅看结果）
                ↑ 黑盒区域：不知道做了什么、花了多久
```

**现在**（云开发）：

```
开发者发起对话（可观测）→ Claude 执行（对话日志可查）→ 推送分支（实时可见）→ 创建 PR（Diff 可审）
   ↑ 每一步都有记录，全程白盒
```

对话内容本身就是"开发日志"。管理者可以查看：
- 开发者给了什么指令
- AI 做了哪些分析和决策
- 修改了哪些文件、为什么这样改
- 总共经过了几轮迭代

### 2.2 数据隔离与应用隔离

云开发提供企业级的安全隔离：

| 隔离维度 | 说明 |
|----------|------|
| **VM 隔离** | 每个会话运行在独立的虚拟机中，由 Anthropic 管理 |
| **网络隔离** | 默认限制网络访问（仅允许白名单域名），可配置为无网络 |
| **凭证隔离** | Git 凭证通过安全代理管理，不存在于沙盒内部 |
| **分支隔离** | 只能推送到当前工作分支，无法直接推送到主分支 |
| **会话隔离** | 每个会话相互独立，不共享状态 |

**这意味着**：
- 代码修改只能通过 PR 合入主分支，不可能绕过 Code Review
- 即使在云端，Claude 也无法访问非授权的仓库或服务
- 敏感凭证受到保护，不会泄露到沙盒中

### 2.3 堵死"假用 AI"的路

管理层最关心的问题：**怎么证明开发者真的在用 AI 开发，而不是说用 AI 实际还是人力？**

云开发从三个维度彻底解决这个问题：

**第一道防线 — 分支标记**

所有云开发产生的分支都有 `claude/` 前缀，一目了然：

```
claude/fix-login-bug-ABCDE          ← 云开发产出
claude/add-user-validation-FGHIJ    ← 云开发产出
feature/manual-refactor             ← 人工开发
```

在 GitHub 仓库中，一眼就能区分哪些是 AI 开发、哪些是人工开发。

**第二道防线 — PR 关联会话**

每个 `claude/` 分支都对应一个可追溯的对话会话。PR 页面可以关联到原始对话，任何人都能查看 AI 的思考过程和决策依据。

**第三道防线 — 自动化周报统计**

项目已配置周报自动生成技能，精确统计每日数据：

> 以下是真实的周报数据（摘自 `doc/weekly-report-2026-W07.md`）：
>
> **总计 239 次提交 | 349 个文件变更 | +43,370 行 / -6,539 行 | 33 个 PR 合并**

周报精确到：
- 每日提交次数和代码行数
- 每个 PR 的标题和分类
- 贡献者分布（Claude vs 人工）
- 功能模块覆盖统计

**以前**：忙几天可能没有任何可量化的产出，只能等结果出来才知道。

**现在**：只要你每天发起一次对话，统计数据就会自动产生。过程可看，产出可量化，精确到每一天。

## 三、快速上手（6 步走）

### 前提条件

| 条件 | 说明 |
|------|------|
| Claude 账号 | Pro / Max / Team / Enterprise（需有 Claude Code 权限） |
| GitHub 账号 | 需要有目标仓库的写入权限 |
| 浏览器 | 任何现代浏览器（Chrome / Edge / Safari / Firefox） |

### 步骤 1：访问 claude.ai/code

打开浏览器，访问 [claude.ai/code](https://claude.ai/code)，使用你的 Claude 账号登录。

### 步骤 2：连接 GitHub 账号

首次使用时，系统会引导你连接 GitHub 账号。点击 "Connect GitHub" 按钮，完成 OAuth 授权。

> **注意**：请确保你连接的是有仓库写入权限的 GitHub 账号。

### 步骤 3：安装 Claude GitHub App

在你需要开发的仓库上安装 Claude GitHub App：

1. 点击 "Install GitHub App"
2. 选择你的 GitHub 组织或个人账号
3. 选择要授权的仓库（建议精确选择，不要 "All repositories"）
4. 确认安装

### 步骤 4：配置环境

选择或创建一个开发环境：

- **环境名称**：如 "prd-agent-dev"
- **网络策略**：建议使用 "Limited"（默认，仅允许白名单域名）
- **环境变量**：如需要，添加键值对（`.env` 格式）

> **提示**：如果项目有特殊依赖，可通过 SessionStart Hook 自动安装（参见第九章注意事项）。

### 步骤 5：提交任务

在输入框中用自然语言描述你的任务。

**好的描述示例**：

```
修复 ChatRunWorker 中的会话超时问题。
当前行为：用户 30 分钟无操作后，Worker 直接终止，未保存中间结果。
期望行为：超时前自动保存当前状态到数据库，并通知用户会话已暂停。
验收条件：1) 超时后数据库有保存记录 2) 用户收到通知 3) 单元测试通过
```

**不好的描述示例**：

```
修一下超时的bug
```

> **建议**：描述中包含：**目标** + **当前行为** + **期望行为** + **验收条件**。

### 步骤 6：审查 Diff → 迭代 → 创建 PR

Claude 完成任务后：

1. 点击 Diff 指示器（如 `+12 -1`）查看文件变更
2. 逐文件审查修改内容
3. 如果有问题，在 Diff 上留评论或继续对话让 Claude 修改
4. 满意后，点击 "Create PR" 创建 Pull Request

恭喜！你已完成第一次云开发闭环。

## 四、对话与开发流程

### 4.1 如何有效描述任务

| 维度 | 好的做法 | 不好的做法 |
|------|---------|-----------|
| 目标 | "在 UserService 中添加邮箱验证逻辑" | "加个验证" |
| 约束 | "使用正则表达式，不引入新依赖" | （不说约束） |
| 范围 | "只修改 UserService.cs 和对应的测试文件" | "改一下后端" |
| 验收 | "单元测试覆盖正常邮箱和异常格式" | （不说怎么验收） |

**进阶技巧**：项目根目录下的 `CLAUDE.md` 文件定义了架构规则（如 AppKey 命名规范、LLM Gateway 调用规则等）。云端会话中的 Claude 会自动遵循这些规则，确保代码风格一致。

### 4.2 Claude 的开发过程

当你提交任务后，Claude 会在云端 VM 中依次执行：

```
1. 克隆仓库         → 拉取最新代码到云端 VM
2. 准备环境         → 安装依赖、运行 SessionStart Hook
3. 分析代码         → 阅读相关文件，理解上下文
4. 制定方案         → 确定修改策略（复杂任务会先进入 Plan Mode）
5. 编写代码         → 创建/修改文件
6. 运行测试         → 执行测试套件，验证修改
7. 自查与修复       → 如果测试失败，自动修复并重新验证
8. 推送分支         → 将代码推送到 claude/xxx 分支
```

在整个过程中，你可以：
- **实时监控**：在 Web 界面看到 Claude 的每一步操作
- **中途引导**：随时发消息调整方向（如 "先别改这个文件，用另一种方案"）
- **断线无忧**：关闭浏览器后会话仍在运行，重新打开即可继续

### 4.3 移动端监控

Claude Code 支持 iOS 和 Android 移动端：

- [iOS App](https://apps.apple.com/us/app/claude-by-anthropic/id6473753684)
- [Android App](https://play.google.com/store/apps/details?id=com.anthropic.claude)

随时在手机上查看任务进度、回复 Claude 的提问、审查结果。

### 4.4 并行开发

你可以同时启动多个云开发会话，每个会话独立运行在不同的 VM 中：

```
会话 A：修复登录 bug          → claude/fix-login-AAAAA
会话 B：添加数据导出功能       → claude/add-export-BBBBB
会话 C：重构日志模块          → claude/refactor-log-CCCCC
```

三个任务同时执行，互不干扰，各自产出独立的 PR。

> **注意**：并行任务会按比例消耗更多速率限制额度。

## 五、版本提交与 PR 流程

### 5.1 分支命名规范

云开发自动生成的分支遵循以下格式：

```
claude/{任务描述}-{会话ID}
```

示例：
```
claude/fix-statistics-panel-values-CEZBO
claude/add-user-validation-1Z28M
claude/refactor-logger-output-FGHIJ
```

- `claude/` 前缀：标识这是 AI 云开发产出
- 任务描述：从你的任务描述自动提取
- 会话 ID：唯一标识，可追溯到原始对话

### 5.3 Diff 审查与迭代

在 Web 界面中，Claude 完成任务后会显示文件变更指示器。你可以：

1. **查看 Diff**：点击变更指示器（如 `+24 -3`），打开 Diff 查看器
2. **逐文件审查**：左侧文件列表，右侧代码变更
3. **留评论**：在 Diff 上留评论，Claude 会根据评论继续修改
4. **多轮迭代**：反复审查 → 评论 → 修改，直到满意
5. **创建 PR**：最终点击 "Create PR"，代码进入正式 Review 流程

这种方式让你无需切换到 GitHub 就能快速迭代。

### 5.4 本地与云端互通

| 操作 | 方式 | 说明 |
|------|------|------|
| 本地 → 云端 | 在 Claude Code 终端输入 `& 你的任务` | 创建新的云端会话 |
| 本地 → 云端 | `claude --remote "你的任务"` | 命令行直接发起 |
| 云端 → 本地 | 在终端运行 `/teleport` 或 `/tp` | 交互式选择云端会话 |
| 云端 → 本地 | `claude --teleport <session-id>` | 指定会话 ID 拉取 |
| 查看进度 | `/tasks` | 查看所有后台任务 |

> **进阶用法**：先在本地用 Plan Mode 讨论方案（`claude --permission-mode plan`），确定方案后用 `& 执行我们讨论的方案` 发送到云端自主执行。

### 6.1 周报自动生成

项目已配置 `weekly-update-summary` 技能，基于 Git 历史自动生成结构化中文周报，内容包括：

| 统计维度 | 说明 |

| 代码行变更 | 新增行数 / 删除行数 |
| PR 列表 | 所有合并的 PR 及分类标签 |
| 贡献者统计 | 各贡献者（包括 Claude）的提交占比 |
| 每日分布 | 哪一天有多少提交（精确到天） |
| 功能分类 | 按模块分类本周完成的工作 |

**使用方式**：在 Claude Code 中输入 `生成周报` 或 `weekly report` 即可。

### 6.2 传统开发 vs 云开发对比

| 维度 | 传统开发 | 云开发 |
|------|---------|--------|
| **过程可见性** | 黑盒 — 只看到最终提交 | 白盒 — 对话全程可观测 |
| **产出追溯** | 依赖开发者自觉汇报 | 自动记录每次提交和 PR |
| **日报/周报** | 手写，信息可能不准确 | 自动生成，数据精确 |
| **代码来源** | 无法区分人工/AI 产出 | `claude/` 分支明确标记 |
| **工作证明** | 主观描述 | 客观数据：提交数、行数、PR 数 |
| **空窗期** | 忙几天没有可见产出 | 每天对话即产出，连续可追踪 |
| **任务粒度** | 按需求/Sprint 粗粒度 | 精确到每个对话会话 |
| **复查成本** | 需要逐行 Review 猜意图 | PR + 对话记录，意图明确 |

## 七、期望值管理

### 7.1 云开发擅长的场景

| 场景 | 说明 |
|------|------|
| Bug 修复 | 有明确报错信息和复现步骤的问题 |
| 功能开发 | 需求明确、验收条件清晰的功能 |
| 代码重构 | 重命名、提取方法、消除重复 |
| 测试编写 | 为已有功能补充单元测试 |
| 文档生成 | 生成 API 文档、代码注释 |
| 多任务并行 | 同时推进多个独立任务 |
| 代码调研 | 分析代码架构、回答代码相关问题 |

### 7.2 云开发不擅长的场景

| 场景 | 原因 |
|------|------|
| UI 精细调试 | 需要视觉反馈和频繁人工确认 |
| 内网服务交互 | 云端网络受限，无法访问公司内网 |
| 需要特殊硬件 | 如 GPU、专用设备等云端 VM 不具备 |
| 极度模糊的需求 | "帮我做个好看的页面" 这类需求效果不好 |
| 需要大量创意决策 | AI 更擅长执行明确指令而非创意设计 |
| 长时间持续调试 | 复杂交互调试可能需要本地环境支持 |

### 7.3 最佳实践

1. **任务拆分**：将大任务拆成多个小任务并行执行，每个任务有清晰的验收条件
2. **明确描述**：提供"目标 + 当前行为 + 期望行为 + 验收条件"的四要素描述
3. **利用 CLAUDE.md**：在仓库根目录配置项目规则，Claude 会自动遵循
4. **先规划后执行**：复杂任务先用 Plan Mode 讨论方案，确定后再发到云端执行
5. **及时审查**：收到完成通知后尽快审查 Diff，有问题立即反馈迭代
6. **小步快走**：宁可多发几个小任务，也不要一次塞太多需求

## 八、好处总结

### 对开发者

1. **效率提升**：从描述到代码产出，AI 执行速度快，且可并行
2. **减少重复劳动**：例行修复、测试编写等可委托 AI
3. **随时随地**：浏览器、手机都能发起和监控任务
4. **会话持久**：关闭浏览器任务继续运行，不丢失进度
5. **灵活切换**：本地和云端可以互通（`&` 发送、`/teleport` 拉取）

### 对管理者

1. **100% 可追溯**：每行代码变更都有对话上下文和决策依据
2. **自动化统计**：周报/日报无需手写，数据精确客观
3. **过程透明**：从黑盒变为白盒，开发过程全程可观测
4. **防止造假**：`claude/` 分支标记 + 对话记录 + PR 关联，三重验证
5. **产出量化**：精确到每日的提交数、代码行数、PR 数量

### 对团队

1. **质量保障**：CI/CD + Code Review 流程不变，AI 代码同样需要审查
2. **知识沉淀**：对话记录本身就是开发文档
3. **规范一致**：`CLAUDE.md` 确保所有人（和 AI）遵循同一套架构规则
4. **快速交付**：缩短从需求到上线的周期

## 九、已知限制与注意事项

### 9.1 平台限制

| 限制 | 说明 |
|------|------|
| **仅支持 GitHub** | 不支持 GitLab、Bitbucket 等其他代码托管平台 |
| **网络受限** | 默认只允许白名单域名（包含主流包管理器和云平台） |
| **通用镜像** | 使用 Anthropic 预装的通用镜像，特殊依赖需通过 Hook 安装 |
| **会话独立** | 每个会话运行在独立 VM 中，会话间不共享状态 |
| **速率限制** | 与其他 Claude 用法共享额度，并行任务消耗更多 |
| **研究预览** | 目前仍处于 Research Preview 阶段 |

### 9.2 环境注意事项

**预装工具链**：云端 VM 预装了常见的开发工具，包括：
- 编程语言：Python 3.x、Node.js LTS、Ruby、PHP、Java、Go、Rust、C++
- 包管理器：pip、npm、yarn、pnpm、bun、gem、cargo、Maven、Gradle
- 数据库：PostgreSQL 16、Redis 7.0
- 可以通过 `check-tools` 命令查看完整列表

**SessionStart Hook**：如果项目有特殊依赖，在 `.claude/settings.json` 中配置自动安装：

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/install_pkgs.sh"
          }
        ]
      }
    ]
  }
}
```

对应脚本示例（`scripts/install_pkgs.sh`）：

```bash
#!/bin/bash
# 仅在远程云端环境执行
if [ "$CLAUDE_CODE_REMOTE" != "true" ]; then
  exit 0
fi

npm install
dotnet restore
exit 0
```

> **注意**：Hook 在本地和云端都会触发，务必通过 `CLAUDE_CODE_REMOTE` 环境变量区分。

### 9.3 安全注意事项

1. **不要在环境变量中存放生产凭证**：云端环境变量应仅包含开发/测试用途的配置
2. **审查 Diff 中的敏感信息**：创建 PR 前检查是否包含密钥、密码等
3. **精确授权仓库**：安装 Claude GitHub App 时选择具体仓库，不用 "All repositories"
4. **共享会话前检查内容**：会话可能包含私有仓库的代码和配置

### 9.4 Teleport 注意事项

从云端拉取会话到本地时，需要满足以下条件：

| 条件 | 说明 |
|------|------|
| 干净的工作区 | 本地无未提交的修改（可先 stash） |
| 同一仓库 | 必须在相同仓库的本地副本中执行 |
| 分支可用 | 云端分支已推送到远程 |
| 同一账号 | 使用创建会话时相同的 Claude 账号 |

## 十一、常见问题 (FAQ)

### Q1：需要什么级别的 Claude 账号？

Pro、Max、Team、Enterprise 均可使用云开发。Team 和 Enterprise 账号支持会话共享和团队可见性。

### Q2：云端环境里没有我需要的工具怎么办？

通过 SessionStart Hook 自动安装。在 `.claude/settings.json` 中配置安装脚本，每次会话启动时自动执行。详见第九章。

### Q3：需要访问公司内网服务怎么办？

云端环境默认不能访问内网。两种解决方案：
1. 将内网依赖的数据 mock 化，用测试数据替代
2. 先在云端完成不依赖内网的部分，再 teleport 到本地完成剩余工作

### Q4：会话中断了（关闭浏览器/网络断开）怎么办？

会话在云端持续运行，不受客户端影响。重新打开 [claude.ai/code](https://claude.ai/code) 即可恢复查看。也可以通过手机 App 随时监控。

### Q5：如何同时处理多个任务？

每个任务创建独立的云开发会话即可。在首页可以看到所有活跃会话。也可以在本地终端用 `&` 前缀快速启动多个云端任务：

```
& 修复登录接口的 token 过期问题
& 为 UserService 添加单元测试
& 将 README 翻译成英文
```

### Q6：手机上能用吗？

可以。Claude 的 iOS 和 Android App 都支持 Claude Code 云开发功能，可以发起任务、监控进度、回复 Claude 提问。

### Q7：如何从本地终端发送任务到云端？

在 Claude Code 终端中，消息前加 `&` 前缀即可发送到云端执行：

```
& Fix the authentication bug in src/auth/login.ts
```

或使用命令行参数：

```bash
claude --remote "Fix the authentication bug in src/auth/login.ts"
```

### Q8：如何把云端会话拉到本地继续？

使用 teleport 功能：

```bash
# 交互式选择
claude --teleport

# 指定会话 ID
claude --teleport <session-id>

# 在 Claude Code 内部
/teleport
```

### Q9：并行任务会不会影响速率限制？

会。云开发与其他 Claude 用法共享速率限制，并行任务会按比例消耗更多额度。建议根据账号额度合理安排并行数量。

### Q10：AI 写的代码质量怎么保证？

通过三层保障：
1. **AI 自查**：Claude 会自动运行测试、检查编译错误
2. **Diff 审查**：在创建 PR 前你可以逐行审查并迭代
3. **Code Review**：PR 进入正常的团队 Review 流程

质量保障流程与人工开发一致，AI 代码同样需要通过 CI 检查和团队 Review。

## 附录：会话共享配置

### Team / Enterprise 账号

- **Private**：仅自己可见
- **Team**：团队成员可见（基于 GitHub 仓库权限验证）

> 推荐设置为 **Team**，让管理者和同事可以查看对话过程。

### Pro / Max 账号

- **Private**：仅自己可见
- **Public**：所有登录用户可见

> 共享前务必检查会话中是否包含敏感信息（代码、凭证等）。

可在 **Settings > Claude Code > Sharing settings** 中配置默认共享策略和仓库访问验证。
