# CDS CLI 蜂群协作 · 指南

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：已落地

## 适用场景

当同一 cdscli 或部署链路问题在两个以上真实项目反复出现，且反馈、修复、复测已经形成多轮往返时，使用并行蜂群协作。单项目单故障继续走普通 issue 修复，不启用蜂群。

## 角色

| 角色 | 责任 | 不负责 |
| --- | --- | --- |
| 协调者 | 维护 tracker、拆分问题、分派、汇总结论 | 直接改代码或替反馈方验收 |
| 反馈方 | 在各自真实项目复现、给证据、复测 | 提实现方案或修改共享代码 |
| 修复方 | 归因、修复、测试、回复 issue | 代替真实项目判断是否可用 |

至少需要两个独立反馈项目；同一环境重复运行不算独立反馈。

## 启动条件

满足任一项即可启动：

- 同一周内两个真实项目命中同类故障。
- issue 从“已解决待验收”被打回至少两次。
- 用户必须在多个 Agent 之间持续转述上下文。

## 操作流程

1. 协调者创建一个 tracker，写清统一问题、参与项目、成功标准和当前 issue。
2. 每个反馈方只在自己的环境复现，提交版本、命令、输入、实际结果和脱敏证据。
3. 协调者合并同根问题，区分共享缺陷与项目特有配置，不让修复方重复修三个表象。
4. 修复方先补行为级测试，再提交最小修复，并在每个 issue 标出对应变更和复测方法。
5. 反馈方使用修复版本重新执行原路径；只回复“好了”不算证据。
6. 全部反馈方通过后，协调者关闭子 issue 和 tracker，并把长期约束更新到规则、设计或债务。

## 证据合同

每次反馈至少包含：

- 项目与分支标识
- cdscli/CDS 版本或 commit
- 可复制的最短操作
- 预期与实际结果
- 日志、HTTP 状态或截图之一
- 是否包含敏感值及脱敏说明

复测必须注明使用的新版本，并对原复现步骤给出通过或失败结论。

## 停止条件

- 所有反馈项目通过同一成功标准。
- 证明问题并非共享缺陷，已分别归入项目配置或外部依赖。
- 连续两轮没有功能净进展且阻塞来自外部凭据、授权或基础设施，此时按熔断规则升级，不继续制造进度消息。

- 无人值守维护：`issues-autofix`
- CDS 部署与诊断：`cds`
