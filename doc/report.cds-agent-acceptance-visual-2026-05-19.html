2. 架构设计

MAP

→

CDS

容器、分支、runtime、sandbox 管理器。发现并调度 managed runtime。

Official SDK Adapter

Claude Agent SDK 执行 agent loop；本仓库只保留最薄 adapter、policy、observability。

7. 是否需要合并 main

结论：当前验收不需要先合 main，因为验收目标是当前工作分支与远端 CDS runtime 的闭环，证据已经来自该分支。

合并前建议：在准备进入主线或 PR 前执行一次非破坏性预检：git fetch origin、检查 origin/main...HEAD、本地 merge/rebase dry-run、跑 one-cycle 或至少 S1/S2/S3/V1 窄验收。

8. 下一阶段规划：Writable Agent Cycle

W1 · 0.5 天
Writable profile 设计
把默认 read-only profile 与 writable profile 分离，定义工具白名单、审批策略、资源限制、审计字段。
W2 · 0.5-1 天
审批与工具策略
Bash/Edit/Write 只在 writable profile 中暴露；危险命令必须进入 MAP approval，不允许默认 auto-run。
W3 · 1 天
文件 diff / 测试 / 产物
把写入结果、diff、测试输出、失败重试和产物索引回传 MAP。
W4 · 0.5 天
PR 或补丁交付
先支持 patch/diff artifact，再接 GitHub PR。PR 权限独立开关，不进入默认 profile。
W5 · 0.5 天
验收
新增 S4 writable smoke：改一个小文件、跑测试、生成 diff、拒绝危险命令、stop/cancel、视觉截图。

11. 验收结论

已完成：默认 hardened read-only CDS Agent 可作为“上手即用”的代码巡检能力，具备可观察、可调试、可停止、可复跑和视觉验收证据。

未夸大：它还不是默认自动改代码/跑测试/提交 PR 的 agent；该能力需要下一阶段 writable profile，并且必须显式审批。

下一步：若目标是“审代码并给报告”，当前即可使用；若目标是“审代码并修改/提 PR”，进入 W1-W5 writable cycle。
