# CDS 容量预算与故障隔离设计 · 设计

> **版本**：v3.0 | **日期**：2026-07-17 | **状态**：已落地

## 一、管理摘要

- **解决什么问题**：分支预览环境无限常驻会耗尽单机资源，多节点场景还需要明确容量派发、节点失联和恢复边界。
- **当前方案**：单节点以分支温池限制常驻容量并按 LRU 降温；集群以 scheduler、executor registry 和 dispatcher 分配执行节点。
- **关键原则**：容量必须可查询，构建峰值与长期常驻容量分开判断，任何降温、迁移和失联都不能破坏项目状态。
- **运维边界**：代码能力已存在不等于目标环境已经启用；实际可用性以运行配置、节点状态和现场验证为准。

## 1. 目标与非目标

### 1.1 目标

- 为每个节点设置可预测的常驻分支上限。
- 按最近访问和用户固定状态选择可降温分支。
- 冷分支再次访问时可恢复，并持续展示唤醒进度。
- 多节点按在线状态、负载或温池余量选择 executor。
- 节点失联时明确标记，不把未知状态当作空闲容量。

### 1.2 非目标

- 不承诺所有分支永远常驻或零冷启动。
- 不用 CDS 自研调度器替代 Kubernetes 等通用平台。
- 不在设计文档固化某台服务器能承载的固定分支数。
- 不把集群注册成功等同于业务分支已完成迁移。

## 2. 单节点分支温池

分支运行状态按调度视角分为 warming、hot、cooling 和 cold。`heatState` 未设置时保留兼容语义，由现有 branch 与 service 状态判断是否运行。

| 状态 | 含义 | 允许动作 |
|------|------|----------|
| warming | 正在构建、启动或等待 readiness | 观察、取消、失败回到 cold |
| hot | 服务可响应，计入常驻容量 | touch、固定、主动降温 |
| cooling | 正在停止运行服务 | 等待完成，不重复停止 |
| cold | 保留项目、分支和 worktree 事实，服务未运行 | 按请求唤醒或预热 |

Scheduler 关闭时不得改变既有运行行为。开启后，代理请求成功会刷新最近访问时间，持久化可节流，但内存中的 LRU 时间必须及时更新。

## 3. 容量与驱逐

`maxHotBranches` 是配置后的常驻容量上限，不由文档根据机器内存自动拍定。运营者应结合基础设施保留量、应用平均内存、构建峰值和故障余量进行设置。

达到容量上限时，只能从非固定的 hot 分支中选择最久未访问者。以下分支视为固定：

- 用户显式固定的分支。
- 当前被颜色标记或处于主动调试保护的分支。
- 项目默认分支。
- 调度配置中明确列出的固定分支。

没有安全候选时，唤醒应返回容量不足，而不是停止受保护分支。构建中的分支不作为普通 LRU 驱逐对象。

## 4. 降温与唤醒

降温只停止可重建的分支运行服务，保留项目定义、分支记录、持久化数据边界和 worktree。数据库、共享基础设施和显式持久化卷不随普通分支降温删除。

唤醒复用 CDS 的构建、启动和 readiness 链路。页面必须展示排队、构建、启动、探活和失败阶段；请求不能在后端长时间无响应而前端只显示静态加载。

并发唤醒同一分支必须合并或串行化，避免重复构建。唤醒失败后恢复为可重试状态，并保留最后错误和阶段。

## 5. 运行时配置

| 配置 | 作用 |
|------|------|
| enabled | 是否启用温池调度 |
| maxHotBranches | 当前节点允许的 hot 分支上限 |
| idleTTLSeconds | 空闲分支进入降温候选的时间 |
| tickIntervalSeconds | 后台检查周期 |
| pinnedBranches | 配置级固定分支 |

Dashboard 可以更新受支持的配置，但配置持久化由服务端负责。前端显示值不构成配置已经应用的证据。

## 6. 集群角色

| 角色 | 责任 |
|------|------|
| standalone | 在单节点同时承载控制面与执行能力 |
| scheduler | 保存 executor 注册、选择节点并派发任务 |
| executor | 运行分支服务，回报心跳、资源和温池状态 |
| hybrid | 加入流程中的临时状态，重启后按持久配置收敛 |

集群 bootstrap 使用短期连接码传递主节点地址和一次性授权材料。连接码必须有有效期，不能作为长期 executor 凭据保存或展示。

## 7. 节点注册与派发

Executor 注册后持续发送心跳。Registry 根据心跳时间区分 online、draining 和 offline；draining 或 offline 节点不能接收新分支。

Dispatcher 支持按分支数量、实时负载或温池容量选择节点：

| 策略 | 适用事实 |
|------|----------|
| least-branches | 节点能力接近，按已分配分支数量均衡 |
| least-load | 以内存和 CPU 等心跳指标判断即时余量 |
| capacity-aware | 以 hot 数量与节点温池上限判断长期常驻余量 |

获取温池快照失败时不能把该节点当作零负载。派发决定应记录候选节点、使用策略和选择原因。

## 8. 故障隔离

| 故障 | 处理原则 |
|------|----------|
| 调度 tick 失败 | 记录错误，下一周期重试，不终止 CDS 主进程 |
| 单分支启动失败 | 只影响该分支，保留其他 hot 服务 |
| state 持久化失败 | 不覆盖最后可用状态，采用原子写和恢复检查 |
| executor 心跳超时 | 标记 offline，停止新派发，等待恢复或迁移决策 |
| scheduler 重启 | 从持久状态和 executor 心跳重建视图 |
| 网络分区 | 避免双重派发；未知执行结果先对账再重试 |

自动迁移必须区分“目标节点重新构建”与“持久数据搬迁”。没有数据迁移证据时，只能恢复无状态服务，不能宣称完整故障转移。

## 9. 安全边界

- bootstrap token 短期有效，加入后换成长期受控凭据。
- executor 管理接口必须鉴权，节点不能自行扩大项目权限。
- 调度器只对 CDS 管理的分支与服务执行停止和唤醒。
- 用户固定状态和默认分支保护由服务端裁决。
- 日志不输出连接码、永久凭据和完整环境变量。

## 11. 验收标准

- 调度关闭时不改变现有分支运行行为。
- 容量满时只降温最久未访问且未固定的分支。
- 冷分支唤醒阶段持续可见，失败不会影响其他分支。
- offline 和 draining executor 不接收新任务。
- 派发记录能解释候选节点、策略和选择结果。
- 集群重启和网络异常后通过对账恢复，不制造重复 owner。
