# CDS 产品能力 · 规格

> **版本**：v2.0 | **日期**：2026-07-15 | **状态**：开发中

CDS 是面向开发、预览和验收的自动交付平台。用户提供代码仓库并批准项目权限，CDS 负责理解项目、准备运行环境、持续展示部署过程、提供预览地址，并保存可重复部署和恢复的成功版本。

本规格描述当前 CDS 产品全貌，取代旧版单项目功能快照。专项数据模型和协议仍以对应 `spec.cds.*`、`design.cds.*` 文档为准。

## 产品定位

### 解决的问题

- 新项目部署前需要手工准备大量配置；
- 不同 Agent 的接入方式不一致，容易复制长期密钥或修改电脑环境；
- 分支、服务、日志和部署状态分散，失败后难以判断原因；
- 相同代码重复构建，新版本失败后恢复成本高；
- 私有仓库、环境变量和项目权限容易混用；
- 预览部署能启动，但缺少真实用户验收证据。

### 标准用户流程

1. 选择 Codex、Cursor 或 Claude Code；
2. 选择已有项目或创建新项目；
3. 在 CDS 页面批准项目权限；
4. 选择 Git 仓库并确认自动识别结果；
5. 发起部署并持续查看过程；
6. 打开最终预览地址完成核心操作；
7. 后续复用成功版本或恢复以前版本。

### 不做什么

- 不修改业务代码来适应托管模式；
- 不强制复杂项目放弃 Compose；
- 不把容器运行等同于用户验收通过；
- 不让长期密钥进入 Agent 对话；
- 不把不同项目的环境、网络和权限混在一起。

## 用户角色

| 角色 | 主要需求 |
| --- | --- |
| 项目成员 | 创建项目、部署分支、看预览、诊断和恢复版本 |
| 验收人员 | 从最终预览地址验证功能并保存证据 |
| CDS 管理员 | 管理系统配置、GitHub App、集群、容量和审计 |
| Agent | 在用户批准的项目范围内扫描、部署、诊断和查询 |
| GitHub App | 在被选择仓库范围内拉取代码、接收事件和回写状态 |

| F3 | 项目识别 | 自动检测技术栈、服务、命令、端口和依赖 |
| F4 | 交付模式 | 托管模式降低配置，Compose 模式保留高级控制 |
| F5 | 分支与自动部署 | 多分支隔离、推送触发、手动部署和状态回写 |
| F6 | 部署记录 | 可恢复的阶段、进度、日志、失败原因和处理建议 |
| F7 | 部署版本 | 成功产物复用、重新部署和回滚 |
| F8 | 应用与基础设施 | 多服务、数据库、缓存、队列、存储和健康检查 |
| F9 | 环境与配置 | 系统配置、项目变量和 Agent 凭据分层 |
| F10 | 预览与路由 | 按项目和分支提供预览地址及多服务路由 |
| F11 | 可观测与诊断 | 构建、容器、HTTP、Webhook、系统事件和失败归因 |
| F12 | 任务调度 | 项目级定时或手动任务、测试和运行记录 |
| F13 | 验收中心 | 验收报告、项目与文件夹归类、直达链接 |
| F14 | 集群与维护 | 执行节点、容量调度、服务自更新和恢复 |
| F15 | CDS Agent 工作台 | 在受控工作区执行代码任务或只读审查 |

## 功能要求

### F1. 身份与授权

| ID | 要求 |
| --- | --- |
| F1.1 | 支持人类用户登录、会话过期和登出 |
| F1.2 | 支持 Codex、Cursor 和 Claude Code 通过同一页面接入 |
| F1.3 | Agent 默认通过页面批准获得项目级权限，不复制长期 Key |
| F1.4 | 新项目使用一次性创建权限，创建成功后自动换成项目专属权限 |
| F1.5 | Agent 凭据只保存在当前项目，不修改系统环境和 Shell 配置 |
| F1.6 | 所有 Agent 操作进入活动记录，并标明主体和项目 |
| F1.7 | 保留旧环境变量和 Key 方式用于历史自动化兼容 |

### F2. 项目与仓库

| ID | 要求 |
| --- | --- |
| F2.1 | 每个项目关联独立仓库、网络、环境、分支和部署历史 |
| F2.2 | 支持公开仓库和 GitHub App 授权的私有仓库 |
| F2.3 | 私有仓库授权默认只选择目标仓库，并可被扫描、构建和部署复用 |
| F2.4 | 读取仓库真实默认分支，不假设一定是 `main` |
| F2.5 | 项目删除、迁移和恢复不得污染其他项目 |

### F3. 项目识别

| ID | 要求 |
| --- | --- |
| F3.1 | 识别应用目录、语言、包管理器、安装和构建方式 |
| F3.2 | 识别前端、后台、后台任务和多入口服务 |
| F3.3 | 推导启动命令、端口、健康检查和常见依赖 |
| F3.4 | 展示识别依据和把握程度，结果默认可编辑 |
| F3.5 | 支持一次性试运行验证，不把临时容器当正式部署 |
| F3.6 | 无法安全判断时要求补充最小信息，不猜测并覆盖特殊逻辑 |

### F4. 交付模式

| ID | 要求 |
| --- | --- |
| F4.1 | 托管模式为常规项目生成最终生效运行方案 |
| F4.2 | Compose 模式保留多服务、网络、镜像和启动顺序控制 |
| F4.3 | 两种模式可以切换，切换不删除原有 Compose 配置 |
| F4.4 | 标准 Compose 缺少启动命令时，从项目自身信息继续推导 |
| F4.5 | 托管模式不修改业务代码 |

### F5. 分支与自动部署

| ID | 要求 |
| --- | --- |
| F5.1 | 浏览、搜索、创建、删除、收藏和标记分支 |
| F5.2 | 分支拥有独立运行环境和预览入口 |
| F5.3 | GitHub 推送、PR 和手动操作可以触发部署 |
| F5.4 | 展示触发来源、提交、目标分支和服务结果 |
| F5.5 | 同一分支防止无意的重复并发部署 |

### F6. 部署记录

| ID | 要求 |
| --- | --- |
| F6.1 | 在实际构建或启动前创建部署记录 |
| F6.2 | 持续记录准备、构建、启动、检查和完成阶段 |
| F6.3 | 页面刷新、暂时离开或短暂断线后仍能恢复同一过程 |
| F6.4 | 每个服务分别记录状态和错误 |
| F6.5 | 失败时给出问题归属、失败阶段和建议处理顺序 |
| F6.6 | AI 解释是辅助能力，基础诊断不依赖大模型 |

### F7. 部署版本

| ID | 要求 |
| --- | --- |
| F7.1 | 构建成功后保存不可变部署版本 |
| F7.2 | 展示代码提交、配置摘要、产物状态和创建来源 |
| F7.3 | 可复用版本重新部署时避免重复构建 |
| F7.4 | 支持选择以前的成功版本恢复 |
| F7.5 | 重新部署和回滚都生成新的部署记录 |
| F7.6 | 产物不可用或不在当前执行节点时明确说明，不伪装为可直接恢复 |

### F8. 应用与基础设施

| ID | 要求 |
| --- | --- |
| F8.1 | 一个项目支持多个应用服务和后台任务 |
| F8.2 | 支持数据库、缓存、消息队列、搜索和对象存储预设 |
| F8.3 | 基础设施连接信息按项目注入并脱敏展示 |
| F8.4 | 支持持久化卷、依赖顺序、健康检查和日志 |
| F8.5 | 运行状态按服务展示，不合并成一个笼统绿色状态 |

### F9. 环境与配置

| ID | 要求 |
| --- | --- |
| F9.1 | CDS 系统配置、项目运行变量和 Agent 项目凭据分开管理 |
| F9.2 | 项目变量按项目隔离，修改后通过重新部署生效 |
| F9.3 | 敏感值在页面、日志、事件和导出中脱敏 |
| F9.4 | Agent 凭据不得注入业务容器或进入 Git |

### F10. 预览与路由

| ID | 要求 |
| --- | --- |
| F10.1 | 为项目分支生成稳定且可区分的预览地址 |
| F10.2 | 支持前端、后台和多服务路径路由 |
| F10.3 | 失败或准备中时展示明确状态页，避免无限触发部署 |
| F10.4 | 文档和交付消息提供最终功能深链，不只给根域名 |

### F11. 可观测与诊断

| ID | 要求 |
| --- | --- |
| F11.1 | 提供部署阶段、构建日志、容器日志、HTTP 日志和 Webhook 记录 |
| F11.2 | 区分代码、配置、CDS、外部服务和未知问题 |
| F11.3 | 长任务持续显示变化，不让用户面对静止等待 |
| F11.4 | 日志和事件包含项目、分支、服务、版本和操作主体 |

### F12. 任务调度

| ID | 要求 |
| --- | --- |
| F12.1 | 支持每天、间隔和仅手动三种调度方式 |
| F12.2 | 支持 HTTP 调用和受控命令动作 |
| F12.3 | 创建前可以测试目标，失败时不创建任务 |
| F12.4 | 保存运行记录、结果和可读错误 |

### F13. 验收中心

| ID | 要求 |
| --- | --- |
| F13.1 | 保存 HTML 或 Markdown 验收报告 |
| F13.2 | 支持按项目和文件夹归类 |
| F13.3 | 提供可以直接打开目标报告的深链 |
| F13.4 | 报告访问遵循 CDS 登录和项目权限 |

### F14. 集群与维护

| ID | 要求 |
| --- | --- |
| F14.1 | 支持单机运行和调度端、执行端分离 |
| F14.2 | 展示执行节点容量、心跳和任务归属 |
| F14.3 | 执行节点不可用时提供明确恢复和迁移信息 |
| F14.4 | CDS 服务自更新可选择分支、重启并等待恢复 |

### F15. CDS Agent 工作台

| ID | 要求 |
| --- | --- |
| F15.1 | 在独立工作区执行受控代码任务，不污染主系统工作区 |
| F15.2 | 持续展示会话、步骤、审批和结果 |
| F15.3 | 完整运行时不可用时提供明确的只读审查模式 |
| F15.4 | 只读模式不得修改文件或执行命令，并明确展示能力边界 |

## 关键状态

### 部署状态

`queued`、`preparing`、`building`、`starting`、`verifying`、`running`、`failed`、`cancelled`。

### 服务状态

`idle`、`building`、`starting`、`running`、`stopped`、`error`。

### 版本可复用状态

- 可直接复用：当前环境具备完整产物；
- 需要重新构建：只有代码和配置记录，缺少可用产物；
- 不可恢复：版本信息不完整或权限已失效。

## 核心验收

| 场景 | 通过标准 |
| --- | --- |
| 新用户接入 | 不复制长期密钥、不修改系统环境即可连接项目 |
| 普通项目部署 | 只确认自动识别结果即可打开预览 |
| 私有仓库 | 单仓库授权后扫描和部署均可复用 |
| 过程恢复 | 页面刷新后继续显示同一部署记录 |
| 版本复用 | 相同成功版本重新部署不重复构建 |
| 回滚 | 选择以前成功版本后预览恢复正常 |
| 失败诊断 | 用户能知道问题归属和下一步 |
| 高级项目兼容 | 原有 Compose 配置和行为不被覆盖 |

## 非功能要求

| 类别 | 要求 |
| --- | --- |
| 安全 | 最小权限、敏感值脱敏、凭据不进 Git 和对话 |
| 隔离 | 项目网络、环境、仓库授权和部署历史互相隔离 |
| 可恢复 | 部署过程、版本和关键事件持久化 |
| 可观察 | 长任务持续反馈，操作主体和阶段可追踪 |
| 兼容 | 旧 Compose、旧 Key 自动化和高级配置可继续使用 |
| 易用 | 普通项目先自动识别，用户只处理无法安全推断的部分 |
| 验收 | 必须通过真实预览地址完成核心业务流程 |
