# 缺陷截图 VLM 预解析 · 设计

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：已落地

## 管理摘要

- **解决的问题**：缺陷描述的关键信息经常只存在于截图中，纯文本润色无法理解红框、箭头、错误弹窗和异常状态。
- **核心方案**：图片加入附件时异步调用视觉模型生成文字描述；润色阶段只把文字描述交给聊天模型。
- **当前状态**：缺陷提交面板和全局提交弹窗均已接入，支持自动分析、结果回填、流式润色和失败降级。
- **成本边界**：一张截图只在加入时调用一次视觉模型，重复润色不重复发送图片。

## 问题背景

用户提交缺陷时通常同时提供文字和截图。截图中的页面位置、标记区域与错误文案对理解问题很重要，但普通聊天模型只收到文字。若每次润色都重新把所有图片交给视觉模型，会增加成本和等待时间，也会把图片分析与文案改写耦合成一个难以降级的步骤。

## 设计目标

1. 图片加入后立即开始分析，不阻塞继续输入和提交。
2. 分析结果对用户可见，并能参与后续润色。
3. 删除中的附件不能在异步返回后污染正文。
4. 模型池未配置或分析失败时，仍允许提交原始缺陷。
5. 润色使用 SSE 持续展示阶段和输出，不出现静止等待。

## 核心决策

| 决策 | 说明 |
|------|------|
| 看图与润色分离 | 视觉模型负责提取事实，聊天模型负责组织缺陷报告 |
| 上传即预解析 | 图片加入附件列表后异步分析，一次结果可多次复用 |
| 文本化传递 | 润色请求只携带截图描述，不重复携带图片 |
| 非阻塞降级 | 分析失败不阻止附件上传、正文编辑和缺陷提交 |
| 服务端权威执行 | 客户端断开不取消已经开始的视觉推理 |

## 前端状态设计

每个附件同时保存文件、预览地址、分析状态和可选描述。

| 状态 | 含义 | 后续行为 |
|------|------|----------|
| `idle` | 非图片或无需分析 | 正常保留附件，不加入截图描述 |
| `analyzing` | 正在调用视觉模型 | 显示持续变化的分析状态 |
| `done` | 已得到描述 | 展示结果并加入润色上下文 |
| `error` | 调用失败 | 显示失败信息，不阻断提交 |

当分析成功时，前端会把识别结果以明确标签合并到现有正文末尾，同时保留用户已经输入的内容。润色时还会从所有 `done` 附件收集描述，作为结构化的 `imageDescriptions` 传给后端。

异步返回前必须再次确认原附件仍在列表中。用户已经删除截图时，不回填描述，也不修改正文。

## 运行流程

1. 用户粘贴、拖入、选择或从系统分享一张截图。
2. 前端创建本地预览并把状态置为 `analyzing`。
3. 前端把图片转换为 Base64，调用图片分析端点。
4. 后端先解析 `defect-agent.analyze-image::vision` 模型池。
5. 视觉模型提取标记区域、错误信息、异常状态和页面上下文。
6. 成功结果回填附件并合并到正文；失败则进入可降级状态。
7. 用户点击 AI 润色，前端通过 SSE 端点发送正文、模板和截图描述。
8. 聊天模型结合文字与截图事实生成完整缺陷报告，用户确认后应用。

## API 契约

| 方法 | 路径 | 输入 | 输出 |
|------|------|------|------|
| `POST` | `/api/defect-agent/images/analyze` | Base64 图片与 MIME 类型 | 简洁的截图缺陷描述 |
| `POST` | `/api/defect-agent/defects/polish/stream` | 原文、模板 ID、截图描述列表 | `phase`、`model`、`thinking`、`typing`、`done`、`error` 事件 |

旧的一次性润色端点仍用于兼容，但新界面的事实入口是流式端点。

## 模型与错误处理

- 图片分析固定使用 `defect-agent.analyze-image::vision`，不得绕过 Gateway 直连模型。
- 润色使用缺陷 Agent 的聊天调用标识，与视觉模型池独立配置。
- 视觉模型池未配置时返回明确错误码，前端将附件恢复为可提交状态并跳过分析。
- 一般分析错误保留在附件状态中，不把失败文本当作截图事实交给润色模型。
- 服务端视觉推理使用独立于浏览器连接的取消策略，避免用户关闭弹窗后浪费已完成一半的推理。

## 安全与体验边界

- 只接受图片内容进入视觉端点，客户端提供的 MIME 类型仍需由服务端按请求契约处理。
- 日志记录调用状态和错误，不记录完整 Base64 图片。
- 自动回填必须标明是 AI 识别内容，不能伪装成用户原始描述。
- 识别结果是辅助证据，最终缺陷内容仍由用户确认。
- 大图限制、压缩和更精细的图片安全检查属于上传基础设施，不在本文重复定义。

## 验证重点

1. 图片加入后出现持续分析状态，成功后正文与附件都能看到结果。
2. 分析中删除图片，异步返回后正文不被污染。
3. 多张图片只把成功描述加入润色请求。
4. 视觉模型池未配置时仍可提交缺陷。
5. 润色流能显示阶段、增量文本、完成和错误状态。
