## 一、管理摘要（30 秒看懂）

团队动态的「行为洞察」原本是一长串接口报错/慢请求列表，像后台日志，非技术同事看不懂、也看不到全局。本次把它重定位为 **用户体验之声（VOC, Voice of Customer）**：

> 用户嫌烦但懒得反馈的地方，系统替他喊出来，并盯着你修完。

核心是三件事：**一张看得懂的全景图（哪里在疼）→ 一份按影响力排序的待办（先修什么）→ 一条修复闭环（修完疼没疼降下去）**。

## 二、为什么不是「VOC 问卷」

严格说，本系统采集的是**隐性 VOC / 行为之声**——用户没开口、但行为暴露的不满（报错、等待、流失、横跳），不是问卷式的显性 VOC。两者互补：

- **行为之声**：`apirequestlogs`（报错/慢）+ `behavior_events`（停留/秒退/横跳）
- **用户之声**（后续波次）：`defect_reports`（用户主动提的缺陷）

二者指向同一件事时即「说做互证」，是最高置信的痛点。这是本系统比纯 APM（Datadog/Amplitude 只有行为、没有缺陷闭环）能多走的一步。

## 三、用户场景

1. 产品负责人打开洞察 tab → 体验全景热力图扫描成形 → 一眼看到哪个模块在「发烧」（红/琥珀块从冷色平静海里跳出来）。
2. 点发烧的块 → 下钻到下方痛点榜对应行 → 看证据、影响人数、AI 建议。
3. 「转为缺陷」→ 接入缺陷管理流水线 → 修复后标记「已修复」→ 痛点指数回落（证据闭环）。
4. 想要一段话的判断 → 点「AI 简报」流式生成，可一键存档到知识库（文档沉淀）。

## 四、核心能力与设计取舍

### 4.1 体验全景热力图（squarified treemap）

- **形态**：WinDirStat 式矩形树图，每块=一个端点，**面积=访问量**，**颜色=健康**。
- **为什么是 treemap 而不是雷达/星图**：雷达点少、信息密度低（显单调）；treemap 密、满、富，面积即信息，红块从一片平静里自己跳出来。
- **配色（ui-ux-pro-max 规范：父级不同色相、子级同色相浅色梯度）**：健康区走蓝→青→绿「冷色平静海」，每个模块一个色相、区内明度递变；暖色（红/琥珀）与紫**只留给告警**，保证痛点是屏幕上唯一的高饱和，自然跳出。
- **名字给中文示意名**：模块/端点路径段（`mark-seen`/`entries`/`authz`）对非技术同事是天书，后端用 `ModuleLabels` + `SegmentLabels` + `LeafLabel()` 压成中文（标记已读/条目/权限鉴权），原始 `METHOD 路径` 保留在悬浮提示给技术同事。
- **真实数据 vs 示意图**：treemap 吃真实 `apirequestlogs`，只显示近 30 天**真被访问过**的端点，所以不会像设计稿那样铺满「全系统」——这是真账，不是 bug。

### 4.2 入场动画（借 react-bits 的交错揭示）

方块从中心缩放+淡入、按索引错开 9ms 依次冒出（`voc-cell-in`），叠加一条**横向扫描光带**（`voc-scan`，4.5s 循环）滑过整图 + 标题「实时扫描中」ping 徽章。三者合成「正在实时扫描成图」的体感，呼应 `artifact-is-experience`（等待期主视觉是产物本身在生长）。

### 4.3 痛点指数 + 声道占比（右侧可折叠）

- **痛点指数（0-100，越低越健康）**：从痛点榜现算的启发式——`Σ 声道权重 × 影响人数 × log(频次)` 过饱和曲线映射；修复后回落，是「修好了」的可视化证据。
- **声道占比**：把洞察按类型归并为 报错/等待/停留/流失/横跳 五声道，按加权占比排条。
- **可折叠**：顶部空间紧张时折起来让 treemap 全宽；折叠态留一条竖向「痛点指数」按钮一键展开。

### 4.4 痛点榜 + 闭环（既有能力，本次保留）

按严重度排序的洞察列表，每条带证据 + AI 建议 + 操作（转为缺陷/确认待改/已修复/忽略），指纹级生命周期持久化（`behavior_insight_state`）。这是 VOC「先修什么 + 修完闭环」的承载。

### 4.5 文档沉淀

AI 简报可一键发布到知识库「行为洞察简报」（幂等：当日同名更新不新建），把一次性分析固化成可检索的存档。

## 五、数据流（端到端）

```
apirequestlogs ─┐                          ┌─ experience-map 端点 → 体验全景热力图(treemap)
behavior_events ─┼─ ComputeInsightsAsync ──┼─ insights 端点 → 痛点榜 + 痛点指数/声道占比(前端现算)
                 │  (规则式聚类+严重度)      └─ insights/brief(SSE) → AI 简报 → 知识库存档
defect_reports ──┘ (后续波次：用户之声并入)
```

- 后端 `GET /api/team-activity/experience-map`：按 `/api/{module}` 分区聚合端点访问量/报错率/慢请求率，叶子 `target` 与 `insights.target` 同口径，支撑前端点击下钻联动。
- 前端 `ExperienceMap.tsx`（squarified treemap + 入场动画 + 扫描光带）、`ExperienceStats.tsx`（仪表盘 + 声道占比）、`InsightsPanel.tsx`（编排 + 痛点榜 + 闭环 + AI 简报）。

## 六、关联设计文档 / 后续波次

- 波次二：把 `defect_reports`（用户主动提的缺陷）并入同一张痛点榜，做成完整「说的 + 做的」双声道 VOC，「说做互证」高亮。
- 波次三：AI 维度下钻（`llmrequestlogs` 的 AppName → AppCallerCode：调用量/token/时延/报错/「选 A 给 B」对账）。
- 波次四：闭环回落证据区（修复前→后的真实回落）、成本预测 + 突增 Webhook 告警。

## 七、风险与边界

- 痛点指数是**启发式**，不是精确 SLA 指标；用于相对趋势（在涨/在跌），不做绝对承诺。
- treemap 仅覆盖被访问端点；流量稀疏时图会偏空，属预期。
- 中文词典 `SegmentLabels` 是增量维护：新端点段未登记会回落英文，按截图补词典即可。
