# 前端移动端适配 · 计划

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：规划中

## 目标

让 prd-admin 的核心用户路径在手机与窄窗口中可完成，而不是只做到“页面能缩小”。移动端必须保持导航可达、主操作可见、表单可提交、长内容可滚动，并与 Desktop 和宽屏 Web 共用业务状态。

Apple 视觉双轨迁移由 `plan.frontend.apple-design-migration.md` 管理；本计划只负责响应式能力和功能可达性。

## 基线原则

- 业务 API、权限和数据模型不因移动布局建立第二套实现。
- 列表在窄屏可转卡片或压缩列，但不能隐藏完成任务所需的信息。
- 模态框、抽屉和全高页面遵守统一滚动与安全区规则。
- 能在一个页面完成的任务不强迫用户跳到桌面端。
- 移动适配变更必须同时验证宽屏，防止响应式修复破坏桌面布局。

## 范围与优先级

### P0：全局壳层

- 顶部导航、底部入口、抽屉和返回行为一致。
- 视口高度、刘海安全区、键盘弹起和滚动锁正确。
- 全局 toast、确认框和权限提示不被遮挡。

### P1：核心工作流

- 登录、首页与 Agent 启动器。
- PRD、缺陷、周报、知识库、视觉创作的列表和详情主链。
- 文件选择、文本输入、提交、进度和结果查看。

### P2：管理与观测

- LLM 日志、模型配置、用户和权限管理。
- CDS、工作流和高级运营页面提供可用的窄屏信息层级。
- 宽表格采用列优先级、详情卡或局部横向滚动，不让整个页面横滚。

## 页面完成门

每个页面只有同时满足以下条件才标为 full：

1. 从移动导航可达，返回路径明确。
2. 首屏说明页面目标，主操作无需横向滚动。
3. 列表、详情、空状态、加载、失败和无权限状态均可用。
4. 表单在软键盘弹起后仍能看到当前字段和提交按钮。
5. 内容滚动发生在正确容器，页面没有双滚动和固定高度魔数。
6. 390px 与一个常见平板宽度完成深浅主题验收。
7. 宽屏回归没有新增溢出、遮挡或信息丢失。

## 执行方式

1. 从导航与高频访问数据生成页面清单。
2. 为每页标记 `full`、`limited` 或 `blocked`，记录唯一阻塞原因。
3. 按 P0、P1、P2 修复，不按组件数量报告进度。
4. 每批用同一组移动视口跑主链、失败态和宽屏回归。
5. 已完成页面从计划移出，长期布局规则进入 `.Codex/rules/` 或相关 design。

## 当前边界

移动日志页已有专门的窄屏信息收敛方案；其它页面不能直接照搬其隐藏规则。当前准确页面状态应由视觉矩阵和实际路由验收维护，本文件不保存易过期的逐组件行号和截图清单。

## 风险

| 风险 | 控制 |
| --- | --- |
| 为移动端复制页面 | 共享路由数据和业务组件，只替换布局层 |
| 隐藏关键字段 | 为字段定义优先级，并提供详情入口 |
| 键盘遮挡提交 | 使用动态视口与安全区实测 |
| 只验首页 | 每个核心流程至少走到提交和结果 |
| 修移动坏桌面 | 每批强制宽屏回归 |
