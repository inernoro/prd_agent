# 应用模型配置左右布局 · 设计

> **版本**：v1.1 | **日期**：2026-07-17 | **状态**：已落地

## 管理摘要

- **问题**：按树形层级查看应用调用者时，运维人员需要反复展开节点，难以同时比较一个应用下的全部功能、模型池和健康状态。
- **决策**：模型应用管理页采用“左侧应用、右侧功能详情”的主从布局；选中应用后在右侧纵向展示全部功能配置。
- **结果**：应用筛选、模型类型筛选、模型池管理和健康状态在一页完成，不改变后端模型绑定契约。

## 用户任务

1. 按名称或应用代码找到目标应用。
2. 选择应用后查看其全部功能和模型要求。
3. 查看每个功能绑定的模型池、模型健康、回退来源和调用统计。
4. 新建、绑定或调整模型池，再由服务端保存到对应功能的模型要求。

## 布局与交互决策

| 区域 | 责任 | 关键交互 |
|---|---|---|
| 左侧应用列表 | 应用发现与选择 | 名称搜索、模型类型筛选、功能数量提示、选中状态 |
| 右侧应用摘要 | 当前上下文 | 展示应用标识、功能数与聚合调用统计 |
| 右侧功能列表 | 模型配置和运行观察 | 展示每个功能的模型池、实际解析模型、健康与失败信息 |
| 模型池选择器 | 创建和绑定 | 统一进入 `ModelPoolPickerDialog`，避免在页面内重复实现选择逻辑 |

选中状态以具体 AppCaller 为准，但右侧展示同一应用下的全部功能。这使左侧保持扁平，右侧仍能完成跨功能比较。

## 数据边界

| 数据 | 读取用途 | 写入边界 |
|---|---|---|
| AppCaller 与功能要求 | 分组、筛选和展示 | 更新显示信息与功能的 `modelGroupIds` |
| 模型池 | 展示绑定与候选池 | 由模型池选择器创建或绑定 |
| 监控数据 | 显示健康、失败和回退原因 | 不在前端派生或修改健康结论 |
| 默认解析结果 | 解释未显式绑定时实际使用的模型 | 由服务端权威返回 |

页面支持一个功能绑定多个模型池；“无绑定”并不等同于不可用，界面应展示服务端解析出的默认或传统配置来源。

## 约束与验收

- 选择应用后，不需要展开树即可看到其全部功能。
- 新建或绑定模型池后刷新对应应用的运行监控数据。
- 健康、失败次数和回退原因只展示服务端结果，不在浏览器计算健康状态。
- 布局遵循当前双皮肤 token 规则，不使用旧设计中的硬编码深色或文字图标。
