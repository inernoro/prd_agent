# 缺陷自动化自治体系 · 规格

> **版本**：v0.2 | **日期**：2026-06-25 | **状态**：草案 | **appKey**：defect-agent

## 背景

缺陷自动化不能停留在“按提示词拉缺陷、修代码、提交结果”的阶段。这样的系统看起来像 Agent，实际仍是一次性脚本：它会执行命令，但不一定理解目标；它会调用工具，但不一定验证事实；它会继续往前跑，但不一定知道什么时候该停。

企业级缺陷处理需要的是一个能承担日常闭环的数字化员工。这个数字员工必须有明确的自治边界、稳定的证据链、可恢复的状态机、可沉淀的经验记忆，以及行动前的监督护栏。

本文定义缺陷自动化 Agent 的上层“宪法”。`spec.defect-agent-automation-protocol` 负责接口和工作流契约；本文负责定义判断原则、自治边界和系统性能力。

## 目标

- 从正式缺陷系统读取缺陷，一次领取并处理一个缺陷。
- 把缺陷描述还原为真实业务目标，而不是盲目执行用户文本。
- 对轻量、低风险缺陷自动修复、验证、提交 commit、创建 PR、回写证据。
- 对过期、虚假、不适合、重量级或高风险缺陷自动评论、跳过或升级人工。
- 在更新中心按 PR/commit 展示关联缺陷、发布状态和证据入口。
- 正式发布后执行验收报告归档和提交人通知，正式发布前不误通知。
- 把稳定步骤下沉为协议和代码，把智能体留给判断、修复和复盘。

## 非目标

- 不追求让 Agent 自动处理所有缺陷。
- 不允许 Agent 绕过 PR、代码审查或发布制度直接改生产。
- 不把测试库或预览库的缺陷当作正式缺陷闭环结果。
- 不把长提示词当作主要约束手段。
- 不用 changelog 文案、日期或人工点击建立缺陷关联。

## 核心原则

### 1. 边界先行

做缺陷 Agent 的第一步不是选模型、写提示词或调工具，而是定义什么能自主、什么必须请示人类。

| 维度 | 自动处理 | 必须停下 |
|------|----------|----------|
| 需求清晰度 | 可复现、范围明确、期望结果清楚 | 描述冲突、缺少关键上下文、无法判断真实目标 |
| 改动范围 | 局部代码、无迁移、无权限模型变化 | 跨服务、涉及数据迁移、权限、安全、计费、发布策略 |
| 风险等级 | 低风险 UI、序列化、文案、状态展示、兼容性修复 | 生产数据写入、鉴权、账户、通知群发、自动发布 |
| 验证能力 | 能通过测试、探针或浏览器验收证明 | 无法构造验证、验证依赖缺失、结果不可观察 |
| 发布状态 | 可创建 PR、可在预览环境验收 | 合并、正式发布、通知用户需要满足外部条件 |

### 2. 目标驱动，不盲从指令

缺陷提交人的描述不等于事实，也不一定等于真实目标。Agent 在执行前必须还原：

- 用户看到的现象是什么。
- 用户真正受阻的业务目标是什么。
- 当前描述是否包含猜测、误判或不完整证据。
- 问题是否属于本系统责任范围。
- 如果用户说法不成立，应该如何用验收报告和证据回复。

判断结果分为：

| 结果 | 动作 |
|------|------|
| 可修复缺陷 | 进入轻重判定和修复流程 |
| 描述不完整 | 评论追问或标记等待补充 |
| 非缺陷需求 | 评论说明并转入需求或建议流 |
| 用户陈述不成立 | 生成验收报告，引用证据回复 |
| 过期或已修复 | 评论说明现状，必要时附正式环境证据 |

### 3. 规划要算账

每次处理前必须形成轻量决策账本：

| 字段 | 说明 |
|------|------|
| 缺陷引用 | 编号、标题、用户原始问题摘录、影响位置 |
| 目标还原 | Agent 认为要解决的真实问题 |
| 轻重判定 | 复用 issue/缺陷技能定义的轻量标准 |
| 风险判断 | 是否涉及权限、数据、迁移、跨服务、通知、发布 |
| 验证路径 | 本地测试、集成测试、CDS 预览、浏览器验收或验收报告 |
| 停止条件 | 什么情况必须 block 或等待人工 |

### 4. 工具调用必须事实核查

工具不是“调了就算”，而是“调完要验证”：

- 拉取缺陷后确认来自正式 `domain`，不是测试库。
- 写回 commit 后确认缺陷结构化字段和 `defect_resolution_traces` 都存在。
- 更新中心展示只按 commit id 关联，不按日期或 changelog 文案猜测。
- 发布后验收报告必须能打开，且包含缺陷引用、PR/commit 证据和验收事项。
- 通知提交人前必须确认正式发布状态和验收结论。

### 5. 记忆沉淀进协议、代码和报告

Agent 不应每天重新靠长提示词学规矩。稳定约束必须沉淀：

| 类型 | 承载位置 |
|------|----------|
| 工作流顺序 | `defect-agent-workflow.v1` 后端接口 |
| 运行状态 | `DefectAutomationRun` 与 run item |
| commit 关联 | `defect_resolution_traces` |
| 每日调度壳 | 缺陷自动化按钮生成的短计划 |
| 执行判断 | `ai-defect-resolve` 技能 |
| 验收门禁 | `create-visual-test-to-kb` 技能和归档脚本 |
| 复盘经验 | `doc/spec.*`、`doc/debt.*`、验收报告知识库 |

### 6. 监督护栏先于行动

系统必须允许 Agent 主动停下：

| 护栏 | 触发条件 | 动作 |
|------|----------|------|
| 轻重熔断 | 超过轻量标准、风险不可控 | `workflow/block` 并评论原因 |
| 授权熔断 | domain/K/connector 自检失败 | 停止，不领取缺陷 |
| 环境熔断 | 预览或 CDS 不可用 | 记录阻塞，不伪造验收 |
| 事实熔断 | 用户陈述与证据冲突 | 生成证据报告后回复 |
| 通知熔断 | 未正式发布或未验收 | 不通知提交人 |

## 缺陷证据契约

每个缺陷闭环必须保留两类证据。

### 修复提交证据

缺陷评论、每日输出和更新中心关联弹窗必须能让用户直接看懂：

- 修的是哪个缺陷：缺陷编号、标题、用户原始问题摘录。
- 为什么这样修：修复目标和处理结论。
- 提交在哪里：PR 地址、commit 地址、commit SHA。
- 是否已发布：发布状态、待人工审核还是已正式发布。
- 如何验收：预览地址、正式发布后的验收报告地址。

### 发布后验收证据

“缺陷修复验收报告”必须在首屏包含：

| 区块 | 必填内容 |
|------|----------|
| 缺陷引用 | 编号、标题、用户原始问题摘录、影响位置、修复目标 |
| 提交与发布证据 | PR 地址、commit 地址、commit SHA、分支、发布状态、验收地址、更新中心关联状态 |
| 验收事项清单 | 从缺陷和提交总结出的 3-8 条验收事项，每条有预期结果和证据位置 |

没有这三段，报告不能证明“这个提交修复了这个缺陷”，只能证明“某个页面被看过”。

## 自动化任务与技能分工

自动化任务只应该做调度壳：

- 固定正式缺陷系统 domain。
- 读取长期授权 K。
- 指定工作目录和目标日期。
- 调用 `ai-defect-resolve`。
- 输出 runId、结果摘要和阻塞项。

技能负责业务闭环：

- 自检连接器。
- 领取缺陷。
- 评论计划。
- 判断轻重。
- 修复和验证。
- 提交和创建 PR。
- 回写 commit/PR/trace。
- 发布后触发视觉验收和通知。

## 完成定义

一次缺陷闭环只有同时满足以下条件才算完成：

| 阶段 | 完成条件 |
|------|----------|
| 领取 | run item 绑定单个正式缺陷 |
| 修复 | 轻量缺陷完成代码修改和至少一条有效自测 |
| 提交 | 中文 commit 已创建，PR 已创建或明确待创建原因 |
| 回写 | `workflow/complete` 写回 commit/PR/trace，并生成带缺陷原文摘录的评论 |
| 展示 | 更新中心 commit 记录能看到关联缺陷入口 |
| 发布 | 正式环境包含该 commit，发布状态可判定 |
| 验收 | 缺陷报告包含缺陷引用、提交证据和验收事项，并归档到“缺陷修复验收报告” |
| 通知 | 仅在正式发布且验收后通知提交人 |

## 与协议的关系

本规格回答“缺陷 Agent 应该如何判断和担责”。`spec.defect-agent-automation-protocol` 回答“系统之间如何通信和记录”。任何新增自动化能力都必须先判断是否属于本自治边界，再决定是否进入协议或代码。
