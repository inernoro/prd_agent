# 多文档知识库与文档类型系统 · 设计

> **版本**：v4.0 | **日期**：2026-07-17 | **状态**：已落地

## 管理摘要

- **解决的问题**：单文档会话无法支撑 PRD、技术方案、设计稿说明和参考资料之间的交叉分析。
- **核心方案**：在 Session 中绑定多份文档及其会话角色，以主文档为锚，按类型优先级和 Token 预算组装上下文。
- **当前状态**：多文档增删、类型调整、文件上传、格式检测、上下文降级和旧会话兼容均已落地。
- **已知边界**：当前是全量加载后按预算截断或摘要，不是基于 Embedding 的检索增强生成。

## 问题背景

实际评审通常需要同时参考产品需求、技术设计、交互规范和历史资料。若所有文档都无差别全文注入，会很快耗尽上下文窗口；若只保留一份主文档，又会失去交叉验证能力。

文档角色也不是文件本身的固定属性。同一份 API 说明在产品评审中可能是参考资料，在技术评审中可能是核心文档。因此角色应属于会话使用方式，而不是永久写入文档实体。

## 设计目标

1. 一次会话可绑定多份文档，并明确主文档和辅助文档。
2. 文档角色影响上下文优先级，但不改变原文档实体。
3. 超出上下文预算时优先保留重要文档和最近对话。
4. 旧单文档会话无需数据迁移即可继续使用。
5. 用户可以直接上传常见办公文档、文本、代码和配置文件。
6. 图片、音视频和无法安全识别的二进制文件应明确拒绝。

## 核心决策

| 决策 | 说明 |
|------|------|
| 主文档是会话锚点 | AI 的回答围绕主文档展开，辅助文档用于补充和校验 |
| 类型属于 Session | 同一文档在不同会话中可以承担不同角色 |
| 兼容旧字段 | `DocumentIds` 为空时回退到旧 `DocumentId` |
| 类型驱动优先级 | `product` 最高，`technical` 与 `design` 次之，`reference` 最后 |
| 预算不足时摘要 | 使用标题、章节目录和前文片段，不额外调用 LLM 生成摘要 |
| 单文档保持旧路径 | 只有多文档会话进入类型加权预算逻辑，降低行为漂移 |

## 数据设计

### Session 绑定

| 字段 | 用途 |
|------|------|
| `DocumentId` | 旧版主文档字段，继续用于兼容 |
| `DocumentIds` | 当前会话绑定的文档顺序 |
| `DocumentMetas` | 文档 ID 与会话角色的映射 |

### 文档角色

| 类型 | 含义 | 默认优先级 |
|------|------|------------|
| `product` | 产品需求或当前主文档 | 最高 |
| `technical` | 技术方案、接口或实现约束 | 中 |
| `design` | 视觉、交互或体验规范 | 中 |
| `reference` | 背景、竞品与一般参考资料 | 低 |

旧会话只有 `DocumentId` 时，该文档视为 `product`。存在多文档列表但缺少元数据时，主文档默认为 `product`，其他文档默认为 `reference`。

## 上下文组装

### 预算分配

当前聊天链路按约 100K Token 上下文预算组织输入：

| 区域 | 预算 | 策略 |
|------|------|------|
| 文档 | 60K | 按文档角色优先全文注入，预算不足时降级摘要 |
| 历史 | 30K | 从最新消息向前保留，达到预算后停止 |
| 当前消息与系统提示 | 约 10K 预留 | 由调用链保留空间，不在文档组装器中裁剪 |

预算是工程估算，不是模型分词器的精确硬上限。字符到 Token 使用近似换算，因此仍需通过模型层的上下文保护处理极端输入。

### 文档注入顺序

1. 加载会话绑定的全部可访问文档。
2. 计算每份文档角色、原始顺序和估算 Token。
3. 按角色优先级决定谁先获得全文预算。
4. 预算充足时注入全文；不足时改为标题、章节目录和前文片段。
5. 最终仍按用户绑定顺序输出，避免上下文位置随预算变化而跳动。
6. 加入按预算裁剪后的最近对话和当前用户消息。

多文档上下文使用明确边界标记区分每份资料，并携带标题、类型和是否摘要，帮助模型识别来源。引用提取和展示逻辑不得依赖未声明的文档顺序猜测来源。

## 文件上传与格式检测

### 处理顺序

文件上传采用三阶段判断：

1. 浏览器提供文件名、扩展名和 MIME；服务端对常见办公格式补充 MIME 推断。
2. PDF、Word、Excel、PowerPoint 等已知二进制文档交给专用内容提取器。
3. 其他文件尝试按 UTF-8 或带 BOM 的 UTF-16、UTF-32 文本读取，并通过空字节与不可打印字符比例排除二进制内容。

提取成功后仍需经过统一 `DocumentValidator`，再保存为文档并追加到会话。

### 支持边界

| 类别 | 处理 |
|------|------|
| PDF、Word、Excel、PowerPoint | 通过专用提取器读取文本 |
| Markdown、纯文本、代码、JSON、YAML 等 | 通过文本探测读取 |
| 图片 | 明确拒绝，不作为知识库文档上传 |
| 音视频 | 明确拒绝 |
| 可执行文件或未知二进制 | 文本探测失败后拒绝 |
| 空文件或无有效文本 | 拒绝 |

单文件上限为 20 MB。扩展名与 MIME 只用于路由，是否能获得有效文本由服务端最终判定。

## API 契约

| 方法 | 路径 | 用途 |
|------|------|------|
| `POST` | `/api/v1/sessions/{sessionId}/documents` | 以文本内容追加文档 |
| `POST` | `/api/v1/sessions/{sessionId}/documents/upload` | 上传文件并追加到会话 |
| `DELETE` | `/api/v1/sessions/{sessionId}/documents/{documentId}` | 从会话移除辅助文档 |
| `PATCH` | `/api/v1/sessions/{sessionId}/documents/{documentId}/type` | 修改会话中的文档角色 |
| `GET` | `/api/v1/sessions/{sessionId}` | 返回文档列表和角色元数据 |

移除文档时至少保留一份文档。若移除当前主文档，服务层选择列表中的下一份作为兼容主文档，并同步旧字段。

## 前端交互

桌面端知识库页面负责：

- 展示会话内全部文档与角色。
- 追加文本或上传文件。
- 修改文档角色。
- 移除非唯一文档。
- 在刷新后从 Session 响应恢复角色映射。

文件状态必须区分读取、上传、成功和失败。客户端的扩展名提示只用于提前反馈，服务端响应才是格式支持的最终结论。

## 风险与边界

| 风险 | 当前处理 | 后续方向 |
|------|----------|----------|
| 文档数量和体积继续增长 | 角色优先级、全文预算和摘要降级 | 引入分块、Embedding、召回与重排 |
| 摘要只保留目录和前文 | 明确标记为截断内容 | 支持按问题检索相关章节 |
| Token 为近似估算 | 预留输出空间并限制输入区域 | 接入与模型一致的分词器 |
| 文档类型粒度有限 | 四种稳定角色 | 在技能上下文中按类型筛选或加权 |
| 引用主要围绕主文档 | 保持既有引用体验 | 扩展多文档来源定位与证据展示 |

RAG、Embedding、语义检索及技能级文档类型过滤仍属于知识库债务，统一记录在 `debt.knowledge-base.md`，不在本设计中标成已完成。

## 验证重点

1. 旧单文档会话打开和对话行为不退化。
2. 新增、上传、改类型和移除文档后，前后端列表与角色一致。
3. 多文档超预算时，高优先级文档保留全文，低优先级文档出现摘要标记。
4. 长历史优先保留最近消息，输出顺序仍从旧到新。
5. 办公文档、UTF 文本和代码文件可提取，图片、音视频与二进制文件被明确拒绝。
6. 无权限用户不能通过会话 API读取或追加文档。
