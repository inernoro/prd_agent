# CDS 三种部署方式 · 指南

> **版本**：v1.0 | **日期**：2026-05-12 | **状态**：已落地

本文用于交给新同学按步骤完成 CDS 项目接入。目标是像 Railway 一样：先看清来源，再执行部署，最后用预览页和日志确认结果。

## 部署前检查

| 检查项 | 怎么看 | 通过标准 |
|--------|--------|----------|
| 仓库权限 | 项目设置的 GitHub 区域 | 已绑定仓库或可用 Git 地址 |
| 默认分支 | 项目卡片、GitHub 仓库页面或远程分支列表 | 使用真实默认分支，不假设一定是 `main` |
| 环境变量 | 项目设置的环境变量区域 | 必填项已填写，没有 TODO 占位 |
| 构建配置 | 项目设置的构建配置区域 | 每个应用服务都有 image、workDir、command、port |
| 基础设施 | 拓扑页的 infra 节点 | 数据库、缓存、对象存储可启动并能看日志 |

## 方式一：从 `cds-compose.yml` 创建

适用：项目已经有 CDS 配置文件，或者团队已经评审过配置。

1. 打开 CDS 项目列表。
2. 点击新建项目，选择已有仓库或项目目录。
3. 确认仓库根目录存在 `cds-compose.yml`。
4. 进入项目设置，查看导入预览。
5. 确认应用服务和基础设施都被识别。
6. 批准导入申请。
7. 回到分支列表，选择目标分支。
8. 点击分支卡片打开详情抽屉。
9. 在部署页签点击重新部署。
10. 部署完成后点击预览地址。
11. 切到日志页签确认构建日志和容器日志没有错误。
12. 切到 Webhook 日志确认本次是否来自 GitHub push；如果没有命中，来源判断应显示为手动操作。

验收标准：

| 项目 | 期望 |
|------|------|
| 分支卡片 | 显示来源、提交、服务端口、最近时间 |
| 详情抽屉 | 显示来源判断、部署记录、服务状态 |
| 预览页 | 返回 200 或显示明确的 CDS 等待/失败页 |
| 日志 | 能看到构建、容器、HTTP、Webhook 四类线索 |

## 方式二：从 CDS 技能扫描创建

适用：项目没有 `cds-compose.yml`，但代码结构能被扫描器识别。

1. 在项目根目录执行 CDS 技能扫描。
2. 阅读扫描结果里的 signals，确认识别到的技术栈、端口、包管理器。
3. 如果扫描出 pending import，进入 CDS 项目设置批准导入。
4. 检查自动生成的 build profile。
5. 对 Node 项目确认 lockfile 对应的包管理器：`pnpm-lock.yaml` 使用 pnpm，`package-lock.json` 使用 npm。
6. 对 Java 项目确认 Maven/Gradle 命令能在仓库根目录运行。
7. 启动基础设施。
8. 触发分支部署。
9. 打开预览页。
10. 通过服务日志确认应用真正监听端口，而不是只有容器进程存活。

验收标准：

| 项目 | 期望 |
|------|------|
| 扫描结果 | 没有把 npm 项目误判成 pnpm 项目 |
| 导入配置 | 没有泄漏 token，没有 TODO 环境变量直接上线 |
| 服务状态 | 前端/后端/infra 分开显示，不混成一个绿色状态 |
| 失败归因 | 代码侧、配置侧、CDS 侧能从日志中区分 |

## 方式三：从 0 创建 Railway 模式项目

适用：用户只知道 Git 仓库地址，希望从 UI 一步步部署。

1. 打开 CDS 项目列表。
2. 点击“一键部署项目”。
3. 输入 Git 仓库地址，或点击“从 GitHub 选择”。
4. 确认默认分支。留空时 CDS 会 clone 后读取远端默认分支；如果仓库默认分支是 `master`，不要手动输入 `main`。
5. 选择运行环境：
   - `自动识别`：仓库里已有 `cds-compose.yml` / `docker-compose.yml`，或希望 CDS 根据代码特征扫描。
   - `Node.js`：Vite、Next.js、Express、NestJS 等项目。
   - `Python`：FastAPI、Flask、Django 或脚本服务。
   - `.NET`：ASP.NET Core 或 Worker 服务。
   - `Java`：Spring Boot 或 Maven 项目。
   - `Go`：Gin、Fiber、Echo 或标准库 HTTP 服务。
   - `Rust`：Axum、Actix Web、Rocket 等 Rust 服务。
   - `PHP`：Laravel、Symfony 或 public 目录站点。
   - `静态站点`：Vite、React、Vue、文档站等前端构建产物。
   - `Dockerfile`：仓库自带 Dockerfile 时进入 Dockerfile / compose 扫描路径。
   - `自定义`：手动填写 Docker 镜像、启动命令和端口。
6. 选择基础设施。可一次勾选 MongoDB、PostgreSQL、MySQL、Redis；CDS 会创建持久化卷和连接环境变量。
7. 创建项目后等待 clone 完成。
8. 环境变量弹窗出现时，补齐必填项；由 CDS 生成的数据库连接串可直接保留。
9. 打开分支页，先刷新远程分支。
10. 选择带“默认分支”标记的远程分支。
11. 点击远程分支创建 CDS 分支。
12. 进入详情抽屉，检查来源判断、提交 ID 和部署目标。
13. 触发部署。
14. 打开预览页。
15. 如果失败，先看部署页签，再看容器日志，最后看项目设置。

验收标准：

| 项目 | 期望 |
|------|------|
| 创建项目 | 用户不用写 YAML 也能完成 clone 和扫描 |
| 分支选择 | UI 能看出哪些是远程分支、哪些已被 CDS 跟踪 |
| 手动部署 | 卡片、抽屉、完整页都能找到重新部署入口 |
| 可观察性 | 可以解释本次部署来自 webhook、手动操作，还是待配置 |

## 手动新增基础设施

适用：项目已经存在，但后来需要新增数据库、缓存或其他容器。

1. 打开项目拓扑页。
2. 在“基础设施”区点击“新增基础设施”。
3. 选择预设：MongoDB、PostgreSQL、MySQL、Redis，或选择自定义。
4. 填写服务 ID。建议使用稳定英文小写，例如 `mongodb`、`postgres`、`redis`。
5. 填写 Docker 镜像和容器端口。
6. 填写环境变量，每行一个 `KEY=value`。
7. 如果希望立刻拉起容器，勾选“创建后立即启动”。
8. 创建后在拓扑图里点击该基础设施节点。
9. 在右侧详情里执行启动、停止、重启、查看日志。
10. 如果是数据库服务，进入数据库操作入口查看库表、运行查询或初始化脚本。

验收标准：

| 项目 | 期望 |
|------|------|
| 拓扑图 | 新增 infra 节点立即出现 |
| 操作入口 | 右侧详情能启动、重启、停止、查看日志 |
| 数据安全 | MySQL、PostgreSQL、MongoDB、Redis 默认带持久化卷 |
| 错误归因 | 启动失败时在节点和详情里显示错误，不和 CDS 自身故障混淆 |

## Railway 对齐点

| Railway 体验 | CDS 当前复刻 | 验收方式 |
|--------------|--------------|----------|
| New Project 先选 GitHub 仓库 | 项目列表“一键部署项目”支持粘贴 URL 和 GitHub 选择器 | 创建项目前能看到仓库来源 |
| Deploy from GitHub 后选择运行环境 | 同一弹窗内选择 Node.js、Python、.NET、Java、Go、Rust、PHP、静态站点、Dockerfile、自定义或自动识别 | clone 后能自动生成 BuildProfile |
| Add Database 在项目画布里直接添加 | 拓扑页“新增基础设施”直接添加数据库或缓存 | infra 节点出现在拓扑图，可启动和看日志 |
| 变量和数据库连接串自动注入 | 创建项目时生成 DATABASE_URL、MONGODB_URL、REDIS_URL 等项目环境变量 | 环境变量弹窗显示为基础设施推导 |
| 服务卡片能看到运行状态和日志 | 分支详情、拓扑节点、Webhook 日志、服务日志分开展示 | 能区分代码错误、配置错误、CDS 错误 |

## 常见失败处理

| 现象 | 优先判断 | 处理方式 |
|------|----------|----------|
| Webhook 日志为空 | 分支可能是手动创建 | 看来源判断，不要直接认定 webhook 坏了 |
| 容器 running 但页面 502 | HTTP 还没就绪或端口不对 | 直连服务端口，再看容器日志 |
| 默认分支不是 main | 仓库可能使用 master 或其他名称 | 看项目卡片的默认分支和远程列表标记 |
| Java 项目缺 SNAPSHOT 依赖 | 项目构建命令不完整 | 在项目根目录先安装模块，再运行服务 |
| npm 项目被 pnpm 启动 | 扫描器包管理器判断错误 | 按 lockfile 修正 build profile |
| 数据库离线 | infra 没启动或健康检查失败 | 在拓扑页直接启动并查看日志 |

## 最低交付口径

一次 CDS 部署不能只说“已运行”。必须同时给出：

| 证据 | 示例 |
|------|------|
| 版本 | 当前 CDS commit 短 SHA |
| 来源 | Webhook、手动操作、待配置 |
| 预览地址 | `https://{branch}-{project}.miduo.org/` |
| 服务明细 | 每个 profile 的端口和状态 |
| 日志入口 | 构建日志、容器日志、Webhook 日志 |
| 下一步 | 失败时给出代码侧、配置侧或 CDS 侧归因 |
