# 缺陷管理标签体系 · 设计

> **版本**：v1.0 | **日期**：2026-05-12 | **状态**：规划中

## 背景

缺陷协作需要一个比状态更轻、更灵活的标记层。状态表示流程位置，例如“待处理、处理中、待验收”；标签表示协作语义，例如“AI 正在跟进、需要补充信息、复现不稳定”。标签的目标不是替代状态，而是让人类和 AI 快速判断“当前谁在看、缺什么、下一步是什么”。

本设计面向 `prd-admin` 与 `prd-desktop` 同步展示，后端作为唯一权威来源。

## 目标

1. 支持任何有缺陷访问权限的人为缺陷打标签、撤销标签、清空标签。
2. 支持新增标签，但新增的是“枚举项”，不是一次性的自由文本。
3. 标签必须枚举化，避免自由输入造成语言不统一。
3. 支持“AI 正在跟进”这类协作标签，让人类知道 AI 已接手。
4. 标签展示在缺陷标题下方第二行，不挤占标题本身。
5. 管理后台和桌面端使用同一份标签枚举与展示规则。

## 非目标

1. 本版本不做复杂工作流自动流转。
2. 本版本不做自定义任意文本标签。
3. 本版本不做跨项目标签权限隔离。
4. 本版本不把标签作为关闭、验收、删除的强约束条件。

## 标签模型

后端在 `DefectReport` 上扩展标签字段：

```csharp
public List<DefectLabel> Labels { get; set; } = new();
```

标签项结构：

```csharp
public class DefectLabel
{
    public string Code { get; set; } = string.Empty;
    public string Name { get; set; } = string.Empty;
    public string Group { get; set; } = string.Empty;
    public string ColorToken { get; set; } = string.Empty;
    public string AddedById { get; set; } = string.Empty;
    public string AddedByName { get; set; } = string.Empty;
    public DateTime AddedAt { get; set; } = DateTime.UtcNow;
}
```

`Code` 是稳定枚举值，前后端以 `Code` 判断逻辑。`Name` 仅用于显示，可跟随语言统一调整。

## 枚举定义

第一版内置标签如下：

| Code | 名称 | 分组 | 用途 |
|------|------|------|------|
| `ai-following` | AI 正在跟进 | 协作 | AI 已接手分析或修复，人类可避免重复处理 |
| `human-following` | 人工正在跟进 | 协作 | 人类已接手处理 |
| `needs-info` | 需要补充信息 | 阻塞 | 缺少截图、日志、复现步骤或账号环境 |
| `cannot-reproduce` | 暂未复现 | 阻塞 | 当前无法复现，不应盲目修改 |
| `ready-to-fix` | 可直接修复 | 判断 | 根因清晰、风险低，可进入修复 |
| `needs-confirmation` | 需要确认 | 判断 | 涉及破坏性、体验取舍或需求边界 |
| `frontend` | 前端 | 模块 | 主要影响前端 |
| `backend` | 后端 | 模块 | 主要影响后端 |
| `desktop` | 桌面端 | 模块 | 需要同步桌面端 |
| `ci-deploy` | 构建发布 | 模块 | 与 CI、发布、缓存或版本可见性相关 |

标签枚举集中定义，建议后端提供 `GET /api/defect-agent/labels/catalog` 返回完整列表，前端仅消费，不重复硬编码业务语义。

## 枚举维护

新增标签不是直接把任意文本贴到某个缺陷上，而是先创建一个全局枚举项，再把该枚举项应用到缺陷。这样既满足“可以新增任何标记”，又保证之后所有人看到的是同一套语言。

新增枚举项字段：

| 字段 | 说明 |
|------|------|
| `Code` | 稳定编码，由后端按名称生成，也允许管理员手动调整 |
| `Name` | 展示名称，例如“AI 正在跟进” |
| `Group` | 分组，只能从协作、阻塞、判断、模块中选择 |
| `ColorToken` | 颜色 token，由分组默认推导 |
| `IsActive` | 是否可用，停用后不再出现在新增菜单，但历史缺陷仍可展示 |
| `CreatedById` | 创建人 |
| `CreatedAt` | 创建时间 |

新增规则：

1. 所有可查看缺陷的人都可以新增枚举标签。
2. 新增时必须选择分组。
3. 后端按 `Name` 做去空格、去重复、同义词提示，避免出现“AI 跟进中”和“AI 正在跟进”两套语言。
4. 已存在同名或近似名称时，前端提示用户复用已有标签。
5. 停用标签只影响后续选择，不删除历史展示。

## 权限规则

1. 可查看该缺陷的人可以添加标签。
2. 可查看该缺陷的人可以撤销任意标签。
3. 可查看该缺陷的人可以清空所有标签。
4. 所有标签操作写入审计字段，并作为缺陷对话系统消息展示。
5. 删除缺陷时标签随缺陷软删除，不单独保留可见入口。

这里有意保持低门槛，因为标签的价值在于快速协作。如果后续出现滥用，再增加“仅管理员可维护标签枚举”，而不是限制打标动作。

## API 设计

```http
GET /api/defect-agent/labels/catalog
```

返回当前可用标签枚举。

```http
POST /api/defect-agent/labels/catalog
Content-Type: application/json

{
  "name": "AI 正在跟进",
  "group": "collaboration"
}
```

新增一个全局枚举标签。

```http
POST /api/defect-agent/defects/{id}/labels
Content-Type: application/json

{
  "code": "ai-following"
}
```

为缺陷添加标签。若已存在同 `Code` 标签，接口幂等返回当前缺陷。

```http
DELETE /api/defect-agent/defects/{id}/labels/{code}
```

撤销指定标签。

```http
DELETE /api/defect-agent/defects/{id}/labels
```

清空全部标签。

所有接口返回 `{ defect }`，前端用返回的缺陷对象更新列表和详情。

## 前端展示

缺陷列表每一行分两层：

1. 第一行：严重级别、编号、标题、状态、人员、时间。
2. 第二行：标签 chip 列表，最多展示 4 个，超出显示 `+N`。

标签 chip 使用统一颜色 token，不使用自由颜色：

| 分组 | 视觉 |
|------|------|
| 协作 | 蓝色系 token |
| 阻塞 | 红色系 token |
| 判断 | 黄色系 token |
| 模块 | 灰色系 token |

管理后台卡片视图、表格视图、看板卡片都使用同一 `DefectLabelChips` 组件。

## 桌面端同步

`prd-desktop` 不维护独立标签枚举，直接复用后端返回字段与共享类型：

1. 缺陷列表第二行显示标签。
2. 缺陷详情标题区显示标签。
3. 标签操作使用同一组 API。
4. 离线或 API 失败时保留原列表，不做本地乐观清空。

桌面端必须覆盖“AI 正在跟进”和“需要补充信息”两个最常用标签，避免桌面用户看不到协作状态。

## 操作入口

1. 缺陷详情标题区增加“标签”按钮，打开枚举菜单。
2. 列表行第二行标签支持点击打开同一个菜单。
3. 菜单内支持勾选/取消勾选。
4. 菜单底部提供“清空标签”按钮。
5. 评论发送区旁可提供快捷按钮：“标记 AI 正在跟进”。

## 与评论的关系

标签用于状态概览，评论用于解释原因。关键标签变更建议自动生成系统消息：

| 操作 | 系统消息 |
|------|----------|
| 添加 `ai-following` | 已标记为“AI 正在跟进” |
| 添加 `needs-info` | 已标记为“需要补充信息” |
| 撤销标签 | 已撤销标签“标签名” |
| 清空标签 | 已清空全部标签 |

这样列表可快速扫，详情可追溯是谁在什么时候打过标记。

## 实施顺序

1. 后端扩展 `DefectReport.Labels`，新增标签枚举与 4 个 API。
2. 管理后台新增共享类型、service、`DefectLabelChips` 组件。
3. 管理后台列表和详情第二行展示标签。
4. 管理后台详情菜单支持添加、撤销、清空。
5. 桌面端同步类型、service、列表与详情展示。
6. 为 `ai-following` 添加快捷入口，并在 AI 修复流程开始时优先打标。

## 验收标准

1. 任意可见缺陷可添加“AI 正在跟进”标签。
2. 刷新页面后标签仍在。
3. 同一标签重复添加不会重复显示。
4. 撤销单个标签后列表和详情同步消失。
5. 清空标签后列表第二行消失或展示为空状态。
6. `prd-admin` 与 `prd-desktop` 展示一致。
7. 标签变更会在缺陷对话中留下系统消息。

## 风险与边界

1. 标签枚举需要产品语言统一，不能让前端随意新增显示文案。
2. 标签操作如果每次都发通知，可能造成噪声；第一版建议只写系统消息，不额外推送通知。
3. “AI 正在跟进”如果 AI 异常退出，可能悬挂；后续可补“超过 24 小时未更新自动转为需要人工确认”。
