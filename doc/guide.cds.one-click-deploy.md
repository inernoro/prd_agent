# CDS 一键可视化部署 · 指南

> **版本**：v2.0 | **日期**：2026-07-15 | **状态**：已落地

一键部署适合手上只有 Git 仓库、希望先把项目跑起来的用户。新版流程是“选择仓库、自动识别、确认结果、观察部署、打开预览”，不再要求用户先准备 Compose、`.env` 文件或 Docker 命令。

## 开始前需要什么

- 一个可以访问的 Git 仓库；
- CDS 登录权限；
- 私有仓库对应的 GitHub App 单仓库授权；
- 对项目核心入口和必要环境信息的基本了解。

不需要把 GitHub Token 或 CDS 长期 Key 粘贴到对话中。

## 第一步：打开一键部署

在项目列表点击“一键部署项目”。

如果希望由 Codex、Cursor 或 Claude Code 代为操作，先点击“接入 Agent”，把无密钥接入口令交给 Agent，并在页面批准。

## 第二步：选择仓库

可以直接粘贴仓库地址，也可以从 GitHub 已授权仓库中选择。

私有仓库授权按项目复用。完成一次单仓库授权后，仓库检测、项目创建和后续部署不需要重复粘贴凭据。

默认分支以仓库真实设置为准，不假设一定是 `main`。

## 第三步：自动识别项目

点击“检测仓库并自动填好配置”。CDS 会分析真实仓库，尝试填好：

- 应用服务和工作目录；
- 前端、后台和后台任务角色；
- 安装、构建和启动方式；
- 端口和健康检查；
- 数据库、缓存、消息队列等依赖；
- 已有 Compose 或 Dockerfile。

识别结果会显示依据和把握程度。把握高时仍应快速检查；把握中或低时应先做试运行。

标准 Compose 项目如果只声明构建方式，没有写启动命令，CDS 会继续从项目清单推导。无法安全判断时才要求用户补充，不会用猜测覆盖复杂项目逻辑。

## 第四步：检查服务和依赖

### 应用服务

每个服务重点确认：

- 名称和角色；
- 工作目录；
- 镜像或运行环境；
- 启动命令；
- 容器端口；
- 健康检查路径。

多入口项目可以保留多个服务；只有单体服务时关闭不需要的默认服务。

### 基础设施

按项目实际需要选择数据库、缓存、消息队列、搜索或对象存储。CDS 会准备持久化卷和对应连接信息。

自动生成的密码不会在普通日志中明文展示。业务自己的环境信息在项目设置中维护，不写进 Agent 接入口令。

## 第五步：试运行验证

点击服务卡片上的“试运行验证”。CDS 会在一次性环境中检查仓库、镜像、命令和端口。

| 结果 | 怎么处理 |
| --- | --- |
| 配置可用 | 直接创建项目 |
| 需确认 | 检查启动耗时、端口和健康路径 |
| 不通过 | 根据失败原因修改目录、镜像或命令后重试 |

试运行结束后临时环境会清理，不会把测试容器当成正式部署。

## 第六步：创建并观察部署

点击创建后，CDS 会连续完成项目准备、构建、启动和检查。页面会持续显示当前阶段和每个服务的结果。

刷新页面或短暂离开后，重新打开分支详情仍然可以继续查看同一条部署记录。如果失败，先按页面给出的问题归属和处理建议修改，不要连续重复点击部署。

## 第七步：打开预览

部署成功后点击“打开预览”，完成真实用户验证：

1. 打开最终功能页；
2. 完成登录或身份校验；
3. 执行一个核心操作；
4. 验证后台接口和依赖；
5. 刷新页面确认结果仍然正确。

## 以后怎么使用

### 推送代码

已连接 GitHub 的项目在推送目标分支后自动部署。CDS 会记录触发来源、提交和部署过程。

### 重新部署同一版本

服务需要恢复但代码没有变化时，在“不可变部署版本”中选择当前成功版本，点击“重新部署此版本”。可复用版本不需要重新构建。

### 恢复旧版本

新版本异常时，选择以前的成功版本并点击“回滚到此版本”。恢复会产生新的部署记录，完成后重新打开预览验证。

## 常见问题

| 问题 | 解答 |
| --- | --- |
| 检测不到项目 | 检查仓库权限、默认分支和工作目录，再手工补充最小信息 |
| 私有仓库反复要求凭据 | 检查项目是否绑定 GitHub App 单仓库授权 |
| Compose 没写启动命令 | 标准项目会从项目清单继续推导；复杂项目保留手工控制 |
| 容器运行但页面打不开 | 检查端口、健康路径和服务日志 |
| 页面刷新后进度不见了 | 从分支详情打开对应部署记录 |
| 新版本有问题 | 选择以前的成功版本恢复，不要手工找旧代码重建 |
| 托管模式不适合项目 | 切回 Compose，原配置不会被删除 |

## 完成标准

- 仓库和默认分支正确；
- 自动识别结果经过确认；
- 试运行或真实部署证明服务可启动；
- 部署过程可以刷新后续看；
- 预览地址完成核心操作；
- 至少有一个可识别的成功版本；
- 出现问题时知道如何恢复。
