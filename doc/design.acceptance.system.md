# 功能验收体系 · 设计

> **版本**：v1.1 | **日期**：2026-05-26 | **状态**：方案评审中（v2 已落地，v3 业界对标待批准融合）
> **关联实现**：`.claude/skills/create-visual-test-to-kb/`（可执行 SSOT）
> **一句话**：把"凭感觉验收 + 随手写报告"升级为业界对标、可追溯、可沉淀的标准化验收体系——并让它成为质量资产，越用越值钱。

---

## 1. 管理摘要

验收是交付的最后一道闸。当开发越来越多由 AI 完成、改动以十倍速涌入时，瓶颈不再是"写代码"，而是"凭什么信这段代码能用"。一句"我测了没问题"无法证伪、无法检索、无法复盘——它不是验收，是口头承诺。

本设计把验收从"口头承诺"升级为"可审计的资产"：用一套**标准**定义测什么、怎么取证、怎么下结论；用一个**模拟人类的取证引擎**走真实用户路径取证；用一套**归档机制**把证据与结论沉淀进知识库，可检索、可对比、可回归。当前 v2 已落地（自评约 90 分），本文同时给出对标 IEEE 829 / ISO·IEC·IEEE 29119-3 / 29148 / ISTQB / BDD 的 **v3 演进**，并展望它如何从"一次性报告"长成"质量资产库"。

读者只需记住一句：**验收不是"我点了一下没报错"，而是"按标准走真人路径、留可追溯证据、给可推导的结论"。**

---

## 2. 立意：验收是交付的最后一道闸，也是信任的凭证

软件交付里有一个长期被默许的漏洞：**验收的质量等于验收者的自觉**。自觉好的人测得细，自觉差的人点一下就过；而无论哪种，结论都只活在那个人的脑子里和聊天记录里——别人无法复核，事后无法追溯。

这个漏洞在 AI 时代被放大成主要矛盾：

- **改动量爆炸**：当 Agent 一天提交几十次，人工逐一手测不可能跟上。验收必须能被机器执行、被人快速复核。
- **信任必须可审计**："相信我，测过了"在一次性交付里勉强能用，在持续交付里就是定时炸弹。信任的载体必须从"人的话"变成"可查的证据"。
- **角色在迁移**：人的精力从"自己测"转向"复核 AI 给的结论"。这要求验收产出**两端友好**——AI 能忠实地产出（走真实路径、不自欺），人能几秒钟读懂（结论 + 证据一目了然）。

所以这套体系的立意不是"写个更漂亮的报告模板"，而是：**把验收变成一种可执行、可审计、可累积的基础设施**，让信任有凭证、让质量可度量、让经验能沉淀。

## 3. 设计哲学：四条原则

四条原则是整套体系的地基。它们不是口号，每一条都对应一个"如果不这样会怎样"。

**一、模拟人类——保真度决定能发现什么。**
一个测试的价值，取决于它多忠实地复现用户的真实路径。绕过导航、用深链直达页面的测试，验证的是"这个函数存在"，而不是"用户能找到并用上它"。后者才是"功能是否真的交付"。坚持登录后点击导航进入，是为了让取证撞上真人会撞的那一类问题（入口没进菜单、路径断裂、跳转丢状态）——这类问题，纯接口测试和深链测试在结构上就发现不了。

**二、人类可读优先——报告是给人做决策的。**
报告的第一消费者是要拍板"能不能上"的人。机器字段服务于工具，但首屏必须让人在几秒内拿到结论：状态词（通过/不通过）、一句话结论、一张速览卡。把 YAML 元数据怼在最前面，是把工具的方便凌驾于人的决策之上——本末倒置。

**三、证据可追溯——没有证据的结论只是观点。**
每一条结论都挂一张写明"这张证明了什么"的截图。这一步把验收从"相信我"变成"你自己看"。它让验收可被任何人复核，也让报告在归档后仍然自证——三个月后回看，证据还在。

**四、定义下限——诚实地说清"保证了什么、没保证什么"。**
没有人能穷举未知缺陷（这近乎停机问题）。但我们可以保证一条**下限**：已知的质量维度（硬约束、档位要求）全部覆盖、不漏检。验收是"覆盖下限"，不是"完备上限"。把这条说破，比假装"测全了"更负责任，也更可执行。

> **原则是踩出来的，不是拍脑袋。** 早期建设中撞过几类具体坑，每一类都固化成了上面原则的一条硬规则——此处只留教训，细节根因见 git 历史与 changelog，不在设计文档里铺陈：
>
> | 踩过的坑（一句话） | 固化成的规则 |
> |---|---|
> | 报告在目录里显示成 `---`，分不清哪份 | 正文必须以 `# 标题` 打头，机读字段挪文末注释 |
> | 深链直达，测不出"入口没进菜单" | 必须点击导航进入；点不到入口本身就是缺陷 |
> | 给一种不存在的主题硬截一张图 | 能力不存在就如实标注，不假装 |
> | 把打不开正文的链接当交付 | 交付前必须自己走一遍查看路径 |

## 4. 系统架构：标准 → 取证 → 归档

整套体系是一条不可分的三段流水线，固化在 `create-visual-test-to-kb` 技能里。三段各司其职，数据单向流动：

```
┌─────────┐      ┌──────────────────┐      ┌─────────────────┐
│  标准    │ ───> │  取证(模拟人类)    │ ───> │  归档(证据→资产)  │
│ 测什么   │ 档位 │ 登录→点击导航→操作 │ 截图 │ 内联证据+结论    │
│ 怎么截   │ 规范 │ →带 caption 截图   │ 清单 │ →写入知识库      │
│ 怎么判   │      │ →输出 manifest     │      │ →出查看路径      │
└─────────┘      └──────────────────┘      └─────────────────┘
   standard-v2.md     harness.mjs              archive_report.py
                      example-driver.mjs       (doc-store / local 双模式)
```

三个架构决策值得展开：

- **项目无关、配置驱动**：所有与具体仓库耦合的东西（预览域名、登录方式、文档空间 API、库名、截图参数）收敛到一个 `acceptance.config.json`。换一个仓库，只改这一处。这让"验收能力"成为可移植的资产，而不是绑死在某个项目里的脚本。
- **双模式落地（doc-store / local）**：有知识库的项目把报告归档进文档空间、可分享；没有知识库的项目退化为写本地 `doc/acceptance/`，零密钥零网络。同一套标准，能力强弱自适应环境——这是"不假定不存在的能力"原则在架构层的体现。
- **取证引擎是"人类模拟器"**：`harness.mjs` 暴露的不是"打开某 URL"，而是 `login / gotoByClick(可见文本) / click / shot / setTheme` 这些**以人的动作为单位**的原语。脚本读起来就是一段"人怎么操作"的剧本，而不是"程序怎么请求"的代码。

方法论上不重造轮子：测什么、怎么分级，**复用** `doc/rule.issues-system.md §5` 的 10 条硬约束、P0-P3、四轴矩阵（视口 × 主题 × 状态 × 交互），与 issues 视觉验收体系共用同一份 SSOT。本体系只补它没有的"取证要走真人路径"和"结论要沉淀成资产"。

## 5. 原理：为什么这套设计成立

讲清三个"为什么"，这套体系才不是一堆规定，而是有内在逻辑的系统。

**原理一：测试保真度（Test Fidelity）决定缺陷可见性。**
把"测试能发现什么缺陷"形式化：一个测试能发现的缺陷集合，受限于它复现真实使用路径的保真度。深链直达的保真度低（跳过了导航注册、登录态初始化、SPA 路由状态），所以"入口缺失/路径断裂"这类缺陷对它**不可见**；点击导航进入的保真度高，这类缺陷才进入可见域。结论：**要发现某类缺陷，取证路径的保真度必须覆盖产生该缺陷的环节**。这是"模拟人类"原则背后的硬道理，不是偏好。

**原理二：证据即数据（Evidence as Data）。**
报告不只是给人看的文章，它同时是结构化数据：标题里的状态、速览卡里的计数、文末注释里的 `report_id / verdict / severity`。这意味着一份报告既能被人"读"，也能被机器"查"。当报告积累成语料，它就从"一次性文档"变成"可聚合的质量数据集"——这是第 9 节"涌现"的物质基础。

**原理三：下限可保证，上限不可承诺。**
"测全了"是无法兑现的承诺；"已知维度全覆盖"是可以兑现的契约。把验收定义为"覆盖下限"而非"完备上限"，带来两个好处：一是诚实（不误导决策者以为零风险），二是可执行（下限是有限集合，可以逐项打勾）。Verdict 的推导也据此而来——它回答的是"是否达到这一档的下限"，不是"是否绝对无 bug"。

## 6. 方法与步骤：从一次改动到一份可审计报告

把原理落成可照做的步骤。一次完整验收分四步，配合按风险选择的档位。

**档位按风险选（不是越重越好）：**

| 档 | 适用 | 取证强度 | 哲学 |
|----|------|---------|------|
| **L0 冒烟** | 任何 push 的最低保障 | 1 用例 1 截图 | 防"白屏级"事故 |
| **L1 单项** | 单个改动点 | ≥3 用例，≥3 维度 | 覆盖该点的主路径 + 边界 |
| **L2 全链路** | 多功能串成用户旅程 | ≥6 用例，含回归/回滚 | 端到端跑通，模拟真实使用 |

为小改动套 L2 是过度工程，给关键链路只做 L0 是失职——**按风险匹配档位**本身就是方法的一部分。

**四步工作流：**

1. **定标准与档位**：读标准，按改动风险定 L0/L1/L2。
2. **写取证剧本**：用 `harness` 原语写一段"真人怎么操作"的 driver——登录、点击导航进入、执行、按关键步骤截图（caption 写"这张证明了什么"）。
3. **读图核对、写报告**：用多模态读回截图逐张核对（这一步抓到过"匿名未登录""按钮没渲染"等真问题），按模板写速览卡 + 各段，推导 Verdict。
4. **归档交付**：截图传 URL 内联，报告以 `# 标题` 打头写入知识库，输出 owner 可靠查看路径。

**Verdict 是推导的，不是感觉的：**

- **通过**：无未决 P0/P1，DoD 全勾，核心用例全 pass。
- **有条件通过**：仅剩 P2/P3，列明整改项。
- **不通过**：存在未决 P0/P1，或 DoD 缺项，或核心用例 fail。

把结论绑定到可核对的事实上，任何人都能复核"凭什么是这个结论"。

## 7. 业界对标：站在巨人肩上，保留独有内核

v2 自评 90 分，差的 10 分在"工业级严谨度"。对照国际标准逐项补齐，方式是在 v2 骨架上**加列、加段、加轴**，而非推倒重来：

| 业界标准 | 它要求什么 | v3 借鉴 |
|----------|-----------|---------|
| **ISO·IEC·IEEE 29119-3** 测试完成报告 | 标识 / 摘要 / 量化结果汇总 / 偏差 Variances / 对照出口准则的综合评估 / 决策依据 | 加"结果汇总"度量表（总/通过/失败/阻塞/通过率/覆盖率）+ "偏差"段 |
| **ISO·IEC·IEEE 29148** 需求质量 | 每条需求可验证、可追溯 | 加**需求可追溯矩阵 RTM**：用例 ↔ 验收准则 ↔ 状态 ↔ 缺陷 |
| **ISTQB / 29119** 入/出口准则 | 明确开始/停止测试的条件 | 形式化入口准则（前置）+ 出口准则（= DoD） |
| **BDD / Gherkin** | Given-When-Then 可执行可读 | 核心场景用 Given-When-Then 表达，精确无歧义 |
| **缺陷管理通行做法** | Severity（技术影响）与 Priority（修复优先级）分离 | 缺陷拆 Severity × Priority 两轴，再映射处置 |

> 这些是业界共识，拿来即用。但有一点业界模板都没有、是我们的独有内核：**它们只规定"文档长什么样"，不规定"证据怎么取"**。"模拟人类走真实路径取证 + 防断头交付"是我们比纯模板更强的地方——模板保证报告好看，我们保证报告**说的是真的**。

v3 报告骨架（29119-3 对齐）与状态前置标题已在标准与配置中固化，细节见技能 `reference/` 与 `acceptance.config.json`。

## 8. 变革：从"我测了"到"可审计的验收资产"

这套体系真正改变的，是团队的交付习惯：

| | 过去 | 现在 |
|---|------|------|
| 验收结论 | "我测了没问题"（口头、不可证伪） | 带证据的报告（可复核、可追溯） |
| 存放 | 聊天记录 / PR 评论 / 散落 HTML | 知识库里的标准条目（可检索） |
| 复核成本 | 别人无法复核，只能再测一遍 | 几秒读懂 Verdict + 证据 |
| 取证可信度 | 截图随手截，不知证明啥 | 每张图说明"证明了什么"，走真人路径 |
| AI 参与 | AI 说"我测了"同样不可信 | AI 走真实路径取证，人只复核结论 |

变革的核心是**信任的载体变了**——从"人的话"变成"可查的证据"。这让验收第一次具备了"基础设施"的属性：可被复用、可被审计、可被累积。

## 9. 涌现：验收报告是质量资产，不只是一次性文档

单份报告是一次交付的快照；但当标准化报告累积成语料，量变会引起质变——它从"文档"长成"质量资产库"。这正是"证据即数据"原理的兑现：

- **回归基线**：一份 L2 报告记录了"上次这个功能用这些用例、这些证据通过"。下次改动后重新跑同一套剧本，差异即回归信号。报告即可执行的回归基线。
- **覆盖热力图**：聚合所有报告的档位与维度，立刻看出"哪些功能有 L2 验收、哪些只做过 L0"——质量的盲区一目了然。
- **场景模式库**：登录、分享流、CRUD、SSE 流式……这些反复出现的验收场景，会自然沉淀成可复用的 Given-When-Then 剧本模板。新功能验收时，不再从空白起步，而是组合已有剧本。
- **经验复利**：报告越多，语料越能回答"我们通常怎么验这类功能"，单次验收的边际成本越低。第一份报告很贵，第一百份几乎免费——这就是涌现带来的复利。

一句话：**我们不是在写报告，是在养一座会自己增值的质量资产库。**

## 10. 未来：验收的演进路线

沿着"基础设施 + 资产库"两条线，验收可以长到比今天大得多。下面每一条都接得上本平台已有的能力，不是空想：

1. **AI 自主验收闭环**：Agent 改完代码 → CDS 自动部署预览 → 取证引擎走真实路径自动验收 → 归档 → 人只复核 Verdict。把"开发→验收"从串行的人工接力，变成 AI 自跑、人抽检的闭环。（接 CDS 自动部署 + Run/Worker）
2. **验收即 CI Gate**：push → 部署 → 自动验收 → Verdict 决定能否合并。把验收报告接进 GitHub check-run，让"未达验收下限"直接挡住合并。（接 PR check-run 集成）
3. **跨 Agent 独立验收**：写代码的 Agent ≠ 验收的 Agent。独立性天然规避"自己测自己、自己骗自己"。（接 issues 视觉验收的多 Agent 工单制）
4. **需求到报告的全栈追溯**：spec（需求）→ design（设计）→ commit（代码）→ 用例 → 报告，串成一条可查询的追溯链。问"这个需求验到什么程度了"，一跳到底。（接 RTM + doc/ 编号体系）
5. **质量趋势看板**：聚合报告元数据（Verdict / Severity / 覆盖率随时间变化）+ LLM 日志，画出质量趋势、易坏区、回归热点。验收数据第一次能驱动"质量度量"而不只是"一次性判断"。
6. **自愈回归**：验收发现回归 → 自动派生修复任务（Run/Worker）→ 修完自动重验。让质量闭环自己转起来，人只在转不动时介入。
7. **验收知识的语义检索**：报告语料 + LLM →"上次分享流是怎么验的、踩过哪些坑"可被语义检索。验收历史从"翻不动的存档"变成"问得到的顾问"。（接知识库 + 涌现探索器）

这些不必一次做完。但方向清楚：**验收会从"交付末端的一道手续"，长成贯穿研发全程的质量神经系统**——它取证、它记忆、它度量、它在出问题时自己拉响警报。

## 12. 风险与边界

- **分享阅读器不渲染 reference markdown 正文**：交付走 owner 授权路径；跨人分享需对方登录或库设公开。属文档空间侧已知缺陷，建议另立 issue 修复。
- **暗色 only 页面**：双主题轴对多数 prd-admin 页面不适用，按"无亮色主题则单张 + 注明"处理。
- **v3 升级的成本约束**：RTM / GWT / Severity×Priority 会增加报告填写成本，按档位启用——L0/L1 轻量、L2 全量，避免为小改动套重模板。
