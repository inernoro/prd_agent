## 1. 功能全景

周报功能是一个完整的团队周报管理系统，包含以下模块：

| 模块 | 说明 | 入口 |
|------|------|------|
| 周报编写 | 创建、编辑、提交周报 | 周报 Tab → 写周报 |
| 每日打卡 | 记录每日工作内容，周报生成时自动汇总 | 周报 Tab → 打卡 |
| 数据采集 | 自动从 GitHub/GitLab/语雀采集活动数据 | 设置 → 我的数据源 |
| 团队管理 | 创建团队、添加成员、设置角色 | 设置 → 团队管理 |
| 模板管理 | 定义周报结构（章节、输入类型） | 设置 → 模板管理 |
| 团队工作台 | 管理/加入分区、周报列表主视图、成员抽屉 | 团队 Tab |
| 团队汇总 | AI 聚合所有成员周报生成摘要 | 团队 Tab → 一键汇总 |
| 数据统计 | 个人/团队 12 周趋势图 | 设置 → 数据统计 |
| 评论系统 | 按段落评论、回复、删除 | 周报详情页 |
| 导出 | 导出周报/团队汇总为 Markdown | 周报详情页 / 团队汇总 |

### 页面结构

进入周报功能后，页面顶部有 **3 个 Tab**：

| Tab | 谁能看 | 内容 |
|-----|--------|------|
| **周报** | 所有人 | 我的周报列表 + 打卡入口 + 数据概览 |
| **团队** | 任一团队成员 | 团队汇总主视图 + 成员抽屉入口 |
| **设置** | 所有人（个人区域）+ 管理员（管理区域） | 数据源、模板、团队配置 |

## 2. 权限体系与角色说明

### 2.1 系统权限（5 项）

周报功能涉及 5 个后台权限，需要系统管理员在「用户管理 → 角色管理」中分配：

| 权限标识 | 中文名 | 谁需要 | 作用 |
|----------|--------|--------|------|
| `report-agent.use` | 周报基础使用 | **所有需要用周报的人** | 访问周报页面、写周报、打卡 |
| `report-agent.template.manage` | 模板管理 | 管理员 | 创建/编辑/删除周报模板 |
| `report-agent.team.manage` | 团队管理 | 管理员 | 创建/编辑/删除团队、管理成员 |
| `report-agent.datasource.manage` | 数据源管理 | 管理员 | 配置团队级 Git 仓库连接 |
| `report-agent.view.all` | 全局查看 | 高级管理员 | 查看所有团队的周报（跨团队） |

### 2.2 团队角色（3 种）

在团队内部，成员有不同角色：

| 角色 | 英文标识 | 能做什么 |
|------|----------|----------|
| **负责人** | `leader` | 审阅/退回周报、标记请假、查看团队面板、生成团队汇总、触发工作流 |
| **副负责人** | `deputy` | 与负责人权限相同 |
| **成员** | `member` | 写周报、打卡、查看自己的周报 |

### 2.3 关键规则

- 一个用户可以属于 **多个团队**，每个团队有独立的周报
- 团队负责人创建团队时会 **自动成为团队成员**（角色为 leader）
- 没有 `report-agent.use` 权限的用户 **看不到** 侧边栏的周报入口
- `report-agent.view.all` 是超级查看权限，可以跨越团队可见性限制

## 3. 首次搭建：管理员配置（必须按顺序）

> **注意**：以下步骤有严格的先后依赖关系。跳过任何一步都会导致后续功能异常。这是历史上最常见的问题来源。

### 步骤 1：分配权限

**操作路径**：侧边栏 → 用户管理 → 角色管理

1. 找到（或创建）一个角色，如"周报管理员"
2. 勾选以下权限：
   - `report-agent.use`
   - `report-agent.template.manage`
   - `report-agent.team.manage`
   - `report-agent.datasource.manage`
3. 将该角色分配给需要管理周报的管理员
4. 对普通用户，只需分配包含 `report-agent.use` 的角色

**验证**：刷新页面，侧边栏应出现"周报管理"（或类似入口）。如果没出现，检查权限是否保存成功。

> **常见错误**：给用户分配了团队成员角色，但忘了给 `report-agent.use` 权限。结果用户完全看不到周报功能入口。

### 步骤 2：初始化系统模板（推荐）

**操作路径**：周报页面 → 设置 Tab → 模板管理

系统预置了标准模板，首次使用时需要初始化：

1. 进入「设置」Tab
2. 点击「模板管理」卡片
3. 如果列表为空，系统会提供"初始化预置模板"按钮，点击它
4. 系统会创建标准模板（如：标准周报、研发周报等）
5. 这些系统模板 **不可删除**，但可以基于它们创建新模板

**如果想自建模板**：

1. 点击「新建模板」
2. 填写模板名称（如"产品团队周报"）
3. 添加章节，每个章节设置：
   - **标题**：如"本周完成"、"下周计划"、"需要协助"
   - **描述**：对章节内容的引导说明
   - **输入类型**：
     - `BulletList`（默认）：逐条列举，最常用
     - `RichText`：长文本自由输入
     - `KeyValue`：键值对形式
     - `ProgressTable`：进度表格
   - **是否必填**：勾选后提交时校验
4. 可以拖拽调整章节顺序
5. 可选绑定特定团队（`TeamId`）或岗位（`JobTitle`）
6. 勾选「设为默认模板」让新建周报时自动选中

**验证**：模板列表中应显示至少一个模板。

### 步骤 3：创建团队

**操作路径**：周报页面 → 设置 Tab → 团队管理

> **前置条件**：当前用户必须有 `report-agent.team.manage` 权限。

1. 点击「团队管理」卡片
2. 点击「新建团队」
3. 填写：
   - **团队名称**：如"产品研发组"
   - **负责人**：从用户列表选择（该用户会自动加入团队）
   - **描述**：可选
   - **周报可见性**：
     - `all_members`（默认）：所有成员可互相查看周报
     - `leaders_only`：仅负责人/副负责人可查看成员周报
   - **自定义打卡标签**：可选，如"需求评审"、"代码复查"（详见第 7 节）
4. 点击保存

**验证**：团队列表中应显示新建的团队，负责人自动成为成员。

### 步骤 4：添加团队成员

**操作路径**：团队管理 → 选择团队 → 成员管理

1. 在团队详情中，点击「添加成员」
2. 从用户下拉列表选择用户
3. 设置角色（默认为 `member`）
4. 可选设置岗位（如"前端开发"、"后端开发"）
5. 点击保存

**验证**：成员列表中应显示新添加的成员。

> **隐含规则**：
> - 被添加的用户必须在系统中存在（`users` 集合状态为 Active）
> - 同一用户不能重复添加到同一团队（返回 `DUPLICATE` 错误）
> - 用户必须有 `report-agent.use` 权限才能实际使用周报功能
> - 团队负责人在创建团队时已自动添加，无需再次添加

### 步骤 5（可选）：配置团队数据源

**操作路径**：周报页面 → 设置 Tab → 团队数据源

> **前置条件**：需要 `report-agent.datasource.manage` 权限 **或** 团队负责人/副负责人角色。

1. 点击「团队数据源」卡片
2. 点击「新增数据源」
3. 填写：
   - **名称**：如"前端仓库"
   - **类型**：`git`（GitHub/GitLab）或 `svn`
   - **仓库地址**：完整的仓库 URL
   - **访问令牌**：GitHub Personal Access Token 或 GitLab Token
   - **分支过滤**：可选，如 `main,develop`
   - **轮询间隔**：最小 10 分钟，默认 60 分钟
   - **用户映射**：将 Git 提交作者邮箱映射到系统用户（关键！）
4. 点击「测试连接」验证配置
5. 保存后点击「同步」拉取数据

**用户映射说明**：

用户映射决定了 Git 提交归属到哪个团队成员。格式为 JSON 键值对：
```json
{
  "zhangsan@company.com": "user-id-of-zhangsan",
  "lisi@gmail.com": "user-id-of-lisi"
}
```
如果不配置映射，提交记录无法与用户关联，在周报中不会显示。

### 配置完成后的检查清单

| 检查项 | 预期结果 | 不通过的原因 |
|--------|----------|-------------|
| 用户能看到周报入口 | 侧边栏显示周报 | 缺少 `report-agent.use` 权限 |
| 管理员能看到设置中的管理区域 | 显示模板管理/团队管理/团队数据源 | 缺少对应管理权限 |
| 用户能看到引导页 | 显示"创建团队 → 选择模板 → 开始编写" | 用户未加入任何团队 |
| 用户能创建周报 | 选择团队和模板后成功创建 | 无团队或无模板 |
| 团队成员能看到团队 Tab | 顶部出现"团队"标签 | 用户未加入任何团队 |

## 4. 普通用户日常使用

### 4.1 写周报

**操作路径**：周报 Tab → 「写周报」按钮

1. 点击周报 Tab 顶部的「写周报」按钮
2. 弹窗中选择：
   - **团队**：如果你属于多个团队，选择要提交的团队
   - **模板**：选择周报模板（如果团队绑定了模板会自动选中）
3. 点击「开始编写周报」
4. 进入编辑器，按章节填写内容：
   - 每个章节有颜色编码的标题
   - 点击「+ 添加一条」输入内容
   - 支持删除已有条目
   - 右上角显示完成进度（如 3/5 项）
5. 随时点击「保存」存为草稿
6. 全部填写完成后点击「提交」

**关键规则**：

- 每个用户在每个团队的每一周 **只能创建一份周报**（唯一约束：`userId + teamId + weekYear + weekNumber`），重复创建会返回 `DUPLICATE` 错误
- 支持补填往期周报：创建时可以指定 `weekYear` 和 `weekNumber`
- 保存不会改变状态，仍然是草稿
- 提交后状态变为"已提交"，**不可再编辑**，除非被负责人退回

### 4.2 查看周报

**操作路径**：周报 Tab

- 默认显示当前周的所有团队的周报卡片
- 周导航器可以切换到历史周
- 每张卡片显示：团队名、状态徽章、完成进度
- 点击卡片进入详情/编辑

### 4.3 查看周报详情

**操作路径**：点击周报卡片 → 详情页

- 顶部显示：用户名、团队、周次
- 内容区按章节展示
- 每个章节下方有评论区
- Tab 切换到「计划对比」可查看上周计划 vs 本周完成
- 右上角「导出」按钮可导出 Markdown

## 5. 团队工作台操作

### 5.1 查看团队面板

**操作路径**：团队 Tab

> 团队 Tab 区分两类：
> - 我管理的团队（leader / deputy）
> - 我加入的团队（member，可主动退出）

面板包含：
- **团队分区**：在“我管理的团队 / 我加入的团队”间切换
- **周导航器**：切换查看哪一周
- **团队选择器**：如果负责多个团队可切换
- **团队周报列表**：默认主视图（展示总人数/已提交/待提交）
- **团队成员按钮**：仅选中团队后显示，点击后在右侧抽屉查看/管理成员
- **AI 团队汇总入口**：从列表页单独进入，支持返回周报列表

可见性规则：
- 我管理的团队：查看完整团队汇总，支持生成汇总
- 我加入的团队：
  - `all_members`：可查看完整团队汇总
  - `leaders_only`：仅查看本人已提交周报汇总

### 5.2 审阅周报

**操作路径**：团队 Tab → 点击成员周报 → 审阅

1. 在团队面板中，点击状态为「已提交」的成员周报
2. 进入周报详情页
3. 阅读各章节内容
4. 可以在各章节下方添加评论
5. 操作：
   - 点击「审阅通过」→ 状态变为"已审阅"
   - 点击「退回」→ 弹窗输入退回原因 → 状态变为"已退回"

**规则**：
- 「已提交」可审阅或退回；「已审阅」可退回
- 审阅后状态变为 `reviewed`
- 退回时必须填写原因；退回后用户可以修改并重新提交

### 5.3 标记请假

**操作路径**：团队 Tab → 成员卡片 → 标记请假

1. 在成员状态网格中，找到请假的成员
2. 点击「标记请假」
3. 可选填写请假原因
4. 系统会创建一条特殊的 `vacation` 状态周报

**规则**：
- 如果该成员本周已有草稿/逾期周报，会被 **自动删除** 后替换为请假标记
- 如果该成员本周已提交或已审阅的周报，**不能** 标记请假（返回 `INVALID_REQUEST` 错误）
- 取消请假会删除 vacation 周报，成员状态恢复为"未开始"

### 5.4 生成团队汇总（独立入口）

**操作路径**：团队 Tab → 团队周报列表 → 「AI 团队汇总」入口

1. 确保团队成员已提交周报（AI 需要内容来生成）
2. 点击「一键汇总」
3. AI 会聚合所有已提交/已审阅的周报，生成结构化汇总
4. 汇总包含多个段落（如：整体进展、重要成果、风险与问题）
5. 可以导出为 Markdown，并通过“返回周报列表”回到主视图

**规则**：
- 仅 leader/deputy 或拥有 `report-agent.view.all` 权限者可生成
- 如果没有任何已提交的周报，生成会失败
- 每个团队每周只有一份汇总，重新生成会覆盖

## 6. 周报生命周期与状态机

### 6.1 状态流转图

```
                    ┌──────────────┐
                    │   未开始      │  （用户还没创建周报）
                    │  not-started  │
                    └──────┬───────┘
                           │ 创建周报
                           ▼
                    ┌──────────────┐
         ┌────────▶│    草稿       │◀──────────────┐
         │         │    draft      │               │
         │         └──────┬───────┘               │
         │                │ 提交                   │
         │                ▼                        │
         │         ┌──────────────┐                │
         │         │   已提交      │               │
         │         │  submitted   │               │
         │         └──────┬───────┘               │
         │                │                        │
         │         ┌──────┴───────┐               │
         │         │              │               │
         │         ▼              ▼               │
         │  ┌──────────────┐ ┌──────────────┐    │
         │  │   已审阅      │ │   已退回      │───┘
         │  │  reviewed    │ │  returned    │  修改后
         │  └──────────────┘ └──────────────┘  重新提交
         │
         │  特殊状态：
         │  ┌──────────────┐
         └──│   请假        │  （负责人手动标记）
            │  vacation    │
            └──────────────┘
```

### 6.2 各状态下的可用操作

| 状态 | 用户可做 | 负责人可做 |
|------|----------|-----------|
| **未开始** | 创建周报 | 标记请假 |
| **草稿** | 编辑、保存、提交、删除 | 标记请假（会删除草稿） |
| **已提交** | 无（只读） | 审阅、退回、添加评论 |
| **已审阅** | 查看（只读） | 查看 |
| **已退回** | 编辑、重新提交 | - |
| **请假** | 查看 | 取消请假 |
| **逾期** | 编辑、提交 | - |

### 6.3 常见疑问

**Q：已提交的周报能修改吗？**
A：**不能**。提交后只有两种路径：
1. 负责人「审阅通过」→ 状态锁定为已审阅，永久只读
2. 负责人「退回」→ 用户可以修改后重新提交

如果确实需要修改已提交的周报，请联系团队负责人先退回。

**Q：已审阅的周报还能退回吗？**
A：**不能**。审阅通过是终态操作，不可撤销。请在审阅前仔细检查。

**Q：能删除周报吗？**
A：只有 **草稿** 状态的周报可以删除。已提交/已审阅/已退回的周报不可删除。

**Q：同一周能创建多份周报吗？**
A：**不能**。每个用户在每个团队的每一周只能有一份周报。但如果你属于多个团队，每个团队各一份。

## 7. 每日打卡系统

### 7.1 功能说明

每日打卡是一个轻量级的工作记录工具，帮助用户随时记录当天做了什么。这些记录会在生成周报时被 AI 自动汇总。

### 7.2 操作流程

**操作路径**：周报 Tab → 「打卡」按钮

1. 点击「打卡」进入打卡面板
2. 在顶部输入框输入工作内容
3. 选择分类标签（点击对应按钮）
4. 按 Enter 或点击发送按钮提交
5. 记录会出现在下方时间线中
6. 悬停可编辑或删除

### 7.3 分类标签

系统预置 **6 个固定分类**（不可修改、不可删除）：

| 分类 | 英文标识 | 图标 | 颜色 |
|------|----------|------|------|
| 开发 | `development` | 代码图标 | 蓝色 |
| 会议 | `meeting` | 用户组图标 | 紫色 |
| 沟通 | `communication` | 消息气泡图标 | 橙色 |
| 文档 | `documentation` | 文件图标 | 绿色 |
| 测试 | `testing` | 试管图标 | 粉色 |
| 其他 | `other` | 更多图标 | 灰色 |

### 7.4 自定义打卡标签

**Q：管理员可以自定义分类吗？（比如替换"其他"或新增分类）**

A：**6 个固定分类不可修改**。但管理员可以在「团队管理」中为每个团队配置 **自定义标签**。

**配置路径**：设置 → 团队管理 → 编辑团队 → 自定义打点标签

1. 在团队编辑弹窗中找到「自定义打点标签」区域
2. 在输入框中输入标签名（如"需求评审"），按 Enter 添加
3. 点击标签旁的 × 可删除
4. 保存团队设置

**显示效果**：
- 6 个系统分类按钮始终显示在第一行
- 团队自定义标签以青色小标签形式显示在下方
- 用户打卡时可以同时选择一个分类 **和** 多个自定义标签

**技术细节**：
- 自定义标签存储在 `ReportTeam.CustomDailyLogTags` 字段（`List<string>` 类型）
- 标签通过 `DailyLogItem.Tags` 字段保存（与 `Category` 字段分开）
- 后端只校验 `Category` 必须是 6 个固定值之一，如果传入无效值会静默回退为 `other`
- `Tags` 字段不做内容校验，接受任意字符串列表

### 7.5 数据源集成

打卡面板右侧会自动显示从个人数据源同步的 Git 提交记录：

- 未记录的提交会以提示形式出现
- 点击单条提交旁的「+」可快速添加为打卡记录
- 点击「一键补充」可批量添加所有未记录的提交

### 7.6 热力图与统计

打卡面板右侧栏包含：

| 区域 | 说明 |
|------|------|
| 本周热力图 | 7 天柱状图，点击可跳转到对应日期 |
| 连续打卡天数 | 火焰图标 + 天数 |
| 快捷分类统计 | 饼图式统计各分类的时长或条数 |
| AI 建议 | 显示未记录的提交，建议补充 |

### 7.7 打卡数据存储

- 每个用户每天一条记录（`userId + date` 唯一）
- 保存操作是 **Upsert**：同一天重复保存会覆盖
- 每条记录包含：内容（`content`）、分类（`category`）、标签（`tags`）、时长（`durationMinutes`）、创建时间

## 8. 数据源系统（个人 + 团队）

### 8.1 两级数据源架构

周报系统有 **两种数据源**，容易混淆：

| 维度 | 个人数据源 | 团队数据源 |
|------|-----------|-----------|
| 配置入口 | 设置 → 我的数据源 | 设置 → 团队数据源 |
| 权限要求 | `report-agent.use`（所有用户） | `report-agent.datasource.manage` 或 leader |
| 数据归属 | 只属于配置者本人 | 属于整个团队 |
| 支持类型 | GitHub / GitLab / 语雀 | Git / SVN |
| 用途 | 个人打卡面板显示提交 | 团队维度的提交统计 |
| 令牌 | 个人 Token | 团队共享 Token |

### 8.2 个人数据源

**操作路径**：设置 → 我的数据源

**添加流程**：
1. 点击「新增数据源」
2. 选择类型：GitHub / GitLab / 语雀
3. 填写显示名称
4. 根据类型填写：
   - **GitHub**：仓库 URL（如 `https://github.com/org/repo`）、Personal Access Token
   - **GitLab**：仓库 URL、Private Token、可选自定义 API 地址
   - **语雀**：空间 ID（SpaceId）、Token
5. 点击「测试连接」
6. 保存
7. 点击「同步」拉取本周数据

**令牌生成方式**：
- GitHub：Settings → Developer Settings → Personal Access Tokens → 勾选 `repo` 权限
- GitLab：Preferences → Access Tokens → 勾选 `read_repository` 权限
- 语雀：个人设置 → 开发者 → 创建 Token

**管理功能**：
- 启用/禁用开关
- 编辑配置
- 删除数据源
- 查看同步状态和错误信息

### 8.3 团队数据源

**操作路径**：设置 → 团队数据源

**用途**：为整个团队配置 Git 仓库连接，自动采集成员提交记录，用于团队统计和活动概览。

**关键配置 —— 用户映射**：

团队数据源的核心价值在于 **用户映射**。Git 提交中的作者邮箱需要映射到系统用户 ID：

```
Git 提交作者: zhangsan@company.com  →  映射到系统用户: 6501a2b3c4d5e6f7
```

如果不配置用户映射，提交记录无法归属到具体成员，在团队统计中不会显示。

**身份映射编辑器**（v2.0）：

在团队成员管理中，还可以为每个成员配置多平台身份映射：

```json
{
  "github": "zhangsan",
  "tapd": "zhangsan@company.com",
  "gitlab": "zhangsan"
}
```

这用于跨平台数据关联。

## 9. AI 功能

### 9.1 AI 填充周报

**操作路径**：编辑周报时 → 「AI 填充」按钮

AI 会综合以下数据源自动生成周报内容：
- 本周的每日打卡记录
- 个人数据源的 Git 提交
- 系统内的活动数据（缺陷、文档、创意等）

**前提条件**：
- 周报状态必须是草稿、已退回或逾期
- 需要有足够的数据（打卡记录和/或提交）才能生成有意义的内容

**注意事项**：
- AI 填充会 **覆盖** 当前周报所有章节的内容
- 填充后需要人工审阅和调整
- 填充完成后周报仍然是草稿状态，需要手动提交

### 9.2 AI 团队汇总

详见第 5.4 节。

### 9.3 活动数据采集

**API 路径**：`GET /api/report-agent/activity`

系统自动采集的活动数据分为以下类别：

| 数据组 | 包含项 | 数据来源 |
|--------|--------|----------|
| 代码 | 提交数 | Git 数据源 |
| 任务 | 缺陷数 | 缺陷管理 Agent |
| 协作 | 协作记录 | 系统内操作 |
| 文档 | 文档数 | PRD Agent |
| AI 工具 | AI 使用量 | 各 Agent 调用 |
| 创意 | 视觉创作、视频 | VisualAgent / VideoAgent |

周报 Tab 的数据概览面板会自动展示这些统计卡片。零值的卡片会自动隐藏（除非标记为 `alwaysShow`）。

## 14. 常见问题与故障排除

### 14.1 权限相关

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 看不到周报功能入口 | 缺少 `report-agent.use` 权限 | 管理员在角色管理中添加权限 |
| 设置中看不到"模板管理" | 缺少 `report-agent.template.manage` | 添加模板管理权限 |
| 设置中看不到"团队管理" | 缺少 `report-agent.team.manage` | 添加团队管理权限 |
| 设置中看不到"团队数据源" | 缺少 `report-agent.datasource.manage` | 添加数据源管理权限 |
| 看不到团队 Tab | 未加入任何团队 | 由管理员将你加入团队 |
| "缺少团队管理权限" | 操作需要 `team.manage` 但当前角色没有 | 联系管理员分配权限 |
| "无权查看该团队周报" | 非团队成员也非 leader | 加入团队或获取 `view.all` 权限 |
| "团队设置仅负责人可查看成员周报" | 团队可见性设为 `leaders_only` | 联系 leader 或修改团队可见性设置 |

### 14.2 创建相关

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 创建周报时找不到团队 | 用户未加入任何团队 | 联系管理员将你添加到团队 |
| 创建周报时找不到模板 | 没有模板或模板未初始化 | 管理员创建模板或初始化预置模板 |
| "该周已存在周报，不能重复创建" | 同一周同一团队已有周报 | 编辑已有周报而非重新创建 |
| "你不是该团队成员" | 用户不在选中的团队中 | 确认选择了正确的团队 |
| "模板不存在" | 选中的模板被删除 | 刷新页面，选择其他模板 |

### 14.3 编辑/提交相关

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| "只有草稿、已提交、已退回或逾期状态的周报可以编辑" | 周报已审阅 | 联系负责人退回 |
| "只能编辑自己的周报" | 尝试编辑他人周报 | 只能编辑自己的 |
| "只有草稿、已提交、已退回或逾期状态的周报可以删除" | 周报已审阅 | 联系负责人退回后删除 |
| 提交后发现内容有误 | 周报尚未审阅 | 作者可直接编辑并保存 |

### 14.4 审阅相关

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| "只有已提交状态的周报可以审阅" | 周报状态不是 submitted | 等待用户提交 |
| "只有团队负责人可以审阅" | 操作者不是 leader/deputy | 确认角色正确 |
| "只有待审阅或已审阅状态的周报可以退回" | 周报状态不是 submitted/reviewed | 先确认周报状态后再退回 |
| 退回时提示"退回原因不能为空" | 未填写退回原因 | 填写必填的退回原因后重试 |

### 14.5 数据源相关

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 测试连接失败 | Token 无效或仓库 URL 错误 | 检查 Token 权限和 URL 格式 |
| 同步后没有数据 | 用户映射未配置 | 配置 Git 邮箱到系统用户的映射 |
| 打卡面板看不到提交记录 | 个人数据源未配置或未同步 | 在"我的数据源"中添加并同步 |
| "不支持的数据源类型" | 类型拼写错误 | 使用 `github`/`gitlab`/`yuque` |

### 14.6 团队管理相关

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| "该团队下存在周报，无法删除" | 团队有关联的周报记录 | 需要先删除/清理所有相关周报 |
| "负责人用户不存在" | 选择的负责人 ID 无效 | 确认用户存在且状态为 Active |
| "该用户已是团队成员" | 重复添加 | 该用户已在团队中 |
| "系统预置模板不可删除" | 尝试删除系统模板 | 系统模板标记为 `isSystem`，不可删除 |

### 14.7 AI 相关

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| AI 填充按钮不可用 | 周报状态不是草稿/退回/逾期 | 只有可编辑状态才能 AI 填充 |
| AI 生成内容为空或质量差 | 打卡记录和提交数据不足 | 先补充打卡记录和数据源 |
| "AI 生成失败" | LLM 服务异常 | 检查 LLM Gateway 配置和模型可用性 |
| 团队汇总生成失败 | 没有已提交的周报 | 至少需要一份已提交的周报 |

## 附录 C：系统模板初始化

调用 `POST /api/report-agent/templates/seed` 会创建系统预置模板。该接口是 **幂等** 的：已存在的 `TemplateKey` 会跳过，不会重复创建。

系统模板标记为 `isSystem = true`，具有以下特殊行为：
- 不可删除（返回 `SYSTEM_TEMPLATE` 错误）
- 可以被管理员编辑（名称、描述、章节）
- 有唯一的 `TemplateKey` 标识
