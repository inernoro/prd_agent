# CDS 环境与凭据 · 指南

> **版本**：v3.0 | **日期**：2026-07-15 | **状态**：开发中

CDS 中有三类容易混淆的信息：CDS 服务自身配置、业务应用运行变量、Agent 项目凭据。三者作用范围、保存位置和使用方式不同，不能放在同一个文件或当成同一把 Key 使用。

## 先看区别

| 类型 | 作用对象 | 用户在哪里管理 | 是否进入业务容器 |
| --- | --- | --- | --- |
| CDS 系统配置 | CDS 服务本身 | 管理员初始化和系统设置 | 否 |
| 项目运行变量 | 某个项目的应用 | 项目设置 > 环境变量 | 是 |
| Agent 项目凭据 | 当前项目中的 Codex、Cursor 或 Claude Code | CDS 页面批准 | 否 |

新版 Agent 接入不依赖 Shell 环境变量。不要为了接入 Agent 修改 `.zshrc`、`.bashrc`、系统环境变量或全局 PATH。

## CDS 系统配置

系统配置决定 CDS 服务如何登录、保存状态、连接 GitHub 和提供域名，由 CDS 管理员维护。

本地或单机环境通过以下命令初始化：

```bash
cd cds
./exec_cds.sh init
./exec_cds.sh start
./exec_cds.sh status
```

初始化结果写入 `cds/.cds.env`。常见配置包括：

| 配置 | 用途 |
| --- | --- |
| `CDS_USERNAME`、`CDS_PASSWORD` | CDS 管理界面登录 |
| `CDS_JWT_SECRET` | 登录会话签名 |
| `CDS_ROOT_DOMAINS` | Dashboard 和预览域名 |
| `CDS_STORAGE_MODE` | CDS 状态存储方式 |
| `CDS_MONGO_URI`、`CDS_MONGO_DB` | MongoDB 状态存储 |
| `CDS_AUTH_BACKEND` | 登录信息存储方式 |
| `CDS_SECRET_KEY` | CDS 内部敏感信息加密 |

系统配置的原则：

- 只用于 CDS 服务本身；
- 不作为 Agent 接入方式；
- 不放进业务项目仓库；
- 修改后按配置类型重启 CDS；
- 生产环境的密码和签名密钥必须使用安全随机值。

## 项目运行变量

项目运行变量属于某个 CDS 项目，并在部署时注入该项目的业务容器。

### 在页面配置

1. 打开目标项目；
2. 进入“项目设置 > 环境变量”；
3. 添加或修改变量；
4. 保存；
5. 重新部署需要获取新值的分支。

变量保存后影响后续部署。已经运行的容器不会自动获得新值，必须重新部署或重新启动对应版本。

### 常见类型

| 类型 | 示例 | 建议 |
| --- | --- | --- |
| 数据库连接 | `DATABASE_URL`、`MONGODB_URL` | 优先使用 CDS 基础设施自动生成的连接信息 |
| 缓存和队列 | `REDIS_URL`、`RABBITMQ_URL` | 按项目隔离，不跨项目复用 |
| 对象存储 | `S3_ENDPOINT`、云存储凭据 | 生产使用持久存储，预览可使用明确的临时方案 |
| 应用身份 | 应用登录或签名配置 | 使用项目专属值 |
| 外部服务 | 第三方 API 地址和凭据 | 只授予业务需要的最小权限 |

### 项目隔离

CDS 按项目保存运行变量。Agent 或管理员操作时必须明确目标项目，不能把一个项目的变量写到全局作用域。

需要少量项目通用默认值时，可以由管理员使用全局作用域；同名项目变量覆盖全局默认值。敏感值应尽量保持项目级，避免扩大影响范围。

### 不要这样做

- 不要把业务密钥写进 `cds-compose.yml`；
- 不要把生产密钥提交到 Git；
- 不要把 GitHub App 私钥作为业务容器变量；
- 不要把 Agent 项目凭据注入业务容器；
- 不要为了方便把所有项目共用一套高权限密钥。

## Agent 项目凭据

Agent 项目凭据只用于让当前项目中的 Agent 操作 CDS，不属于业务运行变量。

### 推荐接入方式

1. 打开“一键部署 > 接入 Agent”；
2. 选择 Agent 和目标项目；
3. 把无密钥接入口令交给 Agent；
4. 在 CDS 页面批准；
5. Agent 自动保存并验证项目凭据。

凭据保存在当前项目的 `.cds/credentials.json`，文件权限为 `0600`，并加入仓库本地排除规则。

这意味着：

- 用户不需要复制长期 Key；
- Key 不进入对话、命令参数或正常输出；
- 凭据默认不能访问其他项目；
- 关闭终端或重新打开 Agent 后仍可在当前项目使用；
- 不会修改用户电脑的 Shell 配置。

### 项目技能文件与凭据不同

Codex、Cursor 和 Claude Code 的项目级技能文件不包含凭据，用于告诉 Agent 如何调用 CDS。它们可能出现在 `git status`，是否提交由团队决定。

`.cds/credentials.json` 包含项目凭据，不能进入 `git status` 和 Git 提交。

### 旧版兼容

历史流程使用 `CDS_HOST`、`CDS_PROJECT_KEY` 或 `AI_ACCESS_KEY` 环境变量。现有自动化可以继续运行，但新用户和交互式 Agent 不再以此作为默认接入方式。

只有明确维护旧自动化时，才使用兼容流程。不要把兼容方式重新写成新手 Quick Start。

## GitHub 仓库权限

GitHub App 授权也不是项目运行变量。

私有仓库应使用单仓库授权：

- 只选择目标仓库；
- CDS 在扫描、构建和部署时复用授权；
- 不在项目变量中保存个人 GitHub Token；
- 不把 GitHub App 私钥下发到业务容器或 Agent。

详细操作见 [CDS GitHub App 接入](guide.cds.github-app.md)。

## 如何判断放在哪里

| 问题 | 应放在哪里 |
| --- | --- |
| 它控制 CDS 服务登录、域名或存储吗 | CDS 系统配置 |
| 它是业务应用运行时需要的吗 | 项目运行变量 |
| 它只让当前 Agent 操作 CDS 吗 | Agent 项目凭据 |
| 它让 CDS 访问某个 GitHub 仓库吗 | GitHub App 仓库授权 |

如果一个信息同时看起来属于两个位置，应先缩小权限和用途，而不是复制两份。

## 修改后的验收

### 修改系统配置

- CDS 可以重新启动；
- Dashboard 能正常登录；
- 项目和部署历史仍然存在；
- 预览域名仍能访问。

### 修改项目运行变量

- 只影响目标项目；
- 重新部署后容器获得新值；
- 页面和核心操作正常；
- 日志不显示敏感值明文。

### 重新授权 Agent

- 当前项目连接成功；
- 其他项目不能复用权限；
- `.cds/credentials.json` 不进入 Git；
- Shell 配置没有被修改。

## 常见问题

| 问题 | 解答 |
| --- | --- |
| 为什么保存变量后旧容器没变化 | 运行中的容器不会自动重新注入变量，需要重新部署 |
| Agent 为什么不能访问另一个项目 | 凭据按项目隔离，需要在另一个项目重新批准 |
| 可以把 Agent Key 放到项目变量吗 | 不可以，两者用途不同，也会扩大泄露范围 |
| 可以把 GitHub Token 放到业务容器吗 | 通常不需要，私有仓库使用 GitHub App 授权 |
| 旧环境变量脚本还能用吗 | 可以兼容，但不再是新用户默认方式 |
| 如何撤销 Agent 权限 | 在 CDS 中撤销目标项目授权，并删除本地项目凭据 |
