# CDS 自动交付与 Agent 接入 · 指南

> **版本**：v3.0 | **日期**：2026-07-15 | **状态**：已落地

> **目标读者**：第一次使用 CDS，以及希望减少部署配置和调试负担的用户

## 先看结论

CDS 的主流程已经从“用户准备好一切，平台负责执行”升级为“用户选择项目并批准，平台和 Agent 负责准备，用户确认结果”。

这次重大更新统一了 Agent 接入、项目识别、部署过程、失败处理和版本恢复。用户不需要先理解密钥、环境变量、构建容器和完整部署配置，才能开始使用 CDS。

新的标准流程只有五步：

1. 选择 Codex、Cursor 或 Claude Code；
2. 选择已有项目或创建新项目；
3. 在 CDS 页面批准一次；
4. 确认 CDS 自动准备的运行方案并部署；
5. 打开预览，之后按需要再次部署或恢复旧版本。

## 这次升级解决了什么

| 用户原来的负担 | 现在的处理方式 |
| --- | --- |
| 不知道技能应该放到哪个 Agent 目录 | 页面根据 Codex、Cursor 或 Claude Code 给出项目级接入方式 |
| 需要复制和保管长期密钥 | Agent 发起申请，用户在 CDS 页面批准 |
| 接入过程修改电脑环境变量 | 凭据只保存在当前项目，不修改系统环境 |
| 第一次部署要先写完整配置 | CDS 自动识别，用户只确认最终结果 |
| Compose 没写启动命令就失败 | CDS 从项目自身信息继续推导可运行配置 |
| 私有仓库已经授权却仍拉取失败 | 复用项目对应的 GitHub App 单仓库授权 |
| 网络短暂超时后要重新开始 | 接入等待自动继续，不丢失原申请 |
| 刷新后不知道部署到哪一步 | 回到同一条部署记录继续查看 |
| 相同版本也要重新构建 | 成功产物可直接再次部署 |
| 新版本出问题时恢复困难 | 从成功版本中选择一个恢复 |
| 失败后只能读大量日志 | 页面先给出问题归属和处理顺序 |

## Quick Start：3 分钟完成第一次接入

### 第一步：选择 Agent

进入 CDS 控制台任意页面，点击左侧常驻的“接入 Agent”。入口会自动携带当前页面上下文；在登录与认证页，还可以直接点击“交给 Agent 配置”。

选择你正在使用的 Agent：

- Codex；
- Cursor；
- Claude Code。

三种 Agent 使用同一套 CDS 能力。区别只在于技能保存在当前项目的哪个目录，不需要为不同 Agent 准备不同权限。

### 第二步：选择项目

已有 CDS 项目时，选择“连接已有项目”，再选择目标项目。

准备部署新仓库时，选择“创建一个新项目”。首次创建使用一次性权限，项目创建成功后会自动切换为该项目专属权限。

### 第三步：把接入口令交给 Agent

复制页面生成的接入口令并交给 Agent。口令除了接入目标，还包含当前页面任务、验收要求和安全边界，Agent 不需要依赖用户转述页面内容。

这条口令只告诉 Agent 去哪里发起申请、申请哪个项目，不包含长期密钥，也不会要求执行 `source`、修改环境变量或重启终端。

### 第四步：在 CDS 页面批准

Agent 发起申请后，CDS 右下角会出现授权申请。确认 Agent 名称和目标项目无误后点击批准。

批准后，Agent 会自动接收项目专属权限、保存到当前项目并验证连接。用户不需要复制任何 Key。

### 第五步：确认凭据没有进入 Git

接入完成后检查目标仓库：

- `git status` 不应出现 `.cds/credentials.json`；
- 项目级技能目录可能显示为新增文件，这些文件不含凭据，是否提交由团队决定；
- 关闭并重新打开 Agent 后，当前项目仍能连接；
- 切换到其他项目时，不应自动继承这份权限。

## Quick Start：完成第一次部署

### 第一步：让 CDS 理解项目

进入目标项目并选择代码分支。常规项目进入“项目设置 > 运行时 > 交付模式”，选择“托管模式”。

CDS 会检查：

- 哪些目录是真正的应用；
- 如何安装依赖；
- 是否需要构建；
- 使用什么命令启动；
- 服务监听哪个端口；
- 如何判断服务已经正常；
- 是否依赖数据库、缓存或其他服务。

标准 Compose 项目只写了构建方式、没有写启动命令时，CDS 会继续读取项目自身的运行信息并补全，而不是立即把配置责任交还给用户。

### 第二步：检查最终生效配置

重点确认五项：

1. 应用和目录是否正确；
2. 安装、构建和启动方式是否合理；
3. 端口和健康检查地址是否正确；
4. 数据库、缓存等依赖是否齐全；
5. 是否有必须由用户提供的环境信息。

全部正确时直接部署。只有自动识别错误或项目结构特殊时，才修改“最小声明”。

### 第三步：开始部署

返回项目分支页，打开目标分支并点击“重新部署”。

页面会持续显示准备、构建、启动和检查阶段。不要在看不到结果时连续点击；当前部署记录就是这次操作的事实来源。

### 第四步：打开预览完成验收

部署成功后打开页面给出的预览地址，至少检查：

- 页面或后台服务能够正常访问；
- 登录和核心操作正常；
- 数据库、缓存等依赖正常；
- 刷新 CDS 页面后，部署记录仍然存在；
- 预览地址与当前项目和分支一致。

## 日常使用

### 发布新代码

1. 把代码推送到目标分支；
2. 打开 CDS 中的对应分支；
3. 查看自动部署，或点击“重新部署”；
4. 在同一条部署记录中查看当前阶段；
5. 成功后打开预览地址验收。

页面可以刷新，也可以暂时离开。再次回来时，仍然显示原来的部署过程和结果。

### 重新运行同一个成功版本

适用于代码没有变化、服务临时停止或需要验证同一版本能否恢复的情况。

打开分支详情，在“不可变部署版本”中选择当前成功版本，点击“重新部署此版本”。标记为“产物可复用”的版本可以省去重复构建。

### 新版本有问题，恢复以前的版本

1. 打开分支详情；
2. 找到“不可变部署版本”；
3. 选择以前成功且可复用的版本；
4. 点击“回滚到此版本”；
5. 等待恢复完成并重新打开预览地址。

恢复会生成一条新的部署记录，因此过程和结果仍然可以追踪。

### 部署失败

先不要重复点击部署。按照以下顺序处理：

1. 查看问题归属：代码、配置、CDS、外部服务或暂未归类；
2. 确认失败发生在准备、构建、启动还是检查阶段；
3. 阅读页面给出的处理建议；
4. 只修改对应问题；
5. 修复后重新部署；
6. 基础说明仍然难懂时，再使用“AI 解释”。

“AI 解释”不可用不会阻止部署。基础问题判断、阶段记录和处理建议仍然有效。

## 权限和电脑环境

### 权限默认只属于当前项目

Agent 获得的是目标项目专属权限，不能因为连接了一个项目就访问其他项目。

首次创建项目的权限只能用于创建一次项目。创建完成后，这份权限自动失效，并换成新项目专属权限。

### 凭据不会进入对话或 Git

项目凭据保存在当前项目的 `.cds/credentials.json`，权限为 `0600`，并加入仓库本地排除规则。

因此：

- 用户不需要把密钥发送给 Agent；
- 凭据不会出现在接入口令中；
- 凭据不会出现在正常的命令输出中；
- 凭据不会进入 Git 提交；
- CDS 不修改 `.zshrc`、`.bashrc` 或全局 PATH。

### 私有仓库只授权被选择的仓库

使用 GitHub App 时，优先选择单仓库授权。CDS 拉取私有仓库时复用该项目的授权，不要求开放整个组织，也不把授权内容显示在命令或日志中。

## 我应该选择哪个交付模式

### 优先使用托管模式

适合：

- 普通网站；
- 前后端应用；
- 单个或少量后台服务；
- 定时任务和常驻任务；
- 希望减少配置和排查工作的项目。

### 继续使用 Compose 模式

适合：

- 服务数量很多，启动顺序特殊；
- 使用特殊网络；
- 必须使用自定义镜像或复杂启动方式；
- 已经有一套稳定运行的高级配置。

托管模式不会删除 Compose 配置。两种模式可以切换，切回 Compose 后原有配置继续生效。

## 从入门到熟练

| 阶段 | 先掌握什么 | 达成标准 |
| --- | --- | --- |
| 入门 | 页面批准 Agent，完成一次自动识别和部署 | 不复制密钥也能打开预览 |
| 日常 | 查看部署阶段和失败建议 | 能处理常见代码或配置问题 |
| 熟练 | 直接再次部署成功版本，恢复旧版本 | 新版本异常时能快速恢复 |
| 高级 | 判断托管模式和 Compose 模式的边界 | 不为了省配置牺牲特殊项目能力 |

## 完整验收清单

| 检查项 | 你应该看到什么 |
| --- | --- |
| 打开“接入 Agent” | 可以选择 Codex、Cursor 或 Claude Code |
| 复制接入口令 | 不包含密钥，也不要求修改终端配置 |
| 页面批准申请 | Agent 自动连接并验证当前项目 |
| 检查仓库 | `.cds/credentials.json` 不出现；技能文件不含凭据，是否提交由团队决定 |
| 选择托管模式 | 页面生成“最终生效配置” |
| 发起部署 | 分支详情持续显示部署阶段 |
| 刷新页面 | 原部署记录和结果仍然存在 |
| 打开预览 | 应用和核心业务流程正常 |
| 再次部署成功版本 | 可复用版本不必重新构建 |
| 恢复旧版本 | 恢复后预览地址正常 |
| 制造一次失败 | 页面显示问题归属和处理建议 |
| 切回 Compose | 原有高级配置继续生效 |

## 常见问题

| 问题 | 解答 |
| --- | --- |
| 为什么页面仍然显示“下载技能包” | 当前环境还没有发布本次重大更新。正式发布后会替换为“接入 Agent”。 |
| 我需要把 Agent Key 发给 Agent 吗 | 不需要。Agent 发起申请，你在 CDS 页面批准即可。 |
| 接入会修改电脑环境变量吗 | 不会。新版只在当前项目保存凭据。 |
| 为什么其他项目不能使用当前凭据 | 权限默认按项目隔离，需要在另一个项目重新批准。 |
| 我需要先懂构建和容器吗 | 常规项目不需要。先让 CDS 自动识别，再检查结果。 |
| Compose 没写启动命令怎么办 | CDS 会继续从项目自身信息推导；无法判断时才要求补充。 |
| 托管模式会修改业务代码吗 | 不会。它准备运行方式，不改写业务代码。 |
| 切换托管模式会删除原配置吗 | 不会。切回 Compose 后原配置继续生效。 |
| 为什么某个版本不能直接恢复 | 该版本没有完整可复用产物，需要从源代码重新部署。 |
| AI 解释不可用会影响部署吗 | 不会。基础判断和处理建议仍然可用。 |

## 当前发布状态

托管交付、可续看的部署记录、成功版本复用和回滚已经进入 CDS 正式主流程。

本次重大更新中的“接入 Agent”、项目级页面批准、无环境变量接入、私有仓库授权复用、网络波动续连和标准 Compose 自动补全，已经完成开发并通过自动测试与真实私有仓库部署验收，但尚未发布到当前 `cds.miduo.org` 主分支。

因此，在本次版本正式发布前，线上环境仍可能显示“下载技能包”。正式发布后，以本指南中的新流程为准；旧环境变量接入只作为历史兼容保留。
