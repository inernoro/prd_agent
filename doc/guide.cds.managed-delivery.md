# CDS 托管交付 · 指南

> **版本**：v2.0 | **日期**：2026-07-10 | **状态**：开发中
>
> **目标读者**：希望更省心地上线、查看和恢复应用的 CDS 用户

## 先看结论

这次更新最重要的变化是：**你不再需要先把“怎么构建、怎么启动、端口是多少、依赖怎么连接”全部配置好，才能开始部署。**

对于常规网站、后台服务和自动任务，你可以选择“托管模式”。CDS 会先理解项目，再准备一份可检查的上线方案。你确认后开始部署；上线失败时，页面直接告诉你问题大致出在哪里、先处理什么；需要恢复时，可以直接重新使用成功版本或回到旧版本。

你获得了五项可以直接使用的新功能：

1. **自动准备上线配置**：CDS 自动识别项目如何安装、构建、启动和检查是否正常。
2. **部署过程不会因刷新丢失**：关闭页面、刷新页面或短暂断线后，仍能继续查看原来的部署进度。
3. **成功版本可以直接再次上线**：没有代码变化时，不必重新构建。
4. **可以回到以前的成功版本**：新版本有问题时，从历史版本中选择一个恢复。
5. **失败原因更容易看懂**：先告诉你属于代码、配置、CDS 还是外部服务问题，再给出处理建议。

## 和以前相比，流程有什么变化

### 以前

一次常规上线通常需要你先完成这些事情：

1. 判断项目使用什么技术。
2. 填写安装和构建方式。
3. 填写启动方式和端口。
4. 配置数据库、缓存等连接。
5. 发起部署。
6. 在多个页面之间查看进度和日志。
7. 失败后自己判断是哪一部分出了问题。
8. 再次上线或回滚时重新走一遍构建流程。

问题不一定出在代码，更多时候出在遗漏配置、配置不一致，或者不知道应该先检查哪里。

### 现在

常规项目的主流程缩短为：

1. 选择“托管模式”。
2. 让 CDS 自动识别项目。
3. 检查生成的上线方案。
4. 点击部署。
5. 成功后直接使用预览地址；失败时按页面建议处理。

日常再次上线时，通常只需要进入分支并点击“重新部署”。如果只是恢复同一个成功版本，可以直接复用，不再重复构建。

| 你要做的事 | 以前 | 现在 |
|------------|------|------|
| 第一次上线 | 先理解并填写大量配置 | CDS 自动识别，你负责确认 |
| 查看进度 | 在状态、日志等入口之间切换 | 在同一份部署记录中持续查看 |
| 页面刷新 | 可能需要重新判断部署到哪一步 | 回来后继续显示原来的过程 |
| 再次上线同一版本 | 重新拉取、重新构建 | 直接使用已经成功的版本 |
| 新版本出问题 | 手工重新部署旧代码 | 选择旧的成功版本恢复 |
| 部署失败 | 从大量日志里自己寻找原因 | 页面先给出问题归属和处理顺序 |

## Quick Start：3 分钟完成首次设置

### 第一步：进入交付模式

打开 CDS 项目列表，进入目标项目，选择：

**项目设置 > 交付模式**

### 第二步：选择托管模式

点击“托管模式”。首次使用时，不需要填写高级内容，保持默认即可。

CDS 会自动检查项目，并生成一份“最终生效配置”。

### 第三步：检查识别结果

重点检查下面几项是否符合预期：

- 找到的应用是否正确；
- 应用目录是否正确；
- 安装、构建和启动方式是否合理；
- 端口和健康检查地址是否正确；
- 数据库、缓存等依赖是否齐全。

如果都正确，直接进入下一步。只有识别错误或项目结构特殊时，才需要修改“最小声明”。

### 第四步：开始部署

返回项目的分支页面，打开要上线的分支，点击“重新部署”。

页面会持续显示当前阶段。部署成功后，打开预览地址进行检查。

### 第五步：完成首次验收

至少确认以下内容：

- 页面或后台服务能够正常打开；
- 登录和核心操作正常；
- 数据库、缓存等依赖正常；
- 刷新 CDS 页面后，部署结果仍然存在。

## 我应该选择哪个模式

### 推荐使用托管模式

以下项目优先选择托管模式：

- 普通网站；
- 前后端应用；
- 单个或少量后台服务；
- 定时任务和常驻任务；
- 希望减少配置和排查工作的项目。

### 继续使用 Compose 模式

以下情况继续使用 Compose 模式更合适：

- 服务数量很多，并且启动顺序非常特殊；
- 使用特殊网络；
- 必须使用自定义镜像或复杂启动方式；
- 已经有一套稳定运行的高级配置，不准备迁移。

两种模式可以切换。切回 Compose 模式后，原有配置继续生效，不会被托管模式覆盖。

## 日常怎么使用

### 场景一：发布新的代码

1. 把代码推送到目标分支。
2. 打开 CDS 中的对应分支。
3. 点击“重新部署”。
4. 查看当前处于准备、构建、启动还是检查阶段。
5. 成功后打开预览地址验收。

你可以刷新页面或暂时离开。再次回来时，页面会继续显示这一次部署，而不是重新开始一条不相关的进度。

### 场景二：重新启动已经成功的版本

适用于以下情况：

- 代码没有变化，只想恢复服务；
- 服务临时停止；
- 希望确认同一版本能否重新运行。

操作方法：

1. 打开分支详情。
2. 找到“不可变部署版本”。
3. 在当前版本上点击“重新部署此版本”。

如果页面显示“产物可复用”，CDS 会直接使用已经成功的版本，省去重复构建时间。

### 场景三：新版本有问题，回到旧版本

1. 打开分支详情。
2. 找到“不可变部署版本”。
3. 选择以前成功的版本。
4. 点击“回滚到此版本”。
5. 等待恢复完成，再打开预览地址确认。

回滚会生成一条新的部署记录，因此你仍然能看到恢复过程和最终结果。

### 场景四：部署失败

部署失败后，先不要立刻重复点击部署。按照下面的顺序处理：

1. 查看页面判断的问题归属：代码、配置、CDS、外部服务或暂未归类。
2. 阅读页面给出的处理建议。
3. 根据建议修复对应问题。
4. 修复后再点击重新部署。
5. 如果仍然看不懂，再使用“AI 解释”把现有结论转成更容易理解的说明。

“AI 解释”是辅助功能。即使没有启用它，基础的问题判断和处理建议仍然可用。

## 页面上新增了什么

### 项目设置中的“交付模式”

这里用于选择托管模式或 Compose 模式。

选择托管模式后，会增加两个区域：

- **最小声明**：自动识别不准确时，才需要在这里补充信息；
- **最终生效配置**：部署前确认 CDS 最终会怎样处理你的应用。

### 分支详情中的“部署事实账本”

这里显示最近几次部署的状态、当前阶段、发生时间和失败建议。

它解决的是“刚才那次部署到底怎么样了”。即使页面刷新，记录也不会消失。

### 分支详情中的“不可变部署版本”

这里显示已经成功保存的版本，并标明哪些版本可以直接再次使用。

你可以：

- 重新部署当前版本；
- 回到以前的成功版本；
- 查看某个版本是否具备直接恢复条件。

### 失败记录中的“AI 解释”

当基础说明仍然难以理解时，可以点击“AI 解释”。它会基于已经确认的信息，用更容易理解的语言说明问题和建议顺序。

## 从入门到熟练的推荐路径

| 阶段 | 先学什么 | 达成标准 |
|------|----------|----------|
| 入门 | 选择托管模式，检查自动识别结果 | 能完成一次成功部署 |
| 日常 | 查看部署阶段和失败建议 | 能自己处理常见代码或配置问题 |
| 熟练 | 重新使用成功版本、回滚旧版本 | 新版本失败时能快速恢复 |
| 高级 | 判断何时使用托管、何时保留 Compose | 不为了省配置而牺牲特殊项目能力 |

## 验收清单

| 检查项 | 你应该看到什么 |
|--------|----------------|
| 选择托管模式 | 页面出现“最小声明”和“最终生效配置” |
| 保存设置 | 页面提示已启用托管模式并生成配置 |
| 发起部署 | 分支详情持续显示部署阶段 |
| 刷新页面 | 原来的部署记录和结果仍然存在 |
| 重新使用当前版本 | 标记可复用的版本可以直接重新部署 |
| 回到旧版本 | 旧版本恢复后，预览地址可以正常访问 |
| 制造一次失败 | 页面显示问题归属和处理建议 |
| 切回 Compose | 原有高级配置继续生效 |

## 常见问题

| 问题 | 解答 |
|------|------|
| 我需要先懂构建和容器吗 | 常规项目不需要。先让 CDS 自动识别，再检查结果即可。 |
| 托管模式会修改我的代码吗 | 不会。它负责准备上线方式，不会改写你的业务代码。 |
| 切换托管模式会删除原配置吗 | 不会。切回 Compose 模式后，原有配置继续生效。 |
| 为什么没有生成最终配置 | 项目需要至少有一个已经准备好的分支，或者你需要补充应用目录。 |
| 为什么某个版本不能直接恢复 | 该版本没有保存完整的可复用产物，需要重新从源代码部署。 |
| AI 解释不可用会影响部署吗 | 不会。基础失败判断和处理建议仍然可用。 |
| 所有项目都应该切换托管模式吗 | 不是。复杂网络、多服务或特殊启动项目继续使用 Compose 更稳妥。 |

## 当前可用范围

本功能已经完成开发并进入 PR 验证阶段。PR 合并并完成 CDS 正式更新后，生产环境中的“交付模式”、部署记录、版本恢复和失败解释入口才会正式开放。

在生产 CDS 完成更新前，新入口可能暂时看不到。这不代表项目配置有问题，也不需要反复修改项目。
