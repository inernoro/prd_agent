# 产品评审员技术 · 设计

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：已落地

## 管理摘要

- **解决的问题**：产品方案缺少统一、可解释、可追踪的评审标准，人工评审难以稳定复用。
- **核心方案**：上传 Markdown 方案后创建评审提交，由 LLM 按可配置维度评分，SSE 实时返回过程，系统再执行确定性校验和分数收口。
- **当前状态**：提交、评审、历史、系统故障重跑、未通过补交、申诉、排行榜、Webhook 和权限控制均已落地。
- **关键边界**：LLM 提供分析与证据，清单真值、资格判断和统计口径由服务端决定。

## 问题背景

纯 LLM 评分容易受输出格式、提示词波动和空泛高分影响。只保存最终分数又无法解释评审依据，也不能区分用户方案未通过与系统调用失败。产品评审员需要把上传、模型推理、确定性校验、救济流程和统计拆成可审计状态机。

## 设计目标

1. 评审维度可配置，但总分、必填项和标识保持合法。
2. 长评审通过 SSE 持续展示阶段、增量文本和分项结果。
3. 客户端断开不取消服务器评审，结果最终持久化。
4. 系统故障重跑不计入用户失败次数。
5. 未通过补交和申诉有明确资格、次数与时间窗口。
6. 排行榜以方案而非原始提交行统计，避免重复提交扭曲通过率。

## 核心决策

| 决策 | 说明 |
|------|------|
| Submission 与 Result 分离 | 提交保存生命周期，结果可保留多次历史 |
| LLM 输出双策略解析 | 优先解析结构化 JSON，失败时使用受控兜底解析 |
| 清单真值服务端派生 | 是否涉及、是否覆盖和是否找到方案按 truth table 计算，不信任模型自填结论 |
| 高分证据门 | 高分必须有足够具体证据；数据密度和总结语义矛盾时执行封顶 |
| 两类重跑分离 | Error 重跑只累计系统重试；未通过补交消耗用户的一次救机会 |
| 方案桶统计 | 同一用户、同一标题的多次提交聚合为一份方案 |

## 数据设计

| 模型 | 职责 |
|------|------|
| `ReviewSubmission` | 提交人、附件、状态、当前结果、重跑与完成信息 |
| `ReviewResult` | 总分、各维度分数、总结、原始输出与调整记录 |
| `ReviewDimensionConfig` | 维度、满分、说明、清单项与启用状态 |
| `ReviewAppeal` | 申诉理由、图片、处理状态与受理结果 |
| `ReviewWebhookConfig` | 通知地址、事件类型、启用状态与凭据配置 |

提交状态使用 `Queued`、`Running`、`Done`、`Error`。结果历史独立保存，重跑或重新上传不会删除旧结果。

## 评审流程

1. 用户上传可提取文本的 Markdown 附件并创建 Submission。
2. 服务端验证所有权和文件内容，将提交置为 `Queued`。
3. 结果页订阅 SSE；服务端以原子方式进入 `Running`，避免并发重复评审。
4. 后端加载当前维度，构建系统提示和方案上下文，经 LLM Gateway 流式调用模型。
5. 增量输出持续发送给前端，并解析结构化维度结果。
6. 服务端重算清单维度，执行证据门、数据密度和总结一致性校验。
7. 保存 Result，更新 Submission 为 `Done` 或 `Error`。
8. 发送站内通知和配置的 Webhook，SSE 返回最终结果。

## 结果收口

系统级校验用于约束模型评分，不替代模型阅读方案：

- 清单类维度按配置项逐条派生通过状态和分数。
- 非清单维度达到高分区间时，评语必须包含足够长度和具体证据标记。
- 原文缺少数字、链接、章节或引用等证据时，高分维度按规则封顶。
- 总结明确表达“未达高分”但总分仍处于高分区间时，执行一致性修正。
- 所有系统调整写入结果记录，便于解释和复核。

## 救济与历史

| 场景 | 资格 | 统计影响 |
|------|------|----------|
| 系统故障重跑 | 仅 `Error` | 累计 `ErrorRetryCount`，不计用户重跑 |
| 故障后重新上传 | 仅 `Error` | 替换附件，不消耗未通过救机会 |
| 未通过补交 | `Done` 且未通过，最多一次 | 累计 `RerunCount`，保留旧结果 |
| 提交申诉 | 未通过、本人、在时限内且无冲突申诉 | 进入独立受理流程 |
| 申诉成功后重传 | 申诉获批 | 重新排队并按新方案首次评审口径处理 |

## API 分区

| 分区 | 主要能力 |
|------|----------|
| 维度 | 获取与管理员全量更新 |
| 提交 | 创建、我的列表、全量列表、详情与提交人筛选 |
| 评审 | SSE 结果流、系统重跑、故障重传、未通过补交、结果历史 |
| 申诉 | 提交、列表、图片上传、批准与驳回 |
| 统计 | 按自然月的提交人和方案排行榜 |
| 通知 | Webhook 增删改查与连通性测试 |

所有路由统一位于 `/api/review-agent`。端点细节以 Controller 的路由特性为准，文档不复制完整请求对象。

## 权限与安全

- 普通用户只能查看和操作自己的提交。
- 全量列表、提交人筛选和申诉受理要求独立权限。
- 维度配置与 Webhook 管理使用管理权限。
- 附件读取、重新上传和申诉图片必须校验当前用户与提交关系。
- Webhook 凭据不在列表响应中返回明文，通知失败不回滚已完成评审。

## 验证重点

1. 同一提交不能因重复订阅 SSE 启动两次评审。
2. JSON 解析失败时能降级，且分数不绕过系统校验。
3. Error 重跑与未通过补交分别更新正确计数。
4. 旧结果在重跑和重传后仍可查看。
5. 申诉资格、时限、次数和处理权限均由服务端验证。
6. 排行榜按方案桶统计，一次性通过率不把系统故障归咎用户。
