# CDS Legacy 前端退场 · 计划

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：规划中

## 目标

在 React 控制台已成为 dashboard 权威的基础上，完成 legacy 功能差距复核、真实路径验收和旧 HTML/JS/CSS 的物理退场。本文只保留尚未完成的退场门禁；现行页面结构由 `cds/web/` 代码和 `guide.cds.web-migration-runbook.md` 说明。

## 当前事实

- `cds/web/` 是 React、Vite、TypeScript、Tailwind 和 shadcn/ui 控制台。
- 项目列表、项目设置、分支列表、分支详情、拓扑和系统设置已经由 React 路由承载。
- `cds/web-legacy/` 不再被正常 dashboard 服务，只作为历史功能对照层保留。
- legacy 的存在仍会诱导新代码复制旧实现，也让规则、测试和维护者误判双栈仍然有效，因此终态必须物理删除。

## Week 4.5：功能差距最终复核

删除前按用户能力而不是文件名逐项比对：

| 能力域 | 必须确认 |
| --- | --- |
| 项目 | 创建、导入、仓库绑定、删除、Agent key、pending import |
| 分支 | 创建、远程来源、构建、部署、停止、重启、删除、日志、预览 |
| 配置 | env 继承与覆盖、BuildProfile、路由、额外服务、数据库隔离 |
| 运维 | 容量、主机、executor、集群、活动流、系统事件、错误归因 |
| 安全 | GitHub 授权、管理员操作、危险确认、密钥只显示一次 |
| 恢复 | factory reset、自更新、配置快照、备份恢复和失败回滚 |

发现缺口时必须登记到 `plan.cds.backlog-matrix.md`，说明“迁移、放弃或被新能力替代”。禁止仅因 legacy 有按钮就机械复制没有真实后端能力的空壳。

## Week 4.6：视觉与主链路验收

- 双主题覆盖项目列表、项目设置、分支列表、分支详情、部署、日志和系统设置。
- 窄屏和长内容下无遮挡、双滚动、固定高度魔数或深色泄漏。
- 新用户从 Git 仓库到首个可访问预览有连续引导，不需要理解 BuildProfile 内部结构。
- 部署中持续显示阶段和日志；失败给出服务端归因与下一动作。
- 所有危险动作有对象、影响范围和不可逆后果说明。

## Week 4.7：部署与运维页签收口

- 部署页统一展示当前 commit、构建模式、服务状态、入口、日志和失败原因。
- 运维、容量、主机、executor 和活动入口不存在空白 tab 或只读占位。
- 列表与拓扑视图共享服务端动作和状态，不产生只能切回另一视图才能完成的暗门。
- 预览入口使用 CDS 的权威 URL，不在前端自行拼接域名。

## Week 5：物理删除 legacy

开始条件：

1. Week 4.5 每项都有结论，P0/P1 缺口已关闭。
2. React backend、web 类型检查、单测、构建和关键浏览器路径全绿。
3. 生产服务确认不再挂载 `cds/web-legacy/`，访问旧路径有明确重定向或 404 契约。
4. 用户已授权删除旧前端代码；本计划本身不构成删除业务代码的授权。
5. 已记录回退版本，不创建新的 `web-legacy-archive/` 目录继续长期保留副本。

执行内容：

- 删除 `cds/web-legacy/` 及只服务旧前端的静态挂载、fallback、构建和测试。
- 删除 `MIGRATED_REACT_ROUTES` 等过渡期概念，React SPA 直接承担现行控制台路由。
- 清理 legacy 专用 CSS、图标、翻译、事件桥和兼容 API；共享业务规则继续保留。
- 更新 `cds/CLAUDE.md`、运行手册和测试，禁止再把旧代码称为有效对照层。

## 验收门

1. 全仓不存在对 `web-legacy` 的运行时读取、静态挂载或构建依赖。
2. 旧书签要么重定向到语义等价 React 页面，要么返回明确退场响应。
3. React 控制台覆盖现行用户能力，重要路径有双主题浏览器证据。
4. 删除后 backend、web、容器镜像和自更新流程全部通过。
5. 新功能只允许进入 React 栈，不恢复第二套页面实现。
