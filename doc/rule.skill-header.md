# 技能头部规范 · 规则

> **版本**：v1.0 | **日期**：2026-06-24 | **状态**：已落地

## 为什么

`.claude/skills/` 下技能数量多，但头部各写各的，在技能列表和文件里都「很难分辨」：

- 有的整文件**没有 frontmatter**（`add-image-gen-model` 直接 `# Xxx Skill` 开头，harness 只能拿 H1 当描述）
- `description` 从空到 900+ 字符不等，有的不带触发词，中英混杂
- 版本写法三套：`## 版本与优先级` 列「当前版本」/ `> 版本：1.1.0` / 完全没有
- 首节标题混乱：目录 / 核心理念 / Overview / Purpose / 何时触发 / 概述 …

本规则把「技能头部」统一成可一眼分辨的格式，对齐 `doc/rule.doc-naming.md` 已有的文档头规范。

---

## 一、Frontmatter（harness 用，决定技能列表里怎么显示）

每个 `SKILL.md` **必须**以 frontmatter 开头，字段顺序固定：

```yaml
---
name: <kebab-case，必须等于目录名>
description: <一句话能力定位（做什么 → 产出什么）。触发词："X"、"Y"、`/cmd`。>
---
```

- `name`：**必填**，kebab-case，与所在目录名一字不差。
- `description`：**必填**，结构 = 「能力定位一句话 + 触发词收尾」。
  - 建议 ≤ 500 字符；超长说明该进正文，不该挤在 description。
  - 中文为主（API/SSE 等技术术语可英文）。
  - 用户可调用的技能，description **必须以触发词结尾**（与正文头部「触发」字段一致）。
- 多行用 YAML `|` 或 `>` 均可，但仍须含触发词。
- 其他 harness 字段（`model` / `allowed-tools` 等）按官方写法，放 `description` 之后。

---

## 二、正文头部（人读，H1 之后第一行）

H1 之后**必须**紧跟一行 blockquote 元信息，对齐 `doc/rule.doc-naming.md` 的文档头：

```markdown
# <技能中文名>[ — 副标题]

> **版本**：vX.Y.Z | **状态**：<状态> | **触发**：`/xxx`、"关键词"[ | **SSOT**：<规则/设计文档>][ | **领地**：<管辖与避让>]
```

| 字段 | 是否必须 | 说明 |
|------|---------|------|
| **版本** | 必须 | 语义化 `vX.Y.Z`；新增能力升次版本，破坏性改协议升主版本 |
| **状态** | 必须 | `草案` / `规划中` / `开发中` / `已落地` / `已废弃`（同 doc-naming） |
| **触发** | 必须 | 斜杠命令 + 关键词，与 frontmatter `description` 的触发词一致 |
| **SSOT** | 可选 | 该技能依据的规则 / 设计文档路径 |
| **领地** | 可选 | 无人值守 / 多 Agent 协作技能标明管辖范围与避让边界 |

版本号是单一事实源放在这一行；正文里不要再散落「当前版本：x」。

---

## 三、首节顺序（推荐，让结构可辨）

新建技能首节按此序排，存量技能逐步对齐：

```
何时用 / 何时别用  →  前置 / 参数  →  流程  →  边界 / 禁止
```

标题用中文（除非技能本身面向英文场景），不要 `Overview` / `Purpose` 这类英文标题与中文正文混搭。

---

## 四、禁止

- 缺 frontmatter（整文件无 `---` 块）
- `description` 为空 / 只复述标题 / 不含触发词
- 版本只散在正文 `## 版本与优先级`，而 H1 下没有元信息行
- 中英标题混搭（中文技能用英文首节标题）
- 头部出现 emoji（CLAUDE.md §0）

---

## 五、落地

- **新建技能**：由 `create-skill-file` 技能按本规则生成头部。
- **存量技能**：逐步对齐。本规则落地首批样板：`ai-defect-resolve`、`issues-autofix`、`add-image-gen-model`。
- 其余存量技能可在触碰时顺手对齐，或集中跑一次头部对齐 sweep。

---

## 历史背景

2026-06-24 用户反馈技能头部「用起来乱七八糟的，很难分辨」。根因是技能从未有过头部规范——`doc/` 文档早有统一头（`rule.doc-naming.md`），技能却放任自流。本规则把同一套头部纪律引到技能侧。
