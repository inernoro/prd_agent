# CDS 部署架构与冲突分析 · 设计

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：已落地

## 管理摘要

- **解决的问题**：CDS、独立部署、宿主机 Nginx 和分支容器共享网络与端口，启动或切换不当可能造成端口冲突、错误路由和误停进程。
- **核心方案**：以 `cds/exec_cds.sh` 为唯一 CDS 运维入口，启动前检查依赖与占用，统一生成 Nginx 配置，并以可验证、可恢复的步骤管理 Master、Forwarder 和分支运行环境。
- **当前状态**：初始化、启停、证书、集群接入、环境迁移、Nginx 渲染和状态诊断均已收口到入口脚本。
- **事实边界**：脚本和自动测试是可执行 SSOT；本文只记录稳定职责和冲突处理原则，不复制数千行脚本细节。

## 问题背景

CDS 同时管理控制面、分支预览和外部入口。宿主机可能还运行独立版 PRD Agent、Docker Gateway 或已有 Nginx。最危险的情况不是“启动失败”，而是脚本误判占用者后杀错进程、覆盖有效配置，或把公网请求路由到错误分支。

## 设计目标

1. 新机器能够从单一入口完成依赖检查、配置生成和首次启动。
2. 重复执行启动、停止和重启保持幂等。
3. 端口、PID、容器和配置文件在修改前验证归属。
4. Nginx 配置先渲染、再校验、最后切换，失败时保留可用配置。
5. CDS 模式与独立部署模式的入口职责明确，不同时争抢同一公网端口。
6. 失败能通过状态、日志和验收脚本定位，不依赖人工猜测。

## 核心决策

| 决策 | 说明 |
|------|------|
| 单一入口 | 根目录 `exec_cds.sh` 只转发到 `cds/exec_cds.sh`，所有逻辑在后者维护 |
| 控制面与流量面分离 | Master 负责管理 API 和 Dashboard，Forwarder 或 Gateway 负责预览流量 |
| 配置生成而非手改 | 分支和等待页 Nginx 配置由脚本或 TypeScript renderer 生成 |
| 先验证再替换 | Nginx 配置必须通过语法与目标检查后才生效 |
| 归属验证 | 停进程、删容器和释放端口前确认对象属于 CDS |
| 生产入口可回退 | 未激活分支或 CDS 不可用时使用明确的 disconnected 或 waiting 页面，不静默转错服务 |

## 组件与职责

| 组件 | 默认职责 |
|------|----------|
| Host Nginx | TLS 终止、域名入口和到 CDS 流量面的反向代理 |
| CDS Master | 控制面 API、Dashboard、项目与分支调度，默认端口 9900 |
| Forwarder 或 Worker Proxy | 接收预览流量并按域名、分支或路由转发，默认流量端口 5500 |
| 分支容器 | 承载具体分支的 Web、API 与依赖服务 |
| Docker Gateway | 独立部署模式的应用入口，与 CDS 流量面互斥使用宿主端口 |
| MongoDB | CDS 持久状态、认证状态或项目业务数据，具体用途由独立配置控制 |

## 启动与切换流程

1. 加载 `.cds.env` 与项目配置，检查命令、目录、端口和凭据是否齐备。
2. 确认当前运行模式及端口占用者，拒绝处理不属于 CDS 的进程。
3. 准备 MongoDB、构建产物、运行目录和生成型页面。
4. 渲染 Nginx 与 Forwarder 配置，完成语法和路径校验。
5. 启动 Master 与流量面，等待健康端点可用。
6. 输出状态、日志入口和下一步操作。
7. 模式切换时先停止旧入口，再启动新入口；Host Nginx 只指向稳定的本机流量端口。

## 冲突处理

| 冲突 | 处理原则 |
|------|----------|
| 9900 已占用 | 验证是否为当前 CDS Master；未知进程时停止并提示，不自动清理 |
| 5500 已占用 | 区分 CDS Forwarder、Worker Proxy 与独立 Docker Gateway，按目标模式显式切换 |
| PID 文件陈旧 | 同时验证进程存在、命令行和工作目录，不能只凭数字发送信号 |
| 容器同名 | 检查标签、网络和项目归属，禁止删除无关容器 |
| Nginx 配置无效 | 保留旧配置并报告校验错误，不 reload |
| 分支目标缺失 | 展示等待或断开页，不把请求随机转发到其他分支 |
| 状态文件损坏 | 从运行时事实重建可恢复部分，并保留诊断信息 |

## 安全边界

- 入口脚本需要提权的操作必须明确发生在证书、系统服务或宿主 Nginx 管理阶段。
- 环境文件、Token 与连接串不得输出到日志或提交仓库。
- 停止和清理操作必须限定在当前 CDS 目录、已登记服务和带项目归属的容器。
- 公网 HTML 与入口资源可达才算部署完成，不能只以进程存在作为成功。
- `exec_dep.sh`、`fast.sh` 和生产 Nginx 的兼容要求受 `.claude/rules/production-release-safety.md` 约束。

## 验证体系

| 层级 | 验证内容 |
|------|----------|
| 脚本单测 | Nginx 模板、Forwarder 同步、环境文件与安装行为 |
| 路由测试 | 分支、项目、资源、集群和外部访问的控制面契约 |
| 服务测试 | 代理恢复、容器网络隔离、调度与状态一致性 |
| 全量验收 | 首次启动、基础设施、API、分支生命周期、网关、幂等、混沌恢复和公网入口 |

历史全量脚本位于 `scripts/cds-acceptance-test.sh`。新增能力优先进入 `cds/tests/` 的可重复测试，避免继续扩大单一 Shell 验收脚本。

## 事实来源

| 文件 | 职责 |
|------|------|
| `exec_cds.sh` | 仓库根目录转发入口 |
| `cds/exec_cds.sh` | CDS 初始化与运维唯一实现 |
| `cds/src/forwarder/` | 预览流量转发 |
| `cds/src/loading-pages/` | 等待与断开页面生成 |
| `deploy/nginx/conf.d/branches/` | 独立、断开与分支路由配置 |
| `docker-compose.yml` | 独立部署 Gateway 与基础服务 |
| `cds/tests/scripts/` | 入口脚本与模板回归测试 |
| `scripts/cds-acceptance-test.sh` | 历史全量验收入口 |
