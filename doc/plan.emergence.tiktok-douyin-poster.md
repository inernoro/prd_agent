# 抖音与 TikTok 真订阅闭环 · 计划

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：规划中

## 目标

在已经落地的短视频解析、创作者作品抓取、ASR、素材重托管和 feed-card 海报链路上，补齐用户授权、定时增量同步和去重通知，让“手工粘贴链接生成海报”升级为真实订阅。

## 已有事实

- Workflow 已有短视频解析、创作者作品抓取、视频转文字、素材重托管和首页海报发布能力。
- TikTok、抖音、B 站、小红书和 YouTube 的响应差异已有适配；这些已完成实现不属于本计划。
- `ad-rich-text`、feed-card、ASR 字幕和最小化交互已经落地，已知边界归 `debt.workflow-agent.md`。
- 当前缺口是抖音授权身份、长 token 生命周期、可靠调度和新增作品去重。

## P1：授权与凭据

1. 明确使用官方开放平台或受支持的第三方授权能力，不保存用户账号密码或浏览器 cookie。
2. 服务端保存 access token、refresh token、过期时间、scope、平台用户标识和授权状态；敏感值加密且不回前端。
3. 支持首次授权、续期、撤销、失效重连和多账号区分。
4. 授权失败要区分用户拒绝、scope 不足、平台限流、token 过期和上游不可用。

## P2：订阅模型与增量同步

- 订阅记录包含平台、创作者稳定 ID、展示名称、同步游标、最近成功时间、失败次数和启停状态。
- 每次抓取只处理游标之后的作品，使用平台作品 ID 作为幂等键。
- 同一作品重复返回时不得重复创建素材、字幕、海报或通知。
- 首次订阅默认只导入有限历史数量，执行前展示预计范围和模型成本。

## P3：可靠调度

- 优先复用已经可用的服务端 timer 或任务调度事实源；`wip` 能力不能作为生产依赖。
- 调度任务需要唯一 owner、租约、超时、重试和取消，服务重启后可恢复。
- 单个创作者失败不能阻断其他订阅；连续失败进入退避并通知用户重新授权或检查来源。
- 手动“立即同步”与定时同步必须走同一服务，不能形成第二套逻辑。

## P4：产品闭环

- 用户可以查看订阅、最近同步、最新作品、失败原因和下一次计划时间。
- 新作品通过既有 workflow 生成素材、字幕和海报，产物可追溯到订阅和原作品。
- 通知只在新作品成功入库后发送；生成失败与抓取失败分别展示。
- 支持暂停、恢复和删除订阅；删除不默认删除已生成的用户资产。

## 验收门

1. 授权、续期、撤销和 token 失效均有端到端测试，前端看不到明文凭据。
2. 连续两次同步同一返回结果不会产生重复数据或通知。
3. 新作品从发现到海报入库和通知可追溯，失败可从记录重试。
4. 服务重启、任务超时、平台限流和单订阅异常不会造成全局停摆。
5. 手动与定时路径行为一致，调度期间有持续进度反馈。
