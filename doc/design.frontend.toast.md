# Toast 通知系统 · 设计

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：已落地

## 一、管理摘要

- **解决什么问题**：轻量成功、失败和提示信息不应使用阻塞对话框打断用户。
- **当前方案**：全局 Zustand 队列管理 Toast，提供 success、error、info、warning、action、loading 和 dismiss 能力。
- **边界**：Toast 只承载短暂反馈；需要确认、长内容、表单修正或不可逆风险时使用 Dialog 或页面内错误。
- **事实入口**：调用 API 和状态以 `prd-admin/src/lib/toast.tsx` 为准，文档不复制组件代码和动画 CSS。

## 1. 使用场景

| 场景 | 反馈方式 |
|------|----------|
| 操作成功且无需后续决定 | success Toast |
| 操作失败但页面仍可继续 | error Toast，并保留页面状态 |
| 输入不完整或存在风险 | warning Toast，必要时聚焦对应字段 |
| 中性状态或复制结果 | info Toast |
| 可撤销、可重试动作 | 带 action 的 Toast |
| 短时异步过程 | loading Toast，完成后 dismiss 并显示终态 |

涉及删除确认、权限授予、覆盖数据、发布和支付等不可逆或高风险操作，必须先用 Dialog 或专用页面确认，不能仅靠 Toast。

## 2. 数据与 API

每条 Toast 包含唯一 ID、类型、标题、可选消息、持续时间和可选操作。Store 负责添加和移除，展示组件订阅队列。

| API | 用途 | 默认倾向 |
|-----|------|----------|
| `toast.success` | 成功反馈 | 较短展示 |
| `toast.error` | 可恢复错误 | 比成功稍长 |
| `toast.info` | 中性提示 | 较短展示 |
| `toast.warning` | 风险与校验提示 | 按可读长度调整 |
| `toast.action` | 撤销或重试 | 给用户足够操作时间 |
| `toast.loading` | 返回 ID 的持续提示 | 必须在终态 dismiss |
| `toast.dismiss` | 移除指定 Toast | 用于 loading 和主动关闭 |

默认时长是实现细节。调用方只有在内容长度、风险或动作需要时才覆盖，不能让普通提示长期占据页面。

## 3. 队列与生命周期

- 新 Toast 追加到全局队列，并由唯一 ID 标识。
- 普通 Toast 到期自动移除，也允许用户主动关闭。
- action 执行后关闭对应 Toast，业务失败可产生新的错误反馈。
- loading 必须保存返回 ID，在成功、失败、取消和组件卸载路径中清理。
- 页面跳转不应制造重复 Toast；同一失败循环需要在调用层去重。

若队列增长过快，展示层应限制可见数量并保留最新或最高优先级信息。不能因隐藏旧 Toast 而丢失需要持久展示的表单错误。

## 4. 内容规则

- 标题直接说明结果，不使用模糊的“提示”。
- message 说明原因或下一步，不重复标题。
- 后端错误优先展示可安全公开的 `error.message`，不直接输出对象或堆栈。
- 成功反馈不加入无意义感叹和装饰符号。
- action 标签使用明确动词，如“撤销”“重试”“查看”。
- 同一操作只产生一个主反馈，避免页面提示、Toast 和 Dialog 同时重复。

## 5. 视觉与可访问性

| 要求 | 约束 |
|------|------|
| 位置 | 不遮挡主导航、保存按钮和移动端底部操作区 |
| 层级 | 高于普通内容，低于需要用户决策的模态框 |
| 类型表达 | 使用颜色、文本和 SVG 图标，不只依赖颜色 |
| 动效 | 快速进入和退出，遵循 reduced motion 设置 |
| 屏幕阅读器 | 使用合适的 live region，区分错误与普通提示 |
| 键盘 | action 可聚焦，关闭操作有可访问名称 |
| 长文本 | 截断或转为页面内错误，不无限扩张 Toast |

## 6. Toast 与其他反馈

| 需要 | 使用 |
|------|------|
| 用户必须做选择才能继续 | Dialog |
| 表单字段需要修正 | 字段旁错误与页面摘要 |
| 长任务持续进度 | 页面进度、SSE 时间线或任务中心 |
| 页面整体无法加载 | 页面错误态与重试入口 |
| 短暂操作结果 | Toast |

静止的 loading Toast 不能替代长任务可视化。超过短时等待的 AI 或批量任务必须展示持续变化的阶段和进度。

## 7. 当前事实入口

| 能力 | 事实入口 |
|------|----------|
| 状态和调用 API | `prd-admin/src/lib/toast.tsx` |
| 全局渲染 | 应用根布局中的 Toast 容器 |
| 调用示例 | `prd-admin/src/pages/` 下各业务页面 |

## 8. 验收标准

- 四种基础类型能展示并自动关闭。
- action 可通过键盘触发，执行后 Toast 关闭。
- loading 在所有终态清理，不残留长时间占位提示。
- 多条提示不会遮挡主要操作或突破移动端视口。
- 需要确认和字段修正的场景没有被错误替换为 Toast。
- 屏幕阅读器能获知新提示，动效尊重 reduced motion。
