# LLM Gateway 控制台产品化与独立目录收拢 · 计划

> **版本**：v1.5 | **日期**：2026-07-12 | **状态**：验收完成

## 一、概述

**目标**：把已经完成租户化、四协议接入和 full-http 切换的 LLM Gateway，从“能力已经存在但普通用户难以理解”的控制台，推进为能看懂、能直测、能解释数据来源，并可在后续独立迁出的产品。

**前序 SSOT**：`plan.platform.llm-gateway-external-platform.md` 已完成并保持归档状态。本计划只承接产品心智、教程、页面分层和物理目录收拢，不重新打开前序五个 PR。

**范围边界**：

- 包含：双主题可发现性、请求记录、Quickstart 直测、密钥身份解释、普通用户与内部运维信息隔离、租户全局首页、学习中心、模型池信息分层、根目录 `llmgw/` 收拢。
- 不包含：模型迁移、模型池调度算法重写、full-http 切换、发布 Gate 重建、批量付费模型测试、对 OpenRouter 的视觉或品牌复制。
- 运行约束：tenant 只从服务端会话或 service key 解析；所有租户查询、唯一索引、日志、预算和审计继续包含 `TenantId`。
- 费用约束：unknown cost 不显示为 0；CNY 与 USD 没有可审计汇率时不相加。
- 测试约束：每类真实协议最多调用一次，其余使用假上游。

## 二、用户心智

控制台应让用户在进入页面后三秒内回答四个问题：

1. 当前租户使用了多少请求、由谁和哪个 appCaller 发起、成功率与速度如何。
2. 如何创建外部接入密钥并立即验证 Gateway 地址、密钥和路由是否有效。
3. “请求记录”记录什么，为什么现在为空，产生请求后去哪里定位错误和费用。
4. 模型池负责什么、每个池承担什么业务、流量如何进入池以及池内模型如何排序。

平台存在两种调用身份，页面必须明确区分：

| 身份 | 用途 | 是否出现在租户密钥列表 | 是否可交给外部系统 |
|---|---|---|---|
| 租户接入密钥 | 外部团队、脚本和应用调用 Gateway | 是 | 是，仅显示一次明文 |
| 平台内部身份 | MAP 等平台内部服务调用 Gateway | 否 | 否 |

“租户密钥为空但 MAP 仍可调用”因此不等于绕过鉴权，但页面必须解释两类身份的边界，避免用户把空列表理解为无鉴权。

## 四、分阶段计划

### P0：先让用户看懂并敢于测试

**目标**：修复不需要重建数据模型即可解决的高频困惑，让新用户能切换主题、理解请求记录、拿到正确地址、创建密钥并安全验证。

**交付物**：

1. 主题默认跟随系统，并在控制台设置中提供“跟随系统、浅色、深色”三种明确选项；用户菜单保留快捷切换。
2. 导航和页面统一使用“请求记录”，英文 Activity 仅作为行业术语说明，不作为主标签。
3. 请求记录空态说明数据产生条件，提供“去快速接入”和“查看示例说明”入口；示例不得伪装为真实租户数据。
4. Quickstart 从服务端公开配置或当前部署入口推导 Gateway 地址，禁止回退到 `gateway.example.com`；页面提供不持久化密钥的安全验证，并明确是否会调用真实上游及可能费用。
5. 接入密钥空态显示“当前租户外部密钥为 0”，提供创建第一把密钥入口，并解释 MAP 内部身份不出现在这里。
6. 普通租户不显示发布 Gate、容器拓扑、迁移状态和“GW 权威”等开发术语；内部运维能力进入有权限的系统运维页面。

**验收标准**：

- [x] 首次打开时主题与操作系统一致，设置页可见三种主题选项，刷新后选择仍生效。
- [x] 桌面与移动导航均显示“请求记录”，空数据时仍能理解用途和下一步。
- [x] Quickstart 地址来自真实部署配置或当前 origin，页面源代码和渲染结果均无 `gateway.example.com`。
- [x] 未输入密钥不能发送测试；输入只保存在当前页面内存，不写入浏览器持久化存储和日志。
- [x] 安全验证能区分“只验证配置”和“真实请求”，不会在用户无确认时触发付费上游。
- [x] 空密钥页能解释平台内部身份，且不能展示或推导平台内部密钥。
- [x] 外部租户通过导航和直接访问均不能看到内部运维页；内部管理员仍可访问。
- [x] 1280×800、1440×900、390×844 下浅色和深色无横向溢出，主操作与空态可见。

### P1：租户全局首页与学习中心

**目标**：把概览变成当前租户的全局使用首页，并提供可随项目发布的教程。

**交付物**：

1. 首页展示请求量、成功率、P95、实时速率、token、费用可信度、活跃用户、Top appCaller、Top 模型和密钥状态。
2. 所有统计都以服务端会话 TenantId 为边界，支持时间范围切换，不接受前端自报 tenantId。
3. 新增学习中心，解释租户、团队、用户、appCaller、密钥、模型池、模型、Provider、Exchange、请求记录和费用口径。
4. 首页第一屏继续优先健康状态、快速接入、最近请求和费用可信度；内部运行状态不回流首页。

**验收标准**：

- [x] 两个租户看到的首页统计、Top 列表和最近请求完全隔离。
- [x] 无数据、少量数据、部分未知费用和多币种场景均有明确表达。
- [x] 用户从首页到首次请求不超过三个明确步骤。
- [x] 教程文档与页面入口随版本发布，页面术语能跳转到对应说明。

### P2：模型池用户心智与高级维护分层

**目标**：先解释模型池承担的业务，再提供维护工具，避免用户先面对大面积批量表单。

**交付物**：

1. 模型池首屏按池展示用途、绑定 appCaller、近期流量、健康、模型组成和优先级。
2. 创建、编辑与成员详情进入抽屉或详情页；批量导入、币种校准和高级筛选默认折叠。
3. 清理持久化描述中的迁移遗留话术；系统生成说明使用面向用户的当前态文案。
4. 将 Provider、模型、Exchange 中的“权威平台”“确认写入”等开发术语替换为“平台配置”“导入到平台”等可执行文案。

**验收标准**：

- [x] 用户无需滚动即可说清每个池负责什么、是否健康、由哪些 appCaller 使用。
- [x] 80% 常用操作在首屏完成，高级维护不遮挡主任务。
- [x] 页面不出现 P3、legacy、stub、full-http、Gate、权威迁移等开发阶段文案。

### P3：根目录 `llmgw/` 收拢

**目标**：像 `cds/` 一样形成可独立构建、部署和迁出的顶层模块，同时保持线上运行语义不变。

**目标结构**：

```text
llmgw/
  console-api/
  web/
  serving/
  deploy/
  docs/
```

**交付物**：

1. 将 `llmgw/console-api`、`llmgw/web` 和 `PrdAgent.LlmGateway` 相关工程按依赖顺序收拢。
2. 同步 solution、Docker、Compose、CI、CDS、脚本和文档路径。
3. 用兼容入口或一次性原子迁移保证调用地址、协议、配置集合、TenantId 语义和发布流程不变。

**验收标准**：

- [x] 根目录只保留一个 `llmgw/` 产品边界，旧目录无双写和影子构建。
- [x] 四协议、租户隔离、密钥撤销、PromptPolicy、日志、费用覆盖率测试无回归。
- [x] CDS 与生产构建使用新路径，线上 URL 和数据库集合语义不变。

## 六、风险与依赖

| 风险或依赖 | 影响 | 缓解策略 |
|---|---|---|
| Gateway 地址与控制台入口并非同域 | 自动推导可能错误 | 服务端公开配置优先，当前 origin 仅作同域回退；无法确认时明确不可测试，不给假地址 |
| 内部运维标记只在前端控制 | 可被直接访问绕过 | 服务端会话产生标记，路由守卫与接口权限同时校验 |
| 浏览器直接测试泄露密钥 | 安全风险 | 仅内存持有、不持久化、不进入 URL、错误日志和分析事件 |
| 模型池说明再次成为静态遗留 | 页面与运行事实漂移 | 用绑定、流量、健康和成员数据动态生成，静态描述只作可编辑补充 |
| P3 路径迁移引起发布回归 | 可能影响线上 | 独立批次、原子路径更新、四协议假上游回归和部署清单逐项验证 |
