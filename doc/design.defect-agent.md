# 缺陷管理 Agent 总体设计 · 设计

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：已落地

> **appKey**：`defect-agent`

## 一、管理摘要

- **解决什么问题**：缺陷提交信息不完整、处理状态不透明、内外部协作与验收证据分散。
- **当前方案**：以缺陷报告为主实体，组合模板、附件、AI 审核、标准状态机、分享修复和自动化运行，形成从提交到关闭的闭环。
- **文档定位**：本文是能力边界和事实入口，不复制 Controller、数据模型、提示词或页面实现。
- **专题拆分**：截图分析、外部分享修复和自动化自治由独立设计文档维护。

## 1. 产品边界

缺陷管理 Agent 负责：

- 创建、编辑、查询、软删除、恢复和永久删除缺陷。
- 使用个人、共享或默认模板组织提交信息。
- 通过 AI 审核发现缺失信息，并保留对话和附件证据。
- 指派处理人，推进处理、解决、验证、关闭和重新打开。
- 生成趋势、人员和状态统计。
- 通过受控分享链接与外部 Agent 协作。
- 承载自动化运行、评论、提交信息、修复状态和验证报告。

本模块不负责通用项目管理、代码托管、消息通知或模型调度；这些能力通过项目、Webhook、Automation Hub、LLM Gateway 等平台边界接入。

## 2. 角色与权限

| 角色 | 主要能力 |
|------|----------|
| 报告人 | 创建和补充缺陷、查看有权访问的记录、执行验收 |
| 处理人 | 接收指派、更新处理信息、提交解决说明和修复证据 |
| 管理者 | 管理模板、范围、指派、状态纠偏和统计视图 |
| 外部协作者 | 在分享和密钥授权范围内读取、评论或回传修复结果 |
| 自动化 Agent | 在显式授权和运行记录约束下推进指定步骤 |

权限判断必须同时考虑应用权限、记录可见范围、项目或团队关系以及分享授权。分享 token 不是内部管理权限的替代品。

## 3. 生命周期

主流程状态包括草稿、审核、待补充、已提交、已指派、处理中、已解决、已拒绝和已关闭。具体枚举以当前模型为准，设计约束如下：

| 阶段 | 进入条件 | 退出要求 |
|------|----------|----------|
| 草稿 | 用户创建或保存未完成内容 | 用户提交审核 |
| 审核或待补充 | AI 检查模板要求和现有证据 | 信息完整后进入已提交 |
| 已提交 | 缺陷内容可供处理 | 管理者或规则完成指派 |
| 已指派或处理中 | 已明确责任人 | 提交解决、拒绝或阻塞原因 |
| 已解决 | 处理人给出修复说明和证据 | 验证通过后关闭，失败则退回 |
| 已关闭 | 验收完成或管理者确认结束 | 有新证据时可重新打开 |

软删除与业务状态分离。进入回收站不等同于关闭，恢复后应保留删除前的业务状态和版本历史。

## 4. 领域模型

模型按职责分为以下家族：

| 家族 | 责任 |
|------|------|
| 模板 | 字段要求、默认模板、个人和共享范围 |
| 缺陷报告 | 编号、结构化内容、状态、严重程度、人员和时间 |
| 消息与附件 | 用户补充、AI 回复、图片和文件证据 |
| 版本与审计 | 内容变更、状态操作和关键字段历史 |
| 分享与修复报告 | 外部访问范围、有效期、回传结果和验收结论 |
| 自动化运行 | 授权、连接器、步骤、评论、提交和验证证据 |
| 项目与团队 | 缺陷归属、协作者范围和统计维度 |

Controller 不应成为领域事实源。状态规则、授权和外部协作契约应由模型与服务表达，页面只消费 API 结果。

## 5. API 能力域

| 能力域 | 包含的操作 |
|--------|------------|
| 模板 | 列表、个人和共享管理、默认模板选择 |
| 缺陷 | 列表、详情、创建、更新、版本历史和严重程度 |
| 回收站 | 软删除、恢复、永久删除 |
| 审核与对话 | 提交 AI 审核、消息、附件、日志和预览代理 |
| 状态流转 | 指派、处理、解决、验证、拒绝、关闭和重开 |
| 统计 | 总览、趋势、人员和状态分布 |
| 协作 | 项目、团队和 Webhook |
| 分享 | 分享链接、外部评论、修复状态、修复报告验收 |
| 自动化 | 授权、运行、下一步、阻塞、完成、提交和验证报告 |

端点明细以 `DefectAgentController` 及其请求模型为准。设计文档只在能力边界变化时更新，不跟随每个路由参数重复维护。

## 6. AI 与附件处理

- AI 审核从模板要求、用户输入和已有证据中识别缺失项，不应凭空补写事实。
- 截图分析先建立受控附件引用，再通过统一视觉模型能力提取可验证观察。
- 所有 LLM 调用经过 Gateway，使用登记的 appCallerCode，并把阶段进度持续推送给前端。
- AI 输出只能建议严重程度、结构化字段或后续动作；权限和状态转换由服务端规则裁决。
- 附件访问必须经过记录权限或有效分享授权，不直接信任客户端 URL。

截图分析的细节由 `design.defect-agent.image-analysis` 维护。

## 7. 外部分享边界

分享链接定义可见缺陷范围和有效期；Agent API key 定义调用身份和用途。二者共同存在时仍需在端点校验实际授权，不能把 scope 文案误认为已经完成强制门控。

外部协作者可以在授权范围内读取缺陷、追加分析评论、更新修复状态或提交修复报告。内部人员负责接受或驳回报告，并据此推进业务状态。

详细契约由 `design.defect-agent.share-skill-architecture` 维护。

## 8. 自动化自治边界

自动化能力必须满足：

- 每次运行有明确缺陷、授权主体和允许动作。
- 下一步、阻塞、完成和失败均形成可查询事件。
- 评论、commit 信息、修复状态和验证报告可追溯到运行。
- Agent 不得越过服务端状态机直接修改最终状态。
- 外部依赖不可用时显式阻塞，不制造虚假进度。

详细运行模型由 `design.defect-agent.automation-autonomy` 维护。

## 9. 前端信息架构

前端围绕三个主要视图组织：缺陷列表、缺陷详情、模板与管理设置。详情页集中展示内容、附件、状态、处理记录、分享修复和自动化证据，避免把同一事实复制到多个孤立页面。

关键交互必须满足：

- 提交入口支持自然语言和直接粘贴附件。
- AI 审核期间持续显示阶段和增量结果。
- 每个状态按钮说明前置条件和操作后果。
- 回收站、分享和自动化区域与正常处理流明确区分。
- 无权限或记录不存在时不泄露缺陷内容。

## 11. 验收标准

- 新建缺陷可从草稿经审核、指派、处理、解决和验收形成完整证据链。
- 权限、记录范围和分享授权在所有读取与写入入口一致生效。
- 回收站操作不破坏业务状态、版本历史和审计信息。
- AI 失败不导致状态被误推进，用户能看到失败阶段和重试入口。
- 外部修复报告必须经过内部验收才能影响最终结论。
- 自动化运行的每个动作都能追溯到授权、输入和结果。
