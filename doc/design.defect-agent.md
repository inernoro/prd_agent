# 缺陷管理 Agent · 设计

> **appKey**：`defect-agent` | **状态**：已实现

## 一、管理摘要

- **解决什么问题**：缺陷提交流程表单复杂、信息不完整率高，指派跟踪缺乏标准工单流转，外部团队无法参与缺陷分析
- **方案概述**：自然语言 + 粘贴附件的极简提交方式，AI 自动审核信息完整性并友好追问，标准工单状态机管理全生命周期，支持外部分享 + AI 修复报告 + 验收闭环
- **业务价值**：降低缺陷提交门槛，AI 审核保障信息质量，分享链接让外部团队参与分析，修复报告沉淀为团队知识资产
- **影响范围**：prd-api（DefectAgentController，3900+ 行）、prd-admin（缺陷管理页面）、Visual Agent（截图 VLM 分析）、Webhook（外部集成）

## 二、用户场景与协同涌现

### 场景 1：测试员的一天

> 测试员小张发现一个 UI 渲染异常。

1. 小张打开缺陷 Agent → 用自然语言描述问题 → 粘贴截图
2. AI 自动审核："缺少复现步骤，请补充在什么页面、什么操作后出现的？"
3. 小张补充后提交 → 系统自动分类严重等级、指派给对应开发者
4. 开发者收到工单 → 修复 → 标记已修复 → 小张验收关闭

**以前**：填 10 个字段的表单，经常漏填，开发者还得反复追问。
**现在**：说两句话 + 贴截图，AI 替你追问，工单自动流转。

### 场景 2：截图智能分析（与 Visual Agent 协同）

> 测试员提交缺陷时附了截图但描述模糊："这个页面有问题"。

1. 缺陷提交 → 系统检测到附件是截图
2. 自动调用 Visual Agent 的 Vision 能力分析截图
3. VLM 识别出：按钮文字溢出、图标缺失、布局错位
4. 分析结果自动补充到缺陷描述中
5. 开发者打开工单就能看到精确的 UI 异常描述

**协同涌现**：Vision 能力是 Visual Agent 的，但通过事件驱动自动服务了缺陷管理流程。

### 场景 3：缺陷分享给外部团队

> 产品缺陷需要外部供应商配合分析。

1. 项目经理生成分享链接（可设密码和过期时间）
2. 供应商打开链接 → 看到缺陷详情 + 截图 + AI 分析
3. 外部 Agent 可对缺陷进行独立分析 → 生成修复报告
4. 修复报告回传到缺陷系统 → 内部团队验收

**协同涌现**：分享链接打通了内外部边界，外部团队无需登录也能参与缺陷分析。

### 场景 4：缺陷驱动自动通知（与工作流协同）

> 关键模块出现高优缺陷时，相关负责人应立即收到通知。

1. 测试员提交缺陷 → 系统发布事件 `defect-agent.report.created`
2. AutomationHub 匹配规则（严重等级=高 + 模块=支付）
3. 触发工作流 → LLM 分析舱生成摘要 → 邮件发送舱通知负责人
4. 同时 Webhook 发送舱 → 更新 TAPD 工单

**协同涌现**：缺陷 Agent 只负责管理缺陷，通知和外部同步全部交给工作流。

## 1. 功能概述

缺陷管理 Agent 是一个极简高效的缺陷提交与跟踪系统，类似工单系统。核心理念：**人类交互极简，AI 辅助审核**。

### 1.1 核心价值

- **极简提交**：自然语言描述 + 粘贴/拖拽附件，无复杂表单
- **AI 审核**：自动检查是否符合模板要求，不完整时友好提醒
- **工单流转**：标准工单状态机，指派→处理→解决→关闭
- **权限清晰**：测试人员提交，Admin 指派，被指派人处理

### 1.2 用户场景

> 作为测试人员，我希望用最简单的方式（打字+粘贴截图）提交缺陷，AI 帮我检查信息是否完整，然后 Admin 指派给开发处理。

### 4.2 状态说明

| 状态 | 说明 | 可操作角色 |
|------|------|-----------|
| `draft` | 草稿，尚未提交 | 报告人 |
| `reviewing` | AI 正在审核 | 系统 |
| `awaiting` | 信息不完整，等待补充 | 报告人 |
| `submitted` | 已提交，等待指派 | Admin |
| `assigned` | 已指派处理人 | 被指派人 |
| `processing` | 处理中 | 被指派人 |
| `resolved` | 已解决 | 报告人（确认）/ Admin |
| `rejected` | 已拒绝（非缺陷/无法复现） | 被指派人 |
| `closed` | 已关闭 | Admin |

## 5. 数据模型

### 5.1 MongoDB 集合

| 集合名 | 说明 |
|--------|------|
| `defect_templates` | 缺陷提交模板（Admin 配置） |
| `defect_reports` | 缺陷报告主表 |
| `defect_messages` | 提交过程中的对话消息 |

### 5.2 DefectTemplate（缺陷模板）

```javascript
{
  _id: ObjectId,
  name: string,              // 模板名称，如"默认模板"
  description: string,       // 模板描述
  requiredFields: [          // 必填信息项
    {
      key: string,           // 字段标识，如 "title", "steps", "severity"
      label: string,         // 显示标签，如 "问题标题", "复现步骤"
      type: "text" | "select",
      options: string[]?,    // select 类型的选项
      aiPrompt: string       // AI 追问时的提示语
    }
  ],
  aiSystemPrompt: string,    // AI 审核系统提示词
  isDefault: bool,           // 是否为默认模板
  createdBy: string,         // 创建人 UserId
  createdAt: DateTime,
  updatedAt: DateTime
}
```

**默认模板必填字段**：
- `title` - 问题标题
- `description` - 问题描述
- `steps` - 复现步骤
- `expected` - 预期结果
- `actual` - 实际结果
- `severity` - 严重程度（阻塞/严重/一般/建议）

### 5.3 DefectReport（缺陷报告）

```javascript
{
  _id: ObjectId,
  defectNo: string,          // 缺陷编号 DEF-2025-0001（自增）
  templateId: string,        // 使用的模板 ID

  // 内容
  rawContent: string,        // 用户原始输入文本
  structuredData: {          // AI 提取的结构化数据
    title: string,
    description: string,
    steps: string,
    expected: string,
    actual: string,
    severity: string,
    // ... 其他模板字段
  },
  attachments: [             // 附件列表
    {
      id: string,            // 附件 ID
      fileName: string,      // 文件名
      fileSize: number,      // 文件大小 (bytes)
      mimeType: string,      // MIME 类型
      cosUrl: string,        // COS 存储 URL
      thumbnailUrl: string?, // 缩略图 URL（图片类型）
      uploadedAt: DateTime
    }
  ],

  // 状态
  status: string,            // draft/reviewing/awaiting/submitted/assigned/processing/resolved/rejected/closed
  missingFields: string[],   // 当前缺失的必填字段

  // 人员
  reporterId: string,        // 报告人 UserId
  assigneeId: string?,       // 被指派人 UserId

  // 处理信息
  assignedAt: DateTime?,     // 指派时间
  resolution: string?,       // 解决说明
  rejectReason: string?,     // 拒绝原因

  // 时间戳
  submittedAt: DateTime?,    // 正式提交时间
  resolvedAt: DateTime?,     // 解决时间
  closedAt: DateTime?,       // 关闭时间
  createdAt: DateTime,
  updatedAt: DateTime
}
```

### 5.4 DefectMessage（对话消息）

```javascript
{
  _id: ObjectId,
  defectId: string,          // 关联的缺陷 ID
  seq: number,               // 消息序号（用于 SSE afterSeq）
  role: "user" | "assistant",
  content: string,           // 消息内容
  attachmentIds: string[]?,  // 本条消息携带的附件
  extractedFields: object?,  // AI 从该消息提取的字段
  createdAt: DateTime
}
```

## 6. 接口设计

### 6.1 模板管理（Admin）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/defect-agent/templates` | 获取模板列表 |
| POST | `/api/defect-agent/templates` | 创建模板 |
| PUT | `/api/defect-agent/templates/{id}` | 更新模板 |
| DELETE | `/api/defect-agent/templates/{id}` | 删除模板 |

### 6.2 缺陷管理

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/defect-agent/defects` | 获取缺陷列表（支持筛选） |
| GET | `/api/defect-agent/defects/{id}` | 获取缺陷详情 |
| POST | `/api/defect-agent/defects` | 创建缺陷（草稿） |
| PUT | `/api/defect-agent/defects/{id}` | 更新缺陷 |
| DELETE | `/api/defect-agent/defects/{id}` | 删除缺陷（仅草稿） |

### 6.3 缺陷状态操作

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/defect-agent/defects/{id}/submit` | 提交缺陷（触发 AI 审核） |
| POST | `/api/defect-agent/defects/{id}/assign` | 指派处理人（Admin） |
| POST | `/api/defect-agent/defects/{id}/process` | 开始处理 |
| POST | `/api/defect-agent/defects/{id}/resolve` | 标记解决 |
| POST | `/api/defect-agent/defects/{id}/reject` | 拒绝缺陷 |
| POST | `/api/defect-agent/defects/{id}/close` | 关闭缺陷 |
| POST | `/api/defect-agent/defects/{id}/reopen` | 重新打开 |

### 6.4 对话消息（AI 审核）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/defect-agent/defects/{id}/messages` | 获取对话消息 |
| POST | `/api/defect-agent/defects/{id}/messages` | 发送消息 |
| GET | `/api/defect-agent/defects/{id}/messages/stream` | SSE 订阅 AI 回复 |

### 6.5 附件上传

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/defect-agent/attachments` | 上传附件到 COS |
| DELETE | `/api/defect-agent/attachments/{id}` | 删除附件 |

## 10. AI 审核逻辑

### 10.1 审核流程

```
用户提交内容
    │
    ▼
AI 分析内容 + 提取结构化数据
    │
    ├── 必填字段完整 ──▶ 状态变为 submitted
    │
    └── 缺少必填字段 ──▶ 状态变为 awaiting
                         │
                         ▼
                    AI 生成友好提醒
                         │
                         ▼
                    用户补充信息
                         │
                         ▼
                    重复审核...
```

### 10.2 AI 系统提示词（默认模板）

```
你是一个缺陷报告审核助手。用户会用自然语言描述遇到的问题。

你的任务：
1. 从用户描述中提取结构化信息
2. 检查是否包含必填字段：标题、描述、复现步骤、预期结果、实际结果、严重程度
3. 如果信息不完整，用友好的语气询问缺失的内容
4. 如果信息完整，确认并总结

注意：
- 保持友好、简洁的语气
- 一次只询问 2-3 个问题，不要一次问太多
- 如果用户提供了截图，参考截图内容
```
