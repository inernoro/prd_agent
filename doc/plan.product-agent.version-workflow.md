# 产品管理智能体版本流程整改计划

## 0. 2026-06-10 首版实现状态

已完成：

- 版本页拆分为“上线 / 立项”两个标签，默认展示上线。
- 立项三步向导：基础信息、Review Agent 文件评审、评审会/负责人决策。
- 负责人同意或选择召开评审会后生成 T 号。
- 上线时回填立项方案和需求，并允许追加需求、选择项目组成员和上线时间。
- 自动生成 V 号；上线公告地址必填后才可完成上线。
- 临时优化需求入口、定义提示和无 T 号的小版本上线。
- 历史立项/上线导入预览，支持 `.xlsx` 和 `.csv`。
- 原 `ProductVersion` 数据保留在“旧版版本数据”折叠区。

当前限制：

- 旧二进制 `.xls` 暂不直接解析，界面会提示另存为 `.xlsx`。
- 本机缺少 .NET SDK，本轮未能执行 API 编译和接口测试。
- 本地浏览器验证到登录门禁，未使用默认账号进入系统。

## 一、背景与目标

团队原先使用 TAPD 管理需求与缺陷、使用语雀维护 T 立项号和 V 上线号，并通过产品委员会流程完成立项、评审、版本申领和上线记录。现改由 MAP 的产品管理智能体统一承接这些能力。

本期整改重点是产品内的“版本”页面：

- 保留现有产品、版本、需求、功能、缺陷及其关联数据，不做破坏性迁移。
- 需求详情页和缺陷详情页后续参考 TAPD 调整，本期不做大改。
- 将现有版本页面拆分为“上线”和“立项”两个子页面，默认展示“上线”。
- 新立项接入简化版 Agent 评审和负责人审批逻辑。
- 常规上线从立项记录发起；临时优化需求允许不关联 T 立项号。
- 支持 Excel 历史数据导入，承接原语雀/TAPD/产品委员会数据。

## 二、本期范围

### 2.1 本期实施

1. 重构单产品“版本”页面的信息架构。
2. 新增立项列表、立项流程和立项详情。
3. 新增上线列表、上线表单和上线详情。
4. 对接产品管理内部需求，不再依赖 TAPD 需求作为新数据源。
5. 简化集成现有 `review-agent` 的方案评审能力。
6. 实现 T 立项号和 V 上线号的服务端生成与唯一性约束。
7. 立项、上线分别支持 Excel 历史导入。
8. 保留并兼容现有 `ProductVersion` 数据。
9. 增加不关联 T 立项号的临时优化需求特殊上线通道。

### 2.2 本期不实施

1. 不全面重构需求详情页。
2. 不全面重构缺陷详情页。
3. 不直接在 MAP 内发布 Baklib 上线公告。
4. 不迁移或删除现有版本和大版本升级申请数据。
5. 不复刻原 `review-agent` 的完整独立页面。
6. 不恢复旧产品委员会系统的日常巡检、语雀发布和 TAPD 自动抓取能力。

1. `上线`：默认选中。
2. `立项`。

标签切换只切换内容区，不改变产品级左侧导航。

## 四、立项业务流程

立项表单采用带进度条的分步流程。

### 4.1 节点一：基础信息

字段：

| 字段 | 必填 | 规则 |
|---|---|---|
| 项目类别 | 是 | 单选：非定制 / 定制；默认非定制 |
| 客户 | 条件必填 | 选择定制后显示；从客户库选择，并允许后续补充客户来源 |
| 方案名称 | 是 | 保存原始名称；生成 T 号后可形成含编号的展示名称 |
| 方案地址 | 否 | URL；可指向知识库、语雀或其他方案地址 |
| 版本类别 | 是 | 大版本 / 中版本 / 小版本 |
| 关联需求 | 是 | 来源为当前产品管理系统内部需求，可多选 |

基础信息保存为草稿时不生成 T 立项号。

### 4.2 节点二：Agent 评审

本节点复用 `review-agent` 的后端评审能力，不嵌入原复杂页面。

用户只需：

1. 拖拽上传方案文件；或
2. 从当前产品/版本知识库选择文件；
3. 点击评审；
4. 查看评分、结论和主要问题。

规则：

- 保存评审输入文件引用、评分、结论、维度结果和评审时间。
- 未通过时显示分数与结果动画，流程停留在本节点。
- 未通过后允许替换文件重新评审，并保留评审历史。
- 达到通过阈值后才能进入下一节点。
- 通过阈值需配置化，首期可使用系统默认值。

### 4.3 节点三：评审决策

用户选择是否需要开评审会。

#### 不需要开评审会

1. 必须选择该产品的主负责人。
2. 记录流转到主负责人。
3. 主负责人选择同意或拒绝。
4. 同意后生成 T 立项号。
5. 拒绝后退回发起人，并要求填写原因。

负责人默认从产品团队成员中选择。是否允许选择全公司用户列为待确认项。

#### 需要开评审会

1. 必须填写预计开会时间。
2. 提交后生成 T 立项号。
3. 首期不强制录入会议结论，但预留会议结论和会议纪要字段。

文案统一使用“预计开会时间”，不使用“逾期开会时间”。

### 4.4 立项状态建议

```text
draft
agent_reviewing
agent_rejected
decision_pending
owner_pending
owner_rejected
approved
cancelled
```

## 五、上线业务流程

上线流程分为“申领上线号”和“填写上线公告”两个阶段。必须先取得正式版本号，再使用该版本号发布公告；公告地址回填后，上线流程才算完成。

### 5.1 常规上线：申领上线号

| 字段 | 必填 | 规则 |
|---|---|---|
| T 立项号 | 是 | 通过勾选方式选择已取得 T 号且未取消的立项 |
| 方案名称 | 自动确认 | 根据 T 立项号调取，展示给人工确认 |
| 项目类别 | 自动确认 | 根据 T 立项号调取，允许按权限修正 |
| 版本类别 | 自动确认 | 根据 T 立项号调取 |
| 产品负责人 | 自动确认 | 根据 T 立项号调取 |
| 方案地址 | 自动确认 | 根据 T 立项号调取 |
| 需求来源 | 是 | 调取该 T 立项号关联的内部需求，同时允许新增需求 |
| 项目组成员 | 是 | 支持多人选择，展示格式如“张三、李四、王五” |
| 上线时间 | 是 | 支持日期或日期时间，展示格式如 `2026.06.10` |

选择 T 立项号后，系统不能只显示编号，必须调取并展示关联资料供人工确认。确认后提交上线号申领，流程走自动审批；自动审批通过后，由后端生成 V 上线号，记录进入“待发布公告”状态。

自动审批首期含义：

- 校验 T 立项号有效且未取消。
- 校验必填资料完整。
- 校验上线版本边界与编号可用。
- 校验需求、成员和上线时间已确认。
- 校验通过后自动生成 V 上线号，不再进入人工审批节点。

### 5.2 取得 V 号后：发布并填写上线公告

生成 V 上线号后，详情页展示：

- 正式 V 上线号。
- `前往 Baklib 编写公告`按钮。
- 上线公告地址输入框。
- `完成上线`按钮。

“前往 Baklib 编写公告”打开：

`https://sso.baklib.com/`

用户使用已生成的 V 上线号发布公告，然后返回系统填写公告地址。上线公告地址为必填；未填写时不能点击“完成上线”，记录保持“待发布公告”状态。

本期不调用 Baklib API。后续接入 API 时，应在同一上线记录中保存公告 ID、发布状态、发布时间和最终 URL。

### 5.3 临时优化需求特殊通道

上线页面提供独立的`临时优化需求`按钮。该通道：

- 不需要选择 T 立项号。
- 按小版本上线资料填写。
- 需求来源允许选择产品管理系统内部需求，也允许现场新增。
- 必须填写项目组成员和上线时间。
- 资料确认后走自动审批。
- 取得编号后，同样必须发布公告并回填公告地址，才能完成流程。

问号提示内容建议：

> 临时优化需求是当月常规产品计划外、需要紧急插入开发的优化需求，通常应满足紧急度高、工作量小、可弹性开发。原则上产品工作量不超过 3 天、开发工作量不超过 5 天；超过范围应走常规立项流程。

同时显示以下补充规则：

- 不要求输出完整 PRD 原型方案，但必须填写清晰的产品规则或需求说明。
- 初始状态为待排期，确认排期后进入待开发，并持续跟踪至上线。
- 公告按“日期-功能优化小报（需求概述）”命名。
- 上线当天必须发布优化小报并回填公告地址。

制度差异待确认：研发管理规范 V2.5 第 3.6.7 条规定“临时优化需求无需分配版本号”；本次产品需求提出“按小版本上线并走自动审批”。实施前需明确采用以下哪种口径：

1. 生成正式 V 小版本号，但不需要 T 立项号；或
2. 不生成 V 号，仅生成临时优化流水号并完成公告闭环。

在该口径确认前，数据模型应支持 `TCode` 为空，不应把临时优化需求伪关联到某条立项记录。

### 5.4 上线状态建议

```text
draft
auto_approving
approval_failed
announcement_pending
completed
cancelled
```

## 六、T/V 编号规则

旧数据表明正式业务编号为三段式：

```text
T主版本.中版本.小版本
V主版本.中版本.小版本
```

历史示例：

- `T3.6.1`
- `T3.7.0`
- `T6.15.1`
- `V3.32.8`
- `V4.23.0`

递增规则：

- 大版本：第一段加一，第二、三段归零。
- 中版本：第二段加一，第三段归零。
- 小版本：第三段加一。

生成要求：

1. 编号必须由后端事务生成，前端不得自行拼接最终编号。
2. 编号必须有数据库唯一约束。
3. 打开或保存草稿时不得提前占号。
4. T 号在立项流程满足条件时生成。
5. V 号在常规上线资料确认并通过自动审批时生成，此时公告可以尚未发布。
6. 历史导入必须保留原始 T/V 编号，不重新编号。
7. 新 V 号原则上与关联 T 号使用相同的三段数字，例如 `T6.15.1 -> V6.15.1`。
8. 若历史数据出现一 T 多 V、跳号或特殊编号，导入时保留并进入异常提示，不静默改写。
9. 临时优化需求是否生成正式 V 小版本号，按第五节待确认口径处理。

## 七、Excel 历史数据导入

立项和上线页面都必须提供“导入历史数据”。

### 7.1 支持格式

- `.xlsx`：必须支持。
- `.xls`：必须支持。
- `.csv`：可同时支持，作为辅助格式。

不要求用户先将 Excel 转成 CSV。

### 7.2 导入流程

```text
上传 Excel
-> 选择工作表
-> 自动识别表头
-> 字段映射预览
-> 数据校验
-> 用户确认
-> 后台导入
-> 展示导入结果
-> 下载错误报告
```

导入前必须预览，不允许上传后直接写库。

### 7.3 Excel 解析要求

1. 自动探测前 20 行中的有效表头，兼容首行说明文字。
2. 支持中文历史列名和新模板列名。
3. 忽略全空行、合并单元格造成的空白尾列和纯说明行。
4. 日期兼容 Excel 日期值、`yyyy-MM-dd`、`yyyy/MM/dd` 和带时间格式。
5. 保留原始行号，错误报告必须能定位到工作表和行。
6. 单个文件可包含多个工作表，由用户选择或按名称自动匹配。
7. 大文件走后台任务，前端显示解析和导入进度。

### 7.4 历史立项导入字段

优先兼容旧 `【T立项号】表格数据.xlsx`：

- 平台
- 系统
- 应用/产品
- 项目类别
- T 立项号
- 版本类别
- 产品立项方案名称
- 项目需求描述
- 所属部门
- 产品负责人
- 第一稿会议时间
- 第二稿会议时间
- 第三稿会议时间
- 立项时间
- 是否需要 UI 设计
- 计划立项时间
- 方案地址
- 开发状态

导入后：

- 原 T 号原样保存。
- 流程状态直接标为历史已立项或按映射状态保存。
- 不重新执行 Agent 评审。
- 旧字段无法映射时保存在 `legacyData` 或原始导入快照中，不能丢弃。

### 7.5 历史上线导入字段

优先兼容旧 V 上线号表和公告记录：

- V 上线号
- T 立项号/内部版本号
- 项目类别
- 版本类别
- 产品立项方案名称
- 产品负责人
- 项目组成员
- 方案地址
- 计划/实际上线时间
- 上线公告地址
- 当前开放范围

导入后：

- 原 V 号原样保存。
- 优先按 T 号关联立项。
- 无法匹配 T 号时仍允许导入，状态标记为“待关联”。
- 后续允许人工绑定，不因关联失败丢弃上线记录。

### 7.6 去重与幂等

每条导入记录保存：

- `sourceType = history-import`
- `sourceSystem`
- `sourceRecordId`
- `sourceFileName`
- `sourceSheetName`
- `sourceRowNumber`
- `importBatchId`
- `importedBy`
- `importedAt`
- 原始行快照

去重优先级：

1. T/V 编号。
2. `sourceSystem + sourceRecordId`。
3. 无编号数据进入人工确认，不自动合并。

同一文件重复导入时，应显示新增、更新、跳过和冲突数量。

### 7.7 校验与错误报告

导入预检至少检查：

- 编号格式不合法。
- 文件内编号重复。
- 与系统现有编号冲突。
- 上线记录找不到关联 T 号。
- 版本类别无法识别。
- 日期格式错误。
- 产品、客户、负责人无法匹配。
- 必填字段缺失。

结果分为：

- 可导入
- 可导入但需提醒
- 必须修正

错误报告使用 Excel 下载，包含原始行、错误字段、错误原因和建议处理方式。

## 八、旧数据兼容和迁移原则

1. 现有 `ProductVersion` 不删除、不覆盖。
2. 现有版本记录可标记为 `legacy`，并在新版上线列表或兼容区继续访问。
3. 新立项流程和上线流程使用独立业务对象，不把审批过程硬塞进 `ProductVersion.FormData`。
4. 立项通过后可创建或绑定 `ProductVersion`。
5. 上线完成后更新关联 `ProductVersion` 的生命周期和发布时间。
6. 现有 `VersionUpgradeRequest` 暂不删除；新版稳定前隐藏旧入口，保留数据读取能力。
7. 所有迁移先预演、输出统计，再执行正式写入。
8. 禁止以新必填规则阻断历史数据导入；历史缺失字段应标记而不是伪造。

## 十、与现有模块的集成

### 10.1 产品管理需求

立项的“关联需求”直接读取当前产品下的 `Requirement`，不再要求 TAPD 链接。历史 TAPD 地址可作为需求的外部来源字段后续迁移。

### 10.2 Review Agent

复用评审服务和评分配置，不复用完整页面。需要抽出可被产品立项调用的评审 API，并保存结果快照，避免评审规则变化后历史结论漂移。

### 10.3 知识库

上传方案支持：

- 本地文件上传到附件/知识库。
- 从产品知识库选择已有条目。
- 后续可自动归档到版本知识库。

### 10.4 用户与产品团队

主负责人优先从产品团队成员中选择。审批权限由后端校验，前端显隐不能替代权限控制。

## 十二、验收标准

1. 原有产品版本数据仍可查询和访问。
2. 上线/立项标签位于版本页面内容区左上方，默认进入上线。
3. Agent 评审不通过时不能进入评审决策节点。
4. 不开会时，负责人未同意不得获得 T 号。
5. 开会时，未填写预计开会时间不得获得 T 号。
6. 上线记录未关联有效 T 号时不得按新流程生成 V 号。
7. 选择 T 立项号后能调取方案、类别、负责人、地址和关联需求供人工确认。
8. 需求来源允许在原关联需求基础上新增。
9. 项目组成员和上线时间未填写时不能提交上线号申领。
10. 自动审批通过后生成 V 号，并进入“待发布公告”状态。
11. 上线公告地址未填写时不能完成上线流程。
12. 临时优化需求入口不要求 T 立项号，并展示规范定义和范围限制。
13. T/V 编号由后端生成且并发下不重复。
14. `.xlsx` 和 `.xls` 能直接上传导入。
15. 导入前可预览字段映射和错误。
16. 历史 T/V 编号原样保留。
17. 无法关联 T 号的历史 V 记录不会丢失。
18. 重复导入同一批历史数据不会产生重复记录。
19. 导入完成后可下载包含行号和错误原因的 Excel 报告。
