# 产品管理智能体版本流程收尾 · 计划

> **版本**：v2.0 | **日期**：2026-07-17 | **状态**：规划中

## 目标

完成已经落地的“立项 T 号、上线 V 号、历史导入和旧版本兼容”流程验证，关闭仍未裁决的业务规则与历史数据迁移缺口。本文不再重复页面字段和已实现的数据模型；现行实现是事实源。

## 已落地

- 版本页分为“上线”和“立项”，默认进入上线。
- 立项流程覆盖基础信息、Review Agent 文件评审、评审会或负责人决策，并由服务端生成 T 号。
- 上线流程可以从立项带入方案和需求，补充需求、项目成员和时间，由服务端生成 V 号。
- 上线公告地址未填写时不能完成上线。
- 临时优化需求支持无 T 号进入特殊上线通道。
- 历史立项和上线支持 `.xlsx`、`.csv` 的预览与导入；旧 `ProductVersion` 数据仍在兼容区可访问。

## 未完成范围

### P1：业务规则定盘

需形成单一配置或规则文档，并补自动化测试：

1. Review Agent 通过阈值及管理员可配置范围。
2. 评审会是否强制会议结论和纪要。
3. 主负责人是否只允许产品团队成员。
4. 一个 T 号是否允许多次上线，以及多个 V 号的编号关系。
5. 临时优化需求是否生成正式 V 号，统一产品诉求与研发规范 V2.5 的冲突。
6. 已生成 T/V 号的取消规则；编号必须永久占用，不得回收复用。
7. 历史导入缺少公告地址时的允许状态与补录责任。

### P2：历史数据导入收口

- 使用正式历史文件执行 dry-run，核对工作表、合并表头、日期、负责人、客户和 T/V 关联。
- 重复导入同一文件或同一业务记录不得生成重复数据。
- 无法关联 T 号的历史 V 记录保留并标记待关联，不得丢弃。
- 导入前展示字段映射和错误；导入后提供带原始行号的错误结果。
- 旧二进制 `.xls` 明确采用“要求另存为 `.xlsx`”或受控转换器，不在 UI 中声称已直接支持。

### P3：验证与兼容

- API 编译、聚焦单元测试和编号并发测试全绿。
- 覆盖评审不通过、负责人拒绝、撤回、取消、重复提交和权限越界。
- 验证旧 `ProductVersion`、大版本升级申请及其需求、功能、缺陷关联无回归。
- 通过真实登录入口验收立项、上线、临时优化和历史导入完整路径。

## 数据权威

- 立项：`ProductInitiation` 及其评审、审批记录。
- 上线：`ProductRelease`。
- 历史导入：导入批次与每行结果；不得只在浏览器内保留错误。
- 旧版本：`ProductVersion` 只承担兼容读取，不作为新流程写入目标。
- T/V 编号必须由服务端原子生成，前端不得计算或预占。

## 验收门

1. 并发申请下 T/V 编号不重复，取消后编号不复用。
2. Agent 评审不通过或负责人未同意时不能生成 T 号。
3. 常规上线缺少有效 T 号、成员或上线时间时不能生成 V 号。
4. 临时优化通道按已定盘规则处理 T/V 号，不与常规上线混用。
5. 上线公告地址缺失时状态保持“待发布公告”。
6. 正式历史文件可预览、导入、重复执行和下载错误结果。
7. 新旧数据可以同时查询，旧关联和详情页不回归。
