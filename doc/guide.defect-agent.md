# 缺陷管理 Agent 使用 · 指南

> **类型**：guide | **适用角色**：测试工程师、开发人员、项目经理 | **最后更新**：2026-06-20
> **预计阅读**：10 分钟 | **难度**：入门

---

## 1. 功能概览

| 能力 | 说明 |
|------|------|
| 多视图看板 | 列表、卡片、看板、统计四种视图 |
| AI 智能分析 | 自动分类、优先级评估、相似缺陷检测 |
| 缺陷自动化 | 一键生成长期 `domain + K` 授权，供每日任务自动拉取、评论、修复和回写 commit |
| 分享协作 | 缺陷链接分享给外部团队，支持外部 Agent 分析 |
| 修复报告 | AI 生成结构化根因分析和修复方案 |
| 模板管理 | 预设缺陷描述模板，统一提交规范 |

## 2. 操作路径

**入口**：首页 → 点击「缺陷管理员」卡片 → 进入缺陷管理页面

页面 Tab 切换：
- **「收到的」** — 分配给你处理的缺陷
- **「我提交的」** — 你提交的缺陷

页面右上角操作栏：
| 按钮 | 功能 |
|------|------|
| **「提交缺陷」**（Plus 图标） | 新建缺陷，**主操作按钮** |
| **「项目管理」**（FolderKanban 图标） | 创建/管理缺陷项目 |
| **「我的模板」**（FileText 图标） | 管理缺陷描述模板 |
| **「分享管理」**（Share2 图标） | 管理分享链接 |
| **「缺陷自动化」**（Bot 图标） | 生成长期 Agent 授权、复制每日计划、查看运行历史和统计 |

## 3. 完整操作流程

### 步骤 1：创建缺陷项目

1. 点击右上角 **「项目管理」**
2. 在弹出面板中点击 **创建项目**
3. 填写项目名称、项目标识（KEY）
4. 创建后，提交缺陷时可以选择归属项目

> **Pro Tip**：项目标识会显示在缺陷编号前，如 `[MAP] #0012`，建议用简短英文缩写。

### 步骤 2：提交缺陷

1. 点击右上角 **「提交缺陷」** 按钮
2. 填写缺陷信息：
   - **标题**：一句话描述问题
   - **详细描述**：复现步骤、期望行为、实际行为
   - **严重程度**：选择优先级
   - **截图/录屏**：上传附件
3. AI 自动补充分类标签和优先级建议
4. 选择处理人
5. 提交

> **Pro Tip**：先在 **「我的模板」** 中创建常用模板（如「UI 缺陷模板」「接口缺陷模板」），提交时一键套用，团队提交格式统一。

### 步骤 3：查看和管理缺陷

四种视图模式（左上角切换）：

| 视图 | 图标 | 适用场景 |
|------|------|---------|
| **列表视图** | List | 快速浏览和批量操作 |
| **卡片视图** | LayoutGrid | 直观查看缺陷详情 |
| **看板视图** | Columns3 | 按状态分列拖拽管理 |
| **统计看板** | BarChart3 | 查看缺陷趋势和分布 |

**筛选功能**：
- **「全部项目」** 下拉框 — 按项目筛选
- **「全部团队」** 下拉框 — 按团队筛选（有团队时显示）

### 步骤 4：处理缺陷

1. 切换到 **「收到的」** Tab 查看分配给你的缺陷
2. 系统自动弹出提示：「有待处理缺陷: #XXXX」或「有 N 个待处理缺陷等待您处理」
3. 点击缺陷卡片查看详情
4. 在看板视图中可以 **拖拽卡片** 切换状态

### 步骤 5：配置每日自动修复

1. 点击页面右上角 **「缺陷自动化」**
2. 点击 **「生成并复制每日任务配置」**
3. 系统会创建或复用名为 **「缺陷处理 Agent 授权」** 的长期 AgentApiKey
4. 复制内容包含：
   - `domain`：缺陷系统访问域名
   - `K`：本次生成的长期授权明文，只显示一次
   - `scope`：`defect-agent:use`
   - `status`：默认拉取 `submitted,assigned,processing`
   - 执行要求：使用 `ai-defect-resolve`，一次只处理一个缺陷
5. 将复制内容加入 Codex 每日计划；触发时间由计划系统配置，可以是 9 点、10 点或其他时间

自动化运行时优先使用 `defect-agent-workflow.v1` 工作流协议：

1. `workflow/start-next` 创建或复用运行记录，并领取一条缺陷
2. 智能体评论修复计划
3. 智能体按轻量规则判断是否自动修复
4. 轻量缺陷由智能体修复、验证并提交中文 commit
5. `workflow/complete` 一次性回写 commit、写入 `defect_resolution_traces`、标记缺陷已修复
6. `workflow/complete` 返回下一次 `workflow/start-next` 入参，再继续下一条
7. 重量级或无法自测时调用 `workflow/block`，记录阻塞原因并默认停止本轮运行

重量级、破坏性、跨服务协议或无法自测的问题不会自动提交，会在缺陷内评论阻塞原因并停在当前缺陷。

### 步骤 6：查看自动化运行结果

在 **「缺陷自动化」** 面板中查看：

| 区域 | 说明 |
|------|------|
| 授权状态 | 当前长期 K 是否可用、是否永不过期、最近使用时间和使用次数 |
| 每日计划内容 | 可复制到 Codex 自动化计划的完整文本 |
| 运行统计 | 待处理缺陷、已拉取、已修复、失败/阻塞数量 |
| 最近运行 | 每次任务处理到的缺陷、状态、commit、预览地址和失败原因 |

## 4. 缺陷生命周期

```
待处理 → 处理中 → 验证中 → 已解决 → 已关闭
                              ↓
                           打回重修
```

| 状态 | 颜色标记 | 说明 |
|------|---------|------|
| 待处理 | 红色 | 新提交，等待分配或开始处理 |
| 处理中 | 蓝色 | 开发人员正在修复 |
| 验证中 | 黄色 | 修复完成，测试验证中 |
| 已解决 | 绿色 | 验证通过 |
| 已关闭 | 灰色 | 最终关闭 |

## 5. 进阶功能

### 缺陷自动化与发布中心

自动修复提交后，Agent 必须调用 `workflow/complete` 回写完整 commit 信息。系统会同时写入缺陷结构化字段和 `defect_resolution_traces`。旧的 `commit-info` 端点仍保留给兼容和排障，但每日自动化优先使用工作流协议。

发布中心/更新中心只读取 commit 与缺陷的关系，不需要人工在更新中心点“关联缺陷”。当正式环境确认发布到包含该 commit 的版本后，系统会把关联 trace 标记为已发布，并进入发布后验收通知队列。

### 发布后的视觉验收与通知

正式发布后，缺陷处理 Agent 会拉取 `published-pending` 队列。每条记录需要执行：

1. 使用 `create-visual-test-to-kb` 对正式环境做视觉验收
2. 验收报告归档到 **「缺陷修复验收报告」** 知识库
3. 回写报告地址、知识库地址和验收结论
4. 验收通过时通知提交人“你的问题已修复”
5. 验收失败时通知“需要继续改进”，不发送已修复文案

正式发布前只在缺陷内评论进度，不提前通知提交人。

### 缺陷分享

1. 在缺陷详情页点击 **「分享」**
2. 生成分享链接（可设置有效期）
3. 发送链接给外部人员
4. 外部 Agent 可通过链接分析缺陷并生成修复报告

### 统计看板

切换到 **统计看板** 视图，查看：
- 缺陷按状态分布的饼图
- 缺陷趋势折线图
- 按严重程度分布统计
- 处理效率指标

## 6. 常见问题

**Q：如何批量操作缺陷？**
A：在列表视图中勾选多个缺陷，使用批量操作按钮。

**Q：外部分享链接安全吗？**
A：分享链接可设置有效期，过期自动失效。仅展示该缺陷的基本信息，不暴露项目内部数据。

**Q：AI 分析缺陷需要多久？**
A：通常几秒内完成。分析过程中有进度提示，AI 的思考过程实时可见。

**Q：每日任务需要我手动提供缺陷链接吗？**
A：不需要。每日任务只需要 `domain + K`，缺陷由 `/api/defect-agent/agent/workflow/start-next` 自动拉取。

**Q：K 丢了怎么办？**
A：后端不会保存明文 K。重新点击 **「缺陷自动化」** 中的 **「生成并复制每日任务配置」**，生成新的长期授权并更新每日计划即可。

**Q：为什么正式发布前不通知提交人？**
A：预览环境修复不等于线上已可用。只有正式环境发布并完成视觉验收后，系统才会通知提交人，避免用户收到过早的“已修复”消息。
