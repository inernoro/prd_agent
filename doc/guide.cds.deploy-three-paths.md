# CDS 部署方式选择 · 指南

> **版本**：v2.0 | **日期**：2026-07-15 | **状态**：开发中

CDS 保留三种项目接入方式，但新版不再把三条路当成同等复杂的选择题。普通项目优先使用可视化托管交付；已有成熟编排的项目继续使用 Compose；需要 Agent 批量操作或自动化时使用项目级技能。

## 先做选择

| 你的项目 | 推荐方式 | 需要准备什么 |
| --- | --- | --- |
| 普通网站、后台、前后端项目 | 可视化托管交付 | Git 仓库地址 |
| 已有稳定的 `cds-compose.yml` 或复杂 Compose | Compose 导入 | 现有配置文件 |
| 希望由 Codex、Cursor 或 Claude Code 操作 | Agent 项目接入 | 页面批准一次 |
| 特殊网络、多容器严格编排、自定义镜像 | Compose 导入 | 明确的高级配置 |

三种方式最终进入同一套部署记录、版本复用、预览和回滚流程，可以混合使用，不需要为新版重建已有项目。

## 方式一：可视化托管交付

这是新项目的默认方式。

### 怎么使用

1. 打开项目列表，点击“一键部署项目”；
2. 粘贴 Git 仓库地址，或从 GitHub 选择仓库；
3. 点击“检测仓库并自动填好配置”；
4. 确认识别到的应用、目录、命令、端口和依赖；
5. 配置把握不高时，先点击“试运行验证”；
6. 创建项目并观察准备、构建、启动和检查阶段；
7. 部署成功后打开预览地址。

### 用户需要确认什么

- 应用目录是否正确；
- 前端、后台和后台任务是否被正确区分；
- 启动命令和端口是否符合项目；
- 数据库、缓存和其他依赖是否齐全；
- 是否存在必须由用户提供的环境信息。

普通项目不需要先写完整构建配置。标准 Compose 只声明了构建、没有启动命令时，CDS 会继续从项目自身信息推导；确实无法判断时才要求补充。

## 方式二：Compose 导入

适合已有成熟编排，或需要精确控制服务、网络和启动顺序的项目。

### 怎么使用

1. 确认仓库根目录存在 `cds-compose.yml` 或受支持的 Compose 配置；
2. 在项目创建或设置中查看导入预览；
3. 检查应用服务、基础设施、端口、环境声明和依赖顺序；
4. 批准导入；
5. 选择目标分支并部署；
6. 在同一条部署记录中查看过程和结果；
7. 打开预览并验证核心操作。

### 新版如何减少 Compose 调试

- 已有配置不会因为切换到托管模式而被删除；
- 标准项目缺少启动命令时，会先尝试从项目清单推导；
- 部署失败会先显示问题归属和失败阶段；
- 相同成功版本可以直接再次部署；
- 新版本异常时可以恢复以前的成功版本。

Compose 仍然是高级控制方式。特殊网络、复杂脚本和自定义镜像不会被托管模式强行改写。

## 方式三：Agent 项目接入

适合希望让 Agent 完成扫描、部署、查看状态、诊断和恢复的用户。

### 怎么使用

1. 打开“一键部署 > 接入 Agent”；
2. 选择 Codex、Cursor 或 Claude Code；
3. 选择已有项目或新项目；
4. 把无密钥接入口令交给 Agent；
5. 在 CDS 页面批准；
6. 让 Agent 继续扫描、部署或诊断。

### 权限和电脑环境

- 不需要把长期 Key 发到对话；
- 不修改 `.zshrc`、`.bashrc`、系统环境变量或全局 PATH；
- 凭据只保存在当前项目；
- 权限默认只覆盖当前项目；
- 新项目创建权限使用一次后自动失效；
- 短暂网络波动不会要求重新发起同一申请。

Agent 接入不是第四套部署系统。它只是操作 CDS 的安全入口，部署结果仍以 CDS 页面里的部署记录和版本为准。

## 已有项目怎么升级

| 现状 | 建议 |
| --- | --- |
| 现有 Compose 已稳定运行 | 保持 Compose，不必迁移 |
| 配置多但大部分是常规网站或后台 | 在设置中试用托管模式，先检查最终生效配置 |
| 仍通过环境变量给 Agent 长期 Key | 改用页面批准；旧方式只保留兼容 |
| 私有仓库每次都重新粘贴 Token | 改用 GitHub App 单仓库授权 |
| 失败后靠反复部署碰运气 | 打开本次部署记录，按失败归属处理 |

切换交付模式前后都应保留一条已验证的成功版本。这样出现识别偏差时，可以立即恢复，而不是重新排查整个项目。

## 部署后的统一操作

无论从哪条方式进入，日常操作都是一致的：

1. 推送代码或手动发起部署；
2. 查看同一条部署记录；
3. 打开预览验收；
4. 代码未变时复用成功版本；
5. 新版本异常时恢复旧版本；
6. 失败时按代码、配置、CDS 或外部服务归因。

## 验收标准

| 检查项 | 通过标准 |
| --- | --- |
| 仓库来源 | 能确认仓库、默认分支和触发来源 |
| 运行方案 | 应用、命令、端口和依赖与项目一致 |
| 部署过程 | 刷新后仍能继续查看同一条记录 |
| 服务结果 | 每个服务状态清晰，不混成一个笼统结果 |
| 用户预览 | 页面或服务入口及核心操作正常 |
| 再次部署 | 能判断成功版本是否可直接复用 |
| 恢复 | 可以选择以前的成功版本恢复 |

## 常见问题

| 现象 | 处理方式 |
| --- | --- |
| 默认分支不是 `main` | 以仓库真实默认分支为准，不手工猜测 |
| 容器运行但预览失败 | 检查端口、健康路径和服务日志 |
| 私有仓库拉取失败 | 检查该项目的 GitHub App 单仓库授权 |
| 自动识别不准确 | 修改最小声明或改用 Compose 高级控制 |
| 相同代码仍在重新构建 | 检查目标版本是否标记为产物可复用 |
| 页面刷新后没有过程 | 回到分支详情并打开对应部署记录 |
