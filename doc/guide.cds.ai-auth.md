# CDS 与业务 API 双层认证诊断 · 指南

> **版本**：v2.0 | **日期**：2026-07-15 | **状态**：开发中

CDS 权限和被部署应用的业务权限是两套独立系统。Agent 能部署项目，不代表它自动拥有业务应用里的用户身份；业务接口返回未授权，也不代表 CDS 接入失败。

## 先分清失败发生在哪里

| 层级 | 保护什么 | 推荐认证方式 | 常见失败 |
| --- | --- | --- | --- |
| CDS 项目权限 | 项目、部署、日志、版本和回滚 | CDS 页面批准的项目凭据 | 当前项目未授权、权限属于其他项目、凭据已撤销 |
| 业务应用权限 | 被部署应用的页面和 API | 应用自己的登录、测试账号或受控 AI 身份 | 用户不存在、业务 Key 错误、权限不足 |

不要把两层的 Key、用户或错误信息混为一谈，也不要为了排查业务接口而扩大 CDS 项目权限。

## CDS 项目权限诊断

### 推荐检查

在目标 Git 仓库目录让 Agent 执行项目连接检查。用户侧只需确认：

- 当前目录对应正确项目；
- CDS 页面曾批准该项目；
- 本地 `.cds/credentials.json` 存在且没有进入 Git；
- 权限没有被撤销；
- 切换项目后没有误用旧项目凭据。

### 新版重新授权

1. 打开“一键部署 > 接入 Agent”；
2. 选择当前 Agent 和目标项目；
3. 把无密钥接入口令交给 Agent；
4. 在 CDS 页面批准；
5. 让 Agent 重新检查连接。

不要让用户把长期 Key 粘贴到对话，也不要通过修改 `.zshrc`、`.bashrc` 或系统环境变量解决项目授权问题。

### 旧自动化兼容

已有 `CDS_HOST`、`CDS_PROJECT_KEY` 或 `AI_ACCESS_KEY` 的自动化仍可运行，但它们是兼容方式，不是新用户默认流程。

如果旧自动化返回未授权，先确认 Key 的作用范围和目标项目，不要直接生成更高权限的全局 Key。

## 业务应用权限诊断

CDS 部署成功后，应用仍然按照自己的认证规则运行。例如业务应用可能要求：

- 用户名和密码；
- JWT 登录态；
- 应用自己的 AI Access Key；
- 真实存在且未禁用的代入用户；
- 管理员角色或特定资源权限。

### 正确的排查顺序

1. 通过预览地址重现问题，记录具体页面或接口；
2. 确认业务容器已经获得所需项目运行变量；
3. 确认使用的业务用户真实存在且未禁用；
4. 阅读业务 API 返回的具体错误；
5. 用应用自己的正常登录方式验证同一路径；
6. 仍无法定位时再查看应用日志。

不要把容器内部请求作为最终验收。容器命令只用于诊断，修复后仍需通过真实预览地址重新验证。

## 常见错误怎么理解

| 现象 | 更可能的原因 | 优先处理 |
| --- | --- | --- |
| CDS 项目列表或部署接口未授权 | 当前项目凭据缺失、失效或项目不匹配 | 在页面重新批准当前项目 |
| 业务 API 提示未配置 AI 认证 | 业务容器没有获得应用自己的运行变量 | 在项目环境变量中配置并重新部署 |
| 业务 API 提示 Key 无效 | 应用 Key 值不匹配 | 核对业务应用配置，不修改 CDS 凭据 |
| 业务 API 提示用户不存在 | 使用了默认猜测用户名 | 选择数据库中真实存在的测试用户 |
| 业务 API 提示用户被禁用 | 测试账号状态不允许访问 | 更换账号或恢复账号状态 |
| 页面登录成功但某个操作无权限 | 业务角色或资源权限不足 | 按应用权限模型处理 |
| 修改变量后行为没变化 | 运行中的容器仍使用旧值 | 重新部署对应分支 |

## 项目环境变量与 Agent 凭据的边界

| 信息 | 保存位置 | 是否注入业务容器 |
| --- | --- | --- |
| CDS Agent 项目凭据 | 当前项目 `.cds/credentials.json` | 否 |
| 业务应用 AI Key | CDS 项目环境变量 | 按业务需要 |
| 业务测试用户名 | 测试数据或业务账号系统 | 否，除非应用明确设计如此 |
| GitHub App 私钥 | CDS 服务系统配置 | 否 |

严禁把 Agent 项目凭据复制为业务应用变量，也不要把 GitHub App 私钥下发到业务容器。

## 活动记录

通过正式 Agent 项目凭据发起的 CDS 操作应出现在 CDS 活动记录中，并能识别为 Agent 操作。

不要使用内部绕过方式代替正式认证。内部请求可能执行成功，但会丢失责任主体、项目权限和审计信息。

## 最小验收清单

| 检查项 | 通过标准 |
| --- | --- |
| CDS 连接 | 当前项目可以查看状态和部署记录 |
| 项目隔离 | 其他项目不能复用当前凭据 |
| Git 安全 | `.cds/credentials.json` 不进入 Git |
| 业务登录 | 通过预览地址使用真实测试身份登录 |
| 核心接口 | 业务操作使用应用自己的权限正常完成 |
| 活动记录 | Agent 的 CDS 操作可追踪 |
| 修复验证 | 重新部署后通过预览地址复测 |

## 常见问题

| 问题 | 解答 |
| --- | --- |
| Agent 能部署但不能调用业务 API，正常吗 | 正常，两层权限独立 |
| 可以让 CDS 项目凭据同时充当业务 Key 吗 | 不可以，这会混淆权限边界并扩大泄露风险 |
| 为什么重新接入 CDS 仍然解决不了业务 401 | 问题发生在应用认证层，应检查应用用户和运行变量 |
| 为什么修改应用 Key 后仍旧失败 | 已运行容器没有自动获得新值，需要重新部署 |
| 可以用内部接口绕过认证做验收吗 | 不可以，内部命令只能诊断，最终必须走真实用户路径 |
