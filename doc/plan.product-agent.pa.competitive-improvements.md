# PA Agent 竞品调研与改进方案

> **版本**：v1.0 | **日期**：2026-05-10 | **状态**：规划中

本文基于公开产品信息与行业文章做的桌面调研，对照当前 `spec.product-agent.pa.md` 所描述的 Phase 1 能力，归纳同类产品的常见做法，并输出可落地的改进方向与分期方案。不构成对任何第三方的评测背书。

---

## 一、调研范围与方法

**对标问题**：用户为何需要「对话 + 结构化任务 + 优先级」的一体化体验，而不是分别打开聊天应用与待办应用。

**竞品谱系（四类）**

| 类型 | 代表方向 | 与 PA Agent 的相似点 |
|------|----------|----------------------|
| A. 四象限 / 优先级原生 | Quadrants、PrioritAI、Focus Matrix 等 | 自然语言进、矩阵出；部分带独立聊天或 Bot |
| B. 日程与每日规划 | Sunsama、Motion、Akiflow、Reclaim 等 | 日视图、容量感、与日历深度结合；部分强调 AI 自动排程 |
| C. 待办底座 + AI 插件 | Todoist Assist、各类「语音转任务」 | 任务模型成熟、拆解与润色、邮件转任务 |
| D. 通用对话 + 定时能力 | ChatGPT Tasks 等 | 强对话、定时触发与提醒；任务结构偏「提示词任务」而非传统看板 |

**方法局限**：未做付费产品的全量实测；功能以官网、应用商店说明与综述类文章为主。实施前应对目标竞品做一次 1～2 周的集中试用校准。

## 二、同类产品常见能力摘要

### 2.1 四象限与 AI 归类（类型 A）

公开宣传中常见要素包括：自然语言或「大脑倾倒」一次性输入后拆成多条任务、AI 建议象限、专注模式、部分支持多端或团队颜色分工。部分产品强调离线可用与隐私。

**启示**：「一句话进、矩阵出」已被市场教育；差异化需在**拆解质量、可解释性、与日历/截止日的联动**上拉开差距。

### 2.2 每日规划与自动排程（类型 B）

Motion 一类强调 AI 根据优先级与会议自动重排时间块；Sunsama 一类强调**晨间/晚间仪式**、手动拖拽与「今日容量」意识；Akiflow 强调**多工具聚合**与键盘优先。

**启示**：用户分化为「要系统自动排」与「要自己掌控感」两派；PA Agent 当前偏「对话生成结构」，下一阶段需明确主打**辅助规划**还是**自动排程**，避免两头不靠。

### 2.3 待办产品内的 AI（类型 C）

Todoist 路线：拆解复杂任务、生成子任务、语音 Ramble、邮件转任务、自然语言过滤器等；能力嵌在成熟任务模型（标签、项目、过滤器、重复规则）之上。

**启示**：**任务元数据**（截止、时长估计、标签、项目）若缺失，AI 再强也难以融入用户既有工作流。

### 2.4 对话产品内的定时任务（类型 D）

ChatGPT Tasks 路线：在对话里用自然语言设定**定时执行的提示**，跨端同步与推送通知，偏「周期性情报与提醒」而非经典项目管理。

**启示**：**主动触达**（到点推送、周期性复盘）可显著提升「助理感」，与当前 PA Agent 的「被动打开才有效」形成互补。

## 三、与 PA Agent Phase 1 的对照

以下根据 `spec.product-agent.pa.md` 与实现常识归纳，用于找差距而非逐条审计代码。

| 能力维度 | 市面常见强点 | PA Agent Phase 1（当前） | 差距性质 |
|----------|----------------|---------------------------|----------|
| 对话与流式 | 各头部产品标配 | 已具备 SSE、多会话 | 基线达标 |
| 任务结构化 | 子任务、标签、项目、截止 | 子任务 + 四象限 | 缺截止、时长、标签、依赖 |
| 日历 / 时间块 | 深集成占竞品核心 | 未强调 | 结构性缺口 |
| 提醒与回访 | 推送、定时、周期性总结 | 未见产品级能力 | 体验缺口 |
| 外部聚合 | 邮件、Slack、多工具导入 | 以附件抽取为主 | 生态缺口 |
| 看板交互 | 拖拽、批量、过滤 | 四象限展示 + 基础操作 | 效率与可编辑性可增强 |
| 专注与仪式 | 今日三件、专注计时 | 未强调 | 可选差异化 |
| 团队 | 分配、视图共享 | 定位个人 | 符合边界，B 端另立项 |
| 模型与隐私 | 可选本地/离线、数据处理声明 | 企业网关 + 权限 | 需强化「企业内数据流」说明 |
| 可解释性 | 为何归此象限、可一键调整 | 有 reasoning 字段 | 可增强 UI 暴露与用户纠正回路 |

## 四、可改进方向（主题级）

### 4.1 时间与承诺：从「优先级」到「可执行日程」

- 为任务增加**截止日期、可选开始时间、预估时长**，并在提示词中要求模型在合理时填充。
- 中期：只读接入日历（或导出 ICS）展示「硬约束」，避免 AI 在已知会议窗口上排不可能的任务。
- 与类型 B 错位竞争：首阶段不做全自动排程，做「**建议时间块 + 用户一键写入日历**」降低复杂度与投诉风险。

### 4.2 任务模型：对齐用户已有习惯

- 增加**标签 / 项目 / 来源会话**字段，支持按场景过滤（工作、学习、家庭）。
- 子任务支持**顺序或简单依赖**（完成后解锁下一项），贴近真实项目推进。
- 看板支持**拖拽改象限**、内联编辑标题与步骤，减少「只能回对话里改」的摩擦。

### 4.3 主动助理：提醒与周期复盘

- 轻量版：任务到点前**浏览器通知 / 邮件**（受限于 Admin 形态可选其一）。
- 与对话结合：允许用户说「每晚 21:00 问我今日 Q1 完成情况」——对标类型 D 的「定时提示词」，但落点在本产品任务状态。
- 周期简报：每周生成「四象限迁移与未完成原因」短文，强化「助理跟进度」的心智。

### 4.4 输入侧：降低「打字负担」

- **语音输入**（浏览器 Web Speech 或上传短音频转写）与竞品 Ramble 对齐。
- 保留附件抽取，增加**结构化模板**（周计划、会议纪要、面试准备）一键带入 System 侧角色与输出格式。

### 4.5 信任与可控：智能任务与 AI 幻觉

- **auto / suggest** 策略可配置：用户选择「仅建议不自动落库」或「仅 Q1/Q2 自动」。
- 展示模型给出的 `reasoning`，并提供「**象限错了，拖到正确区**」的显式纠正；可选「用纠正数据做偏好学习」留到后期。
- 对 JSON 解析失败、字段缺失做**降级**：仍展示正文，提示「未生成看板条目，可一键重试结构化」。

### 4.6 体验：年轻用户与「主流聊天」心智

- **今日焦点**：首页或侧栏显示「建议今日完成的 1～3 条」来自 Q1/Q2。
- **快捷指令**扩展为可配置 Chips，与场景模板联动。
- 视觉与动效保持克制（符合项目无 emoji 与品牌规范），用**信息层级与微交互**替代装饰性元素。

### 4.7 生态与集成（中长期）

- 导出 Markdown / CSV，或只读同步 Todoist / Apple Reminders 等**单向出口**，避免一上来做多向同步的工程与冲突地狱。
- 与 PRD Agent、周报 Agent 的**浅链接**：从文档一键「生成行动项到 PA」作为差异化，比泛集成更贴合本仓库定位。

## 五、改进方案：分期与交付物

### Phase 2（建议 4～8 周，依人力裁剪）

| 包 | 内容 | 验收要点 |
|----|------|----------|
| P2-A 任务元数据 | 截止日、预估时长、标签；提示词与 API 扩展；看板过滤 | 新任务可设截止；看板可按标签/逾期筛选 |
| P2-B 看板可编辑 | 拖拽象限、标题与子步骤内联编辑、删除确认 | 主要操作无需回对话 |
| P2-C 信任设置 | 用户级「自动落库」开关；reasoning 在卡片上可展开 | 减少误创建投诉 |
| P2-D 今日焦点 | 侧栏或顶部「今日 3 件事」启发式排序 | 打开页面即可行动 |

### Phase 3（建议接续 6～12 周）

| 包 | 内容 | 验收要点 |
|----|------|----------|
| P3-A 提醒 | 到期提醒 + 可选每日一问（最小定时任务） | 用户能收到至少一种渠道提醒 |
| P3-B 日历只读 | 拉取忙闲或导出 ICS，对话与拆任务时引用 | 排程建议不与已知会议冲突（规则级） |
| P3-C 语音与模板 | 语音转写进输入区；3～5 个场景模板 | 移动端/桌面 Web 可用 |
| P3-D 周期简报 | 周报式总结卡片，可分享为 Markdown | 强化留存与「助理」心智 |

### Phase 4（可选 / 与研究课题）

- 多工具单向导入、与内部其他 Agent 深度联动、轻量团队只读看板。

## 六、风险与原则

- **范围蔓延**：日历同步与多向待办同步极易变成独立大项目；坚持「先导出、再只读、最后双向」。
- **AI 可靠性**：自动落库越多，纠错成本越高；Phase 2 优先「可编辑 + 可配置 auto」再扩场景。
- **合规**：语音、提醒、外发邮件涉及隐私与营销合规，需在 `spec.product-agent.pa` 中单独补一节数据处理说明。
