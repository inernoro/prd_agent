# debt.pm-agent — 项目管理智能体工程债务台账

> 类型：debt（工程债务台账）｜状态：active｜最后更新：2026-05-29

记录项目管理智能体（pm-agent）已知边界、后续可补项与留尾。

## 已知边界（Phase 1 MVP 范围）

Phase 1 交付「执行层」：立项 + 任务 CRUD + 看板/列表/甘特图 + AI 需求拆解。
Phase 2 交付「评价层」：干系人(权力利益矩阵) + NPSS 结案评价。
以下「组织治理层」能力按方案分期，尚未实现：

| 能力 | 所属阶段 | 状态 | 说明 |
|------|----------|------|------|
| 干系人管理 + 权力利益矩阵 | Phase 2 | 已完成 | 四象限分类 + 管理策略；角色加权（受益方 2×） |
| NPSS 评价（单项目） | Phase 2 | 已完成 | 干系人 0-10 加权打分 → 满意度(0-100) → 等级（成功/平庸/失败） |
| 组织级 NPSS 仪表盘 + M.O.R.E 自评 | Phase 3 | 已完成 | NPSS = 成功占比 − 失败占比，对比全球基线 36；M.O.R.E 为自评展示 |
| 项目奖金计算 + 奖金配置 | Phase 3 | 已完成 | 基数 × 价值系数 × (满意度/100)，<60 归零；整改/督办无奖金；基数走配置 |

## Phase 3 留尾

1. **NPSS 仪表盘聚合全部已评价项目**，未按财年/季度筛选；且当前对所有 pm-agent.use 用户可见（未做 PMO-only RBAC）。Phase 4 补时间维度筛选 + 角色收敛。
2. **M.O.R.E 为组织自评**（0-100 滑杆，存 reward-config），不自动从项目数据推导，不参与 NPSS 计算 —— 避免无根之木式的伪计算。
3. **奖金基数默认值**（S 10万/I 5万/O 3万）为占位，正式数值由 PMO 细则在看板内配置。

## 干系人 + 结案评价重构（已完成）

- 干系人一律为 MAP 系统用户（UserSearchSelect 选择）；外部客户改为选一位 MAP 用户作「代表」并必填备注（说明代表谁）。
- 干系人界面重做：以权力利益矩阵为主，格子内「+」加人、点头像卡进入单卡编辑（不再在矩阵下方整列展开）。
- 一致性：干系人变更（保存）后自动失效进行中的评价轮 + 前端保存触发全量 reload，避免矩阵/列表/评价轮快照不一致。
- 结案评价改为多人协作：立项人/Leader 发起 → 每位干系人各自打分（互相不可见，逐人提交、可更新）→ 全部评完后立项人汇总出加权 NPSS。
- 干系人（系统用户）自动获得该项目访问权以提交评分（FindAccessibleProject 增 stakeholder.UserId 匹配）。

留尾：
1. 暂无站内"待我评分"聚合入口/通知，干系人需进入项目内打分；后续可加待办提醒。
2. 评分汇总要求全部评完（按用户选择）；如需"可随时截止"再加开关。
3. `PmStakeholder.Score` 字段已废弃，仅为兼容旧数据保留反序列化。

## Phase 4 留尾

1. **财年归属按"评价时间(EvaluatedAt)"判定**；若 PMO 口径应按"结案时间"则需调整 `FiscalYearOf` 取 ClosedAt。

## 留尾项（Phase 1 内的简化）

1. **Leader 职级强校验未落地**：方法论要求「战略级 S≥L3 / 运营级 O≥L2 担任 Leader」。
   当前 `User` 模型无职级字段，Controller 暂不强制校验。补法：User 增加 JobLevel 字段后，在 `PmAgentController.CreateProject` 加校验。
2. **甘特图为只读时间线**：横向滚动展示，暂不支持拖拽改期 / 依赖连线绘制（仅 tooltip 标注依赖数）。
   故未涉及 `gesture-unification` 画布手势规则。后续若升级为可拖拽画布需对齐该规则。
3. **任务编辑入口有限**：看板/列表支持改状态、删除、快速新增；详细编辑（负责人/排期/依赖）暂走 API，UI 编辑面板待补。
4. **PMO 注册关联文档**：模型预留 ProposalRef/PlanRef/SummaryRef 字段，UI 关联立项方案/计划/总结文档待补。

## 任务模块优化（2026-06-08）已知边界

1. **进度 roll-up 口径**：父任务进度 = 子任务进度均值（done=100/cancelled=0/其余取各自 ProgressPercent）。未按 `EstimateDays` 加权——子任务工时差异大时均值会失真。补法：`PmTask.ComputeParentProgress` 改为按 EstimateDays 加权（有 xUnit 守护，改动局部）。
2. **两级层级硬约束**：仅支持父→子两级。挂载已有任务为子任务时，若该任务自身已有子任务、或目标父已是子任务，后端 `ValidateParentForTwoLevelAsync` 返回 400 拒绝；多级 WBS 暂不支持。
3. **工作日志与周报未联动**：`pm_task_work_logs` 字段已对齐 `ReportDailyLog`（Date/Content/DurationMinutes/Category）并预留，但本期不自动汇总进个人日报/周报，后续单独一期做。
4. **工作日志权限**：项目成员均可写日志，仅作者本人可编辑/删除；未做"仅处理人可写"的收敛（任何成员可代填）。
5. **进度与状态轻微不一致**：status=done 时展示恒为 100%，但 ProgressPercent 字段值不一定被改写为 100（展示层兜底）。如需字段强一致，可在置 done 时落库 100。
6. **详情页主题**：全屏路由用 `var(--bg-*)` token 跟随全局主题；未像 ReportDetailPage 在独立路由内单独同步 documentElement（依赖 AppShell 外层已设 data-theme，正常进入无问题）。

## 项目简报（2026-06-12）已知边界

1. **匿名分享端点直出 HTML**：`GET /api/pm/briefings/shared/{token}` 直接返回 text/html（非前端路由壳），无 PV 统计、无密码/有效期；token 可撤销但不轮换历史。需要更强分享治理时参照 `web_page_share_links` 体系升级。
2. **保存到网页托管为快照复制**：每次点击都新建一个 HostedSite（重新保存不会覆盖旧站点，旧站点需用户在网页托管里自行删除）；`PmBriefing.HostedSiteId` 只记录最近一次。
3. **简报内容生成后不可编辑**：HTML 由服务端模板渲染，LLM 内容不满意只能重新生成；后续可加「编辑结构化内容再渲染」。
4. **模板单款式**：`PmBriefingRenderer` 仅一套浅色对外模板；多模板/品牌色定制留待需求驱动。
5. **简报权限**：生成/分享/托管限 owner、leader（或创建者），项目成员可查看列表与详情；未做干系人级别的更细粒度控制。
