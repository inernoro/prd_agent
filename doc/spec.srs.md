# PRD Agent 系统需求规格 · 规格

> **版本**：v4.0 | **日期**：2026-07-17 | **状态**：已落地

本文档定义 PRD Agent 平台级需求边界和跨模块共性约束。各 Agent 的详细产品行为以对应 `spec.*` 为准，技术方案以 `design.*` 为准，实施顺序以仍有效的 `plan.*` 为准。本文不复制控制器、数据模型或部署脚本。

## 1. 产品目标

PRD Agent 是面向产品、研发、测试和管理协作的智能体平台。系统应把文档理解、内容生成、质量审查、任务执行和知识沉淀组织成可追踪的工作流，并允许不同应用在统一身份、权限、模型调度和存储能力上独立演进。

核心目标：

- 用户能够通过 Web、桌面端或开放接口使用已授权的 Agent。
- 长任务能够持续展示阶段、进度和结果，不让用户面对静止等待。
- Agent 产生的运行记录、附件、引用和结果可以追踪、复用与审计。
- 平台能力有单一事实源，业务 Agent 不重复实现身份、权限、模型和存储基础设施。

## 2. 用户与角色

| 角色 | 核心诉求 | 权限边界 |
|---|---|---|
| 普通用户 | 使用 Agent 完成文档与业务任务 | 仅访问本人或被授权的数据 |
| 团队成员 | 协作、共享和复用团队资产 | 受团队与应用权限约束 |
| 运营人员 | 配置应用、模型、市场和内容 | 不默认拥有系统管理权限 |
| 系统管理员 | 管理用户、权限、模型与运行环境 | 所有高风险操作必须可审计 |
| 开放平台调用方 | 通过 API 或 SDK 调用能力 | 受应用身份、密钥和限流约束 |

## 3. 平台能力

### 3.1 身份与权限

- 每个应用必须有稳定的 `appKey` 和调用身份。
- 服务端负责权限判定，前端不得用隐藏入口代替授权。
- 用户数据、团队数据和应用数据必须按既定作用域隔离。
- 管理操作、密钥操作和外部授权必须留下审计记录。

### 3.2 模型调用

- 所有业务模型调用统一经过 LLM Gateway 和应用调度配置。
- 调用方不得绕过模型组、应用身份、计量和审计链路。
- 流式能力优先使用 SSE；批量或长任务必须提供阶段和进度事件。
- 模型不可用时应返回可操作的原因，不能静默降级到未经配置的模型。

### 3.3 会话、运行与任务

- 对话型 Agent 应持久化会话和消息，并支持恢复。
- 长任务采用 Run/Worker 或等价的服务端权威模式，页面断开不得直接取消服务端任务。
- 任务状态由服务端持有，前端只展示和发出用户意图。
- 重试、取消、超时和失败必须有明确且可查询的终态。

### 3.4 文档、附件与知识

- 用户应能上传常见办公文档和图片，系统负责提取、引用与权限校验。
- 回答中的引用必须能定位到来源；缺失来源时不得伪造引用。
- 文档空间、分享链接和知识库条目必须遵循访问控制和生命周期规则。
- 验收样本、截图和临时报告不进入 `doc/`，应进入验收知识库。

### 3.5 Agent 生态

- 新 Agent 必须登记应用身份、权限、路由和导航入口。
- 可共享配置通过市场发布、复制和撤回，不直接共享所有权。
- Agent 可复用工作流、Bridge、附件、模型和文档空间能力。
- 跨 Agent 调用必须保留调用链和原始用户授权边界。

## 4. 关键业务需求

| 编号 | 需求 | 验收结果 |
|---|---|---|
| FR-001 | 用户可发现并进入有权限的 Agent | 导航、百宝箱和直接路由行为一致 |
| FR-002 | 用户可发起对话或任务并持续看到进度 | 运行期间持续有流式内容或阶段更新 |
| FR-003 | 用户可上传附件并在结果中追踪来源 | 附件状态、提取结果和引用可核对 |
| FR-004 | 用户可恢复历史会话和运行 | 刷新或重新登录后可继续查看 |
| FR-005 | 用户可取消或重试允许操作的任务 | 服务端终态与页面状态一致 |
| FR-006 | 管理员可配置应用权限和模型组 | 配置生效且有审计记录 |
| FR-007 | 开放平台调用方可在授权范围内访问能力 | 密钥、限流和应用身份均生效 |
| FR-008 | 共享资产可发布、复制和撤回 | 复制后所有权独立，撤回不破坏既有副本 |

具体产品模块的需求编号由各自 `spec.*` 维护，不在本文件重复展开。

## 5. 外部接口约束

### 5.1 Web 与桌面端

- 页面必须提供清晰的首次使用引导、空状态和错误恢复入口。
- 宽屏页面应使用统一布局规则，弹窗、滚动和主题样式遵循前端规则文档。
- 能上传的场景不强迫用户手工粘贴；无法自动获取的输入应说明获取方式。

### 5.2 HTTP 与 SSE

- HTTP 接口使用统一认证、响应包装和错误语义。
- SSE 事件应区分状态、增量内容、进度、结果、错误和心跳。
- 客户端重连不得重复执行非幂等业务动作。
- 开放接口变更应保持兼容，破坏性调整必须有迁移说明。

### 5.3 外部系统

- Git、TAPD、文档平台、对象存储和模型服务等连接必须经受控配置。
- 外部凭据不得进入日志、文档或前端包。
- 外部服务失败时，应区分用户配置问题、权限问题、限流和平台故障。

## 6. 数据要求

| 领域 | 要求 |
|---|---|
| 标识 | 业务实体使用稳定唯一标识，不以展示名称作为关联键 |
| 隔离 | 用户、团队、应用和环境作用域必须显式 |
| 审计 | 关键配置、授权、发布和管理操作可追踪 |
| 保留 | 会话、运行、附件和日志按各自策略保留与清理 |
| 迁移 | 数据结构变化必须提供兼容读取或受控迁移 |
| 索引 | 高频查询、唯一约束和过期清理应有对应索引 |

具体集合与字段以 Model、数据字典和数据库索引指南为事实源。

## 7. 非功能性需求

### 7.1 可用性

- 普通页面操作应在可感知时间内反馈。
- 超过两秒的 AI 等待必须展示持续变化的内容。
- 页面断开、刷新和网络抖动不应让服务端任务进入未知状态。

### 7.2 安全

- 默认拒绝未授权访问，敏感操作执行服务端校验。
- 密钥、令牌、用户上传和模型输入不得被无意泄漏。
- 分享链接、代理接口和外部回调必须防止越权与重放。

### 7.3 可维护性

- 平台事实源保持单一，禁止在多个文档复制实现清单。
- 新 Model、Service 和 Agent 按仓库规则对照现有实现。
- 代码、测试、文档和导航变更必须能够相互追踪。

### 7.4 可观测性

- 请求、模型调用、Worker 和外部依赖应具备关联标识。
- 失败日志包含可定位原因，但不得输出秘密或完整敏感内容。
- 关键成功率、延迟、队列和资源状态可被监控。

## 8. 需求可追溯矩阵

本节是需求可追溯矩阵（RTM）的方法论事实源。产品设计可引用“附录 D”这一历史名称；后续新增条目直接维护本节，不再复制矩阵模板。

| 需求编号 | 产品规格 | 技术设计 | 实现位置 | 验证证据 |
|---|---|---|---|---|
| FR-001 | 对应 Agent 的 `spec.*` | 导航与应用设计 | 路由及注册表 | 导航覆盖测试 |
| FR-002 | 对应 Agent 的运行需求 | Run/Worker 与 SSE 设计 | 控制器、Worker、页面 | 集成测试与浏览器验收 |
| FR-003 | 附件与知识规格 | 存储与引用设计 | 上传、解析、引用服务 | 上传及引用测试 |
| FR-006 | 应用与模型配置规格 | 应用身份和 Gateway 设计 | 注册表、权限、调度服务 | 权限与调度测试 |
| FR-007 | 开放平台规格 | 鉴权、代理与限流设计 | 开放平台接口 | API 冒烟与负面测试 |

RTM 的维护规则：

1. 每项用户可感知需求必须落到产品规格。
2. 跨层或高风险需求必须关联技术设计。
3. 实现位置引用目录或事实源，不复制代码。
4. 验证证据必须能证明行为发生，不能只列编译结果。
5. 删除或替换需求时同步处理设计、实现和验证引用。

## 9. 文档分工

| 文档类型 | 在需求链中的职责 |
|---|---|
| `spec.*` | 定义用户、场景、范围、需求和验收口径 |
| `design.*` | 解释架构、数据流、边界和技术决策 |
| `plan.*` | 记录尚未完成工作的顺序、责任和退出条件 |
| `rule.*` | 约束跨任务必须遵守的长期规则 |
| `guide.*` | 指导操作者完成当前有效流程 |
| `report.*` | 记录周报或受引用保护的历史结果 |
| `debt.*` | 维护尚未解决且仍需追踪的工程债务 |

## 10. 变更原则

- 平台级需求变化先更新本规格或对应专项规格，再进入设计和计划。
- 已落地实现细节不回填为大段代码，只保留可验证的行为和事实源路径。
- 被专项规格覆盖的模块不在本文件复制，避免平台总规格再次膨胀。
- 历史修订通过 Git 查询，不在正文维护不断增长的流水账。
