# Codex 本地连调规则

本规则只约束 Codex 在本仓库内进行本地连调、视觉修复、接口排查、CDS 部署验证时的工作方式。它不是产品业务规则，不同步到 Claude、Cursor 或用户端交互规范。

## 适用范围

- CDS 控制台、预览页、部署页、分支页、项目卡片等本地和远端连调。
- 前端视觉、交互热区、明暗模式、状态语义、实时流面板等体验修复。
- 后端接口、proxy、forwarder、webhook、SSE、构建和部署链路排查。
- Codex 提交前的自测、部署、回归测试和收口报告。

## 强制原则

1. 先看真实系统，再写代码。UI 问题看真实页面，接口问题打真实接口，部署问题查真实日志。
2. 先分析差异、根因、影响范围，再动手修改。禁止只凭截图、猜测或单点审美调整。
3. 修复必须系统化：优先改统一入口、共享组件、状态机、请求分类、数据口径，避免特殊 case 补丁。
4. 每修一个问题必须举一反三，检查同类链路、同类页面、同类组件，并补回归测试。
5. 提交前必须至少跑通一条真实验收路径：CDS、预览域名、Bridge、curl、浏览器或集成测试。
6. 失败状态必须清楚告诉用户错误来源：前端、后端、构建、部署、上游不可达或权限问题。
7. 长等待不能空白，必须有阶段、进度、日志、实时流或可手动刷新入口。
8. UI 信息层级以用户关心为先：名称、分支、状态、容器类型、错误原因、更新时间优先，无关指标降噪。
9. 视觉调整必须遵守统一 spacing、字号、对比度、状态色和明暗模式，不做局部污染。
10. 交互热区必须符合用户直觉，按钮只保护自身行为，其他空白区域应尽量触发主操作。
11. 状态语义必须互斥清晰，文案、颜色、图标、按钮不能表达相互冲突的信息。
12. 仓库名、项目名、项目 ID、分支名、slug 的展示口径必须统一。
13. 页面慢必须定位具体请求、重复刷新和缓存策略，不能笼统归因。
14. 完成后输出收口表：问题、改动、测试、真实验证、残余风险、同类排查结果。
15. 不提交无关文件，不覆盖用户改动，不把本地连调偏好写进产品业务规则。

## 连调流程

### 1. 获取最新状态

- 开始前先执行 `git status --short` 和 `git pull --ff-only origin main`。
- 若存在用户未提交改动，只读识别，不擅自回滚、不擅自提交。
- 若 pull 后出现大规模变化，先重新审视相关文件，避免基于旧结构继续修改。

### 2. 真实观测

- UI 问题：优先使用浏览器、Bridge、截图或远端预览域名确认真实状态。
- 接口问题：用 curl 或测试直接测接口耗时、状态码、content-type、响应体。
- 部署问题：查 CDS 控制台、systemd、构建日志、容器日志和预览域名。
- 性能问题：定位到具体请求、重复刷新、缓存失效、上游慢或前端渲染慢。

### 3. 系统性修复

- 先找共同入口：组件、hook、service、proxy、forwarder、状态机、API 聚合层。
- 不允许只为当前项目名、分支名、截图状态写特判。
- 如果临时兜底不可避免，必须标明原因、范围和后续移除条件。
- 修复一条链路后必须检查兄弟链路，例如 master proxy 与 forwarder、暗色与浅色、列表与详情、构建页与预览页。

### 4. 视觉与交互标准

- 信息优先级：对象名称 > 分支或容器类型 > 当前状态 > 错误原因 > 更新时间 > 次要指标。
- 视觉比例基于 4px / 8px spacing、清晰字号层级、足够对比度和稳定密度。
- 明暗模式必须同时检查，尤其是背景、icon、边框、透明度、浮层和状态色。
- 点击热区要贴合用户直觉；按钮只保护自身动作，其他空白区域尽量触发主操作。
- 动效应克制、可解释，不靠一瞬间闪烁传达关键状态。

### 5. 错误与等待表达

- 构建失败、部署失败、上游不可达、前端资源加载失败必须显示不同语义。
- 页面导航可以返回等待或失败页面；JS、CSS、图片、字体等资源请求必须返回匹配 MIME 的降级响应。
- 预览页不得在已知失败状态下无限重建或反复刷新。
- 长任务至少展示阶段、最近更新时间、可展开日志或手动刷新入口。

### 6. 测试与回归

- 反复出现的问题必须新增回归测试。
- 测试覆盖正常态、失败态、边界态和同类链路。
- 前端视觉改动至少验证暗色与浅色；关键页面需截图或浏览器验证。
- proxy、forwarder、SSE、webhook、SPA fallback 等基础链路要优先用集成测试锁住。

### 7. 提交与交付

- 提交前列出实际运行过的命令和真实验证结果。
- 推送后按仓库规则给出预览地址。
- 完成批量修复时输出收口表，逐项说明用户反馈是否完成、如何验证、是否有残余风险。
- 不把未跟踪的用户文件、无关格式化、无关重构带入提交。

## 收口表模板

| 检查项 | 结论 | 证据 |
|---|---|---|
| 最新代码 | 已同步 / 未同步 | commit 或 pull 结果 |
| 真实观测 | 已完成 / 未完成 | 页面、接口、日志或截图 |
| 系统性修复 | 已完成 / 临时兜底 | 共同入口或状态机说明 |
| 同类排查 | 已完成 / 待补 | 兄弟链路和同类组件 |
| 明暗模式 | 已验证 / 待验证 | 具体页面 |
| 错误语义 | 已区分 / 待区分 | 前端、后端、构建、部署、上游 |
| 回归测试 | 已覆盖 / 待补 | 测试文件和用例 |
| 真实验收 | 已通过 / 未通过 | CDS、curl、Bridge、浏览器或预览域名 |
| 残余风险 | 无 / 有 | 风险与后续动作 |
